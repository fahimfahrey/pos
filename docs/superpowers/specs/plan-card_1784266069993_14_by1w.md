# Plan: Append-only audit log, deterministic seed script, trustworthy test suite, CI

## Overview

This card has four parts: (1) an append-only `AuditLog` entity written from services at nine specific trigger points, plus a read-only audit UI gated to `OWNER`/`VIEWER`; (2) a deterministic `scripts/seed.ts` seed script through the storage layer (two orgs, three branches, 500 products, a month of sales); (3) bringing the test suite (unit/integration/conformance/e2e) to trustworthy coverage; (4) a GitHub Actions workflow that blocks merge on failure, plus a `CONTRIBUTING.md`.

**Critical, load-bearing finding, verified before writing a line of this plan:** the repository does **not** currently pass its own quality bar. Running the actual commands right now:

```
npx tsc --noEmit          → 325 errors across 53 files
npx vitest run            → 94 failed / 298 passed (404 total), 11 failed test files, 18 unhandled-rejection errors
```

"Bring the suite to trustworthy coverage" and "CI green" (the card's own acceptance line) are therefore not achievable by only adding new tests — a large share of this card's work is diagnosing and fixing **pre-existing, already-broken code**, most of it predating this card (left behind by the `Order`→`Sale` and `CatalogItem`→`Category`/`Product`/`ProductVariant` migrations that earlier cards performed without updating every caller). This plan treats that stabilization as **Phase 0**, a hard prerequisite, and treats the four items the card actually asks for as Phases 1–4. Every root cause below was independently reproduced and traced to an exact file/line — this is not a guess; do not re-diagnose these at implementation time, just apply the fix and move on. A long tail of ~30 more files with the *same class* of error (calling a repository method or entity field that no longer exists) is enumerated by category with a systematic sweep procedure rather than file-by-file, since re-deriving each one here would not be more reliable than running `tsc`/`vitest` yourself during implementation.

**Skills**: no `.agents/skills/` directory exists in this repository (confirmed: `ctx_glob('.agents/skills/**/SKILL.md')` and `.agents/**` both return zero files). There are no project-specific skills to apply. Per `AGENTS.md`, this is a modified Next.js (16.2.10) whose APIs may differ from training data — the only new UI in this card (`src/app/admin/audit/page.tsx`) reuses the exact server-component/session/session-gating pattern already proven in `src/app/admin/enum-values/page.tsx`, so no new Next.js API surface is introduced; skim `node_modules/next/dist/docs/` only if that precedent turns out to be insufficient.

## Architecture Decisions

1. **Retention is a query-time visibility window, not physical deletion.** `AuditRepository` gains no `delete`/`purge` method — the interface has never had one (two earlier plan cards for this same domain independently arrived at "append-only... no update/delete" as a hard invariant), and this card doesn't break that. `settings.audit.retentionDays` is enforced entirely inside `listAuditEntriesForOrg` (Phase 1) — an entry older than the window is simply not returned to the UI/API, but remains physically in storage. A real compliance-driven purge job (if ever needed) is a distinct, explicit follow-up that would need its own product decision about whether "retention" means "hide" or "destroy" — conflating the two now would be scope creep on a one-line settings field.
2. **`recordAuditEntry`/`listAuditEntriesForOrg` are plain functions, not an `AuditService` class.** Every one of the nine trigger points lives in a different, already-existing service (`CatalogService`, `FinalizeSaleService`, `RefundService`, `InventoryService`, `ZReportService`, `EnumRegistryService`) plus two brand-new ones (`VoidSaleService`, the role/settings actions). Making audit-writing a constructor-injected class dependency would force a constructor-signature change (and every existing call site of `new XService(...)`) across six files for a two-line side effect. A plain function taking the `ids`/`clock` each caller already has as constructor fields keeps the blast radius to "one new line at each of the nine call sites," matching this codebase's existing preference for small pure functions (`resolveReportScope`, `resolveTaxRate`, `rowsToCsv`) living alongside classes rather than wrapping everything in a service.
3. **The audit-UI role gate is an explicit `OWNER || VIEWER` equality check, not a `requireRole` hierarchy check.** `MEMBERSHIP_ROLE` is `OWNER=1, ADMIN=2, MEMBER=3, VIEWER=4`, and `hasAtLeast`/`requireRole` treat *lower* numbers as *more* privileged (`userRole <= requiredRole`). A hierarchy check anchored at `VIEWER` (`requireRole(role, MEMBERSHIP_ROLE.VIEWER)`) would pass for **every** role, since all four numbers are `<= 4` — the opposite of "OWNER and VIEWER only." This is not a hierarchy at all; it's a two-role allowlist, and the codebase already has exactly this precedent in `src/domains/reporting/services/report-access.ts` (`if (role === MEMBERSHIP_ROLE.OWNER || role === MEMBERSHIP_ROLE.VIEWER) { ...org-wide... }`). Reuse that same explicit-equality pattern; do not introduce a new role-checking helper for it.
4. **`FinalizeSaleService` is wired up, not rewritten.** It already existed, fully implemented, before this card — just never called from any action (confirmed: zero non-self references in the entire codebase). Rather than building a new finalization path, this card adds the one missing action (`finalize-sale.ts`) and fills in the three `'unknown'` placeholders the service left for its caller to supply. This mirrors exactly how the immediately-preceding "Reporting" card found `ShiftService.closeShift` unused and added `close-shift.ts` to give it a real entry point — same category of gap, same category of fix.
5. **The legacy `Order`/direct-Dexie layer is deleted, not migrated.** `container.ts`, `dexie-db.ts`, `dexie-sales-repository.ts`, `dexie-inventory-repository.ts`, `sales-service.ts`, `create-order.ts`, and `complete-sale.ts`(+test) form a self-contained island that (a) already fails to compile today because its own `Order` entity file was deleted by an earlier migration and never replaced, (b) bypasses the `StorageDriver`/adapter abstraction that every other domain and every conformance test in this repository is built around, and (c) has exactly two live external callers (`create-order.ts`'s own action, `list-products.ts`), one of which (`list-products.ts`) is trivially rewritten against the current `InventoryRepository`. Migrating this layer forward instead of deleting it would mean maintaining two parallel, incompatible sales/inventory persistence paths — deleting it is the smaller, safer, and only architecturally-consistent option.

## State Management Strategy

- **Server-side / persistence state**: all new writes (`AuditEntry`, `Sale` void transitions, membership role changes, organization settings changes) go through `StorageProvider.withTransaction`, exactly like every existing write in this codebase — no new persistence pattern introduced. `AuditEntry` itself is write-once per row (no update method exists on `AuditRepository`), so there is no update-conflict/staleness concern for audit data specifically.
- **Audit-UI client state**: `src/app/admin/audit/page.tsx` is a server component (matching `enum-values/page.tsx`) — filters (branch/action/date-range) are plain `<form>` GET params read via `searchParams`, triggering a server re-render; no client-side state management library, no new client component beyid a possible small filter form using native `<form>`/`useActionState` if interactivity beyond full-page GET params turns out to be needed (defer that decision to implementation; start with the simplest `searchParams`-driven approach, matching this codebase's general bias against unnecessary client state).
- **Seed script state**: `build-seed-data.ts` takes an explicit `Clock`/`IdGenerator` (never reads global `Date.now()`/`Math.random()`/`crypto.randomUUID()` directly), so all "state" (the sequence of ids/timestamps produced) is a pure function of the inputs — this is what makes the determinism test in Phase 2 possible at all.

## Accessibility Considerations

- `src/app/admin/audit/_components/audit-log-table.tsx` is a plain HTML `<table>` with real `<th scope="col">` headers (matching whatever pattern `enum-values-table.tsx` already uses — inspect it at implementation time and follow it exactly for consistency) — no custom widget, so native screen-reader table semantics apply for free.

## Risks and Edge Cases

If it remains unbuilt when this card is implemented, either build the minimal product-creation UI as part of this card (scope increase) or write the e2e spec against the server actions directly (`saveProduct`/`saveVariant`) with a thin smoke-test UI assertion — decide at implementation time based on current repo state, and note the decision in the PR description.
- **Threading `audit: AuditRepository` through `adjustStock`/`addValue`'s `repos` parameter is a public-interface change** for those two methods — grep every call site (tests included) before changing the signature, since `RepositorySet`/`RepositoryContext` already include `audit`, so most callers passing the full `repos` object need no change at all; only callers passing a narrowed `{ inventory: ... }`-shaped object need updating to also pass `audit`.
- **The seeded "month of sales" must keep `ReconciliationService.reconcileBranch` at zero drift.** Every seeded stock decrement must go through `InventoryService.recordSale`/`recordReturn` (never a raw `saveStockLevel` bypass), or the seed script will silently produce a demo dataset that fails its own consistency invariant — assert this with a dedicated check in `build-seed-data.test.ts`, not just eyeballing the numbers.
- **`fixed-sequence-clock.ts`'s naive monotonic-increment approach must still produce plausible calendar spread** (a month of sales across 3 branches, not all sales crammed into the same millisecond) — pick an increment (e.g. +11 minutes per `now()` call) that, over the total number of seeded events, actually spans ~30 days; verify the seeded dataset's min/max `createdAt` spread directly in `build-seed-data.test.ts` rather than assuming the arithmetic works out.
