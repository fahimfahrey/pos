# Plan: domains/payments — PaymentGateway adapter, split payments, store credit, refunds

Card: `card_1784266069993_10_tw01`

## Overview

The repo already ships one swappable-adapter subsystem end-to-end: **storage**. A minimal `StorageDriver` port (`get/getAll/put/delete` + `transaction`) lives in `src/infrastructure/storage/core/driver.ts`, concrete engines register themselves into a module-level `Map` via `registerEngine()` (`src/infrastructure/storage/core/engine-registry.ts`), and `src/infrastructure/storage/index.ts` imports every adapter purely for its registration side-effect. `docs/storage-adapters.md` documents "one adapter = one file + one config change." This card builds the **identical shape** for payments: a minimal `PaymentGateway` port, a `registerPaymentGateway()` map, adapter folders under `src/infrastructure/payments/adapters/*`, and a side-effect-registering `src/infrastructure/payments/index.ts`.

Today `src/domains/payments/entities/payment.ts` and `repository.ts` are a thin placeholder (string-union `PaymentStatus`, no gateway concept, no refund-status/idempotency/split-payment support). `src/domains/sales/services/finalize-sale-service.ts` (built by the previous card, `plan-card_1784266069993_9_2td4.md`) already has the *shape* of split payments (`input.payments: Array<{amount, method}>`) but creates `Payment` rows inline with a hardcoded `status: 'completed'` — there is no gateway abstraction to swap. This card:

1. Turns `PaymentStatus`/`RefundStatus` into integer enums (mirroring the existing `MEMBERSHIP_ROLE` numeric-enum pattern used for ordered role comparison) and records every status change as an append-only `PaymentStatusEvent` row — the same "ledger + materialized current value" shape the inventory domain already uses (`StockMovement` ledger → materialized `StockLevel`, from `plan-card_1784266069993_2_vuy5.md`-era work). `Payment.status` is the materialized value; `PaymentStatusEvent` rows are the ledger.
2. Defines `PaymentGateway` (charge/refund) and ships two real, local implementations — `CashGateway` (tendered amount, change calculation, drawer-kick event) and `StoreCreditGateway` (debits/credits a new `Customer.storeCreditBalance` via its own ledger, `StoreCreditTransaction`) — plus a `FakeGateway` used only by tests to prove the contract is sufficient for a network-backed provider.
3. Adds `PaymentService.chargeSplitPayments()` so `FinalizeSaleService`'s payment loop becomes a single call into the payments domain instead of inline `Payment` construction — this is the concrete proof that a future Stripe/bKash/SSLCommerz adapter requires **zero changes to checkout code** (`finalize-sale-service.ts` changes exactly once, in this card, to delegate; never again).
4. Adds `RefundService.refund()`: ADMIN+-gated (reuses the existing `requireAdminMembership`), supports full and partial refunds, optionally reverses stock for specific returned sale items via the existing `InventoryService.recordReturn`, and runs inside the existing `StorageProvider.withTransaction` UnitOfWork (no new transaction abstraction — same decision the sales card already made).
5. Documents, in `docs/payments.md`, exactly which pieces (webhook signature verification, an idempotent `GatewayEventStore`) only activate once a server deployment exists to receive webhooks — the port is defined now, nothing is wired into storage/schema for it yet.

- **New** `services/payment-service.ts` — `PaymentService`:
  - `chargeSplitPayments(repos: Pick<RepositorySet,'payments'|'customers'>, ctx: {clock, ids}, input: {saleId: string; currency: string; total: number; requests: Array<{id: string; amount: number; method: PaymentMethod; gateway: string; tendered?: number; customerId?: string; idempotencyKey?: string}>}): Promise<Payment[]>`
    - Idempotency: for each request, `existing = await repos.payments.findPaymentById(request.id)`; if present, push it unchanged and skip charging (mirrors `FinalizeSaleService`'s idempotent-by-ULID check).
    - Validates `assertSplitPaymentsCoverTotal(requests, total)` up front.
    - For each new request: `resolvePaymentGateway(request.gateway).charge(...)`, map `ChargeResult` → `Payment` (status defaults to `PENDING` if the gateway didn't set one), `repos.payments.savePayment(payment)`, `repos.payments.appendStatusEvent({fromStatus: null, toStatus: payment.status, ...})`.
  - `recordStatusTransition(repos, paymentId: string, toStatus: PaymentStatus, opts?: {reason?: string; actorId?: string}): Promise<Payment>` — loads the payment, calls `isValidPaymentStatusTransition`, throws `InvalidPaymentStatusTransitionError` if illegal, otherwise updates `Payment.status`/`updatedAt`, saves, and appends the `PaymentStatusEvent`. Used by both the charge path (implicitly, via `chargeSplitPayments`) and `RefundService`.
- **New** `services/refund-service.ts` — `RefundService`:
  - Constructor: `(clock: Clock, ids: IdGenerator, inventory: InventoryService, paymentService: PaymentService)`.
  - `refund(repos: Pick<RepositorySet,'payments'|'sales'|'inventory'|'customers'>, input: RefundInput): Promise<Refund>`:
    1. `payment = await repos.payments.findPaymentById(input.paymentId)`; throw `NotFoundError` if missing.
    2. `alreadyRefunded = sum of COMPLETED refunds already on this payment` (via `listRefundsForPayment`).
    3. If `input.amount > payment.amount - alreadyRefunded` → `RefundExceedsPaymentError`.
    4. `gateway = resolvePaymentGateway(payment.gateway)`; call `gateway.refund({paymentId, gatewayRef: payment.gatewayRef, amount, currency: payment.currency, idempotencyKey: input.id, reason}, ctx)`.
    5. Build and save the `Refund` row (`status` from the gateway result).
    6. If `input.items?.length`: for each item call `this.inventory.recordReturn(repos, {orgId, branchId, variantId: item.variantId, quantity: item.quantity, reference: refund.id, createdBy: input.initiatedBy, allowOversell: true})` — same UnitOfWork, no separate transaction.
    7. Recompute total refunded including this one; call `paymentService.recordStatusTransition(repos, payment.id, totalRefunded >= payment.amount ? PAYMENT_STATUS.REFUNDED : PAYMENT_STATUS.PARTIALLY_REFUNDED, {reason: input.reason, actorId: input.initiatedBy})`.
    8. Return the saved `Refund`.
  - `RefundInput`: `{ id: string; paymentId: string; saleId: string; orgId: string; branchId: string; amount: number; reason: string; items?: RefundedItem[]; initiatedBy: string }` (caller supplies `id` — client/action-generated, same idempotency pattern as everything else in this codebase).
- **New** `actions/create-refund.ts` (`'use server'`) — mirrors `src/domains/catalog/actions/save-product.ts` exactly:
  - zod-parse input (`orgId`, `branchId`, `paymentId`, `saleId`, `amount`, `reason`, optional `items[]`).
  - `session = await getCurrentSession()`; 401 if absent.
  - `provider.withTransaction(async (repos) => { await requireAdminMembership(repos.organization, orgId, session.userId); ... new RefundService(...).refund(repos, {...}) })` — permission check happens **inside** the transaction, same as `save-product.ts`, so a failed check throws before any write.
  - `toErrorResponse` on catch; `provider.close()` in `finally`.

- **New** `adapters/cash/gateway.ts` — `CashGateway implements PaymentGateway`:
  - `id = 'cash'`, `requiresServer = false`.
  - `charge(request, ctx)`: `request.tendered` must be `>= request.amount` (throw `ValidationError` otherwise — this is a programmer/UI error, not a decline); compute `changeDue` locally with integer-cent math (`Math.round(tendered*100) - Math.round(amount*100)`, converted back — **not** imported from `@domains/sales/lib/money`, see Architecture Decisions #3); return `{ status: PAYMENT_STATUS.CAPTURED, changeDue }`; publish `{ type: 'drawer.kick', paymentId: request.paymentId, saleId: request.saleId }` and `{ type: 'payment.captured', ... }` on `paymentsEventBus`.
  - `refund(request, ctx)`: returns `{ status: REFUND_STATUS.COMPLETED }` immediately (handing back cash is a physical/manual act — the gateway's job is just to record it) and also publishes `drawer.kick` (opening the drawer to retrieve cash for the refund).
- **New** `adapters/cash/index.ts` — `registerPaymentGateway('cash', () => new CashGateway(paymentsEventBus))`.
- **New** `adapters/store-credit/gateway.ts` — `StoreCreditGateway implements PaymentGateway`:
  - `id = 'store-credit'`, `requiresServer = false`.
  - `charge(request, ctx)`: requires `request.customerId`; calls `new StoreCreditService(ctx.clock, ctx.ids).redeem(ctx.repos, {customerId, amount: request.amount, reference: request.saleId, createdBy: 'system'})` — throws `InsufficientStoreCreditError` on insufficient balance (propagates up through `PaymentService.chargeSplitPayments`, uncaught — a decline here is exceptional, not a soft-fail `ChargeResult.status`, because it's caught and surfaced to the UI as a validation failure before the sale ever finalizes, matching how `FinalizeSaleService` already lets `InsufficientStockError` propagate). Returns `{ status: PAYMENT_STATUS.CAPTURED }`.
  - `refund(request, ctx)`: calls `StoreCreditService.issue(ctx.repos, {customerId: <looked up from the original Payment, passed via RefundRequest — see note below>, amount: request.amount, reference: request.paymentId, createdBy: 'system'})`, returns `{ status: REFUND_STATUS.COMPLETED }`.
  - **Note**: `RefundRequest` doesn't carry `customerId` today. Add it as an optional field (`customerId?: string`) to `ports/payment-gateway.ts`'s `RefundRequest`, populated by `RefundService.refund()` from `payment.customerId` before calling `gateway.refund(...)`.
- **New** `adapters/store-credit/index.ts` — registration side-effect.
- **New** `adapters/fake/gateway.ts` — `FakeGateway implements PaymentGateway`, **not** auto-registered (see Architecture Decisions #5):
  - `id = 'fake'`, `requiresServer = true` (simulates an online provider on purpose, so the contract test proves the interface works for the async/server case too).
  - Constructor takes a `behavior` option: `'always-capture' | 'always-decline' | 'always-pending'` (default `'always-capture'`) plus an in-memory `Map<idempotencyKey, ChargeResult>` for idempotency simulation.
  - `charge()`: if an entry exists for `request.idempotencyKey`, return it unchanged (proves idempotent-retry semantics an HTTP gateway would need); otherwise synthesize a `gatewayRef` (`fake-ch-<random>`), apply `behavior`, store, return.
  - `refund()`: symmetric, keyed by its own idempotency map.
- **New** `index.ts` — `import './adapters/cash'; import './adapters/store-credit'; export * from './gateway-registry'` (fake is **never** imported here — it is only registered explicitly inside test setup, so it can never end up in a production bundle).

## Architecture Decisions

1. **`PaymentGateway.charge`/`refund` return a result object; they do not throw for a normal decline.** A `FAILED` `ChargeResult.status` is a legitimate business outcome (card declined, gateway down), not an exceptional control-flow event — `PaymentService` persists a `FAILED` `Payment` row either way so the failed attempt is auditable. Gateways *do* throw for programmer/setup errors (missing `tendered`, missing `customerId` for store-credit, unregistered gateway id) since those can never be shown to a cashier as "declined."

2. **Payment status transitions are a fixed, explicit graph**, enforced by `isValidPaymentStatusTransition`: `PENDING → {AUTHORIZED, CAPTURED, FAILED}`, `AUTHORIZED → {CAPTURED, FAILED, VOIDED}`, `CAPTURED → {PARTIALLY_REFUNDED, REFUNDED, VOIDED}`, `PARTIALLY_REFUNDED → {REFUNDED}`; `FAILED`/`REFUNDED`/`VOIDED` are terminal. This exists so a future async gateway's webhook handler can call `recordStatusTransition` blindly and trust it to reject nonsensical replays (e.g. a `CAPTURED` webhook arriving after a `REFUNDED` state) instead of silently corrupting `Payment.status`.

3. **`payments` never imports from `sales`, even though `sales` already imports from `payments`.** `FinalizeSaleService` (in `sales/services`) already imports `Payment`/`PaymentsRepository` from `@domains/payments` — that's the correct direction (checkout depends on payments, not the reverse). `CashGateway` needs a change-due calculation identical in technique to `calculateChangeDue` in `src/domains/sales/lib/money.ts`, but it is re-implemented locally (≈5 lines of integer-cent math) rather than imported, so that `payments` has zero dependency on `sales`. This isn't enforced by `eslint-plugin-boundaries` today (that config partitions by *layer* — `services`/`entities`/`repositories`/etc. — not by domain name, so a `payments→sales` import would currently pass lint), but a payments domain that can be reused by a future non-POS product (or tested/shipped independently) must not silently acquire a `sales` dependency just because no linter catches it yet.

4. **`PaymentGatewayContext` always carries the transaction-scoped `RepositorySet`, even for gateways that don't need it.** `CashGateway` ignores `ctx.repos` entirely; `StoreCreditGateway` uses `ctx.repos.customers` to debit/credit inside the *same* `provider.withTransaction` call as the rest of finalize/refund (so a mid-transaction failure elsewhere correctly rolls back the store-credit ledger write too — no separate commit). A future real network gateway also receives `ctx.repos` but is expected to ignore it for the charge/refund call itself (network I/O has no business sharing a driver transaction) and only use `ctx.ids`/`ctx.clock` for constructing its own request records. This is flagged, not solved, in `docs/payments.md` — see Risks.

5. **`FakeGateway` is never auto-registered.** `src/infrastructure/payments/index.ts` (the production wiring point, imported once from wherever `src/infrastructure/storage/index.ts` is imported) only pulls in `adapters/cash` and `adapters/store-credit`. Tests that need `fake` call `registerPaymentGateway('fake', () => new FakeGateway(...))` explicitly in `beforeEach`/test setup. This guarantees a test double can never ship in a production bundle by accident — the equivalent risk doesn't exist for the storage layer's `memory` engine because `memory` is a legitimate SSR/Node runtime fallback, not a test double.

6. **`FinalizeSaleService` changes exactly once, in this card.** Before: inline `Payment` construction with a hardcoded status. After: a single call to `PaymentService.chargeSplitPayments`. This is the acceptance-bar reading of "adding one later touches zero checkout code" — the proof isn't a claim in this document, it's that a fourth gateway (e.g. Stripe) requires only a new `adapters/stripe/*` folder plus one line in `infrastructure/payments/index.ts`; nothing in `sales/**` changes again.

7. **Webhook verification and the idempotent event store are interface-only in this card.** `GatewayEventStore` is defined so the *type* is stable for whoever builds the webhook route later, but no collection, repository, or schema version is added for it — an unused collection with no writer would be dead weight and dead weight in the Dexie version chain is a real migration cost (every future schema bump replays the whole chain on existing browsers). `docs/payments.md` is the single source of truth for what's designed-but-inactive versus shipped.

8. **Refunds are permission-gated at the action layer (`requireAdminMembership`), not inside `RefundService`.** This matches every other domain in the codebase (`InventoryService`, `CatalogService`, `FinalizeSaleService` never call authorization code themselves — `save-product.ts` does). `RefundService.refund()` trusts `input.initiatedBy` as already-authorized; it only *records* that actor, it doesn't check their role.

9. **`Refund.refundedItems` is optional-in-effect (`[]` default), not a separate "refund type" enum.** A refund with no items is a pure monetary adjustment (goodwill credit, price correction) — no stock reversal, no `InventoryService` call. A refund with items reverses exactly those items' quantities via `InventoryService.recordReturn`. This avoids inventing a `RefundKind` enum for something the presence/absence of one array field already expresses unambiguously.

10. **`PaymentStatusEvent` is a ledger only for `Payment`, not `Refund`.** The card asks for "integer-enum status transitions recorded on every payment" — `Refund` has exactly two useful states in practice (`PENDING`→`COMPLETED`/`FAILED`) and its authorization trail already has a home: the existing `AuditRepository` (used nowhere yet for payments, but exactly the mechanism the codebase already has for "who did this admin-gated thing and when" — `AuditEntry.entityType: 'Refund'`, `action: 'refund.created'`, `actorId: input.initiatedBy`). `create-refund.ts` appends one `AuditEntry` per refund inside the same transaction. Building a second parallel ledger type for `Refund` would duplicate `AuditEntry` for no benefit.

## State Management Strategy

This card adds no UI and no client components — it is domain/service/infrastructure only, consumed today by one service (`FinalizeSaleService`, server-side) and one server action (`create-refund.ts`). There is no new client-side state to manage. The only "state" this card introduces is server-side and transaction-scoped:

- `PaymentGatewayContext.repos` is the already-open `RepositorySet` for the current `provider.withTransaction` call — never held across requests, never cached, discarded when the transaction resolves (identical lifetime discipline to every existing service in the codebase).
- `FakeGateway`'s in-memory idempotency map is process-lifetime, test-only state — explicitly not a pattern for `CashGateway`/`StoreCreditGateway`, whose idempotency is enforced by `PaymentService` checking `findPaymentById` against durable storage, not gateway-local memory.
- When the previous card's `/pos` UI (`payment-modal.tsx`, per `plan-card_1784266069993_9_2td4.md`) is eventually built, it is the natural future caller of `chargeSplitPayments` (via a new `finalize-sale.ts` action) and of `create-refund.ts` (from an eventual `/app/sales/[saleId]/refund` admin screen) — neither exists yet and building them is out of scope for this card. Where they land: the payment method list a cashier picks from should merge `PAYMENT_METHOD` static values with `EnumRegistryService.resolveValues(orgId, 'paymentMethod')` (the same call `catalog`'s admin enum-values screen already exercises), and each selected method needs an explicit `gateway` id (this card does not build the UI-level mapping from "method the cashier picked" to "gateway id" — see Risks).

## Accessibility Considerations

No UI ships in this card, so there is nothing to audit directly. The one place accessibility constraints bind future work: `docs/payments.md`'s "Adding a new gateway" checklist should note (and does, per the Files section) that any payment-method/tender UI built against this domain must follow the accessibility rules already established for `/pos` in `plan-card_1784266069993_9_2td4.md` (44×44px touch targets, `aria-live="polite"` for total/error changes, keyboard-navigable tender selection) — this card does not re-derive those rules, it just doesn't violate them since it produces no UI.

## Risks and Edge Cases

- **`conformance/fixtures.ts` has pre-existing drift unrelated to this card.** `makePayment` currently references `orderId` (a field that no longer exists on `Payment`) and the file's top-level `import type { Order, OrderLine } from '@domains/sales/entities/order'` points at an entity file the previous sales card's own plan says to delete. This strongly suggests the conformance suite currently does not compile/pass, independent of anything in this card. This plan fixes only the `Payment`/`Refund` fixtures (required for this card's own conformance suite additions to work) and explicitly does not touch the `Order` import — implementers should expect `npm run typecheck`/`test:conformance:*` to surface pre-existing sales/catalog fixture errors that are not this card's to fix; do not silently "fix" them by guessing at the deleted `Order`/`CatalogItem` shapes, flag them back instead.
- **`getCurrentUser()`/`requireUser()` (`src/domains/auth/actions/session.ts`) call `provider.getRepositorySet(tx)` inside `provider.withTransaction`, but no `getRepositorySet` method exists on `StorageProvider`** (`src/infrastructure/storage/core/storage-provider.ts` only exposes `withTransaction`). This is a pre-existing bug the previous card's plan already flagged and explicitly chose not to fix. `create-refund.ts` in this card uses `getCurrentSession()` directly (not `requireUser()`), matching `save-product.ts`'s established workaround — but if a future caller reaches for `requireUser()` from anywhere in the payments flow, it will hit this bug. Do not copy `requireUser()` into any new payments action; use `getCurrentSession()` + manual `session?.userId` null-check instead, exactly as `save-product.ts` does.
- **A real network gateway calling `charge()`/`refund()` inside `provider.withTransaction` holds a driver transaction open across a network round trip.** Every existing service in this codebase (`CatalogService`, `FinalizeSaleService`, and now `PaymentService`) already does all I/O — including, implicitly, anything a future gateway's HTTP call would do — inside one transaction; there is no existing precedent in this codebase for splitting "external I/O" from "the transaction" at all. This card does not solve that for online gateways (only `cash`/`store-credit`, both zero-network, ship real implementations); `docs/payments.md` flags it as the open question whoever implements Stripe/bKash/SSLCommerz must resolve, likely via an `authorize()`-before-transaction / `capture()`-inside-transaction split that does not exist in the interface today.
- **Store-credit redemption during `chargeSplitPayments` and a later split in the same call failing does not self-heal within `PaymentService`** — rollback is entirely the responsibility of the enclosing `provider.withTransaction`. This is correct (matches every other service in the codebase) but is easy to misremember when reading `payment-service.test.ts`'s explicit test for it — that test exists specifically so this isn't just a comment nobody re-verifies.
- **No UI-level mapping from "payment method the cashier picked" to "gateway id" is built in this card.** `Payment.method` (business-facing tender type) and `Payment.gateway` (technical routing key) are deliberately separate fields, but nothing in this card decides, e.g., that `method: 'card'` routes to `gateway: 'stripe'` once Stripe exists — that mapping is presumably an org setting (`ResolvedSettings`) added when the first real online gateway is built. For now, callers of `chargeSplitPayments` must supply `gateway` explicitly per split (only `cash` and `store-credit` are legal values until a fourth adapter ships).
- **`fast-check` is a new devDependency.** If the team's dependency-approval process requires a heavier justification than "standard vitest-adjacent property-testing library," flag it before implementation rather than after — the alternative (hand-rolled pseudo-random loops with a fixed seed and no shrinking) is possible but strictly worse for the one acceptance criterion that explicitly asks for property testing.
- **Schema version 7 is a breaking Dexie migration for local dev data** in the `payments`/`refunds` stores' index shape (adding `gateway`/`status` to already-populated stores is non-destructive for existing rows' *data*, but any dev/test IndexedDB predating v7 needs the migration to run cleanly once). Consistent with the precedent already set by v4/v6 (`docs` note there's no production data yet), acceptable pre-launch.
