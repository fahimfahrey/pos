# Plan: Reporting domain — aggregate repository methods, role-scoped reports, Z-report, CSV export

## Overview

This card builds a new `src/domains/reporting` domain following the exact architectural pattern already established by every other domain in this codebase (`repository.ts` interface → `Core*Repository` infra implementation → domain `services/` → `'use server'` `actions/`), plus a `CoreReportingRepository` that does **in-adapter aggregation** (fetch via `Collection.getAll()`/`filter()`, reduce in TypeScript) — this is what "in-adapter aggregation over indexes for IndexedDB now" means concretely in this codebase: `Collection<T>.filter()` already reads the whole store into memory and filters/reduces in JS (verified in `src/infrastructure/storage/core/collection.ts` — there is no native IndexedDB cursor-range aggregation here, and Dexie index strings are cosmetic today, not used by `Collection`). A future Postgres adapter implements the same `ReportingRepository` interface with real `GROUP BY` SQL; the interface is the contract both must satisfy, which the conformance suite enforces with identical seeded numbers.

## Architecture Decisions

1. **Repository methods take `orgId` as an explicit first parameter**, not just `(range, branchId)` as the card's illustrative example shows. Every existing repository in this codebase (`listCategoriesByOrg(orgId)`, `listSalesByBranch`, etc.) scopes by `orgId` explicitly — multi-tenant correctness requires it. The card's `sumSalesByCategory(range, branchId)` is read as shorthand for "range and branch are the primary axes," not a literal two-argument signature.

2. **`ReportFilter` is one shared narrowing object (`{ branchId?, shiftId?, categoryId?, productId? }`)**, reused by all nine aggregate methods and both Sales-shaped and non-Sales-shaped reports, rather than a bespoke parameter list per method. This keeps the interface small enough that a future Postgres adapter's `WHERE` clause builder can be generic across all nine methods.

3. **Role scoping lives in one function, `resolveReportScope`, called by `ReportingService` before every repository call — not in the repository, not duplicated per action.**
   - **Owner** (`MEMBERSHIP_ROLE.OWNER`) and **auditor** (`MEMBERSHIP_ROLE.VIEWER`): org-wide by default; an optional `branchId`/`shiftId` in the request further narrows results but is not required. Auditor gets identical read access to owner — "read-only" is satisfied structurally: `ReportingService` exposes no write methods to callers at all (the only write path, `ZReportService.closeShiftAndGenerateReport`, is invoked exclusively via `actions/close-shift.ts`, a completely separate action never exposed as a "report" action), so there is nothing for a `VIEWER` to write even if role-checked more loosely.
   - **Branch manager** (`MEMBERSHIP_ROLE.ADMIN`): must supply a `branchId`; `resolveReportScope` loads `organization.listAssignmentsForMembership(membership.id)` and throws `ForbiddenError` unless the requested `branchId` is in that set. (Multi-branch admins must call once per branch — no cross-branch aggregation is offered, matching the card's singular "own branch.")
   - **Cashier** (`MEMBERSHIP_ROLE.MEMBER`): must supply a `shiftId`; `resolveReportScope` loads `sales.findShiftById(shiftId)` and throws `ForbiddenError` unless `shift.cashierUserId === userId`. The resolved filter is forced to `{ shiftId, branchId: shift.branchId }` — a cashier can never request another branch's or shift's data even if they pass one.
   - Base gate for all four roles: `requireRole(membership.role, MEMBERSHIP_ROLE.VIEWER)` (i.e. `userRole <= VIEWER`, which is true for every role since `VIEWER` is the loosest/highest number) plus `assertActive(membership)` — reusing `src/domains/auth/services/authorization-service.ts` exactly as every other domain does, not reinventing role-checking.

4. **Margin costing uses "most recent prior purchase cost" per `(variantId, branchId)`, not a true weighted-average or FIFO ledger.** `StockMovement.unitCost` is only populated for `movementType: 'purchase'` (established by the prior "purchase StockMovements and cost snapshots" card). For a given sale, COGS per unit = the `unitCost` of the latest `purchase` movement for that `variantId`+`branchId` with `createdAt <= sale.createdAt`; if none exists, the row is flagged `costUnavailable: true` and excluded from the aggregate margin total (never silently treated as zero cost, which would overstate margin). This is a documented simplification — a true moving-average costing engine is out of scope for this card — called out again in Risks.

5. **`countSalesByHour` takes an explicit IANA `timezone` string parameter and converts with `Intl.DateTimeFormat`, never host-local time.** This matches the established convention in `src/domains/promotions/lib/promotion-engine.ts` (`isTimeWindowActive`) — explicit timezone parameters, not implicit system time, so heatmap tests are deterministic regardless of the machine/CI runner's timezone. Callers pass `ResolvedSettings.timezone` (from `resolveSettings`).

6. **`ZReportService.closeShiftAndGenerateReport` composes `ShiftService` by constructor injection (`new ZReportService(clock, ids, shiftService)`), the same way `FinalizeSaleService` already takes a `paymentService: PaymentService` constructor dependency (established by the payments card).** `reporting` depends on `sales`/`payments` (it already needs their repository/entity types for aggregation); `sales` is never modified and never imports from `reporting`. Z-report generation is **synchronous and transactional**, not event-driven like receipt printing (`sale.finalized` → `salesEventBus`, established by the printing card) — deliberately different, because a Z-report is a financial reconciliation record: a shift must never end up "closed" with no corresponding Z-report (an async/best-effort listener could drop it), whereas a missed/delayed receipt print is a UX-only concern the printing card explicitly designed to never block checkout. `closeShiftAndGenerateReport` calls `repos.reporting.findZReportByShift(shift.id)` first and throws `ConflictError` if one already exists — defense-in-depth immutability, since `ShiftService.closeShift` already throws on a non-`OPEN` shift, but this makes the "persisted immutably" requirement hold even if `closeShiftAndGenerateReport` is ever called directly outside that guard.

7. **`Sale`-level aggregates (period/branch/cashier) sum `Sale.subtotal`/`discount`/`tax`/`total` directly — item-level aggregates (category/product/margin) join through `SaleItem`.** `Sale` already carries the aggregated financial fields per sale; only category/product/margin dimensions require descending to `SaleItem` (`variantId`) and joining to `catalogProductVariants` → `catalogProducts` → `categories`. `SaleItem.name`/`sku` are already denormalized at sale time for receipts, but carry no `categoryId`, so the category join is unavoidable for that one report family.

8. **CSV export is a plain, dependency-free string builder (`rowsToCsv`), matching the existing hand-rolled `parseCsv`/`buildCatalogCsvTemplate` in `src/domains/catalog/services/csv.ts`** — no new npm dependency. Proper RFC-4180 quoting (wrap in `"..."` and double internal quotes) is required because report values include free-text names (category/product names, cashier display names) that may contain commas.

## State Management Strategy

Not applicable in the frontend/client-state sense — this card is domain/service/infra/action layer only, no UI. The relevant "state" is persistence state inside `StorageProvider`:
- All new repository/service methods are read-only except `saveZReport`, which is written exactly once per shift, enforced by the `findZReportByShift` guard in `ZReportService` (Architecture Decision 6) — "persisted immutably" is implemented as "write-once, no update method exists on the interface at all."
- `ReportingService` and `CoreReportingRepository` hold no instance state across calls — every method takes `repos`/`orgId`/`range` explicitly, matching the stateless-service convention already used everywhere else (`ShiftService`, `PurchasingService`, etc.).
- `close-shift.ts` is the only action in this card that opens a `provider.withTransaction` for a write; all report/export/read actions open a transaction too (it's the only entry point `StorageProvider` exposes) but perform no writes inside it.

## Accessibility Considerations

Not applicable — this card adds no UI, components, screens, or forms. It is domain/service/infrastructure/action layer only, matching the explicit scope of the card text (repository methods, role scoping, Z-report persistence, CSV export) and the precedent set by the immediately preceding "Customers, promotions, purchasing" card, which was also backend-only. A future card wiring a reports UI would consume `actions/reports.ts` and `actions/export-csv.ts`.

## Risks and Edge Cases

- **The IndexedDB migration-ladder gap (Overview point 1) is real, pre-existing, and affects every collection added since version 7** — this plan fixes only the version-8 block needed to unblock the adapter generally and adds a clean version-9 block for `zReports`. If a future card adds another collection without a matching `db.version(N).stores()` block, the same class of failure will recur; that's a latent risk in the `CollectionName = string` typing (no compile-time enforcement that `COLLECTION_INDEXES`/`buildVersionChain` cover every key in `COLLECTIONS`), explicitly not fixed by this card.
- **`seedAll`/`crud.ts`'s legacy `describe('sales', ...)` block remain broken** after this card (Overview point 2) — `npm run test` (the full suite) will **not** be fully green because of this pre-existing, unrelated issue. This plan's acceptance bar is scoped to: the new reporting conformance suite passes on both adapters, and all new reporting test files pass — not "the entire pre-existing suite is green," which was already false before this card started.
- **Margin cost-unavailable rows must never silently coerce to `0` cost** — a variant sold before its first recorded purchase movement (e.g. initial inventory seeded via `stocktake`/`adjustment` rather than `purchase`) would otherwise show an inflated, wrong margin. The conformance suite includes an explicit cost-unavailable case (Test Plan) to prevent this regressing silently.
- **Floating-point summation** — tax/discount/total sums use plain JS number addition across many rows; the fixed dataset above is chosen so every intermediate and final value is exact to the cent in binary floating point (verified: `0.48+0.20=0.68`, `9.18+3.24=12.42`, `9.18+8.10=17.28`, etc. — no repeating-binary-fraction totals), so tests can use `toBe`/exact equality rather than `toBeCloseTo`. If a future report adds inputs that don't sum exactly, switch that specific assertion to `toBeCloseTo`, not the whole suite.
- **`countSalesByHour` timezone conversion via `Intl.DateTimeFormat` must use the passed `timezone` argument, never the host/CI machine's local timezone** — the fixed dataset's `createdAt` values are chosen at UTC hour boundaries (`09:15`, `14:30`) specifically so a `timezone: 'UTC'` test is simple and deterministic; a follow-on card wiring a UI could pass `ResolvedSettings.timezone` instead without changing the repository contract.
- **Branch-manager multi-branch aggregation is explicitly not offered** (Architecture Decision 3) — an admin assigned to three branches must call each report three times, once per `branchId`. If a future request wants "all my branches at once," that's a new, additive repository parameter (`branchIds: string[]`), not a breaking change to this interface.
- **`ReportingService`/actions perform no audit logging of report views** — the `audit` domain (`AuditRepository`) tracks entity mutations, not read access; "auditor: read-only" in the card is read as an access-level constraint (the `VIEWER` role can read but never triggers the one write path, shift-close), not a requirement to log every report view. If audit-logging of report access is actually wanted, that's a distinct, explicit follow-up.
- **`Payment` has no `shiftId` field** (only `saleId`) even though the IndexedDB `COLLECTION_INDEXES['payments']` index string lists `shiftId` as an index key — a pre-existing, harmless drift (verified: `Collection.filter` never uses Dexie indexes, it reads via `getAll()` and filters in JS, so the stale index string causes no runtime bug). `sumPaymentsByMethod`/Z-report cash calculations correctly join `Payment` to `Sale` via `saleId`, never via a nonexistent `Payment.shiftId`.
