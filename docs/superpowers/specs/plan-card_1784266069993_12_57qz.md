# Plan: Customers, Promotions, Purchasing — close out contract (tests + infra completeness)

## Overview

**This plan supersedes an earlier draft of the same card.** That draft assumed the three domains were skeletal (no `LoyaltyService`, no `promotion-engine.ts`, no `GoodsReceipt` entity). Re-reading the repository now shows the domain layer has since been built out substantially: entities, repository interfaces, and services (`LoyaltyService`, `StoreCreditService`, `PurchaseHistoryService`, `promotion-engine.ts` + `PromotionEngineService`, `PurchasingService`) already exist and match the shapes the card describes (loyalty accrual/redemption, store credit, promotion stacking/combo/happy-hour/coupon logic, supplier/PO/goods-receipt flow). This plan is scoped to what's **actually** still missing, verified by reading the current source, not re-derived from the old assumption.

Concrete gaps found:

1. **`CorePromotionsRepository` (infra) does not implement `PromotionsRepository` (domain interface).** Missing `listActiveForOrg`, `findByCodeForOrg`, `saveRedemption`, `countRedemptions`, `countRedemptionsForCustomer`; its `redemptionCollection` field is declared but never constructed. This fails TypeScript structural typing wherever the class is assigned to the interface (`build-repository-set.ts`). `PromotionEngineService.evaluate`/`.redeem` cannot run against real storage until this is fixed.
2. **`CorePurchasingRepository` (infra) does not implement `PurchasingRepository` (domain interface).** Missing `listSuppliersByOrg`, `listPurchaseOrdersByOrg`, `saveGoodsReceipt`, `findGoodsReceiptById`, `listGoodsReceiptsForPurchaseOrder`; no `goodsReceipts` collection wired in the class. `PurchasingService.receiveGoods` cannot persist a `GoodsReceipt` against real storage until this is fixed.
3. **No cost snapshot on `StockMovement`.** `GoodsReceiptLine.unitCost` is already captured, but the stock ledger (`StockMovement`) has no cost field, so a future margin report (which reads the ledger) has nothing to join purchase cost against. `InventoryService.receivePurchase`/`applyMovement` don't accept or store a cost today.
4. **No server action wraps `PurchasingService.receiveGoods` in a transaction.** `PurchasingService` is storage-agnostic — it takes `repos` as a parameter, exactly like `FinalizeSaleService`/`sales/actions/complete-sale.ts` — so atomicity is the caller's job. Purchasing has no `actions/` folder yet; nothing currently drives `receiveGoods` end-to-end inside one `provider.withTransaction`.
5. **Conformance suite + fixtures don't cover the new entities.** `loyaltyTransactions`, `storeCreditTransactions`, `promotionRedemptions`, `goodsReceipts` are already registered in `schema.ts#COLLECTIONS` — so `exportAll`/`importAll` already handle them generically, **no export-import code changes are needed**, this plan only needs to prove that with a test. `conformance/fixtures.ts` has no `make*` factories for these four, `seedAll` doesn't seed them, and `conformance/suites/crud.ts` has no CRUD coverage for them.
6. **No tests exist for any of the three domains today.** `promotion-engine.ts`, `loyalty-service.ts`, `store-credit-service.ts`, `purchasing-service.ts` all have zero coverage.

This plan fixes gaps 1–4 (infra completeness the domain code needs to actually run) and then delivers exactly what the acceptance line asks for: a promotion-engine unit suite, a loyalty accrual/redemption integration test, a goods-receipt atomicity test, and conformance/fixture coverage for the four new entity types — verifying (not assuming) that JSON export already works generically.

**Out of scope, explicitly:** wiring loyalty/promotion evaluation into `finalize-sale-service.ts` (checkout flow), any UI/screens, and a margin-reporting read API. The card's "feeding margin reporting" is satisfied by making cost available on the stock ledger; computing/reporting margin is a separate future card. `finalize-sale-service.ts` is already committed, already tested (`sales-service.test.ts`, `discount-policy.test.ts`, `tax-resolver.test.ts`, `complete-sale.test.ts`), and rewiring it risks regressing that surface for a change the acceptance criteria doesn't require.

**Skills**: no `.agents/skills/` directory exists in this repository (verified — zero matches walking the repo for `SKILL.md`). There are no project-specific skills to apply here.

## Architecture Decisions

1. **Atomicity via `provider.withTransaction`, not a bespoke `UnitOfWork` object.** `StorageProvider.unitOfWork()` is an explicit stub (`throw new Error('UnitOfWork interface not yet fully implemented')`) and is used nowhere in the codebase. The established pattern (`complete-sale.ts`, `finalize-sale-service.ts`) is: domain services are storage-agnostic and take `repos` as a parameter; the caller (a server action) opens `provider.withTransaction(repos => ...)`; the in-memory driver (`memory-driver.ts#runTransaction`) stages all writes into a scratch copy and only commits it to the backing store if the callback resolves without throwing. Follow this exact pattern for `receive-goods.ts`. Do not implement `unitOfWork()` — that's a larger, separate change outside this card.

2. **Cost snapshot lives on `StockMovement`, not a new entity.** `StockMovement` is the append-only ledger the card means by "purchase StockMovements and cost snapshots ... feeding margin reporting." Adding `unitCost?: number` (populated only for `movementType: 'purchase'`) lets a future margin report join sale movements against purchase movements by `variantId`/`branchId` directly on the ledger, without a second lookup into `GoodsReceipt`. Purely additive — `recordSale`/`recordReturn`/`adjustStock`/`transferStock` call sites are unaffected since the field stays `undefined` for them.

3. **`receive-goods.ts` mirrors `complete-sale.ts` exactly** (zod schema, `requireUser()`, `createDefaultStorageProvider()`, try/finally `provider.close()`, `toErrorResponse` on catch, plus a `*WithProvider` test seam) — the one established action convention in the repo, not a new one.

4. **Fixture factories follow the existing `make*(overrides)` + `FixtureOverrides` convention** exactly (destructure `id`/`orgId` out of overrides, spread `rest`, use `FIXED_DATE`). `makeGoodsReceipt` inlines a default `GoodsReceiptLine`, same as `makePurchaseOrder` inlines a default `PurchaseOrderLine`.

5. **Promotion-engine unit tests are pure-function, no storage** — matches `discount-policy.test.ts`/`tax-resolver.test.ts`. Timezone cases pass explicit IANA zone strings and fixed UTC `Date` instants (never system-local time or `vi.useFakeTimers`) since `isTimeWindowActive`/`isPromotionActive` already take `now`/timezone as explicit parameters.

6. **Loyalty tests are integration tests against a real `StorageProvider`** (memory engine) per the card's explicit wording ("loyalty accrual/redemption **integration** test") — `LoyaltyService` is thin orchestration over `CustomersRepository`; the interesting behavior (balance persistence, insufficient-balance rejection leaving no partial state, transaction history) only shows up against real storage, not a mocked repo.

7. **Goods-receipt atomicity is proven the same way `conformance/suites/unit-of-work.ts` already proves generic rollback**: seed a PO via one `provider.withTransaction`, run `receiveGoods` inside a second `withTransaction` that is forced to fail after the service call completes, then reopen a third transaction and assert the `PurchaseOrder`, `StockLevel`, `StockMovement` list, and `GoodsReceipt` collection are all byte-for-byte unchanged from before the failed attempt. This is the storage-transaction guarantee the card means by "in one UnitOfWork" — it is *not* testing that `receiveGoods` throws partway through on its own (it doesn't, internally, in the happy path); it's testing that the transaction boundary wrapping the whole operation is all-or-nothing.

## State Management Strategy

Not applicable in the frontend sense — this is backend domain/service/storage work only. The relevant "state" is persistence state inside `StorageProvider`/`StorageDriver`:
- New/fixed service and repository methods keep the existing convention of taking `repos: { ... }` explicitly rather than holding storage state on the instance — services stay stateless across transactions.
- Atomicity boundary is one `provider.withTransaction(...)` call per logical operation (goods receipt = PO update + goods-receipt write + N inventory movements, all inside one call), matching `finalize-sale-service.ts`'s one-transaction-per-sale boundary.
- No new caching or module-level mutable state is introduced.

## Test Plan

**`src/domains/promotions/lib/promotion-engine.test.ts`** (unit, pure functions):
- `isPromotionActive`: inactive → false; before `validFrom` → false; after `validTo` → false; active with no window → true; `happy_hour` with `activeWindow` outside window → false, inside window → true.
- Cross-timezone case: a promotion with `activeWindow: { startTime: '17:00', endTime: '19:00', daysOfWeek: ['Fri'] }`, `timezone: 'America/New_York'`, evaluated at a UTC instant that is Friday 21:30 UTC (17:30 EST) → active; the same UTC instant evaluated against `timezone: 'Asia/Dhaka'` (23:30 local, outside any window) → inactive. This is the explicit "time windows across timezones" acceptance case.
- `evaluatePromotions` stacking: two `stackable: true` promotions both apply; second is computed off the running subtotal after the first (assert the exact compounded number, not just "some discount").
- `evaluatePromotions` exclusivity: two `stackable: false` promotions — only the higher-discount one applies; tie broken by lower `priority` value (read directly from `selectBestExclusivePromotion`'s comparison — verify the direction before asserting it, and note in the test which way "lower priority number wins" goes).
- Exclusive + stackable mix: exclusive applies first, stackable promotions apply on top of the reduced subtotal.
- Combo: `kind: 'combo'` with `comboLines` unmet by the cart → not applied at all (never partial); met → applied per `comboDiscountType`.
- Coupon-gated promotions (`requiresCode: true`) simply aren't in `candidates` unless resolved — this is `PromotionEngineService.resolveCandidates`'s concern (already implemented, throws `InvalidCouponCodeError` for bad codes), not `promotion-engine.ts`'s; a `promotion-service.test.ts` covering that is optional/nice-to-have, not required by the acceptance line (which says "promotion engine").
- Total discount clamps to cart subtotal (`Math.min` in `evaluatePromotions`).
- Percentage discount rounding: `Math.floor(x * 100) / 100` — test a case that exposes floating-point rounding (e.g. 33.33% of $10), and a fixed discount that exceeds subtotal (clamped).

**`src/domains/customers/services/loyalty-service.test.ts`** (integration, real `StorageProvider`, memory engine):
- Seed a customer with `loyaltyPointsBalance: 0` via `provider.withTransaction`.
- `resolveEarnedPoints`/`pointsToAmount` pure cases: `loyalty.enabled: false` → 0; `pointsPerDollar: 2` on $50 → 100 (floor applied); `pointsToAmount` with `redemptionValuePerPoint: 0.01` and 100 points → $1.00.
- `accrue(100)` persists `loyaltyPointsBalance: 100` and a `LoyaltyTransaction` (`type: 'accrual'`, `balanceAfter: 100`) retrievable via `listLoyaltyTransactions`.
- `redeem(40)` after the accrual leaves balance `60`, transaction `type: 'redemption'`, `points: -40`.
- `redeem` beyond balance throws `InsufficientLoyaltyPointsError`, and — the integration-test-specific assertion — re-reading the customer and transaction list in a fresh transaction shows **no** change from the throw.
- Two sequential operations produce a correctly ordered/chained transaction history.
- Settings-driven end-to-end: resolve `ResolvedSettings` via `resolveSettings`/`resolveSettingsFrom` (`@domains/organization/services/settings-resolver`) with a custom `loyalty.pointsPerDollar`, feed `resolveEarnedPoints(settings.loyalty, subtotal)` into `LoyaltyService.accrue`, assert the final balance matches — this is the "driven by settings rules" behavior the card names.
- Recommended (not required): mirror the same accrue/redeem/insufficient-balance shape for `StoreCreditService` in the same file or a sibling `store-credit-service.test.ts`, since it shares the identical balance-underflow risk and the card lists store credit as in-scope.

**`src/domains/purchasing/services/purchasing-service.test.ts`** (integration, real `StorageProvider`):
- Seed a `Supplier` and a `PurchaseOrder` (`status: 'submitted'`, one line, `quantity: 50`, `receivedQuantity: 0`).
- Happy path: receive 50 → PO becomes `received`, `receivedAt` set, a `purchase`-type `StockMovement` with `quantity: 50` and the passed-through `unitCost` exists, `StockLevel` reflects +50, `GoodsReceipt` persisted and retrievable via `findGoodsReceiptById`/`listGoodsReceiptsForPurchaseOrder`.

- Over-receipt: receiving 60 of 50 throws `OverReceiptError` without `allowOverReceipt`; succeeds with it.
- Invalid status: receiving against an already-`received`/`cancelled` PO throws `InvalidPurchaseOrderStatusError`.
- Cost snapshot: resulting `StockMovement.unitCost` matches the corresponding `GoodsReceiptLine.unitCost`.
- **Atomicity (the card-required "goods receipt atomicity test")**: call `purchasingService.receiveGoods(repos, input)` inside a `provider.withTransaction(async repos => { await purchasingService.receiveGoods(repos, input); throw new Error('forced rollback') })`, then in a fresh transaction assert the `PurchaseOrder`, `StockLevel`, `StockMovement` list, and `GoodsReceipt` collection are all unchanged from their pre-attempt state — proving PO update + N stock movements + goods-receipt write commit or fail together, exactly the pattern already established in `conformance/suites/unit-of-work.ts`.

## Risks and Edge Cases

- **`PurchaseOrder.orgId` may not exist yet** — required by `listPurchaseOrdersByOrg`. Verify at implementation time (Step 2); adding it is small and additive but touches the `makePurchaseOrder` fixture default, same as `Supplier`/`Promotion` already do.
- **`provider.withTransaction` always stages all `COLLECTIONS`**, so atomicity tests should go through it (not a narrower `driver.transaction(...)` call) to get the real rollback guarantee under test.
- **`selectBestExclusivePromotion` tie-break direction**: lower `priority` value wins on an equal discount amount (read directly from the current implementation) — confirm this reading is still correct at implementation time before asserting it in a test.
- **Floating-point discount rounding**: `computeDiscount`'s `Math.floor(x * 100) / 100` needs a non-round-number test case to actually exercise the rounding path.
- **`LoyaltyNotEnabledError` is currently unused** — `LoyaltyService.accrue`/`.redeem` don't check `settings.loyalty.enabled` themselves; only the free function `resolveEarnedPoints` does (for "how many points to award," not "post this transaction"). Don't add an enabled-check inside `LoyaltyService` as part of this card — that's scope creep toward checkout wiring, which is explicitly out of scope here.
- **The atomicity test must inject the failure** (wrap-and-throw after the service call succeeds) rather than relying on a "natural" partial failure inside `receiveGoods` — document this in the test itself so it's clear what's actually under test: the transaction boundary's atomicity, not error-handling inside the service.
- **`GoodsReceiptLine.variantId` vs `PurchaseOrderLine.productId`** — `receiveGoods` already maps `poLine.productId` → `goodsReceiptLine.variantId`; this naming mismatch predates this plan and isn't this card's to fix — just be consistent with it in test fixtures (both `'product-001'` in the shared fixtures already).
- **Timezone test determinism**: use `Intl.DateTimeFormat` with explicit `timeZone` and fixed `Date` instants only, never `new Date()`/system-local time, so the cross-timezone test is deterministic regardless of CI runner timezone.
