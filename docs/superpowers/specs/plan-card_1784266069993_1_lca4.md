## Overview

This card builds the **engine-swappability foundation**: interfaces plus one reference implementation. Three deliverables:

1. **Repository interfaces for every bounded context** — `catalog, inventory, sales, payments, customers, purchasing, promotions, organization, auth (users), audit` — each declared in the canonical file `src/domains/<context>/repository.ts`, with **narrow, intent-named methods** and **plain-TypeScript entity types only** (no Dexie/IndexedDB/engine types anywhere in `src/domains/**`). `sales` and `inventory` already have interfaces under `repositories/<name>-repository.ts`; these are **relocated** to `repository.ts` so there is one convention across all ten contexts.

2. **Engine-agnostic storage core** in `src/infrastructure/storage/core/`:
   - **`StorageProvider`** — the bound result of the factory for the engine named in config: `{ repositories: RepositorySet, withTransaction, unitOfWork, migrate, exportAll, importAll, close }`.
   - **`createStorageProvider(config)`** — the factory: resolves an engine name (`env.NEXT_PUBLIC_STORAGE_ENGINE`, e.g. `indexeddb`) to a registered driver and returns a bound `StorageProvider`.
   - **`UnitOfWork`** — `begin/commit/rollback` semantics; **every repository write happens inside a transaction scope**, committed atomically or fully rolled back.
   - **`SchemaVersioner`** — an **ordered, engine-agnostic** list of migration steps (`SchemaStep[]`) that each engine adapter interprets through the shared collection API; core provides the generic runner.
   - **Canonical JSON export/import contract** — `exportAll()` / `importAll()` with **per-entity versioned envelopes**, lossless for `Date`, used for backup **and** for migrating data between engines.

3. **An IN-MEMORY reference adapter** (`src/infrastructure/storage/adapters/memory/`) that implements the full contract, **`docs/storage-adapters.md`** (exactly what a new engine must implement + the effort target), and a **reusable conformance test** the memory adapter passes.

**Key architectural lever — how "one adapter, one config change, nothing else" is achieved.** The per-engine surface is deliberately tiny: a new engine implements only a **`StorageDriver`** primitive — a set of named collections, each a key/value store, plus one **atomic `transaction`** primitive and a **persisted schema-version slot**. *Core builds everything domain-facing generically on top of that primitive*: the concrete `RepositorySet` (all ten interfaces), the `UnitOfWork`, `withTransaction()`, `migrate()`, and `exportAll/importAll`. Therefore:

- Adding a new **engine** = write one `StorageDriver` implementation + add one line to the engine registry. Nothing in `src/domains/**` or the rest of core changes.
- Adding a new **bounded context** = add one interface + entities + one thin core repository class + register its collection. No engine changes.

**Acceptance.** The in-memory adapter implements the full `StorageProvider` contract and passes `tests/storage/memory-adapter.test.ts`, which invokes the reusable `runStorageConformance()` suite (CRUD across contexts, transaction commit/rollback atomicity, UnitOfWork begin/commit/rollback, schema migration up to target version, and export→import round-trip including `Date` fidelity).

Layering (unchanged from the scaffold; storage core lives entirely in `infrastructure`):

```
app → actions → services → repositories (INTERFACES: src/domains/*/repository.ts) / entities
                                   ▲
infrastructure/storage ── implements every interface generically over a per-engine StorageDriver ─┘
shared / constants  ← importable by any layer
```

## Architecture Decisions

**1. Canonical interface location `src/domains/<ctx>/repository.ts`.** Mandated by the card and applied uniformly (sales/inventory relocated). One file per context, interface + entity imports only. Forces the eslint pattern change in section H.

**2. The per-engine primitive is `StorageDriver` — nothing more.** Collections are named key/value stores of JSON-serializable records keyed by `string` id. The driver exposes: open/upgrade to a `SchemaDescriptor`, an atomic `transaction(collections, mode, work)`, `getSchemaVersion`/`setSchemaVersion`, and `close`. Everything a domain sees (typed repositories, queries, UoW, migrate, export/import) is built in core over this. This is the mechanism that makes a new engine "one file."

```ts
// core/driver.ts
export type CollectionName = string
export interface SchemaDescriptor { version: number; collections: CollectionName[] }
export interface DriverTransaction {
  get<T>(c: CollectionName, id: string): Promise<T | undefined>
  getAll<T>(c: CollectionName): Promise<T[]>
  put<T extends { id: string }>(c: CollectionName, value: T): Promise<void>
  delete(c: CollectionName, id: string): Promise<void>
}
export type TxMode = 'readonly' | 'readwrite'
export interface StorageDriver {
  readonly engine: string
  open(schema: SchemaDescriptor): Promise<void>
  transaction<T>(collections: CollectionName[], mode: TxMode,
                 work: (tx: DriverTransaction) => Promise<T>): Promise<T>
  getSchemaVersion(): Promise<number>
  setSchemaVersion(version: number, tx?: DriverTransaction): Promise<void>
  close(): Promise<void>
}
export type DriverFactory = () => StorageDriver
```

**3. Generic repositories via `Collection<T>` — queries are `getAll`+predicate in the reference layer.** Each core repository class implements its domain interface by delegating to a `Collection<T>` bound to the active transaction. Intent-named queries (`listOpen`, `findBySku`, `listByEntity`) are expressed as `collection.filter(pred)` / `collection.find(pred)`. This keeps the driver tiny and every engine correct out of the box (no secondary-index contract to implement). Index-backed optimization is an *optional* future per-engine concern (documented as such) and never a correctness requirement — so it does not expand the mandatory adapter surface.

```ts
// core/collection.ts
export class Collection<T extends { id: string }> {
  constructor(private tx: DriverTransaction, private name: CollectionName) {}
  get(id: string) { return this.tx.get<T>(this.name, id) }
  getAll() { return this.tx.getAll<T>(this.name) }
  put(value: T) { return this.tx.put<T>(this.name, value) }
  delete(id: string) { return this.tx.delete(this.name, id) }
  async find(pred: (v: T) => boolean) { return (await this.getAll()).find(pred) ?? null }
  async filter(pred: (v: T) => boolean) { return (await this.getAll()).filter(pred) }
}
```

**4. All writes inside a transaction scope; `UnitOfWork` is the explicit form, `withTransaction` the scoped sugar.** Both route every repository operation through a single `DriverTransaction`. Repositories are **only** obtainable bound to a transaction (via `RepositoryContext`), so it is structurally impossible to write outside a UoW. `withTransaction` opens a driver transaction across all needed collections, builds the `RepositorySet` on that tx, runs the callback, and commits on resolve / rolls back on throw. `UnitOfWork.begin()` returns the bound repositories and defers commit/rollback to the caller.

```ts
// core/unit-of-work.ts
export interface UnitOfWork {
  readonly repositories: RepositorySet
  commit(): Promise<void>
  rollback(): Promise<void>
}
// StorageProvider.unitOfWork(): Promise<UnitOfWork>   // begin
// StorageProvider.withTransaction<T>(fn: (r: RepositorySet) => Promise<T>): Promise<T>
```

**5. Atomicity is the driver's job; the memory driver uses snapshot-and-restore.** `transaction()` in `MemoryStorageDriver` deep-clones (structured-clone) the touched collections into a staging area, runs `work` against staging, and on success swaps staging into the live store atomically; on throw it discards staging (original store untouched) and rethrows. This yields real begin/commit/rollback semantics for the reference engine and defines the contract IndexedDB's native transactions satisfy naturally. Records are cloned on read and write so callers cannot mutate stored state by reference.

**6. `SchemaVersioner`: ordered, declarative, engine-agnostic steps.** A `SchemaStep` has a target `version`, a `description`, and a `migrate(ctx: MigrationContext)` that operates through the same collection API (create/populate/transform records). Core's runner reads the driver's persisted version, runs every step with `version > current` in ascending order inside one transaction per step, and calls `setSchemaVersion` after each. Adapters "interpret" steps only insofar as they provide `open()` (ensure collections exist) and the transaction primitive — the step bodies are identical across engines. `MigrationContext` exposes `collection(name)`, `createCollection(name)`, and `renameField`-style helpers built on the primitive.

```ts
// core/schema-versioner.ts
export interface MigrationContext { collection<T extends {id:string}>(n: CollectionName): Collection<T>; createCollection(n: CollectionName): Promise<void> }
export interface SchemaStep { version: number; description: string; migrate(ctx: MigrationContext): Promise<void> }
export const schemaSteps: SchemaStep[] // ordered; step 1 creates the baseline collections for all 10 contexts
```

**7. Export/import: per-entity versioned envelopes + Date-safe codec.** `exportAll()` reads every collection in one readonly transaction and emits a `StorageExport` whose `entities` array holds one `EntityEnvelope` per collection (`{ entity, version, records }`). `importAll(data, { mode })` validates `format`/`formatVersion`, then within one readwrite transaction applies each envelope (`replace` clears the collection first; `merge` upserts by id). Cross-engine migration = `dst.importAll(await src.exportAll())`. `Date` fidelity is preserved by a `SerializationCodec` that tags dates as `{ "$date": "<ISO>" }` on export and revives them on import — lossless and engine-independent (no reliance on `JSON.stringify`'s lossy Date→string).

```ts
// core/export-import.ts
export interface EntityEnvelope<T = unknown> { entity: CollectionName; version: number; records: T[] }
export interface StorageExport {
  format: 'pos.storage.export'; formatVersion: 1;
  engine: string; schemaVersion: number; exportedAt: string; entities: EntityEnvelope[]
}
export type ImportMode = 'replace' | 'merge'
```

**8. `StorageProvider` + factory + registry.** `createStorageProvider` resolves the engine name against `engine-registry`, constructs the driver, `open()`s it to `CURRENT_SCHEMA_VERSION`, runs pending migrations, and returns the bound provider. Unknown engine → `UnknownEngineError` listing registered names.

```ts
// core/storage-provider.ts
export interface StorageConfig { engine: string }
export interface StorageProvider {
  readonly engine: string
  withTransaction<T>(fn: (r: RepositorySet) => Promise<T>): Promise<T>
  unitOfWork(): Promise<UnitOfWork>
  migrate(): Promise<void>
  exportAll(): Promise<StorageExport>
  importAll(data: StorageExport, opts?: { mode?: ImportMode }): Promise<void>
  close(): Promise<void>
}
export function createStorageProvider(config: StorageConfig): Promise<StorageProvider>
```

**9. Config var name: `NEXT_PUBLIC_STORAGE_ENGINE` (default `indexeddb`).** The card's example is `STORAGE_ENGINE=indexeddb`. Because production storage (IndexedDB) is **browser-side**, the selector must be readable in client bundles, so it must carry the `NEXT_PUBLIC_` prefix and live in `env.ts`'s `clientSchema`. Documented in `docs/storage-adapters.md`. Tests select the reference engine by passing `{ engine: 'memory' }` directly to `createStorageProvider` (no env dependency in tests).

**10. SSR safety / lazy construction.** No engine is constructed at module import. Drivers are built only inside `createStorageProvider` (called from client code or tests). The memory driver is environment-agnostic. The engine-registry uses side-effect imports for registration but registration only stores a factory function — it never opens a database. This preserves Next.js server-render safety and keeps the boundary rule (`no-restricted-globals: indexedDB`) satisfiable everywhere outside `src/infrastructure`.

**11. Errors extend `@shared/errors`.** Reuse the existing `AppError` hierarchy for consistent error surfacing; storage errors carry stable `code`s (`UNKNOWN_ENGINE`, `MIGRATION_FAILED`, `IMPORT_FORMAT_INVALID`).

## State Management Strategy

- **No UI/React state** — this card is pure domain + infrastructure. "State" here means **persisted storage state** and **transaction state**.
- **Single source of truth per record:** collections keyed by `id`. Repositories never hold their own caches; they read/write through the active `DriverTransaction`.
- **Transaction-scoped mutation:** all writes occur inside a `UnitOfWork`/`withTransaction` scope. Commit applies atomically; rollback (default on throw) discards. There is no ambient/global mutable store the app touches directly.
- **Schema version state:** persisted in a driver-owned version slot; advanced only by the `SchemaVersioner` runner, never by application code.
- **Isolation:** memory driver clones records on read and write so no caller retains a live reference into the store (prevents accidental shared-mutable-state bugs and makes rollback trustworthy).
- **Provider lifecycle:** a `StorageProvider` is created lazily (Decision 10) and `close()`d by its owner. No module-level singleton is created in this card; wiring a shared instance into `container.ts` is deliberately out of scope (foundation only).

## Accessibility Considerations

No user-facing UI, DOM, or ARIA surface is introduced by this card, so WCAG/interaction a11y is **not applicable**. The equivalent "accessibility" concern for this layer is **developer and operator ergonomics**, addressed by:
- **Actionable errors:** `UnknownEngineError` lists the registered engine names; `ImportFormatError` states the offending `format`/`formatVersion`; `MigrationError` names the failing step version and description.
- **Human-readable export format:** stable, indented, self-describing JSON envelopes (`entity`, `version`) so a backup file is legible and diff-able, and the Date codec keeps timestamps human-readable ISO strings.
- **Documentation clarity:** `docs/storage-adapters.md` gives a copy-pasteable checklist and effort estimate so a new contributor is not blocked. When any future UI consumes this storage, standard a11y review applies there, not here.

**Reusable conformance suite — `tests/storage/conformance.ts` (`runStorageConformance(name, makeDriver)`):** the contract every adapter must satisfy. A future engine gets coverage by adding one `*.test.ts` that calls this with its driver factory ("one adapter, one test line"). Cases:
1. **CRUD across contexts** — for each of the 10 repositories: put → `findById` returns equal record; list/query intent-methods return expected sets; `findById` of a missing id → `null`.
2. **Query semantics** — e.g. `sales.listOpen` returns only `status==='open'`; `catalog.findItemBySku`; `audit.listByEntity`; `auth.findUserByUsername`.
3. **Transaction commit atomicity** — `withTransaction` writing to ≥2 contexts commits both; both visible afterward.
4. **Transaction rollback atomicity** — a `withTransaction` that writes then throws leaves the store **unchanged** (no partial writes in any touched collection).
5. **Explicit UnitOfWork** — `unitOfWork()` → write → `commit()` persists; a second UoW → write → `rollback()` discards.
6. **Isolation** — mutating a record object returned from a read does not alter stored state on a subsequent read.
7. **Schema migration** — a fresh driver migrated to `CURRENT_SCHEMA_VERSION` reports that version; all baseline collections exist and are writable; re-running `migrate()` is a no-op (idempotent).
8. **Export/import round-trip** — seed multiple contexts → `exportAll` → `importAll` (mode `replace`) into a **second, empty** provider of the same engine → every record deep-equals the source, including `Date` fields (proves Date fidelity and cross-provider portability).
9. **Envelope shape** — export has `format:'pos.storage.export'`, numeric `formatVersion`, `engine`, `schemaVersion`, ISO `exportedAt`, and one envelope per collection with a numeric `version`.
10. **Import validation** — importing a malformed object (wrong `format`) rejects with `ImportFormatError`; `merge` mode upserts by id without clearing.

**`tests/storage/memory-adapter.test.ts` (ACCEPTANCE):** `runStorageConformance('memory', () => new MemoryStorageDriver())`. Must pass in full.

**`tests/storage/export-import.test.ts`:** focused unit tests for the `SerializationCodec` (Date tag/revive, nested Dates, null/undefined handling), `merge` vs `replace`, and **cross-engine simulation** (export from memory instance A, import into memory instance B) as the proxy for engine-to-engine migration until a second real engine lands.

**Regression:** existing suites must stay green — `tests/boundaries/boundaries.test.ts` (relies on the sales-service import path swap being valid) and `src/domains/*/services/*.test.ts`. Run `npm run test` + `npm run typecheck` + `npm run lint`.

**Not in scope:** Playwright/e2e (no UI); a real IndexedDB adapter.

## Risks and Edge Cases

- **`boundaries/no-unknown` on the new file location.** A `repository.ts` directly under `src/domains/<ctx>/` is unclassified until the eslint element pattern is widened (section H). *Mitigation:* Step 4 does this before relying on lint; verify with a lint run on a domain repo file.
- **Relocation breaking importers.** Moving the sales/inventory interfaces changes 4 import paths (services + dexie repos). *Mitigation:* grep for old paths before deleting; update all; keep the retained method signatures identical so no downstream code changes behavior.
- **`Money` duplication vs breakage.** Consolidating `Money` into `@shared/types` while `order.ts` re-exports it avoids breaking existing `@domains/sales/entities/order` importers. *Mitigation:* re-export; do not remove the named export.
- **Date loss through naive JSON.** `JSON.stringify(new Date())` yields a string that silently deserializes as a string. *Mitigation:* the tagged `SerializationCodec`; conformance case 8 asserts `instanceof Date` and equal time after round-trip.
- **`noUncheckedIndexedAccess`.** Collection lookups and `records[i]` are `T | undefined`. *Mitigation:* code defensively (`?? null`, guards); the existing repo already follows this convention.
- **Rollback must be truly atomic.** A half-applied transaction is the worst failure mode. *Mitigation:* memory driver stages a clone and swaps only on success; conformance case 4 asserts no partial writes across multiple collections.
- **`verbatimModuleSyntax`.** Type-only imports must use `import type`. *Mitigation:* follow the existing files' style (all entity/interface imports are `import type`).
- **SSR / `indexedDB` global.** Importing the storage barrel on the server must not touch a browser global. *Mitigation:* lazy driver construction (Decision 10); memory driver is universal; registration stores factories only.
- **Unknown engine at runtime.** A misconfigured `NEXT_PUBLIC_STORAGE_ENGINE`. *Mitigation:* factory throws `UnknownEngineError` listing registered names; `indexeddb` default is registered by a later card — until then only `memory` is registered, so document that selecting `indexeddb` now throws by design.
- **Query cost of `getAll`+filter.** Acceptable and correct for the reference/in-memory engine and small POS datasets; *documented* as the baseline with per-engine index optimization optional and non-mandatory, so it never grows the adapter contract.
- **Append-only audit.** `AuditRepository` intentionally omits update/delete; ensure the interface and core class expose no mutation beyond `append`.
