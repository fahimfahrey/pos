# Plan — Pay Flow, Receipt Template, and Customer Display Redesign

Card: `card_1784591953082_6_nkir`

## Overview

This card redesigns the three surfaces a sale touches after the cart is built: the **payment sheet** in `/pos/checkout`, the **receipt template** (80mm thermal, PDF, on-screen), and the **customer-facing display** (`/pos/display`). All three currently exist only as unstyled or generic-Tailwind scaffolding (`bg-gray-900`, `bg-blue-600`, hardcoded `'en-US'`/`'USD'`) predating the design-token system (`2c7881d`) and component library (`4a42a11`); `docs/design-direction-pos.md:306` explicitly scoped payments/receipts as **out of scope** for that doc, so this card originates the visual language for these three surfaces — it is not adapting an existing spec.

Three facts, verified directly against the live tree, anchor the scope decisions below:

1. **The payment flow is a stub that bypasses the real domain services.** `register-layout.tsx`'s `handleFinalizeSale` hand-writes a `Sale`/`SaleItem`/`Payment` directly (comment: `// Simplified sale creation (full service integration in future card)`), never calls `FinalizeSaleService`, never calls `PaymentService`/a gateway, hardcodes `taxRate: 10` for every line regardless of category, and sets `Payment.status: 'completed'` (a string, not a `PAYMENT_STATUS`). There is **no gateway-pending, failed, or refunded state possible today** because nothing ever calls `gateway.charge()`. Building the six-state payment sheet the card asks for is impossible without wiring real charging — this plan does that wiring, scoped tightly to the payment/finalize path (see Architecture Decision 1).
2. **A pre-existing, unrelated compile-blocking bug sits directly in this card's path.** `pos-cart-context.tsx`, `cart-row.tsx`, and others import `CartLine` from `@domains/sales/entities/cart-line` — **this file does not exist** (`src/domains/sales/entities/` contains no `cart-line.ts`; confirmed by directory listing). Separately, `cart-pricing-service.ts` imports `Variant` from `@domains/catalog/entities/variant` — **also does not exist** (the real entity is `product-variant.ts`, interface `ProductVariant`, which has no `sellingPrice` field at all; pricing resolution from variant → price is an unimplemented catalog-domain gap). `cart-pricing-service.ts` has zero importers anywhere in the codebase — it is dead, never-compiled-against scaffolding. This means `/pos/checkout` almost certainly fails `npm run typecheck` today, independent of anything in this card. Fixing catalog→price resolution is **out of scope** (a catalog-pricing card, not a pay-flow card). Fixing the missing `cart-line.ts` entity is a one-file, ~10-line unblock with a shape already fully determined by every consumer in the checkout module — this plan includes it as a Step 0 prerequisite so the rest of the work is buildable and testable (see Architecture Decision 2).
3. **The customer display is not wired to the register at all.** `register-layout.tsx` never calls `useCartBroadcast(...).publish(...)`. `/pos/display` subscribes to a channel that nothing publishes to. "Polish the customer display as a mirror of the cart" requires first making it an actual mirror — this plan wires the publish side, not just the visual layer.

**No `.agents/skills/` directory exists in this repository** (`find . -iname SKILL.md`, `find . -maxdepth 4 -iname "*skill*"` both return zero hits, confirmed at the start of this planning session and consistent with the same finding recorded in `docs/superpowers/specs/plan-card_1784591953082_5_77la.md`). There is no project skill to apply. Implementation follows the conventions already established by `src/shared/components/ui/*` (task 4/5, commit `4a42a11`), the token system in `src/app/globals.css` (task 3, commit `2c7881d`), and `docs/design-direction-pos.md`'s register conventions (task 2, commit `1caa72d`) — verified by direct file reads, not skill lookups.

### What "warm aesthetic" and "semantic status colors from task 4" mean concretely here

`src/app/globals.css` defines exactly three semantic status tokens: `--success` (sage), `--danger` (brick), `--warning` (amber) — no `--info` token exists, and none should be invented (the design system is a closed three-color semantic palette by deliberate choice, confirmed by `docs/ui-audit.md`'s repeated "no new colors" language). The six payment-sheet states map onto this closed palette as follows:

| State | Token / Badge variant | Rationale |
|---|---|---|
| Cash (method) | neutral — no status color, it's a method not an outcome | Cash settles synchronously; the sheet shows tender/change UI, not a status badge |
| Store credit (method) | neutral — same as cash | Settles synchronously against `StoreCreditService` |
| Gateway-pending | `--warning` (amber), `Badge variant="warning"` | "Needs attention / in progress," matches the amber semantic already used for warnings elsewhere |
| Succeeded | `--success` (sage), `Badge variant="success"` | Matches `PAYMENT_STATUS.CAPTURED` |
| Failed | `--danger` (brick), `Badge variant="danger"` | Matches `PAYMENT_STATUS.FAILED` |
| Refunded | `Badge variant="secondary"` (neutral border/surface, no color) | Deliberately **not** `--danger` — a refund is a completed, correct outcome, not an error; reusing the existing neutral `secondary` badge variant (already defined in `badge.tsx`) avoids inventing a fourth status color while keeping refunded visually distinct from both success and failure |

### New payment gateway adapter (simulated card processor)
- `src/infrastructure/payments/adapters/card/gateway.ts` (**new**) — `CardGateway implements PaymentGateway`, `id = 'card'`, `requiresServer = true`. Mirrors `FakeGateway`'s `behavior` constructor pattern for testability: `constructor(private behavior: 'capture' | 'decline' | 'pending' | 'random' = 'random', private latencyMs = 1200, private pendingResolutionMs = 4000)`. Two distinct async shapes, both achievable through the existing `PaymentGateway.charge()` contract (no port changes needed):
  - `'capture'` / `'decline'`: `await delay(latencyMs)` then resolve `CAPTURED`/`FAILED` directly — this is the ordinary "card swipe, wait ~1.2s" in-flight spinner case.
  - `'pending'`: resolves **immediately** with `PAYMENT_STATUS.PENDING`, recording `{ idempotencyKey, resolvesAt: now + pendingResolutionMs, outcome }` in an in-memory `Map`. A **second** `charge()` call with the same `idempotencyKey` (the payment sheet's "Check status" retry / auto-poll) checks the map: before `resolvesAt` → still `PENDING`; after → returns the cached final `outcome`. This gives a genuine, revisitable pending state (matches `payment-status-machine.ts`'s already-modeled `PENDING → CAPTURED | FAILED` transition) rather than a purely cosmetic spinner.
  - `'random'` (default): weighted per attempt — 80% `'capture'`, 10% `'decline'`, 10% `'pending'`.
- `src/infrastructure/payments/adapters/card/index.ts` (**new**) — `registerPaymentGateway('card', () => new CardGateway())`, following the exact pattern of `adapters/cash/index.ts`.
- `src/infrastructure/payments/adapters/manual/gateway.ts` (**new**) — `ManualGateway`, `id = 'manual'`, `requiresServer = false`. For `check`/`other`: synchronous `CAPTURED`, computes `changeDue` the same way `CashGateway` does when `tendered` is provided, otherwise `changeDue = 0`. No `InsufficientStoreCreditError`-style failure path — these are manual/offline tenders the cashier attests to.
- `src/infrastructure/payments/adapters/manual/index.ts` (**new**) — `registerPaymentGateway('manual', () => new ManualGateway())`.
- `src/infrastructure/payments/index.ts` — add `import './adapters/card'` and `import './adapters/manual'` alongside the existing `cash`/`store-credit` imports.
- `src/infrastructure/payments/adapters/card/gateway.test.ts` (**new**) — runs `runPaymentGatewayConformance('card', ...)` (existing suite in `payment-gateway-conformance.ts`) plus behavior-specific tests for each of the four `behavior` values, including the two-call pending→resolved sequence.

## Architecture Decisions

**1. Charging moves inside `FinalizeSaleService.finalize()`, not into a separate pre-step.** The alternative — charge via `PaymentService` first, then call `finalize()` with already-settled payments — would either double-create `Payment` rows (finalize already creates its own) or require threading pre-built `Payment` objects through `FinalizeSaleInput`, which changes its contract more invasively than fixing the one block that's wrong. Charging inside `finalize()` keeps the "only persist a `PAID` sale if every payment captured" invariant enforceable in one place, and keeps `PaymentService.chargeSplitPayments` (already correct, already handles idempotency) as the thing `finalize()` calls rather than duplicating its logic.

**2. Step 0 fixes only the missing `cart-line.ts` type, not the missing catalog pricing.** These are two independent bugs found during research. The missing `CartLine` entity is a pure type hole — every field it needs is already used consistently by five+ files; writing it down is mechanical and unblocks compilation for the whole checkout module. The missing `variant.sellingPrice`/catalog pricing resolution is a real, unimplemented cross-domain feature (how does a variant get a price at all — direct field? price list? which price list for which channel/branch?) that this card's brief does not ask about and that guessing at would risk building the wrong thing. `CartPricingService` (which depends on it) is dead code with zero importers — leaving it unused and unfixed changes nothing observable. Flagged prominently in Risks so it isn't silently rediscovered later as "why does checkout still not typecheck."

**3. Per-line tax rate at finalize time uses `settings.taxRules[0]?.rate` (the same no-category fallback `resolveTaxRate()` already uses), not full per-category resolution.** True per-category tax (what `resolveTaxRate(settings, category)` is built for) requires a `Category` lookup keyed off the variant, which requires the same catalog-pricing gap from Decision 2 to be closed first (today's ad-hoc `CartLine` carries no `categoryId`). This plan's fix is strictly better than today's hardcoded `10` (respects tenant-configured rates, correctly produces a single-rate breakdown for single-tax-rule tenants — the common case) while being honest that multi-category multi-rate breakdowns remain blocked on the catalog gap. This is called out as a known limitation in Risks, with the receipt's tax-breakdown *rendering* (which is rate-count-agnostic — it already groups by whatever rates are present) being correct regardless of how many distinct rates ever actually reach it.

**4. "Fast to clear" success state uses a fixed dwell + one dominant action, not an instant auto-close.** An instant auto-close risks the cashier not seeing the sale actually succeeded (a real failure mode in POS UX — "did that go through?"). A fixed ~1.5s glance window on `payment-status-view.tsx`'s per-tender success, followed by a `sale-complete-view.tsx` screen that requires exactly one tap ("New sale") to clear, balances "unambiguous" against "fast": the cashier sees confirmation, then one press returns to a scan-ready register. No confirmation dialog on "New sale" — a completed sale is already durably persisted, so there's nothing destructive to protect against.

**5. Thermal receipt hierarchy stays within the existing monospace/plain-CSS renderer; it does not adopt `font-display`/Fraunces.** `render-receipt.ts` produces an HTML string via `renderReceiptHtml()` shared verbatim by the browser-print path and the PDF path (`pdf/printer.ts` wraps the same `renderReceiptHtml()` output in its own `<html>` shell with `THERMAL_RECEIPT_CSS`) — a real 80mm thermal printer has no access to `next/font`-loaded Fraunces (it rasterizes CSS, but only if the physical printer driver/firmware supports arbitrary web fonts, which cheap ESC/POS thermal printers generally do not; monospace is the safe, universal choice already made here). Hierarchy is built with size/weight/border/spacing within the monospace family instead of introducing a font dependency that would silently fall back and look broken on real hardware — this is the actual reason cheap thermal receipts industry-wide use bold/size/rules for hierarchy rather than font pairing, and it directly serves "legible when printed on cheap thermal paper."

## State Management Strategy

**`use-payment-sheet.ts`** — a `useReducer` state machine, colocated with the existing `pos-cart-context.tsx` reducer pattern already used in this module (consistency with the surrounding code, not a new pattern):

```
type Tender = {
  id: string                 // client-generated, becomes the idempotencyKey
  method: PaymentMethod
  amount: number              // major units, remaining-due-capped
  tendered?: number            // cash only
  customerId?: string          // store credit only
  status: 'draft' | 'charging' | 'pending' | 'captured' | 'failed'
  failureReason?: string
  gatewayRef?: string
  chargeStartedAt?: number
}

type SheetState = {
  phase: 'collecting' | 'charging' | 'declined' | 'complete'
  tenders: Tender[]
  remainingDue: number        // derived, recomputed on every tenders change
  saleResult: FinalizeSaleResult | null
}
```

- `addTender` / `removeTender` mutate the `'draft'` list; `remainingDue` is always `total - sum(captured tenders) - sum(in-flight draft/charging/pending amounts)`, recomputed, never stored redundantly to avoid drift.
- `chargeCurrent(tenderId)` moves one draft tender through `'charging'` → (`'captured'` | `'pending'` | `'failed'`) by calling the gateway indirectly via a server-callable path — since gateway adapters live in `src/infrastructure` and the checkout module is a client component tree, the actual charge call goes through the same `createDefaultStorageProvider()` + repo pattern already used by `handleFinalizeSale` today (client-side IndexedDB provider, no network boundary in this app's current architecture — confirmed by every existing `register-layout.tsx` handler using this exact pattern).
- Once every tender is `'captured'` and `remainingDue === 0`, `use-payment-sheet.ts` calls `finalizeSale()`, which invokes `FinalizeSaleService.finalize()` with the captured tenders as `input.payments`. On `outcome: 'paid'` → `phase: 'complete'`. On `outcome: 'declined'` (should not normally happen here since tenders are pre-verified captured, but the finalize-time gateway call could still race — e.g., a second browser tab finalizing the same idempotency key) → `phase: 'declined'`, surfaced via `payment-status-view.tsx`'s failed state on the offending tender.
- **Why the sheet re-verifies rather than trusting client-side "already captured" state blindly:** `finalize-sale-service.ts` re-runs the charge inside its own transaction-shaped block (Architecture Decision 1) rather than accepting pre-captured `Payment` objects as-is, so the sheet's local `'captured'` tender status is provisional UI state, and the authoritative outcome is always what `finalize()` returns. This double-charge risk is mitigated by idempotency keys (`tender.id` reused as `idempotencyKey` end to end) — `PaymentService.chargeSplitPayments` already short-circuits on `existing` payment lookup by id.
- **Interruption safety, scoped honestly:** the sheet blocks accidental dismissal mid-charge and mid-split (Files list, `payment-sheet.tsx`). It does **not** persist across a full page reload — `register-outbox.ts` is explicitly in-memory-only today ("Does not persist across page reloads," its own doc comment), and building durable cross-reload payment-sheet recovery is a materially larger, separate piece of work (a real outbox/sync engine) that this card's brief does not ask for. This is called out in Risks, not silently solved or silently ignored.

**`use-resolved-settings.ts`** — plain `useState` + one-shot `useEffect` bootstrap (same shape as `use-register-session.ts`), no reducer needed (settings don't change within a session).

**Cart broadcast** — no new state management; `register-layout.tsx` gains one `useEffect` calling the already-existing `useCartBroadcast(registerId).publish(...)`, keyed off `[pricedCart, state.lines]`.

## Accessibility Considerations

- **Payment sheet fully keyboard-operable (explicit acceptance criterion).** Use `Sheet`/`SheetContent` (Radix-backed, per `dialog.tsx`/`sheet.tsx`) instead of the current hand-rolled overlay divs precisely because Radix supplies the focus trap, initial-focus-on-open, and Escape-to-close (overridden per Architecture Decision above only during active charging) for free — `payment-panel.tsx` and `manager-override-modal.tsx` currently forfeit all of this by not using `DialogContent`. `Tabs` (method selector), `NumberField` (amount/tender), and `Button` (denominations, actions) are all already keyboard-operable per their existing implementations (task 4/5) — this card's job is to not break that by reaching for raw `<div onClick>` anywhere. `quick-denomination-grid.tsx` buttons need a visible, logical tab order (left-to-right, ascending denomination) and each needs an accessible name beyond the currency symbol alone (`aria-label="Add $20"`) since screen-reader users shouldn't have to parse a bare "$20" as an action vs. a value.
- **`change-display.tsx` / running-total-style blocks** carry `aria-live="polite"` on the change amount specifically (not the whole sheet) so a screen-reader user hears the change update as tendered amount changes, without the entire sheet being re-announced on every keystroke.
- **`payment-status-view.tsx`**: `pending` and `failed` states use `role="status"`/`aria-live="polite"` for pending, `role="alert"` (assertive) for failed — a declined card is exactly the kind of event that shouldn't wait for polite-queue timing, matching the `toast.tsx` convention already established for danger variants (per `plan-card_1784591953082_4_ymou.md`'s note on Radix Toast defaults, verified not just assumed).
- **Badges are never color-only.** Every status `Badge` (success/danger/warning/secondary) pairs with a text label ("Succeeded," "Failed," "Pending," "Refunded") — the component library's `Badge` already renders children as text, so this is a usage discipline (always pass a label), not a component change.
- **Receipt (`receipt-view.tsx`)**: the `.receipt-container` inner HTML is generated via `dangerouslySetInnerHTML` from a pure-string builder (`renderReceiptHtml`) with no interactive elements inside it — no ARIA concerns inside the receipt body itself, but `receipt-actions.tsx`'s Print/PDF buttons get proper `aria-label`s (already present) preserved through the `Button` component swap, plus `aria-live="polite"` region for the printing/success/error status text (already present, preserved).
- **Customer display (`/pos/display`)**: not an interactive surface (no inputs), but the "Thank you" transient state (Architecture Decision on `CartSnapshot.status`) should not use `aria-live` at all — this display typically runs unattended/muted-role on a consumer-facing screen and abrupt live-region announcements aren't the target use case; large-type visual legibility is the actual requirement (`text-display-2xl`+ throughout, matching the "meter away" legibility bar the payment sheet is held to).
- **Focus-visible**: every new interactive element relies on the existing global `*:focus-visible { outline: var(--focus-ring-*) }` rule (`globals.css`) — no component may suppress it with a bare `outline-none`.

## Risks and Edge Cases

- **Pre-existing broken imports beyond this card's fix.** Step 0 fixes `cart-line.ts`. If `npm run typecheck` (or `npm run dev`) still surfaces other pre-existing errors unrelated to payments/receipts/display when this work begins, do not silently expand scope to fix them — note them and proceed; they predate this card (confirmed: `cart-pricing-service.ts`'s broken `Variant` import is a second, independent instance of the same class of bug, deliberately left unfixed per Architecture Decision 2).
- **Catalog pricing gap blocks true per-category tax breakdown.** Documented in Architecture Decision 3. If a tenant configures multiple `taxRules` for different categories, this card's receipt will still show a correct-but-single-rate breakdown (using the tenant's first/default rule) rather than a true multi-rate split, until a separate catalog-pricing card resolves variant→price(→category) properly. Flag this limitation to the user/reviewer explicitly rather than letting "correct tax breakdown per rate" silently mean "correct assuming one rate."
- **Card gateway is simulated, not a real processor integration.** `CardGateway`'s `'random'` behavior is a deliberate design choice to make the gateway-pending/failed states exercisable in a demo/dev environment with no real payment processor credentials available. If/when a real processor (Stripe Terminal, Square, etc.) is integrated, `CardGateway` is the file to replace — the `PaymentGateway` port contract, `resolveGatewayForMethod` mapping, and the payment sheet's state machine should all remain unchanged, since they were built against the port interface, not the fake implementation's internals.
- **Split payment partial-capture-then-abandon.** If a cashier captures a card tender, then cancels the sale before store credit covers the rest, the card tender is already `CAPTURED` (money has moved) but no `Sale` gets persisted. `payment-sheet.tsx`'s cancel-confirmation (Architecture Decision, `payment-sheet.tsx` entry) warns about this, but does **not** auto-refund the captured tender — auto-refund-on-cancel is a meaningfully different, riskier feature (silently moving money a second time) that isn't part of this card's brief; the confirmation copy should tell the cashier to manually void/refund via existing tooling instead of the sheet silently doing it.
- **Interruption safety is scoped to in-session, not cross-reload**, per the State Management Strategy section — explicitly not "durable" in the sense a payments engineer might assume from the word. If the user wants true reload-survival, that's `register-outbox.ts` needing to become a real persisted queue — a separate, larger card.
- **`ManualGateway` has no failure mode.** Check/other tenders always succeed by design (the cashier is attesting to having received payment, there's no external system to decline). If a future requirement wants a manager-approval gate on `other` (a common real-world control against till fraud), that's an addition to `finalize-sale-service.ts`'s discount-override-style gate, not something this card's `ManualGateway` needs to anticipate.
- **Thermal print proof is a manual step.** No CI environment has a physical 80mm printer; the acceptance checklist's "reviewed at actual 80mm width" item is inherently a human-in-the-loop check (browser print-preview at unscaled 80mm, or a real printer if available) — call this out explicitly to whoever verifies acceptance so it isn't skipped as "covered by the render-receipt unit tests" (it isn't; those test data correctness, not physical legibility).
- **`CartSnapshot.status` is additive and optional** specifically so any other code that constructs a `CartSnapshot` today without a `status` field keeps compiling and behaving as `'cart'` — verify no other publisher of this channel exists before assuming this is risk-free (grep found none beyond `register-layout.tsx`, but re-check at implementation time since this plan doesn't lock the codebase).
