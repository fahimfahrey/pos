# Plan: First-Class Non-Happy States for Every Screen

## Overview

Every route in the app currently handles its unhappy paths ad hoc — a generic spinner (`src/app/loading.tsx`), a bare `<h1>Something went wrong</h1>` (`src/app/error.tsx`), inline `text-gray-500` "No X found" strings scattered through server components, and exactly one route (`/pos/checkout`) with anything resembling a considered offline/error/empty design. `docs/ui-audit.md` (2026-07-21) already establishes that every screen except `/pos/checkout` still renders in raw Tailwind gray/blue, not the warm token system from `docs/design-direction.md` / `docs/design-direction-pos.md`.

This card does **not** attempt the full token migration of every screen's happy path — that is the pre-existing, separately-tracked "wholesale migration" gap called out in `docs/ui-audit.md`'s Summary of Findings §1. This card's scope is narrower and orthogonal: give every route's **loading, empty, error, and offline** states a first-class, warm-language design, built from the existing token set and the existing `EmptyState`/`Skeleton`/`Toast`/`Badge` primitives in `src/shared/components/ui/`. Non-happy-state UI is added/upgraded in place on each screen without redesigning the happy-path content around it (with one necessary exception: the dashboard's static placeholder numbers are wired to real (currently-absent) data reads, because you cannot design a truthful empty/loading state for data that is hardcoded — see Architecture Decision 6).

**Local-first framing (governs every loading decision):** `NEXT_PUBLIC_STORAGE_ENGINE` defaults to `indexeddb`; per `docs/storage-adapters.md`, reads are in-process against a local database — there is no network round-trip on the common path. The brief's instruction to "reserve skeletons for genuine waits" is taken literally: pages whose data comes from a single local `withTransaction` read do **not** get a `loading.tsx` Suspense skeleton (Next.js would flash it for a handful of milliseconds, which reads as jank, not polish). Skeletons are reserved for the two genuine-wait cases in this codebase: (a) first paint before the IndexedDB connection opens (register bootstrap, `useRegisterSession`), and (b) route-level payload assembly that chains multiple sequential transactions/joins (receipt page: sale → items → payments → org → branch → settings, each currently a separate `await`).

**Offline framing:** Per `docs/design-direction-pos.md` and `use-online-status.ts`/`register-outbox.ts`, offline is already a first-class concept on the register screen — it does not block finalize (writes go to local IndexedDB regardless of connectivity) and is surfaced via an online/offline pill plus a pending-outbox count. This card generalizes that pattern (an `OfflineBanner` + a shared `usePendingSyncCount`) to every authenticated screen, and adds it, for the first time, to `/login` (which already has offline-capable session checking in `src/domains/auth/client/offline-auth.ts` that nothing in the UI currently surfaces).

## Architecture Decisions

### 1. Skeletons are opt-in per route, not a blanket default

Blanket-adding `loading.tsx` to every segment would violate the brief directly ("reserve skeletons for genuine waits"). Decision rule: a route gets a `loading.tsx` (or an in-component skeleton for client-fetched data) only if its data path does one of:
- Opens a fresh IndexedDB connection before any read can happen (register bootstrap).
- Chains ≥2 sequential dependent transactions/reads before first paint (receipt: 6 sequential reads; dashboard: 3 independent reads that can run concurrently — see below).
- Static/local-only pages (marketing, login, signup, admin hub, display's registerId-gate) get none.

Dashboard's 3 reads (today's sales, order count, recent orders) are independent, not chained — implement them with `Promise.all` inside one `withTransaction`, which keeps the total wait in the same "instant local read" bucket. Its `loading.tsx` is added anyway as a defensive UX floor for the (rare) first-ever-load-before-IndexedDB-populated case, sized small enough to not read as "designed for waiting."

### 2. Two error kinds, not one generic error screen

`RouteError`'s `kind` prop is not cosmetic — user and system errors need different actions:
- `kind="user"`: something the person can fix right now (wrong credentials, no org selected, missing URL param, insufficient role). Copy names the fix. No "Try Again" retry button (retrying won't help; the action is "go fix the input"), instead a contextual action (a link, a role-appropriate message).
- `kind="system"`: something broke underneath them (storage threw, network fetch failed, unexpected exception). Copy is honest about not knowing exactly what happened but confirms local data safety where true, and offers "Try Again" (`retry`) because retrying often *does* help against transient IndexedDB contention.

This distinction is the direct implementation of the brief's "distinguishing user-fixable from system errors."

### 3. Storage errors get a plain-language mapping table, not raw messages

`src/infrastructure/storage/adapters/indexeddb/errors.ts` already types four distinct failure modes (`StorageQuotaError`, `StorageUpgradeBlockedError`, `StorageCloneError`, `StorageUnavailableError`) plus a generic fallback. Add a small mapping (co-located with `RouteError` or in `src/shared/errors/`, decide during implementation based on whether other consumers need it) from `error.code` (the `AppError`/`StorageError` `code` string, e.g. `'STORAGE_QUOTA_EXCEEDED'`) to a plain-language `RouteError` message, e.g.:

| code | Message shown to user |
|---|---|
| `STORAGE_QUOTA_EXCEEDED` | "This device is out of storage space. Free up space (clear browser data for other sites, or delete old receipts) and try again." |
| `STORAGE_UPGRADE_BLOCKED` | "Another open tab is updating the local database. Close other tabs of this app and try again." |
| `STORAGE_UNAVAILABLE` | "This browser doesn't support local storage in its current mode (e.g. private/incognito browsing). Switch to a normal browser window." |
| `STORAGE_CLONE_UNSUPPORTED` | (Internal/programmer error — should not reach a user; if it does, generic system message + logged detail.) |
| anything else | Generic "Something went wrong saving/loading your data. Your existing data on this device is safe." + retry |

Every server-component `catch` block that currently does `console.error(...); notFound()` or swallows to `[]` must instead check `error.code` against this table before deciding notFound vs re-throw vs empty (see the audit/enum-values/receipt file notes above).

### 4. One `RouteError` and one `OfflineBanner`, not per-route bespoke components

`error-state.tsx` (checkout-local) and the ad hoc blocks in `error.tsx`, `login/page.tsx`, `admin/audit/page.tsx`, `admin/enum-values/page.tsx`, `pos/display/page.tsx` are all the same concept implemented five different ways today. Consolidating into one `src/shared/components/ui/route-error.tsx` (Storybook-covered, token-only) is what makes "every screen" tractable instead of five independent redesigns — and it's what the existing `EmptyState`/`Skeleton`/`Toast` components already establish as this codebase's pattern (shared primitive in `ui/`, screen-specific composition in each route).

### 5. Promote `useOutboxCount` to a shared `usePendingSyncCount`

`register-outbox.ts` is checkout-local today (`src/app/pos/checkout/_lib/register-outbox.ts`), but the acceptance criteria requires a pending-sync count wherever the offline indicator appears — i.e., potentially every authenticated screen, not just checkout. Move the outbox singleton and its two hooks (`useOutboxCount` → rename `usePendingSyncCount`, `useOldestPendingAge`) to `src/shared/lib/` (or `src/shared/hooks/`, match existing conventions — check for a `src/shared/lib` or similar directory during implementation) with no behavior change; update the checkout import path. This is a move + rename, not a rewrite — `register-outbox.ts`'s implementation (in-memory FIFO, `localStorage`-mirrored per its own doc comment — verify the mirroring is actually implemented, since the code read above shows only in-memory arrays with no `localStorage` calls; if the doc comment is aspirational/stale, fix the comment rather than the behavior, since implementing real persistence is out of this card's scope) is otherwise correct as-is.

Only checkout currently produces outbox entries (`finalize-sale`/`hold-cart`/`resume-cart`). Other screens' `OfflineBanner` instances will show the online/offline dot but a pending count of 0 until/unless those screens ever enqueue outbox entries — that's correct and expected, not a gap to fill in this card.

### 6. Dashboard gets minimal real data wiring, not a full analytics feature

`/app`'s "Today's Sales: $0.00", "Active Sessions: 1", "No orders yet" are hardcoded JSX, not a real empty/loading/error state — there is nothing to design non-happy states *for* until the page reads real data. The narrowest correct fix: read today's sales total and order count from `repos.sales` (mirror the aggregation approach already used in `src/domains/sales/lib/money.ts` if a suitable helper exists — check during implementation; otherwise a simple `listSalesForBranch`-and-reduce is acceptable and can be optimized later) and the 5 most recent orders, inside one `withTransaction`, matching the `createDefaultStorageProvider()` pattern already used in `admin/audit/page.tsx`. This is intentionally the smallest read that makes "empty" (`0` sales / no recent orders) a real, honest state rather than a fake one. "Active Sessions" has no backing concept anywhere in the domain layer (grep confirms no session-count entity) — drop that stat card rather than inventing a fake data source for it; this is a scope reduction, not a deferred TODO, and should be called out to the user/reviewer explicitly since it changes visible page content beyond state design.

## State Management Strategy

- **No new global state.** Every state introduced here is either (a) server-component conditional rendering based on an `await`ed result (dashboard, audit, enum-values, receipt), or (b) existing client hooks (`useOnlineStatus`, the promoted `usePendingSyncCount`) that are already the established pattern in `pos/checkout`.
- **Loading state ownership stays with Next.js's Suspense convention** (`loading.tsx` per segment) rather than component-level `isLoading` flags, except where a screen is already client-rendered with its own bootstrap hook (`useRegisterSession`'s `status: 'loading' | 'error' | 'needs-shift' | 'ready'` union stays exactly as-is — only its rendering of the `'loading'`/`'error'` branches changes, not the state machine itself).
- **Error kind is carried as data, not inferred in the view.** Every server action / repository call site that can fail in more than one user-visible way returns or throws something that already carries (or is mapped in Architecture Decision 3 to carry) enough information to pick `kind`. Views must not guess `kind` from string-matching error messages.
- **Offline/pending-sync state is global and cross-cutting** (`usePendingSyncCount`, `useOnlineStatus`), consistent with it being a device-wide fact, not a per-screen one — this is why it's promoted out of `pos/checkout/_lib` in Architecture Decision 5.

## Accessibility Considerations

- `RouteError` (`kind="system"`): the outer container gets `role="alert" aria-live="polite"` (matching the existing, already-correct pattern in `(auth)/login/page.tsx`); `kind="user"` errors are not full alerts if they render as part of normal server-rendered page content (an alert role on every render of "no org selected" would be over-announcing on initial page load) — reserve `role="alert"` for errors that appear *in response to* a user action (e.g., failed login submit), and use a plain, well-labeled region (heading + text) for errors that are simply the initial state of a page load.
- `OfflineBanner`: `role="status" aria-live="polite"` (same convention `sync-progress-indicator.tsx` already uses) so the offline transition is announced once, not repeatedly; must not use color alone to convey online/offline (keep the existing text label "● Online"/"● Offline" pattern — the dot is decorative, the text is the actual signal, satisfying WCAG 1.4.1 exactly as `docs/design-direction-pos.md` already requires for the Pay button).
- `Skeleton`/`SkeletonRows`: already `aria-hidden="true"` in the existing `skeleton.tsx` — preserve this; the loading region itself should be wrapped once with `aria-busy="true"` at the container level (Next.js `loading.tsx` fallback boundary) so assistive tech doesn't read placeholder content as real content.
- `EmptyState`: no special ARIA needed beyond its existing heading-level prop; ensure every new usage passes an appropriate `headingLevel` so the empty-state heading doesn't break the page's heading hierarchy (e.g., dashboard's recent-orders empty state should be `h3` under the page's `h1`/section `h2`, not default to `h2`).
- **Reduced motion:** any new transition (offline banner slide-in, skeleton shimmer) must use `--motion-fast`/`--motion-base` and respect `prefers-reduced-motion`, per both design-direction docs — the existing `skeleton.tsx` keyframe (`skeleton-loading`, opacity pulse) already has no translation/scale, so it's cheap to leave as-is; confirm it's wrapped in the same reduced-motion collapse used elsewhere (check `src/app/globals.css` for the global `prefers-reduced-motion` rule and confirm the injected `<style>` tag in `skeleton.tsx` is covered by a global selector, not scoped in a way that excludes it).
- **Focus:** `RouteError`'s retry/action buttons must be reachable and visibly focused with the existing `outline-accent` focus-ring convention (`docs/design-direction.md` Example 3) — audit forms currently use `focus:ring-blue-500` (browser default), which this card's `RouteError`/`OfflineBanner` must not repeat, even though fixing *every* pre-existing `focus:ring-blue-500` on happy-path elements is out of scope (only new/touched elements must be correct).

## Risks and Edge Cases

- **AGENTS.md directs reading `node_modules/next/dist/docs/` before writing Next.js code; that directory does not exist in the installed `next@16.2.10` package** (confirmed empty during this planning pass). Since the existing codebase already has working `loading.tsx`/`error.tsx`/`not-found.tsx`/route-segment conventions checked into `src/app/`, treat those as the authoritative reference for this Next version instead, and verify with a real `npm run dev` pass (per Test Plan) rather than doc lookup — flag this doc/reality mismatch to the team since it may block other cards that rely on the same AGENTS.md instruction literally.
- **The audit/enum-values "swallow error as empty list" bug** (`result.ok ? result.data : undefined) || []`) is a pre-existing correctness bug this card surfaces and must fix, not merely restyle around — if left as-is, the new `EmptyState` would be *actively wrong* (telling an owner "no audit entries" when storage actually failed), which is worse than the current unstyled-but-at-least-vague `"No audit entries found."` text. This must not be treated as out-of-scope cleanup; it's required for the error state to be honest.
- **Dashboard's dropped "Active Sessions" stat** changes visible page content, not just its states — call this out explicitly in the PR rather than silently removing a stat card (see Architecture Decision 6).
- **`register-outbox.ts`'s doc comment claims `localStorage`-mirroring that the read source does not show implemented** — resolve the discrepancy by fixing the comment (cheapest, in-scope) rather than adding real persistence (out-of-scope scope creep); flag if implementation reveals the mirroring does exist elsewhere and the exploration above simply missed it.
- **Onboarding wizard's internal async/error handling is unverified** — the exploration pass for this plan did not read `onboarding-wizard.tsx`'s body, only confirmed its existence and the audit's note about its progress-bar color; implementation must read it fully before deciding whether its current error handling needs a full swap to `RouteError` or already has adequate (if unstyled) handling.
- **`display-cart-view.tsx`'s initial-null-cart handling is unverified** for the same reason (map-mode read only surfaced `page.tsx`'s registerId gate, not the component's internal render logic) — must be confirmed/read before implementing the "no active cart yet" empty state.
- **Onboarding is the one screen where "offline is calm, not scary" is deliberately *not* fully true** — org/branch/register creation is a first-write flow with no existing offline path; the plan blocks submission with a clear message rather than pretending parity with the register's always-works-offline behavior. This is a considered exception to the acceptance criteria's "offline ... not a warning" framing for the app broadly, and should be flagged to the reviewer as such rather than silently deviating.
- **Concurrent `Promise.all` reads for the dashboard (Architecture Decision 1) inside one `withTransaction`** — confirm the storage core's transaction primitive actually permits concurrent `get`/`getAll` calls issued in parallel within one `work` callback (per `docs/storage-adapters.md`, the contract is `get`/`getAll`/`put`/`delete` on a single `tx` object per transaction — verify this is safe to call concurrently, or serialize with sequential `await`s instead if the driver requires it, which would move the dashboard read from "instant" to "borderline genuine-wait" and change the Architecture Decision 1 classification).
- **Playwright offline test flakiness**: `context.setOffline(true)` combined with a service-worker-registered PWA (`service-worker-register.tsx`, `public/sw.js`) can behave inconsistently across browsers if the SW itself caches/intercepts requests; scope the new e2e test to Chromium only if cross-browser flakiness appears, and note this as an acceptable trade-off rather than blocking the card on full cross-browser offline E2E coverage.
