**Verified remaining gaps — this plan's actual scope:**

1. **Tailwind `@theme inline` only maps color and radius tokens.** Motion (`--motion-fast`,

   these must reach for arbitrary-value syntax (`duration-[var(--motion-fast)]`), which is not caught
   by the lint rule and does not satisfy "consumed through Tailwind theme config."
2. **No systematic AA sweep of the base semantic palette.** Existing tests check the brand-derived

## Architecture Decisions

1. **Tailwind theme extension stays additive and CSS-native.** There is no `tailwind.config.js` in
   this project (confirmed absent) — Tailwind v4's `@theme inline` block in `globals.css` is the only
   place tokens become utility classes, and all new mappings follow the existing
   `--<category>-<name>: var(--source);` pattern already used for color and radius. Recognized
   category prefixes: `--duration-*` → `duration-*`, `--ease-*` → `ease-*`, `--z-*` → `z-*`,
   `--shadow-*` → `shadow-*`, `--text-*` → `text-*` (font-size utilities). The raw `--motion-fast`
   etc. variables stay the single source of truth in `:root`; `@theme inline` only aliases them,
   exactly like `--accent` already backs `--color-accent`.
2. **Resolve real computed values in the AA sweep, never hand-copy hex.** `tenant-theme-style.test.tsx`
   already proves the pattern of rendering into jsdom and reading resulting CSS text; this plan
   generalizes it. Because jsdom does not run PostCSS/Tailwind, `design-tokens.ts`'s resolver parses
   `globals.css`'s literal `:root` / `:root[data-theme='dark']` / `:root[data-theme='light']` rules
   with the `postcss` package (already a transitive dependency via `@tailwindcss/postcss`) and
   resolves `var()` chains by hand until only literal hex/rgba values remain. This keeps the test
   honest against the actual shipped file with zero hand-copied values and avoids depending on
   Tailwind's build pipeline inside jsdom.
3. **The token-swap fixture is a fixture, not a new product screen.** "Renders a screen" does not
   require a production route. A throwaway fixture under `__design_token_fixtures__/` (an existing,
   lint/boundary-exempt convention) avoids coupling the test to evolving production markup and avoids
   scope creep into building real `/pos/checkout`-style UI.
4. **Narrow the ESLint ignore instead of special-casing the test's ESLint instance.** The fixtures
   directory is currently in the top-level `ignores` array, which makes it invisible to *any* flat
   config object, including the rule it exists to test. Rather than construct a second `ESLint`
   instance with an inline `overrideConfig` (which would test a config the app doesn't actually run),
   remove `**/__design_token_fixtures__/**` from the top-level `ignores` array and instead add an
   explicit `{ files: ['**/__design_token_fixtures__/**'], rules: { ... } }` override block that keeps
   the fixtures directory exempt from unrelated rules (boundaries, restricted-globals) while leaving
   the `no-restricted-syntax` design-token block (already scoped to `src/app/**/*.{ts,tsx}`, which
   includes this directory) active against it. This is the smallest change that makes the regression
   test assert against the real project config.
5. **No new settings/entity fields.** `ThemeSettings { colorScheme, brandColor }` plus the existing
   `logoUrl` fields already cover "brand color, logo, and light/dark preference." "Defaulting to the
   Claude-inspired theme when unset" is already satisfied structurally (`DEFAULT_SETTINGS.theme =
   { colorScheme: 'system' }`, `TenantThemeStyle` returns `null` when `brandColor` is unset, so the
   static `globals.css` cascade applies unchanged) — this plan adds a test for that path (Implementation
   Step 3) rather than new fields.

## State Management Strategy

No new client or server state is introduced:

- Theme resolution remains **server-side only**, computed once per request in `layout.tsx` from the
  session → org/branch chain (already implemented, unchanged by this plan). No client-side theme
  store, no `localStorage`. `data-theme` on `<html>` plus the injected `<style>` override are the
  entire runtime mechanism; this plan changes only what feeds it (Tailwind utility coverage for
  motion/z-index/shadow/typography) and how it's verified (tests), never the mechanism itself.
- `TenantThemeProvider`'s context continues to carry only `{ logoUrl }` — no new fields needed here;
  brand color/scheme are consumed purely through CSS custom properties, which is correct for
  something that must apply before hydration with zero runtime JS cost.
- The new screen-level test (`token-swap-screen.test.tsx`) simulates "two tenants" by manually setting
  `document.documentElement`'s `data-theme` attribute and passing different `derivedTokens` props — it
  does not re-exercise the server session/org-resolution path (already covered by
  `theme-resolver.test.ts`), keeping the new test fast and focused on the DOM-output layer.

## Accessibility Considerations

- **AA contrast is a hard gate.** Step 4's sweep must fail the build — not just warn — if any semantic
  pairing drops below 4.5:1 (3:1 for the documented large-text-only dark-mode accent exception),
  directly satisfying "contrast checker passes AA for every semantic pairing in both light and dark."
- **Focus-ring distinguishability against its background, not just its own text pairing.**
  `--focus-ring-color: var(--accent)` recolors per tenant, but the ring's relevant contrast is against
  whatever background sits behind the focused element, not against the accent's own foreground. Step
  3 adds an explicit `background/accent` pairing at the WCAG 2.2 non-text/UI-component 3:1 minimum so
  a low-saturation tenant brand color can't produce an invisible focus ring even while passing as a
  *text* pairing elsewhere.
- **`prefers-reduced-motion` handling in `globals.css` is untouched and must stay that way.** Its
  `!important` overrides on `animation-duration`/`transition-duration` continue to apply regardless of
  the new `duration-fast`/`duration-base` utilities added in Step 1 — verify manually, no new
  automated test needed since the mechanism itself doesn't change.
- **`overlay` is exempt from text-contrast rules** (it's a non-text scrim) but content stacked on it
  must still use an AA-verified pairing (typically `surface`/`foreground`) — call this out explicitly
  in the Step 2 docs update so nobody tries to retrofit a contrast-ratio test onto the scrim itself.

## Risks and Edge Cases

- **jsdom does not run a real CSS engine or process `@import`/`@theme`.** The biggest implementation
  risk: tests cannot `getComputedStyle(el).getPropertyValue('--accent')` after importing
  `globals.css` and expect Tailwind's `@theme` resolution to have happened, because jsdom never runs
  PostCSS/Tailwind. Mitigation (Architecture Decision 2): parse the literal `:root` rules with
  `postcss` and resolve `var()` chains manually. If this proves too brittle for the
  `:root:not([data-theme='light']) { @media {...} }` nesting pattern specifically, fall back to
  resolving only the unconditional `:root[data-theme='dark']` block (byte-identical in content to the
  media-query block by construction in the current file) rather than attempting to parse nested
  `@media` rules inside a selector block.
- **The `prefers-color-scheme` media-query dark block is not reachable via `data-theme` attribute
  alone**, and jsdom does not emulate `window.matchMedia` by default. Step 3's resolver targets
  `[data-theme='dark']` specifically (attribute-reachable, and a verified duplicate of the media-query
  block's values) — document in `design-tokens.ts` that this covers the *values*, not the
  *media-query mechanism*, so a future reader doesn't assume the sweep exercises OS-driven dark mode.
- **New `@theme inline` names could collide with a Tailwind v4 built-in utility of the same name.**
  Before merging Step 1, grep the compiled CSS before/after the change to confirm none of
  `duration-fast`, `duration-base`, `ease-base`, `shadow-card`, `shadow-elevated`,
  `text-display-2xl/xl/lg`, `text-body/label/caption` shadow an existing Tailwind default utility in a
  way that would change existing pages' rendering.
- **Narrowing the ESLint `ignores` array (Step 5) could newly expose the fixtures directory to rules
  it was never meant to be checked against** (e.g. `boundaries/no-unknown`). Mitigated by adding the
  explicit exemption block described in Architecture Decision 4 rather than simply deleting the
  ignore entry — verify `npm run lint` reports only the three intentional violations, nothing else,
  after this change.
- **The dark-mode accent/accent-foreground pairing is intentionally large-text-only (documented at
  ~3.8:1 in `docs/design-direction.md`).** A future palette tweak could silently break this exception
  if the sweep's `largeText: true` marking for that one pairing is ever "fixed" by loosening the test
  instead of the palette. Comment the test with a pointer to the doc line so the reason is legible in
  place.
- **Default (`brandColor` unset) path must render zero `<style>` override, confirmed current behavior
  of `TenantThemeStyle`.** Step 7's third scenario is the literal "defaulting to the Claude-inspired
  theme when unset" acceptance path and is not yet covered at screen granularity (only at the isolated
  `TenantThemeStyle` unit level today).
