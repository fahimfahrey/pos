# Implementation Plan ÔÇö IndexedDB Storage Adapter (Dexie)

Card: `card_1784266069993_3_ydz4`

## Overview

Implement `src/infrastructure/storage/indexeddb/` (registered under
`src/infrastructure/storage/adapters/indexeddb/`, matching the existing `adapters/memory`
layout and the authoring guide in `docs/storage-adapters.md`) as a **Dexie-backed
`StorageDriver`**. The driver plugs into the already-built storage core
(`driver.ts`, `unit-of-work.ts`, `schema-versioner.ts`, `export-import.ts`,
`storage-provider.ts`, `engine-registry.ts`) with zero changes to the core contract, and
must pass the **full provider conformance matrix**
(`src/infrastructure/storage/conformance/`) both in Node (fast signal, via
`fake-indexeddb`) and ÔÇö the acceptance gate ÔÇö in a **real Chromium IndexedDB driven by
Playwright** (Vitest browser mode, whose `playwright` provider is the "Playwright browser
context" named in the card).

The driver implements only the four `DriverTransaction` primitives (`get`, `getAll`,
`put`, `delete`) plus `open`/`transaction`/`getSchemaVersion`/`setSchemaVersion`/`close`.
Everything else (typed repositories, `withTransaction`/UnitOfWork, migrations,
export/import) is already built on top of that primitive by core, so getting those green
is a matter of making the primitive behave correctly against real IndexedDB.

Scope of deliverables:

1. Dexie driver with **one object store per entity**, indexes mirroring the repository
   access patterns (`barcode`, `sku`, `orgId`/`[orgId+branchId]`, `saleId`=`orderId`,
   `shiftId`, `createdAt`, ÔÇŽ), and an internal `__meta__` store for the logical schema
   version.
2. **Transactions implementing UnitOfWork** across every store touched in a scope
   (atomic commit/rollback that survives `await` interleaving), via
   `db.transaction('rw'|'r', tables, work)`.
3. **Dexie `version()` chain derived from the engine-agnostic schema module**
   (`CURRENT_SCHEMA_VERSION` + `schemaSteps`), with logical data migrations executed by
   the core `SchemaVersioner` at runtime.
4. **`exportAll`/`importAll`** already produce/consume the canonical JSON ÔÇö add a
   browser backup module that **streams** that JSON to/from disk via the **File System
   Access API** (with `<a download>` / `<input type=file>` fallback).
5. IndexedDB realities handled **inside the adapter only**: quota Ôćĺ typed error,
   blocked upgrade (multi-tab) handling, structured-clone (`DataCloneError`) Ôćĺ typed error.
6. `STORAGE_ENGINE=indexeddb` as the **default binding** in the storage provider
   (env already defaults to `indexeddb`; wire registration + an SSR-safe default helper).
7. `docs/backup.md` documenting the manual export/import round-trip.

## Architecture Decisions

1. **Dexie over raw `idb`.** The UnitOfWork conformance suite
   (`suites/unit-of-work.ts`) asserts full rollback across `await Promise.resolve()`
   interleaving inside a single logical transaction. Raw IndexedDB auto-commits a
   transaction the moment control returns to the event loop with no pending request;
   bridging `async/await` correctly is exactly what Dexie's transaction zone
   (`db.transaction(mode, tables, work)`) does ÔÇö it keeps the IDB transaction alive across
   awaits as long as the awaited work stays in-zone (our repo ops are all Dexie table ops)
   and only microtasks intervene. Dexie also gives us `db.on('blocked')` /
   `db.on('versionchange')` and normalized abort/quota errors. Dexie 3.2.7 is already a
   dependency; ESLint `boundaries/external` permits importing `dexie` from
   `src/infrastructure/**` only (adapter lives there), and `no-restricted-globals:
   indexedDB` is disabled for infrastructure.

2. **Physical schema (Dexie `version().stores()`) vs. logical schema (`SchemaVersioner`)
   are two axes, wired from one source.** Dexie's `version(n).stores({...})` defines the
   *structural* schema (object stores + secondary indexes). The core `SchemaVersioner`
   drives *logical/data* migrations (e.g. the conformance `migrationLadder` adds/removes
   `testField`/`derivedField` on records) at runtime via `driver.transaction` +
   `setSchemaVersion`. The card's phrase "Dexie's `version()` chain driven by the
   engine-agnostic SchemaVersioner steps" is realized by deriving the Dexie chain's
   version numbers and store set from the same `@infra/storage/core/schema` module
   (`CURRENT_SCHEMA_VERSION`, `COLLECTIONS`) that the `SchemaVersioner` reads, so the two
   never drift. Today that is a single `db.version(CURRENT_SCHEMA_VERSION).stores(spec)`.
   When a future `SchemaStep` needs a new store/index, append a
   `db.version(n).stores({...})` link **and** bump `CURRENT_SCHEMA_VERSION`; `buildVersionChain()`
   documents that append point.

3. **Logical schema version stored in an internal `__meta__` store, decoupled from
   Dexie's structural version.** `getSchemaVersion()` reads `__meta__:{id:'schemaVersion'}`
   (default `0`, matching the memory driver's initial version and the schema suite's
   `expect(version).toBeLessThanOrEqual(3)`); `setSchemaVersion(v, tx?)` writes it. The
   schema suite migrates logical `1Ôćĺ3Ôćĺ1` while the structural Dexie version stays `1`
   (those steps change record fields, not stores), so the two axes must be independent.
   `__meta__` is **not** in `COLLECTIONS`, so `exportAll`/`getAll` never touch it.

4. **Every `transaction()` scope implicitly includes `__meta__`.** The core
   `SchemaVersioner` calls `driver.transaction([...COLLECTIONS], 'rw', tx => { migrate;
   driver.setSchemaVersion(v, tx) })`. To make the version bump *atomic with the
   migration*, the Dexie transaction must have `__meta__` in scope. So
   `IndexedDBDriver.transaction()` always appends `__meta__` to the Dexie table list, and
   `setSchemaVersion(v, tx)` ÔÇö when `tx` is our `IndexedDBTransaction` ÔÇö writes through
   that same Dexie transaction (`tx.dexieTx.table('__meta__').put(...)`); when called
   without a `tx` it opens its own tiny `rw` transaction on `__meta__`.

5. **Lazy open.** The schema-versioning suite obtains a *raw* driver from the factory
   (`resolveEngine(adapter.engine)()`) and never calls `open()`. So `open()` is advisory;
   `ensureOpen()` runs on first `transaction`/`getSchemaVersion`/`setSchemaVersion`,
   constructs the Dexie DB from `COLLECTIONS` (ignoring whatever subset `open(schema)`
   passed), wires blocked/versionchange handlers, and `await db.open()`. This also honors
   the guide's principle #4 (browser drivers instantiated lazily, never at import) and
   keeps SSR safe.

6. **Per-instance isolated database for conformance.** The memory adapter gets a fresh
   store on every `new MemoryStorageDriver()`; conformance `beforeEach` and the
   export/import suite (which runs `provider` and a *fresh* `provider2` simultaneously and
   expects `provider2` empty before import) depend on that isolation. IndexedDB is global
   by DB name, so we replicate isolation with a **unique DB name per driver instance** in
   an `ephemeral` mode whose `close()` also `Dexie.delete(name)`s the database. Production
   uses a **stable** name (`pos-storage`) so data persists across reloads. We register two
   engines off the same class: `indexeddb` (stable, app) and `indexeddb-conformance`
   (ephemeral, tests-only, registered from `conformance/adapters.ts`). The conformance
   adapter advertises `name: 'indexeddb'` while resolving the ephemeral engine.

7. **`serializableTransactions: false`.** IndexedDB serializes overlapping-scope `rw`
   transactions, but we do not *claim* SERIALIZABLE; the concurrency suite's non-
   serializable branch only asserts safety invariants (non-negative integer stock, Ôëą1
   movement), which hold regardless. Advertising `false` avoids over-claiming.

8. **No storage-layer serialization codec; rely on structured clone.** IndexedDB stores
   structured clones natively, preserving `Date` objects on read ÔÇö so the CRUD/UoW/Date
   conformance assertions (`expect(found).toEqual(product)` with `Date` fields, isolation
   via fresh clones on each `get`) pass without manual cloning. The `SerializationCodec`
   (`{$date}`) stays in core, used only at the JSON export/import boundary.

9. **`getAll` returns primary-key order.** Dexie `table.toArray()` yields records ordered
   by primary key (`id`), i.e. lexicographic ÔÇö exactly what the ULID-ordering suite
   expects (`allIds` deep-equals the sorted ULIDs). The memory driver's insertion order
   coincides because ULIDs are monotonic; IndexedDB satisfies it structurally.

10. **IndexedDB realities confined to the adapter** (card requirement): `errors.ts` +
    `mapIndexedDbError` translate `QuotaExceededError` Ôćĺ `StorageQuotaError`,
    `DataCloneError` Ôćĺ `StorageCloneError`, blocked upgrades Ôćĺ `StorageUpgradeBlockedError`,
    missing `indexedDB` Ôćĺ `StorageUnavailableError`; all extend core `StorageError`
    (500, so callers catch generically). Core and domains stay ignorant of IndexedDB.

### Step 4 ÔÇö Driver (`adapters/indexeddb/driver.ts`)

- `IndexedDBTransaction implements DriverTransaction`:
  - `constructor(public readonly dexieTx: Transaction)`.
  - `get(name, id)` Ôćĺ `this.dexieTx.table(name).get(id)` wrapped in
    `try/catch Ôćĺ mapIndexedDbError`; return `undefined` when Dexie returns `undefined`.
  - `getAll(name)` Ôćĺ `this.dexieTx.table(name).toArray()`.
  - `put(name, value)` Ôćĺ `this.dexieTx.table(name).put(value)` (keyPath `id`); map
    `DataCloneError`/quota.
  - `delete(name, id)` Ôćĺ `this.dexieTx.table(name).delete(id)`.
- `IndexedDBDriver implements StorageDriver`:
  - `readonly engine = 'indexeddb'`.
  - `constructor(private opts: { databaseName: string; ephemeral?: boolean })`.
  - `private db: Dexie | null`.
  - `private async ensureOpen()`: if `typeof indexedDB === 'undefined'` throw
    `StorageUnavailableError`. If `db` open, return. Else `new Dexie(this.opts.databaseName)`,
    register `db.on('blocked', () => { ÔÇŽ })` (record a flag / reject with
    `StorageUpgradeBlockedError`), `db.on('versionchange', () => db.close())` (yield to
    another tab's upgrade), `buildVersionChain(db)`, `await db.open()` (catch Ôćĺ
    `mapIndexedDbError`; if blocked Ôćĺ `StorageUpgradeBlockedError`).
  - `open(schema)` Ôćĺ `await this.ensureOpen()` (schema arg advisory; stores come from
    `COLLECTIONS`).
  - `transaction(collections, mode, work)`:
    - `await ensureOpen()`.
    - `const tables = Array.from(new Set([...collections, META_STORE]))`.
    - `const dmode = mode === 'readwrite' ? 'rw' : 'r'`.
    - `return this.db!.transaction(dmode, tables, async (trans) => work(new
      IndexedDBTransaction(trans)))` wrapped so a thrown error aborts (Dexie default) and
      is rethrown unchanged (UoW suite asserts the original message);
      map quota/clone/abort via `mapIndexedDbError` on the *rejection* while preserving
      caller-thrown domain errors (only remap `DOMException`/Dexie errors, pass through
      plain `Error`).
  - `getSchemaVersion()` Ôćĺ `await ensureOpen()`; read `META_STORE`
    `{id: SCHEMA_VERSION_KEY}`; return `rec?.value ?? 0`.
  - `setSchemaVersion(version, tx?)`:
    - if `tx instanceof IndexedDBTransaction` Ôćĺ `await tx.dexieTx.table(META_STORE).put({
      id: SCHEMA_VERSION_KEY, value: version })`.
    - else Ôćĺ `await ensureOpen(); await this.db!.transaction('rw', META_STORE, async t =>
      t.table(META_STORE).put({ id: SCHEMA_VERSION_KEY, value: version }))`.
  - `close()` Ôćĺ if `db`: `db.close()`; if `opts.ephemeral` Ôćĺ `await Dexie.delete(
    this.opts.databaseName)`; `db = null`.

### Step 7 ÔÇö Backup module (`adapters/indexeddb/backup.ts`)

- `exportBackupToFile(provider: StorageProvider, opts?: { suggestedName?: string }):
  Promise<'saved' | 'cancelled'>`:
  - `const data = await provider.exportAll()` (canonical `StorageExport`).
  - Prefer File System Access API: if `'showSaveFilePicker' in window`, get a handle
    (`suggestedName: pos-backup-<ISO>.json`, `types: [{accept:{'application/json':
    ['.json']}}]`), `const w = await handle.createWritable()`, **stream** the JSON ÔÇö
    `JSON.stringify(data)` sliced into ~1 MB chunks written sequentially (avoids one giant
    string in the pipe for large datasets) ÔÇö then `await w.close()`.
  - Fallback: `Blob([json], {type:'application/json'})` Ôćĺ object URL Ôćĺ `<a download>` Ôćĺ
    click Ôćĺ `URL.revokeObjectURL`.
  - `AbortError` from a cancelled picker Ôćĺ return `'cancelled'` (not an error).
- `importBackupFromFile(provider, opts?: { mode?: ImportMode }): Promise<'imported' |
  'cancelled'>`:
  - Prefer `showOpenFilePicker` Ôćĺ `getFile()` Ôćĺ read via `file.stream()` +
    `TextDecoderStream` accumulated to text (streaming read), else `<input type=file>`
    fallback Ôćĺ `file.text()`.
  - `JSON.parse`, then `provider.importAll(parsed, { mode: opts?.mode ?? 'replace' })`
    (core validates `format`/`formatVersion` and throws `ImportFormatError`).
- SSR guard: both throw `StorageUnavailableError` if `typeof window === 'undefined'`.

## State Management Strategy

- **Single source of durable state:** one IndexedDB database (`pos-storage` in prod;
  unique per instance in conformance) with one object store per entity + `__meta__`.
  There is **no** in-memory cache in the driver ÔÇö every `get`/`getAll` reads a fresh
  structured clone from IndexedDB, which is what gives the isolation guarantee for free.
- **Transaction/UnitOfWork state:** owned by Dexie's transaction zone. A logical scope =
  one `db.transaction(mode, tables, work)`; all writes commit together on resolve or abort
  together on throw. `IndexedDBTransaction` is a thin, stateless wrapper over the Dexie
  `Transaction`; it holds no buffered writes (unlike the memory driver's staging map) ÔÇö
  Dexie/IndexedDB provide atomicity natively. `__meta__` is always in scope so a schema-
  version bump commits atomically with its migration.
- **Schema-version state:** the logical version lives in `__meta__:{id:'schemaVersion'}`,
  read/written only through `get/setSchemaVersion`; default `0` on a fresh DB. The
  structural Dexie version is a separate constant (`CURRENT_SCHEMA_VERSION`) baked into the
  version chain at open time.
- **Connection lifecycle:** lazily opened on first use, one live `Dexie` handle per driver
  instance, closed (and, if ephemeral, deleted) on `close()`. On `versionchange` from
  another tab the handle self-closes to let the upgrade proceed; the next operation
  re-`ensureOpen()`s.
- **App-level provider:** `createDefaultStorageProvider()` is the single entry the app
  constructs (client-side); it should be memoized by the caller (a client
  provider/context) rather than reconstructed per render ÔÇö no global singleton is created
  by the storage module itself, preserving SSR safety and testability.

## Accessibility Considerations

The driver is non-visual, so a11y applies only to any UI that triggers backup:

- Backup **export/import must be keyboard-operable**: trigger via real `<button>`
  elements (not `div` click handlers); the File System Access pickers and the
  `<input type=file>` fallback are natively focusable/keyboard-accessible ÔÇö do not hide the
  fallback input in a way that breaks keyboard/label association (use a visually-hidden but
  focusable `<input>` tied to a `<label>`/button).
- **Status must be announced, not just visual:** surface "Backup saved",
  "Import complete", and error states (quota exceeded, upgrade blocked, invalid file) in
  an `aria-live="polite"` region (errors `assertive`), with text ÔÇö not color ÔÇö conveying
  outcome; meet WCAG contrast.
- **Long export/import:** if a progress indicator is shown, give it
  `role="progressbar"`/`aria-busy` and keep focus management sane (return focus to the
  trigger after completion); a cancelled picker (`AbortError`) must not trap focus or fire
  an error announcement.
- Typed error `.message` strings are written to be human-readable so they can be shown
  directly to assistive tech.

(No backup UI component is in scope for this card beyond the callable functions; these
notes constrain the future UI and the messages the functions return.)

## Risks and Edge Cases

- **Premature transaction commit across `await` (highest risk).** Real IndexedDB commits
  when the event loop is reached with no pending request. Mitigation: run *all* logical
  work inside `db.transaction(mode, tables, work)` so it stays in Dexie's zone; the UoW
  suite's `await Promise.resolve()` is a microtask that Dexie survives in Chromium. Verify
  specifically in the **browser** run, not just `fake-indexeddb` (which is more lenient).
- **19-store `rw` transaction per operation.** `withTransaction`/`exportAll` scope every
  store (+`__meta__`). Correct but heavy; POS datasets are small (guide's stated
  assumption). Overlapping same-scope `rw` transactions serialize ÔÇö fine for the
  concurrency suite; do not `Promise.all` app writes expecting parallelism.
- **`getAll` ordering.** Relies on Dexie/IDB primary-key ordering to satisfy ULID tests;
  confirmed by spec. `fake-indexeddb` also key-orders. No manual sort needed.
- **Multi-tab blocked upgrade.** If tab A holds an old connection while tab B opens a
  higher version, B's open is `blocked`. Handling: A listens for `versionchange` and
  `db.close()`s (unblocking B); if B still can't proceed, surface
  `StorageUpgradeBlockedError`. Since `CURRENT_SCHEMA_VERSION` rarely changes, this is a
  cold path ÔÇö documented in README + `backup.md`, and unit-tested via a simulated
  `blocked` event where feasible (full multi-tab is a manual check).
- **Quota exceeded.** Large imports may hit `QuotaExceededError`; `mapIndexedDbError`
  surfaces `StorageQuotaError`; `importAll`'s single transaction rolls back so no partial
  state. Documented.
- **Structured-clone limits.** Entities are plain (primitives + `Date`) and clone cleanly;
  `DataCloneError` only if a non-cloneable value (function/DOM node/class instance) is
  ever put ÔÇö mapped to `StorageCloneError`. Defensive, unit-tested.
- **SSR / private-mode / disabled IndexedDB.** `defaultStorageEngine()` falls back to
  `memory` when `indexedDB` is undefined; direct use of the `indexeddb` engine on a
  browser with IndexedDB disabled throws `StorageUnavailableError` at `ensureOpen`.
- **Legacy `pos-db` collision.** The legacy `dexie-db.ts` uses DB `pos-db` with 2 stores;
  the new adapter uses `pos-storage` ÔÇö no shared name, no schema clash. Container migration
  is out of scope.
- **`@vitest/browser` dependency.** Not currently installed; `playwright@1.61.1` is
  present. Add `@vitest/browser@^1.6.1` (matches `vitest@1.6.1`). If a browser binary is
  missing in CI, `npx playwright install chromium` is required (note in README). If adding
  the dep is undesirable, the fallback is a bespoke Playwright `e2e/` spec driving a dev
  page that runs a `describe/it/expect` shim over the suites ÔÇö more code, so browser-mode
  is preferred.
- **`fake-indexeddb` shadowing in browser mode.** The browser config must **not** load
  `vitest.setup.ts` (which imports `fake-indexeddb/auto`), or the "real browser" run would
  silently test the fake. Enforced via a dedicated `setup.browser.ts`.
- **Dexie error re-wrapping in UoW.** Only remap `DOMException`/Dexie infra errors;
  caller-thrown domain `Error`s must propagate unchanged so
  `rejects.toThrow('Simulated transaction error')` passes.
