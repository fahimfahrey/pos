## Overview

Deliver **one reusable, adapter-agnostic conformance suite** — a parameterized Vitest spec that any `StorageProvider` implementation must pass — plus the **auto-expanding matrix** that runs it against every registered adapter, so a future Postgres/Mongo adapter is proven equivalent the moment its registry entry exists.

The suite lives in a **new** directory `src/infrastructure/storage/conformance/` (per the card), authored as a library function `runStorageProviderConformance(adapter)` that a single thin `conformance.test.ts` iterates over a **conformance adapter registry**. Adding an adapter = adding one registry entry; the in-process matrix and CI coverage expand automatically — no per-adapter test file and no YAML edit required.

**Coverage (exactly the card's list):**

1. **CRUD round-trips for every repository interface** — all ten contexts (`catalog, inventory, sales, payments, customers, purchasing, promotions, organization, auth, audit`), driven through `provider.withTransaction(repos ⇒ …)` using each interface's real intent-named methods.
2. **ULID key preservation** — records keyed by canonical 26-char Crockford-base32 ULIDs round-trip **byte-for-byte**, keep **lexicographic == chronological** ordering under `getAll`, and survive export→import unchanged. (App IDs are currently UUID; the suite proves the *adapter* never mangles/normalizes/reorders string keys — a ULID-shaped key is the strictest such probe.)
3. **Tenant scoping** — records tagged with `orgId` for two orgs coexisting in the same provider; a scoped read never returns another org's rows (positive + negative assertions), and isolation survives export→import. (See Architecture Decision 4 for how this is modeled given the domain has no `orgId` yet.)
4. **UnitOfWork atomicity** — a multi-repository write that throws mid-transaction rolls back **completely** (zero partial writes across all touched collections). Expressed through the atomic boundary the system actually provides today: `provider.withTransaction` (Architecture Decision 5).
5. **Concurrent-write behavior for stock decrement** — sequential read-modify-write decrements are exact; parallel decrements are asserted against a **capability-gated** contract (strict serializable equality for adapters that advertise it; a safety invariant — never negative, never corrupt — for those that don't). Memory advertises non-serializable and stays green (Architecture Decision 6).
6. **SchemaVersioner upgrade/downgrade across ≥3 versions** — a 3-step migration ladder (v1↔v2↔v3) run up then down through the **real** `SchemaVersioner`, asserting the version slot persists and each step's data transform applies/reverses. Requires a small additive extension to `SchemaVersioner` (downgrade + injectable steps — Architecture Decision 7).
7. **exportAll/importAll fidelity** — export from adapter A, import into adapter B, **deep-equal** the full normalized dataset (all collections, all records, Date fidelity, key preservation), run for every ordered adapter pair the registry yields.

**Wiring:** a single `conformance.test.ts` loops the registry so the in-process matrix expands per adapter; an optional GitHub Actions `matrix` (documented, since `.github/**` is deny-guarded in this environment) parallelizes per adapter in CI. A `test:conformance` npm script targets the suite directly.

**Acceptance:** suite green against the in-memory adapter (`npm run test`), and the CI matrix is configured to expand per adapter (registry-driven, zero-edit for new adapters).

Layering (unchanged; conformance is test infrastructure inside `infrastructure`, importing domain types only for fixtures):

```
conformance suite ── drives ──▶ StorageProvider (core) ──▶ StorageDriver (per adapter)
        │
        └── imports domain entity *types* for typed fixtures (allowed: infra → domains types)
```

## Architecture Decisions

1. **New directory `src/infrastructure/storage/conformance/`, not `tests/`.** The card names this exact path. It also makes the suite importable by any future adapter's own test file and by the matrix runner, and keeps domain-type imports on the infra side of the boundary. The single `.test.ts` inside it is what Vitest actually executes; everything else is a library.

2. **Parameterize by `StorageProvider`, not `StorageDriver`.** The pre-existing `tests/storage/conformance.ts` tested the low-level driver primitive. The card requires "CRUD round-trips for every *repository interface*" and "UnitOfWork atomicity" — provider/repository-level concerns. So the suite is parameterized by a factory `() => Promise<StorageProvider>` and drives everything through `provider.withTransaction`, `exportAll`, `importAll`, and (for schema) the driver reachable via a provider-exposed migration entry point. This is strictly higher-fidelity and subsumes the old driver-level checks.

3. **Registry-driven, auto-expanding matrix.** `defaultConformanceAdapters` is the single source of truth. `conformance.test.ts` loops it; CI (optionally) shards over the same names. Adding Postgres/Mongo = append one `ConformanceAdapter`. This is exactly the card's "proven equivalent the moment it exists" and "matrix configured to expand per adapter," achieved in code so it can't drift from a hand-maintained YAML list.

4. **Tenant scoping is verified as an adapter-substrate contract (records carry `orgId`; scoped read = scan + `orgId` filter).** The domain has **no `orgId`** today (verified across all entities and repository interfaces) and this card's scope is the test suite, not a tenancy feature — so we do **not** mutate ten entity types or rewrite ten core repositories. Instead the suite seeds records tagged with an extra `orgId` property (fixtures accept `orgId`; drivers store the whole object, so the field persists), then reads a collection and partitions by `orgId` through a small in-suite `scopedList(repos, coll, orgId)` helper — the exact pattern a tenant-aware repository will use. Assertions: org A's scoped list contains only A's rows; a cross-org id lookup under A's scope yields nothing; the partition survives export→import. This gives a **row-level** guarantee meaningful for shared-database engines (Postgres/Mongo) — a false-pass trap that per-provider-instance isolation would hide. **Documented follow-up:** when repositories become tenant-aware (thread `orgId` through `buildRepositorySet`), delete the in-suite helper and point the same assertions at the real scoped query — the test bodies barely change. Rationale recorded in Risks.

5. **UnitOfWork atomicity is asserted through `provider.withTransaction`.** `provider.unitOfWork()` currently **throws** (`'UnitOfWork interface not yet fully implemented'`) and the stub `UnitOfWork` only flips booleans — it is not wired to a real driver transaction, so it cannot demonstrate rollback. The atomic boundary the system genuinely provides is `withTransaction(fn)`: it builds the `RepositorySet` over a single driver transaction and the driver commits all-or-nothing. So the suite simulates a mid-transaction error inside `withTransaction` (write to inventory + sales + audit, then `throw`) and asserts **every** collection is unchanged. This is precisely "multi-repository write rolls back completely on failure." A `// TODO` note in the sub-suite marks that, once `unitOfWork()` wraps a real transaction, an additional block should assert the same invariant through explicit `begin/commit/rollback`. Extending `unitOfWork()` itself is **out of this card's scope**.

6. **Concurrency contract is capability-gated; capabilities live in the conformance registry, not core.** The memory driver stages a per-transaction snapshot and swaps whole collections on commit → **last-writer-wins**, so parallel `Promise.all` decrements lose updates (verified by reading `memory-driver.ts`: staging clones happen synchronously at `transaction()` entry, before any `await`, so two concurrent transactions both observe the pre-decrement value). Asserting strict `final = initial − N` under `Promise.all` would therefore **fail on memory** and violate acceptance. Resolution: each adapter advertises `capabilities.serializableTransactions`. The suite always runs the **sequential** read-modify-write scenario (exact, green everywhere) and runs the **parallel** scenario with a branched assertion — strict equality when `serializableTransactions === true`, else a safety invariant (`0 ≤ final ≤ initial`, integer quantity, ≥1 movement recorded, no corruption). Memory sets `false`; a future SERIALIZABLE Postgres adapter sets `true` and must pass the strict form. Capabilities sit on `ConformanceAdapter` (test infra), leaving the core `StorageProvider`/`StorageDriver` interfaces untouched.

7. **`SchemaVersioner` gains additive downgrade + injectable steps.** The card requires testing upgrade **and** downgrade across ≥3 versions through `SchemaVersioner`, but core ships one v1 step and upgrade-only logic. Rather than duplicate migration logic in tests (which would test a fake, not the real class) or invent three fake *production* domain migrations, we extend the real class additively and backward-compatibly: optional `SchemaStep.down`, an injectable `steps` constructor arg (default `schemaSteps`), and `migrateTo(driver, target)` that walks up or down. Production `schemaSteps` stays a single v1 step; the 3-step ladder is a conformance fixture injected via the new constructor arg. Existing `migrate(driver)` behavior is preserved. **Fallback** (if a reviewer wants zero core change): implement a conformance-local bidirectional runner over the driver's `getSchemaVersion`/`setSchemaVersion` + `transaction`; documented in Risks. Primary plan is the additive core extension because the card names `SchemaVersioner` explicitly.

8. **CI expands in-process first, YAML matrix second.** Because `conformance.test.ts` loops the registry, one `npm run test` already covers every registered adapter — CI correctness needs no per-adapter YAML. The documented GitHub Actions `matrix` (keyed on adapter name via `CONFORMANCE_ADAPTER`) is an optional speed/isolation optimization. This keeps the "matrix expands per adapter" guarantee robust against the environment's `.github/**` write restriction and against YAML drift.

9. **Deterministic fixtures + deterministic ULIDs.** All fixture Dates and ULIDs are fixed/seeded so runs are reproducible and export deep-equality is stable. No `Date.now()`/`Math.random()` in fixtures. No new runtime dependency (ULID generator is ~30 lines of Crockford base32).

10. **Boundary compliance.** The suite imports **types** from `@domains/*/entities/*` (allowed: infrastructure may reference domain types) and the storage core from `@infra/storage`. It imports no engine internals except through registered factories. Follow the existing `// eslint-disable-next-line boundaries/no-unknown` convention only if the linter flags a cross-context type import, matching `repository-set.ts`.

## State Management Strategy

- **No app/UI state** — this is test infrastructure. "State" here is test-fixture data lifecycle and transactional isolation.
- **Provider lifecycle:** one fresh `StorageProvider` per test (`beforeEach createProvider()`, `afterEach close()`), so no cross-test bleed. Memory driver's `close()` clears its store; other engines must isolate per instance (documented adapter obligation).
- **Determinism:** all Dates and ULIDs are fixed/seeded (no `Date.now`/`Math.random`), making export deep-equality and ordering assertions stable.
- **Transactional boundaries under test:** atomicity and concurrency sub-suites deliberately exercise the driver's transaction semantics; the suite never reaches around the provider to mutate stores directly (except the schema sub-suite, which drives a raw driver by design).
- **Tenant/`orgId`:** carried on fixture records only; the suite owns the scoping convention until repositories adopt it.

## Accessibility Considerations

Not applicable in the usual sense — this card produces **no UI, no components, no rendered output**; it is headless test infrastructure. The accessibility-relevant obligations are **developer-experience**:

- **Legible failures:** each assertion targets one property with a descriptive `it(...)` name and message so a failing adapter reports *which* contract broke (e.g. `it('preserves ULID keys byte-for-byte through export/import')`), not an opaque deep-equal dump.
- **Deterministic, reproducible runs:** fixed fixtures/ULIDs mean failures reproduce identically, aiding anyone (including screen-reader users reading CI logs) to diagnose without flaky noise.
- **Structured describe hierarchy:** `Storage conformance: <adapter> › <area> › <case>` gives a scannable, navigable test tree in Vitest and CI output.

No WCAG/ARIA/keyboard/contrast surface exists in this deliverable.

## Test Plan

The deliverable **is** the test suite; "tests to write" = the suite's cases. Meta-verification that the suite itself is correct:

- **Self-check against memory (acceptance):** `npm run test:conformance` → all seven areas green for `memory`. This is the acceptance gate.
- **CRUD:** every repository method exercised with round-trip equality incl. Date instances and `null`-on-miss.
- **ULID:** byte-for-byte key round-trip; lexicographic == chronological ordering; keys unchanged through export/import.
- **Tenant scoping:** positive (own rows present) + negative (foreign rows/id absent under scope) + export/import preservation.
- **UnitOfWork atomicity:** mid-transaction throw ⇒ zero partial writes across ≥3 collections; post-interleave throw variant.
- **Concurrency:** sequential exact; parallel capability-gated (strict vs safety-invariant). Confirm memory takes the safety-invariant branch and passes.
- **Schema up/down×3:** version slot persists both directions; per-step transforms apply and reverse; no version skips.
- **Export/import fidelity:** normalized deep-equal A→B for every ordered registry pair; Dates revive as `Date`.
- **Negative/adapter-mismatch proof (optional, recommended):** a throwaway deliberately-broken in-memory adapter (e.g. one that drops `orgId`, or reorders keys, or commits on rollback) should make the corresponding sub-suite **fail** — proving the suite has teeth. Keep as a commented reference or a skipped test, not in the default registry.
- **Matrix expansion:** temporarily add a second `memory`-clone registry entry and confirm the describe tree doubles and both pass — proving auto-expansion — then remove it.
- **Green bars elsewhere:** `npm run typecheck`, `npm run lint`, full `npm run test` (including untouched `export-import.test.ts` and boundaries test).

## Risks and Edge Cases

- **Tenant scoping models a substrate contract, not repo enforcement (accepted).** Because no `orgId` exists in the domain, the suite proves the adapter *preserves and can partition by* `orgId`, not that repositories *enforce* it. Risk: a future reader assumes repo-level enforcement is tested. Mitigation: explicit `@deprecated`/TODO on the in-suite helper + this note; when repos become tenant-aware, repoint the same assertions. This is the honest maximum without expanding scope into a tenancy feature.
- **Concurrency false-pass/false-fail.** Strict serializable equality would fail on memory (last-writer-wins). Capability gating avoids a false fail while still holding serializable adapters to the strong contract. Risk: an adapter mis-declares `serializableTransactions: true` — then it must genuinely pass the strict test, which is the point. Document the flag's meaning in `types.ts`.
- **`unitOfWork()` currently throws.** Atomicity is proven via `withTransaction`. If a reviewer expects the literal `UnitOfWork` object tested, note it's a stub (not wired to a real transaction) — out of scope; TODO left for when it's implemented.
- **`SchemaVersioner` core change.** Additive and backward-compatible, but still a core edit beyond "just a test suite." Justified because the card explicitly requires up/down×3 through `SchemaVersioner`. Fallback: conformance-local bidirectional runner (zero core change) if the additive edit is rejected in review.
- **Driver access for the schema sub-suite.** `createStorageProvider` hides the driver; the sub-suite instead resolves a raw driver via the registry's `engine` name + `resolveEngine`/`open`. Edge: an engine that can't be opened twice concurrently — mitigated by using a separate driver instance and closing it in `afterEach`.
- **Memory transaction isolation subtlety.** `withTransaction` uses *all* collections, so concurrent transactions contend on the whole store; the snapshot-swap can also clobber collections a transaction didn't intend to touch if two overlap. The concurrency sub-suite's safety invariant tolerates this for memory; a real adapter must scope writes to declared collections (documented adapter obligation, already in `docs/storage-adapters.md`).
- **Export normalization.** `getAll` order is engine-defined; deep-equal must sort records by `id` and drop volatile `engine`/`exportedAt`. Edge: records without a string `id` are skipped by `importAll` — fixtures must always set `id`.
- **`.github/**` write restriction + plan-only mode.** The agent cannot commit CI YAML in this environment; the plan supplies exact content and the in-process matrix guarantees coverage regardless, so CI correctness doesn't depend on the YAML landing.
- **Vitest glob capture.** Ensure only `conformance.test.ts` matches `*.test.ts`; sub-suite libraries must not carry `.test.` in their names, or they'll run standalone without a provider and crash.
- **`isolatedModules`/`verbatimModuleSyntax`.** All type-only imports (`StorageProvider`, entity types) must use `import type`. ULID/fixture helpers are value exports.
- **Deterministic ULID monotonicity.** If the generator's counter overflows within a single millisecond bucket, ordering could break; cap N per timestamp or advance the seed timestamp per ULID to keep monotonicity trivially correct.
