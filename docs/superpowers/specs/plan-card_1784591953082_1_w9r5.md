## Overview

`docs/design-direction.md` establishes the calm, editorial "dashboard" register of the product's visual language (warm neutrals, generous whitespace, muted secondary text, a single restrained coral accent). The POS checkout/register screen is a different *register* of the same language, not a different product: a cashier working a tablet at a counter needs to read prices at a glance, hit large targets without looking, and never squint at low-contrast text — while a customer or reviewer looking at both screens side by side should immediately recognize them as the same system.

This card produces **`docs/design-direction-pos.md`**, a companion document to `docs/design-direction.md` that:

1. States explicitly what carries over unchanged from the base direction doc (palette family, type families, corner radii, motion doctrine, anti-goals).
2. Specifies exactly how five aspects bend for the register screen — contrast floor, density, hit-target size, running-total hierarchy, accent scarcity — each backed by a concrete number (contrast ratio, px, or layout budget), not a vague adjective.
3. Provides three side-by-side worked examples (button, running total, cart row) showing the *same component* in "dashboard calm" and "register adapted" form, so a reviewer can diff them visually.
4. Gives a short reviewer checklist for "does this still read as the same product."

No `/pos` checkout page exists yet in the codebase (only `src/app/pos/display` and `src/app/pos/receipt` exist today — confirmed by directory scan). This doc is therefore the specification a later implementation card will build the checkout screen against, in the same way `docs/design-direction.md` preceded any component built on its tokens. Non-goals: building the checkout page/components, adding new CSS custom properties to `globals.css`, changing `docs/design-direction.md` itself.

## Architecture Decisions

1. **Reuse only already-verified tokens/ratios from `docs/design-direction.md`; do not invent new hex values.** Every contrast claim in this doc must trace to a pairing already computed and AA-verified in the base doc's Color System tables (`--foreground` on `--background`/`--surface` = 15.9:1/16.2:1; `--accent-foreground` on `--accent` = 6.5:1; `--foreground-muted` on `--background` = 4.7:1). This card raises the register's contrast floor by choosing *which existing tokens* apply where, not by minting new colors — keeps the two docs from drifting into two different palettes and avoids unverified arithmetic.

2. **Contrast floor: retire `--foreground-muted` (4.7:1) from the register entirely.** The base doc's muted text token is AA-compliant but sits at the "calm, quiet" end of the range (4.7:1, barely above the 4.5:1 floor) — appropriate for a dashboard's secondary labels, wrong for a register where every glance has to land. Rule: **all register text — cart line items, unit prices, qty, tax line, running total — uses `--foreground` (15.9:1 on `--background`, 16.2:1 on `--surface`)**, the same near-black already used for primary text on the dashboard. Where a visual hierarchy between "primary" and "secondary" register text is still needed (e.g. an item modifier note under a line item), express it with type **scale and weight** (smaller size, regular vs. semibold), never with a lighter color. This satisfies "text... sits at AA-plus, not the calm muted end" using a number already proven, not a new one.

3. **Running total: same two tokens, inverted role, to make it the single highest-salience element.** The register total uses `--foreground` as a solid **background** block with `--surface` (or `--background`) as the text color on top of it — i.e. the exact `--foreground`/`--surface` pair already verified at 16.2:1, just with roles swapped. Inverting a light-on-dark block against an otherwise light, cream page makes the total the one dark mass on screen — this is what makes it "the single largest, highest-contrast element" both numerically (16.2:1, same ceiling as the base doc's best pairing) and visually (only element with inverted polarity). No new token, no new ratio to defend.

4. **Hit targets: 56px maps exactly onto the existing Tailwind spacing scale — no arbitrary values.** Tailwind v4's default scale is 4px-based, so `56px = 14 × 4px` lands exactly on the `14` step (`h-14`/`w-14`/`size-14`). Register primary controls (Pay, scan-confirm, qty +/−) use `min-h-14 min-w-14` (56px), one full step above the dashboard's typical `px-6 py-2` button (~40px tall). This is called out explicitly so an implementer doesn't reach for an arbitrary-value class (`h-[56px]`) when the scale already provides it.

5. **Accent scarcity: `--accent`/`--accent-strong` styling is reserved for the Pay button only.** Scan-confirm and qty +/− controls use the base doc's existing **secondary** button treatment (`border-border`, `bg-surface`, `text-foreground`) at the enlarged 56px size — the same visual family as the base doc's "Adjust" button, just bigger. This is a deliberate rule, not an oversight: introducing a second accent-weight color for "confirm" actions would dilute Pay's unmissability and contradict the base doc's anti-goal of a single restrained accent. No new color is introduced for scan-confirm; a checkmark icon plus label text carries the "positive action" meaning (WCAG 1.4.1 — never color-only).

6. **Density is a layout budget, computed against a concrete viewport, not an adjective.** "Fits without scrolling on a tablet" is unverifiable as prose. This doc fixes a floor viewport (1024×768 landscape, the common minimum tablet/iPad-class logical resolution and already this codebase's `lg` breakpoint) and shows the arithmetic: header/status bar (~56px) + running-total block (~96px, deliberately tall since it's the highest-hierarchy element) + outer padding (~24px) leaves ~592px for the cart list; a register cart row sized to comfortably contain a 56px qty control plus 8px vertical padding is ~64px tall, giving **~9 visible rows before scrolling** on the floor viewport — vs. the dashboard's spacious row (~56px tall table row at `py-3`/16px gaps) which would only fit ~6 in the same space. The doc states this as a target/budget with the caveat that carts longer than ~9 items still scroll (or a later implementation may add virtualization) — it does not claim unconditional zero-scroll.

7. **What does not change.** Palette hue family (warm ivory/cream/terracotta — no new hues), type families (Fraunces for display/numerals, Inter for body/UI), corner radii (`--radius-button`/`--radius-card` = 10px, `--radius-input` = 6px — unchanged even though targets grow), motion tokens and easing (`--motion-fast`/`--motion-base`, `cubic-bezier(0.4,0,0.2,1)`), and every anti-goal from the base doc (no neon, no skeuomorphic register-terminal styling, no rainbow of accent colors). This list is stated explicitly and first in the doc's body, ahead of the deltas, so a reader's first impression is "same system" before "here's what's different."

## Implementation Steps

**Step 1 — Re-read `docs/design-direction.md` in full** (already done during planning) to pull forward the exact token names, hex values, and contrast ratios this doc will cite — no re-deriving numbers.

**Step 2 — Draft `docs/design-direction-pos.md`** in this section order:

1. **Purpose & Relationship to the Base Direction** — one paragraph: this doc is a *register* of `docs/design-direction.md`'s language for the checkout/POS screen, not a fork; link back to it by relative path.
2. **What Carries Over Unchanged** — bulleted: palette family, type families + licensing, radii, motion, anti-goals (per Architecture Decision 7).
3. **What Bends for the Register** — five subsections, one per Architecture Decision 2–6, each stating the rule, the concrete number, and the one-line rationale ("why" — cashier glance-and-go, not browsing).
4. **Layout Budget** — the viewport/row-height arithmetic from Architecture Decision 6, presented as a small table (element → height budget) summing to the floor viewport height.
5. **Side-by-Side Worked Examples** — three subsections (Button, Running Total, Cart Row), each with two fenced code blocks placed under `**Dashboard calm**` / `**Register adapted**` headings so they can be scanned side by side, plus a short annotation naming exactly which token/size changed and why. Content per example:
   - **Button** — dashboard's existing "Void Sale" primary button (`bg-accent`, `px-6 py-2`, ~40px tall, from the base doc's Example 3) next to a register Pay button: same `bg-accent`/`text-accent-foreground`/`rounded-[var(--radius-button)]` tokens, but `min-h-14 min-w-14 px-8 text-lg font-semibold`. Annotate: token set identical; only size/scale changed; contrast unchanged at 6.5:1 (already exceeds AA for both normal and large text). Include the register's secondary-style scan-confirm/qty buttons alongside as a second pair (dashboard "Adjust" vs. enlarged 56px neutral register control), to demonstrate Architecture Decision 5.
   - **Running Total** — dashboard's stat card (base doc Example 1: `bg-surface`, `font-display text-5xl`, `text-foreground`, muted label) next to the register total: inverted block (`bg-foreground text-surface`), larger scale (`text-6xl` or `text-7xl`), full-width, no muted label — annotate the inversion and the 16.2:1 ratio reuse (Architecture Decision 3).
   - **Cart Row** — dashboard's receipt/line-item table row (base doc Example 2: `py-3 px-4`, `border-b border-border`, `text-foreground` for name/qty/price already — note this row *already* avoids muted text, so the delta here is purely density) next to the register row: reduced vertical padding (`py-2`), tighter line-height, and an inline 56px qty stepper control replacing any implied smaller control — annotate the row-height math from the Layout Budget section.
6. **Accessibility & Contrast Summary** — a small recap table: token pair → ratio → AA/AAA note → where it's used in the register (all values pulled from the base doc, per Architecture Decision 1 — no new arithmetic).
7. **Reviewer Checklist ("Still the Same Product?")** — a short bulleted checklist a reviewer can run through: same hue family present (terracotta accent visible, still only on Pay), same serif on the total's numerals, same 10px button radius despite the size increase, same motion easing on press states, no new colors introduced, anti-goals from the base doc still hold. This directly serves the card's acceptance criterion "a reviewer confirms the register still reads as the same product as the dashboard."

**Step 3 — Cross-check every cited number against `docs/design-direction.md`** (token hex values, contrast ratios, radius/motion token values) before finalizing, so nothing in the new doc silently drifts from the source of truth it's adapting.

**Step 4 — Proofread for the one hard rule this doc keeps repeating**: every deviation must have a number attached (px, ratio, or row count) — reject any sentence that says "increase contrast" or "make it denser" without the accompanying figure, since that's the specific failure mode this card exists to avoid (the base doc already did this once for the dashboard; this doc must hold itself to the same bar).

## Accessibility Considerations

- **Contrast floor is the core deliverable, not a checkbox**: the doc must show, with numbers, that register text (15.9–16.2:1) and the Pay button (6.5:1) both clear AA by a wide margin, and explicitly state that `--foreground-muted` (4.7:1) is out of scope for the register.
- **Color independence still applies at the larger scale**: scan-confirm must be labeled/iconed, not identified by a color shift alone (Architecture Decision 5) — carry the base doc's WCAG 1.4.1 rule forward explicitly rather than assuming it's implied.
- **56px targets satisfy WCAG 2.5.5/2.5.8 (Target Size)** with margin — the doc should note this is comfortably above the 44×44px AA minimum and the 24×24px AAA-adjacent floor, which is the point of choosing 56px rather than the smaller minimums.
- **Density must not defeat the target-size rule**: the Layout Budget section's row height (~64px) is sized specifically to *contain* the 56px control plus padding, not to shrink the control to fit a denser row — call this ordering out explicitly (target size is the fixed input, row height is derived from it, not the reverse).
- **Focus visibility and reduced motion**: note that both carry over unchanged from the base doc (accent-colored focus ring, `prefers-reduced-motion` collapse) — the register doesn't get a different focus/motion treatment, only different static sizing/contrast.

## Risks and Edge Cases

- **Temptation to invent new hex values for "AA-plus" register text.** Mitigated by Architecture Decision 1/2: reuse `--foreground` at its existing 15.9–16.2:1 ratio rather than minting a new near-black shade — avoids unverified arithmetic and a second near-black token to maintain.
- **56px targets crowding a real cart row once implemented.** The Layout Budget math (~9 rows on a 1024×768 floor viewport) is a planning-time estimate based on assumed header/total heights; a later implementation card may find real chrome (nav, order status) consumes more vertical space than budgeted here. Mitigate by stating the budget as a target with explicit inputs (so a later card can recompute with real measured heights) rather than a guaranteed outcome.
- **"No scrolling" over-promised.** Explicitly scope the density claim to "~9 line items on the floor viewport, longer carts still scroll" rather than an unconditional guarantee — overpromising here would make the doc wrong the first time a cashier rings up a 12-item order.
- **Accent-scarcity rule conflicting with a future "second urgent action"** (e.g. a void/refund control that also wants visual urgency). Flag this as a known tension for a later card: the base doc already reserves `--danger` (brick red) for genuinely destructive actions, so an urgent-but-not-destructive register action has no obvious token under this doc's rules — note it as an open question rather than silently inventing a third accent color now.
- **Doc/doc drift**: nothing enforces that this doc stays in sync if `docs/design-direction.md`'s base tokens change later (e.g. if the accent hex is retuned). Flag as a known gap, same class of risk the base doc already flagged for doc/CSS drift — no automated check is in scope for either.
