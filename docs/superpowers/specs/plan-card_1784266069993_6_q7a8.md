# Runtime-Extensible System Enums — Implementation Plan

`card_1784266069993_6_q7a8`

## Overview

Today every domain enum (`OrderStatus`, `PaymentMethod`, `MembershipRole`, `TaxMode`, `BarcodeSymbology`, plus the inline unions `StockMovement.movementType` and `Promotion.kind`) is a hard-coded `as const` object under `src/constants/enums/`, exported through the `enums/index.ts` barrel and rendered via a `Record<Enum, string>` label map under `src/constants/maps/`. That static registry is the correct source of truth for values every deployment must support, but it cannot answer "a shop in Dhaka wants to accept bKash" or "a shop in Berlin wants Girocard" without a code change and a deploy.

This card adds a **runtime layer on top of the static registry**, scoped per organization:

1. Fill out `src/constants/enums/` so it is the single import point for all seven registry categories named in the card: sale status (already `order-status.ts`), payment method (already `payment-method.ts`), discount type (new), unit of measure (new), stock movement type (new), shift status (new), membership role (already `membership-role.ts`). Add label maps for the four new ones.
2. Introduce a `SystemEnumValue` entity, persisted through the existing storage layer (`src/infrastructure/storage/core`, `src/infrastructure/storage/adapters/*`) exactly like every other entity — same `Collection<T>` abstraction, same `RepositorySet` wiring, same conformance suite coverage.
3. Add a `system-enums` domain (`src/domains/system-enums/`) holding the entity, the repository port, and a service that merges `static registry values ∪ org's active runtime values` and hands back both a plain string array and a Zod schema built from that merged set.
4. Wire that merged Zod schema into the one place the card asks us to prove it end-to-end: completing a sale with a payment method. A new `src/domains/sales/actions/complete-sale.ts` action validates the submitted `method` against the org's merged payment-method set (not the static one), so a runtime-added method like `bkash` is accepted while a made-up one is rejected.
5. Build an OWNER-gated admin UI at `/admin/enum-values` for CRUD on org-level `SystemEnumValue` rows, reusing the existing `MembershipRole` + `requireRole`/`hasAtLeast` authorization primitives (`OWNER = 1`, and `requireRole(role, MEMBERSHIP_ROLE.OWNER)` already means "exactly OWNER" because nothing outranks 1).

No `.agents/skills/` directory exists in this repository (verified: `.agents` is absent at the repo root), so no packaged skill applies. The conventions this plan follows instead are the repository's own established architecture, reverse-engineered from the existing code during research: the `Collection<T>` / `RepositorySet` / `StorageProvider` storage stack, the per-domain `entities/repository.ts/services/actions` layering enforced by `eslint-plugin-boundaries`, the `fixtures.ts` + `suites/*.ts` conformance pattern, and the `'use server'` + `useActionState` UI pattern already used by `login`/`signup`.

- `src/domains/system-enums/services/enum-registry-service.ts` **[new]** — the merge logic; this is the only place that builds a *runtime* Zod schema from static + org values:
  ```ts
  import { z } from 'zod'
  import type { SystemEnumValueRepository } from '../repository'
  import { STATIC_ENUM_VALUES, type EnumRegistryKey } from '@constants/enums'
  import { ConflictError } from '@shared/errors'

  export class EnumRegistryService {
    constructor(private repo: SystemEnumValueRepository) {}

    /** Static registry ∪ org's active runtime values for one category. */
    async resolveValues(orgId: string, key: EnumRegistryKey): Promise<string[]> {
      const runtime = await this.repo.listForOrgAndKey(orgId, key)
      const runtimeValues = runtime.filter((v) => v.active).map((v) => v.value)
      return Array.from(new Set([...STATIC_ENUM_VALUES[key], ...runtimeValues]))
    }

    /** A Zod enum schema over the merged set — used at every validation boundary. */
    async schemaFor(orgId: string, key: EnumRegistryKey): Promise<z.ZodEnum<[string, ...string[]]>> {
      const merged = await this.resolveValues(orgId, key)
      return z.enum(merged as [string, ...string[]])
    }

    /** Add a runtime value; rejects a duplicate (orgId, registryKey, value). */
    async addValue(input: {
      id: string
      orgId: string
      registryKey: EnumRegistryKey
      value: string
      label: string
      createdBy: string
      now: Date
    }) {
      if (STATIC_ENUM_VALUES[input.registryKey].includes(input.value)) {
        throw new ConflictError(`"${input.value}" is already a static ${input.registryKey} value`)
      }
      const existing = await this.repo.listForOrgAndKey(input.orgId, input.registryKey)
      if (existing.some((v) => v.value === input.value)) {
        throw new ConflictError(`"${input.value}" already exists for this organization`)
      }
      const record = {
        id: input.id,
        orgId: input.orgId,
        registryKey: input.registryKey,
        value: input.value,
        label: input.label,
        active: true,
        createdAt: input.now,
        updatedAt: input.now,
        createdBy: input.createdBy,
      }
      await this.repo.save(record)
      return record
    }
  }
  ```

### Completing a sale with a runtime payment method

- `src/domains/sales/entities/order.ts` — **no change**; `Order.paymentMethod?: PaymentMethod` already exists as an optional field, but per-org runtime values are strings outside the static `PaymentMethod` union, so `complete-sale.ts` will widen the write path to `string` at the point of persistence (see Architecture Decisions — "Why `Order.paymentMethod` stays typed `PaymentMethod`").
- `src/domains/sales/actions/complete-sale.ts` **[new]** — `'use server'`:
  ```ts
  'use server'
  import { z } from 'zod'
  import { createDefaultStorageProvider } from '@infra/storage/default-provider'
  import { UuidIdGenerator } from '@infra/adapters/uuid-id-generator'
  import { SystemClock } from '@infra/adapters/system-clock'
  import { requireUser } from '@domains/auth/actions/session'
  import { EnumRegistryService } from '@domains/system-enums/services/enum-registry-service'
  import { ENUM_REGISTRY_KEY, ORDER_STATUS } from '@constants/enums'
  import { toErrorResponse } from '@shared/errors'

  const baseInputSchema = z.object({
    orderId: z.string().min(1),
    orgId: z.string().min(1),
    method: z.string().min(1),
    amount: z.number().positive(),
    currency: z.string().default('USD'),
  })

  export async function completeSale(input: unknown) {
    try {
      const parsed = baseInputSchema.parse(input)
      const user = await requireUser()
      const provider = await createDefaultStorageProvider()
      const idGen = new UuidIdGenerator()
      const clock = new SystemClock()

      try {
        const result = await provider.withTransaction(async (repos) => {
          const enumService = new EnumRegistryService(repos.systemEnums)
          const methodSchema = await enumService.schemaFor(parsed.orgId, ENUM_REGISTRY_KEY.PAYMENT_METHOD)
          const method = methodSchema.parse(parsed.method) // throws ZodError if not in static ∪ runtime set

          const order = await repos.sales.findById(parsed.orderId)
          if (!order) throw new Error('Order not found')

          const payment = {
            id: idGen.next(),
            orderId: order.id,
            amount: parsed.amount,
            currency: parsed.currency,
            method,
            status: 'completed' as const,
            createdAt: clock.now(),
            updatedAt: clock.now(),
          }
          await repos.payments.savePayment(payment)

          const updatedOrder = { ...order, status: ORDER_STATUS.PAID, paymentMethod: method as any, updatedAt: clock.now() }
          await repos.sales.save(updatedOrder)

          return { order: updatedOrder, payment }
        })
        return { ok: true, data: result }
      } finally {
        await provider.close()
      }
    } catch (error) {
      return { ok: false, error: toErrorResponse(error) }
    }
  }
  ```
  `requireUser()` establishes *who*; `parsed.orgId` is trusted only because it's cross-checked implicitly by `enumService.schemaFor` (an org can only "unlock" its own runtime values — a forged `orgId` just gets a different, still-legitimate merge, never a bypass of validation itself). Full membership/ownership cross-check of `orgId` against the session is intentionally out of scope for this card (no such check exists anywhere else in the codebase yet — see Risks).

## Architecture Decisions

**New domain (`system-enums`) instead of folding into `organization`.** `OrganizationRepository` already owns six entity families; `SystemEnumValue` is conceptually adjacent (org-scoped config) but has an independent lifecycle (added/deactivated by OWNER, read by every domain that validates an enum-shaped field). A dedicated domain keeps `OrganizationRepository` from becoming a dumping ground and gives the conformance suite's existing one-`describe`-block-per-`RepositorySet`-key convention a natural home (`describe('system-enums', ...)` next to `describe('organization', ...)`).

**Merge logic lives in a service, not in `constants/`.** `constants/` may only import `constants/` (boundaries rule) and has no I/O — it can hold `STATIC_ENUM_VALUES` (pure data) but not `resolveValues`/`schemaFor`, which need a repository call. `EnumRegistryService` is a `services`-layer class taking `SystemEnumValueRepository` by constructor injection, exactly like `SalesService` takes `SalesRepository` — no direct `infrastructure` import, so it is unit-testable with a hand-rolled fake repo and reusable from any action regardless of which storage engine is live.

**Zod schema is built per-request, not module-level.** Every other Zod schema in this codebase (`createOrderSchema` in `create-order.ts`) is a static top-level `z.object(...)`. A per-org merged enum cannot be static — it depends on runtime data — so `EnumRegistryService.schemaFor()` returns a freshly constructed `z.ZodEnum` each call. This is the one deliberate deviation from the "static schema" convention, and it's isolated to `EnumRegistryService`; every other new Zod schema in this plan (`systemEnumValueInputSchema`) stays static, matching existing style.

**`STATIC_ENUM_VALUES` coerces `MembershipRole` (numeric) to strings.** `MEMBERSHIP_ROLE` values are numbers (`OWNER: 1`, ...), but `SystemEnumValue.value` and every Zod enum built from `STATIC_ENUM_VALUES` are strings (org-added runtime values are always free-text strings, e.g. `'bkash'`). `MEMBERSHIP_ROLE` is included in the registry table (the card lists it explicitly) purely for introspection/labeling in the admin UI dropdown; the admin UI does **not** allow adding runtime membership roles (`MembershipRole` drives `hasAtLeast`/numeric comparisons throughout `authorization-service.ts` — making it runtime-extensible would break that ordering-based comparison and is out of scope). The `createEnumValueAction` form should omit `MEMBERSHIP_ROLE` from the "add new value" `<select>` (still show it in the read-only static list) — call this out explicitly in the UI implementation step.

**Why `Order.paymentMethod` stays typed `PaymentMethod` in the entity, but `complete-sale.ts` writes a wider string.** Changing `Order.paymentMethod?: PaymentMethod` to `string` would ripple into every existing consumer of `Order` (`sales-service.ts`, fixtures, the conformance suite) for a field that, outside this one runtime-extensibility path, is still meaningfully constrained to the static set. `complete-sale.ts` accepts the type-widening with a narrow, commented `as any` at the single write site rather than loosening the entity's public type — the same trade-off `onboarding.ts` already makes for `plan: (input.plan ?? ORGANIZATION_PLAN.FREE) as any`. If a later card needs `paymentMethod` to be provably string-typed everywhere, that's a follow-up, not part of this one.

**No new "shift" or "discount" entity.** The card asks for the *enum categories* to exist in the registry (so they're introspectable and extensible), not for full `Shift`/discount-line features. `stock-movement-type` is the one exception wired into an existing entity (`StockMovement.movementType`) because the union already matches exactly and the change is a one-line type swap with zero behavior change — see Files list.

**Admin route placement: `/admin/enum-values`, not inside the shared `AdminLayout` role gate.** `AdminLayout` (`src/app/admin/layout.tsx`) today only calls `requireSession()` — it is the shell for *all* admin pages, not all of which need to be OWNER-only. Gating the whole layout to OWNER would be a wider change than the card asks for (and no other admin page exists yet to know if that's even desired). Instead, `enum-values/page.tsx` performs its own OWNER check, and both server actions re-check independently (defense in depth, per the Next.js Server Actions security guidance already read in `node_modules/next/dist/docs/01-app/02-guides/server-actions.md`).

## State Management Strategy

- **Server-authoritative, no client-side enum cache.** The merged enum set is resolved fresh, server-side, on every read (`page.tsx` render) and every write (`complete-sale.ts`, the admin actions) via `EnumRegistryService`. There is no client-side store, no `localStorage` cache, and no `context` provider for enum values — this avoids the classic stale-cache bug where a POS terminal keeps offering a payment method an OWNER just deactivated. The tradeoff (a DB round-trip per validation) is consistent with how every other write path in this codebase already works (`createOrder`, `createInvite`, `completeOnboarding` all open a fresh `StorageProvider` per call); no new pattern is introduced.
- **Admin UI list/create/deactivate all revalidate via `revalidatePath('/admin/enum-values')`**, not client state. `createEnumValueAction`/`deactivateEnumValueAction` return `{ error }` on failure for `useActionState` to render inline (matching `logInAction`'s `{ error }` shape); on success they call `revalidatePath` so the same server-action response includes the freshly rendered table (per the Next.js Server Actions doc: a `revalidatePath` call inside the action means the re-render ships in the same round-trip — no follow-up fetch, no client-side list mutation to keep in sync).
- **`Order`/`Payment` writes stay inside a single `provider.withTransaction(...)` call** in `complete-sale.ts` (enum resolution + order update + payment save), so a failed validation never leaves a half-completed sale — consistent with `completeOnboarding`'s all-or-nothing transaction shape.
- **No global "current org" state exists yet in this codebase** (`SessionClaims.orgId` is optional and nothing currently enforces it is set) — `complete-sale.ts` and the admin actions both take `orgId` as an explicit parameter/form field rather than inventing a new implicit-org mechanism. This is called out again in Risks because it means the immediate cross-org trust boundary is thinner than ideal, matching the rest of the codebase's current maturity level, not a regression introduced by this card.

## Accessibility Considerations

- `create-enum-value-form.tsx`: every input has an associated `<label htmlFor>` (matching `login/page.tsx`'s pattern exactly — no placeholder-as-label). The `registryKey` `<select>` gets a visible label, not just an `aria-label`, since sighted keyboard users benefit too.
- Error banner uses `role="alert" aria-live="polite"`, identical to `login/page.tsx` — screen readers announce validation/authorization failures (including "Owners only" and Zod rejection messages) without requiring focus to move.
- Submit buttons set `aria-busy={pending}` and `disabled={pending}` during the `useActionState` pending phase, matching existing forms, so assistive tech doesn't announce a stale "Add" affordance mid-submit.
- `enum-values-table.tsx`: static vs. runtime values are visually distinguished (badge color/icon) but the distinction must also be conveyed in text (e.g. a `<span className="sr-only">(built-in)</span>` / `(custom)`), not by color alone, per WCAG 1.4.1.
- Each per-row "Deactivate" `<form>` button gets an accessible name that includes the value's label (`aria-label={\`Deactivate ${value.label}\`}`), so screen reader users navigating by button list can tell rows apart — a bare "Deactivate" repeated N times is not sufficient.
- The "Owners only" gated state on `page.tsx` is rendered as a normal heading + paragraph (not just a redirect), so it's reachable and announced by screen readers rather than silently bouncing the user.

## Risks and Edge Cases

- **`orgId` trust boundary is thin everywhere in this codebase today, not just here.** `SessionClaims.orgId` is optional and nothing currently cross-checks a caller-supplied `orgId` form field against the session's actual org membership beyond `requireOwnerMembership`'s `findMembership(orgId, userId)` lookup (which *does* fail closed if the user has no membership in that org — so a forged `orgId` the user has no membership in is already rejected). What is **not** checked: if a user has *multiple* org memberships, nothing stops them from submitting a different `orgId` they legitimately belong to and adding an enum value there, or completing a sale for an order that belongs to a different org than the one implied by the UI. This mirrors the exact gap already present in `organization/actions/invite.ts` and `onboarding.ts` (`ownerUserId` defaults to a placeholder, no session cross-check). Flagging it, not fixing it — fixing session→org binding end-to-end is a separate, larger card.
- **`ConflictError` on duplicate value is a race, not a hard guarantee, under `memory`/`indexeddb` adapters.** Both drivers stage per-transaction snapshots (see `MemoryStorageDriver.transaction`'s clone-on-touch behavior); two concurrent `addValue` calls for the same `(orgId, registryKey, value)` could both pass the pre-check and both write, last-write-wins. The existing conformance `suites/concurrency.ts` pattern would be the right place to add a case if this needs a hard guarantee later — out of scope for this card since no other repository in the codebase has a uniqueness constraint enforced beyond application-level check-then-write either (e.g. `findOrganizationBySlug` has the same shape of race).
- **Deactivating a runtime value that's already referenced by historical `Payment`/`Order` rows must not corrupt history.** `deactivateEnumValueAction` sets `active: false`, which removes it from `resolveValues()`'s *future* merge set, but existing `Payment.method` / `Order.paymentMethod` strings already written stay as-is (they're plain strings on those entities, not foreign keys) — no cascading update, no data loss. Worth a one-line UI note ("existing sales made with this method are unaffected") but no code change needed beyond what's already planned.
- **`STATIC_ENUM_VALUES` becoming stale if a future card adds an eighth enum category and forgets `registry.ts`.** Low risk, but note it for the next engineer: `registry.ts` is the one file that must be touched whenever a new enum is added to `constants/enums/` if it should also be runtime-extensible. Not every enum needs to be (e.g. `TaxMode`, `BarcodeSymbology`, `OrganizationPlan`, `OrganizationStatus`, `InviteStatus`, `MembershipStatus` are deliberately excluded from `ENUM_REGISTRY_KEY` — they're not in the card's list of seven, and status-transition enums in particular should not be runtime-extensible without also updating the state machines that branch on them).
- **`value` regex (`^[a-z0-9_]+$`) could reject legitimate needs like `bKash` as a *value* (vs. label).** This is intentional — `value` is the machine key (stored on `Payment.method`, compared with `===`), `label` is the free-text display string. The admin form should visibly separate the two fields with helper text ("value: machine key, e.g. `bkash`" / "label: what staff see, e.g. `bKash`") to avoid user confusion; call this out in the UI copy during implementation.
- **IndexedDB version-chain mistakes are silent until a browser test runs.** `db.version(3).stores(...)` typos (e.g. reusing an index name that collides with `[orgId+registryKey]`'s compound-index syntax) won't fail typecheck or the `memory`-adapter conformance run — only `npm run test:conformance:browser` (fake-indexeddb / real browser) exercises `buildVersionChain`. Step 9 of Implementation Steps explicitly calls out running both conformance variants for this reason; do not skip the browser one.
- **`Object.values(MEMBERSHIP_ROLE).map(String)` ordering.** `Object.values` on a numeric-valued object returns values in the order the keys were defined in `MEMBERSHIP_ROLE` (`OWNER, ADMIN, MEMBER, VIEWER` → `1,2,3,4` → `['1','2','3','4']`), which is stable and fine for `STATIC_ENUM_VALUES` display purposes, but do not repurpose this array for anything that depends on numeric comparison — it's strings now, `'2' <= '1'` is not the same check as `hasAtLeast`'s numeric one. `requireOwnerMembership` uses the original numeric `MEMBERSHIP_ROLE.OWNER`, never this stringified table, for exactly this reason.
