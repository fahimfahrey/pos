# Implementation Plan — `domains/auth` (TAHFIUM auth: credentials, JWT sessions, guard, PIN re-auth, rate limiting)

Card: `card_1784266069993_5_773g`

## Overview

Implement the authentication domain following the existing TAHFIUM layering (`entities` → `repository` → `services` → `actions` → `app`, plus ports in `shared` and adapters in `infrastructure`). Deliverables:

1. **Credentials**: `bcryptjs`-hashed passwords (and cashier PINs) stored through the existing engine-agnostic `AuthRepository` (`users`/`sessions` collections). No engine-specific code in the domain.
2. **Sessions**: `jose`-signed JWT (HS256). Two transports over **one shared verification code path**:
   - **Server variant** — token in an `HttpOnly` cookie (set/cleared via `next/headers` `cookies()` in server actions; verified in `proxy.ts` and in a server-side Data Access Layer).
   - **Local/offline variant** — token persisted client-side (localStorage + `sessions` collection) for pure-offline PWA/terminal builds, verified by the **same** `verifyToken` function.
3. **Route guard**: `src/proxy.ts` (Next 16's renamed `middleware`) guards `/app/*` and `/admin/*` — redirect unauthenticated users to `/login?returnTo=<path>`, clear invalid cookies, and redirect already-authenticated users away from auth pages.
4. **Authorization**: role checks in a pure `services/authorization-service.ts` using the numeric `MembershipRole` enum.
5. **Cashier PIN quick re-auth** for POS terminals, with **session timeout configurable from organization settings**.
6. **Rate limiting**: in-memory limiter behind a `RateLimiter` port so a Redis limiter can slot in unchanged.

**Acceptance**: auth e2e (signup, login, expiry+returnTo, PIN re-auth, redirect-off-auth-pages, invalid-cookie clearing); a repository-level test proving credentials round-trip through the adapter (parametrized over `memory` and `indexeddb`) with **no engine-specific code**.

## Architecture Decisions

1. **`proxy.ts`, not `middleware.ts`.** Next 16 deprecated/renamed `middleware`→`proxy` (v16.0.0). Rename `src/middleware.ts`→`src/proxy.ts`, function `middleware`→`proxy`. Proxy now runs on the Node.js runtime, so `jose` verification works inside it. The card's intent ("middleware guards …") is fulfilled by the renamed convention. Keep the existing security-header logic intact.

2. **Proxy is an optimistic guard only.** Per the Next docs, do not make proxy the sole authz. Proxy does cookie-presence + JWT-signature/expiry verification for redirect UX; **real protection** is the server DAL (`requireSession()` in layouts) and **role checks live in services** (`authorization-service`). Documented in `docs/auth.md`.

3. **Ports + adapters (mirrors `Clock`/`IdGenerator`).** `Hasher`, `TokenSigner`, `RateLimiter`, `SessionStore` are interfaces in `src/shared/ports`; concrete impls in `src/infrastructure`. Services depend on ports only (satisfies the boundaries plugin: `services` may import `shared`/`constants`/`repositories`/`entities`, **not** `infrastructure`). **Actions** (which may import `infrastructure`) inject the concrete adapters into the service.

4. **`bcryptjs` + `jose` = one code path everywhere.** Both are pure-JS and run in Node (server actions, proxy) and the browser (offline PWA). The offline variant reuses `token.ts::verifyToken` verbatim — the only difference is the `SessionStore` transport (cookie vs localStorage).

5. **Keep `User.password` as the hash column.** The storage conformance fixture (`fixtures.ts::makeUser`) sets `password`. Renaming would break the passing conformance suite (out of scope). `password` stores the bcrypt hash; add optional `pinHash`. No schema-version bump needed (JSON docs; `users`/`sessions` collections already exist at `CURRENT_SCHEMA_VERSION=2`).

6. **Numeric `MembershipRole` ordering.** `OWNER=1 < ADMIN=2 < MEMBER=3 < VIEWER=4` (lower = more privilege). `hasAtLeast(role, required) => role <= required`. `MEMBER` maps to "Cashier" per `membership-role-labels.ts`, so cashier-terminal checks target `MEMBER`.

7. **Server-side persistence singleton.** `createStorageProvider` builds a fresh (empty) memory driver each call, so per-request providers lose data between signup and login under Node. `getServerStorageProvider()` caches one provider promise for the process lifetime — required for e2e and single-node server correctness. Documented limitation: multi-instance servers need a real shared engine (the abstraction already allows it).

8. **Rate limiter must outlive a request.** `authRateLimiter` is a module-level singleton in `auth-container.ts` (in-memory Map). Behind the `RateLimiter` port, a `RedisRateLimiter` can replace it without touching services/actions. Keys: `login:<emailLower>`, `login:ip:<ip>`, `pin:<userId>`.

9. **Session timeout from settings.** Add `ResolvedSettings.security { sessionTimeoutMinutes, pinReauthTimeoutMinutes, maxAuthAttempts, authWindowMinutes }`. `session-timeout.ts` reads the resolved settings; `AUTH_SECRET`/defaults in env are fallbacks only.

10. **Cookie attributes.** `HttpOnly`, `SameSite=Lax`, `Path=/`, `Secure` when `NODE_ENV==='production'`, `maxAge = sessionTimeoutMinutes*60`. Cookie name centralized in `constants/auth.ts`.

- `authorization-service.ts`: pure `hasAtLeast`, `requireRole` (throws `ForbiddenError` from `@shared/errors`), `assertActive(membership)`.
- `auth-service.ts`: constructor `(repo: AuthRepository, hasher: Hasher, signer: TokenSigner, clock: Clock, ids: IdGenerator, limiter: RateLimiter)`. Methods:
  - `signUp(input)`: reject if email/username taken; hash password; create `User` (`active:true`, `roles`); create + persist `Session`; sign token; return `{ token, session, user }`.
  - `logIn(input, meta)`: `limiter.check('login:'+emailLower)` (and ip key) → throw `RateLimitedError` if blocked; find by email; `hasher.verify`; on fail increment (implicit via check) and throw `UnauthorizedError`; on success reset limiter, create session + token.
  - `verifySession(token)`: `signer.verify` → load session → ensure not revoked/expired vs `clock.now()`.
  - `setPin(userId, pin)` / `reauthWithPin(userId, pin)`: bcrypt hash/verify `pinHash`; rate-limit `pin:<userId>`; issue short-lived re-auth token (`pinReauthTimeoutMinutes`).
  - `logOut(sessionId)`: `repo.revokeSession`.
- `session-timeout.ts`: `sessionTtlSeconds(settings)`, `pinReauthTtlSeconds(settings)`.

**Step 7 — server actions.** Each action: `getServerStorageProvider()` → within `withTransaction` build `new AuthService(repos.auth, hasher, tokenSigner, clock, ids, authRateLimiter)` → call method → `CookieSessionStore.set/clear` → `revalidatePath`/`redirect` as needed. Validate inputs with zod (return field errors for `useActionState`). `session.ts` DAL: read cookie, `verifyToken`, load user/membership; `requireSession()` redirects to `/login` when absent.

For `/login`,`/signup`: if valid token → redirect to `/app`. Extend `config.matcher` to include these plus the existing header-only catch-all.

**Step 9 — UI.** Wire `login/page.tsx` (form `action`, `returnTo` hidden input from `searchParams`, `useActionState` error region). Add `signup/page.tsx`. Add PIN re-auth UI (small client component under `src/app/_components/pin-reauth.tsx`) usable on POS terminals. **Step 10 — offline variant + docs.** `client/offline-auth.ts` (client login using browser IndexedDB provider + `LocalSessionStore`, reusing `verifyToken`). Write `docs/auth.md` (both variants, the shared verification path, single-node persistence caveat).

## State Management Strategy

- **Persisted server state**: `users` + `sessions` collections via `AuthRepository` inside `withTransaction` on the singleton `getServerStorageProvider()`.
- **Session token**: JWT is the source of truth for identity; the `sessions` record enables server-side revocation and active-session listing (`listActiveSessionsForUser`).
- **Transport state**: `HttpOnly` cookie (server) or localStorage (offline) — never expose the token to JS in server mode.
- **Form state**: React `useActionState` on login/signup/PIN forms for pending + field-error state (per `server-actions.md`).
- **Rate-limit state**: in-process Map in the `authRateLimiter` singleton (swap for Redis later).
- **Ephemeral in-memory**: proxy holds no state — it re-verifies the cookie each request.
- **Settings-derived TTLs**: resolved per request from `ResolvedSettings.security`; no caching beyond the request.

## Accessibility Considerations

- Login/signup/PIN forms: every input has an associated `<label htmlFor>`; the existing login markup is a good baseline.
- Error region: `role="alert"` + `aria-live="polite"`, associated to inputs via `aria-describedby`; set `aria-invalid` on failed fields.
- PIN entry: `inputMode="numeric"`, `autoComplete="one-time-code"` (or `off`), `type="password"` to mask; large touch targets for POS terminals.
- Submit buttons expose pending state (`aria-busy` / disabled + visible text change) from `useActionState`.
- Preserve visible focus styles; ensure redirect-to-login moves focus to the form heading (`tabIndex={-1}` on `<h1>` + focus on mount).
- Color-independent error signalling (icon/text, not color alone); contrast ≥ 4.5:1.
- `returnTo` links/redirects keep keyboard users in a logical order; no focus traps.

## Risks and Edge Cases

- **`middleware`→`proxy` rename**: leaving both `middleware.ts` and `proxy.ts` is unsupported — delete `middleware.ts` when adding `proxy.ts`. Verify the security headers still apply on all routes after the switch.
- **Server persistence under memory engine**: without `getServerStorageProvider()` singleton, signup/login data vanishes between requests. Multi-instance servers still need a shared engine — document clearly; do not claim horizontal-scale persistence.
- **Proxy is optimistic only**: never rely on it as the sole gate (server functions are POSTs to their route and a matcher change can silently drop coverage). Keep `requireSession()` in layouts and role checks in services.
- **`AUTH_SECRET` management**: `.env*` reads/writes are policy-denied for the agent; the plan only declares the variable and a dev default — the human sets the production secret. Rotating the secret invalidates all tokens (acceptable; document).
- **Cookie `Secure` in dev**: must be off on `http://localhost` or Playwright/`webServer` (built prod `next start`) will drop the cookie — gate `Secure` on `NODE_ENV==='production'` and confirm the e2e prod build over http still sets the cookie (or run e2e with `Secure` conditional honoured).
- **`await cookies()` is async** in this Next version — all cookie reads/writes must be awaited; forgetting throws.
- **Clock skew / expiry**: use the injected `Clock` everywhere (never `Date.now()` directly in services) so expiry tests are deterministic; add a small `jose` clock tolerance if needed.
- **PIN brute force**: PINs are low-entropy — enforce per-user rate limiting + lockout and require an existing valid (possibly soft-expired) session context before PIN re-auth; never allow PIN as a primary login.
- **Timing attacks**: always run `bcrypt.compare` even when the user is not found (compare against a dummy hash) to avoid user-enumeration timing leaks.
- **Boundaries plugin**: services importing infrastructure will fail lint — keep adapter injection in actions. Cross-domain `OrganizationRepository` use from auth is avoided by keeping `authorization-service` pure (operates on passed-in `Membership`).
- **Offline variant scope**: exercised by unit tests on the shared `verifyToken` path, documented in `docs/auth.md`, but not driven by e2e (which uses server mode) — state this explicitly so it isn't mistaken for untested.
- **User entity field name**: keep `password` (bcrypt hash) to avoid breaking the passing conformance fixture; a future rename to `passwordHash` is a separate, coordinated change.
