# Plan — Register Adaptation Implementation (`/pos/checkout`)

## Overview

This card **builds the register/checkout screen** by applying the visual and interaction rules from `docs/design-direction-pos.md` ("task 2," the register adaptation of `docs/design-direction.md`). It turns the register from a spec into working UI at `src/app/pos/checkout/`.

Two facts, re-verified directly against the live tree immediately before writing this plan, must anchor every decision below:

1. **No register/checkout screen exists yet.** `src/app/pos/` contains only `display/` (customer-facing mirror) and `receipt/[saleId]/` (printable receipt). `IMPLEMENTATION_PROGRESS.md` (lines 118–167, 190–203) and `docs/ui-audit.md` (Sections 3 and 5) both confirm this in detail.
2. **None of the four conventions this card is asked to "preserve exactly" currently exist as UI.** A repo-wide grep for `onKeyDown|keydown|autoFocus|useHotkeys|shortcut` across `src/` returns zero matches (re-confirmed; matches `docs/ui-audit.md` Section 5's finding). There is no scan input, no shortcut-key map, no hold/resume UI, no discount-modal UI. The task brief's "preserve … exactly" language cannot be honored literally — there is nothing to preserve. This plan therefore **originates** the first implementation of all four conventions, using the architectural commitments already recorded as fact in the domain layer and in `IMPLEMENTATION_PROGRESS.md`'s acceptance criteria (hidden barcode input w/ HID timing heuristic, keyboard-navigable interactions, hold/resume via `ParkedCart`, discount limits via `DiscountPolicyService`) as the source of truth for what "preserve" should mean once this card ships. This deviation from the literal brief is flagged here rather than silently assumed — see Risks and Edge Cases.

**No `.agents/skills/` directory exists anywhere in this repository** (re-verified via `find . -iname "*skill*"` — zero hits, no `find-skills` binary on PATH). There is therefore no project skill whose guidance applies to this card. Implementation should follow the conventions already established by the component library (`src/shared/components/ui/*`, added in the most recent commit "Establish the component foundation and Storybook") and the two design-direction docs, verified by direct file reads rather than skill lookups.

### What "apply the register adaptation rules" concretely means here

Every visual rule in `docs/design-direction-pos.md` maps to a specific implementation decision in this plan:

| Doc rule | Implementation |
|---|---|
| No muted text on register (§1) | `--foreground` only, never `--foreground-muted` / `text-foreground-muted`, anywhere under `src/app/pos/checkout/` |
| ~9 visible rows, layout budget (§2) | `RegisterHeader` capped at `h-14` (56px), `RunningTotal` block ~96px, cart list `flex-1 overflow-y-auto`, row height 64px via `min-h-16` |
| 56px hit targets (§3) | All primary controls use `Button` `size="register"` (`h-14`, already defined in `src/shared/components/ui/button.tsx:24`) |

| Accent scarcity — coral only on Pay (§5) | `variant="primary"` (accent) reserved for the Pay button only; every other register control uses `variant="secondary"` |

## Architecture Decisions

### 1. Business/cart data is read and written **client-side**, directly against the local-first store; auth/PIN stays server-side

Evidence this is the only viable design, not just a preference:

- `src/infrastructure/storage/default-provider.ts:11` — `defaultStorageEngine()` returns `'indexeddb'` only when `typeof indexedDB !== 'undefined'` (i.e., in the browser); it falls back to `'memory'` in Node. A `'use server'` action calling `createDefaultStorageProvider()` therefore **always** hits an in-memory (or, if `STORAGE_ENGINE` is set, SQLite) store that is a completely different database from what the cashier's browser has in IndexedDB. The existing `src/domains/sales/actions/complete-sale.ts` action demonstrates this exact trap — it operates on server-side storage disconnected from any client data.
- `src/app/pos/receipt/[saleId]/page.tsx:15` calls `createStorageProvider({ engine: 'indexeddb' })` **from a server component**, which cannot work in a Node runtime (`indexedDB` is not a global there) — this is a pre-existing bug/gap in the codebase (consistent with `IMPLEMENTATION_PROGRESS.md` marking the e2e journey "Not yet started"), not a pattern to copy.
- The product title itself is "Local-first Point of Sale System" (`src/app/layout.tsx:31`), and `findVariantByBarcode` / cart pricing / sale finalization all need to be fast enough to hit a **<100ms scan-feedback budget** — a Next.js server-action network round trip cannot reliably guarantee that, while a same-process IndexedDB read can.

**Decision:** `RegisterShell` and its hooks call `createDefaultStorageProvider()` from `@infra/storage` (the client-safe barrel — `src/infrastructure/storage/index.ts`, which registers only `memory` + `indexeddb`, never imports `better-sqlite3`) directly inside `'use client'` code, exactly as `use-cart-broadcast.ts` and `display-cart-view.tsx` already do for the display mirror. `FinalizeSaleService`, `ShiftService`, `CartPricingService`, and `DiscountPolicyService` are plain framework-agnostic classes (constructor-injected `Clock`/`IdGenerator`, no `'use server'` marker) — they are instantiated and called directly from the client component, exactly as `docs/ui-audit.md`'s "domain-only" tier for hold/resume and discount gates anticipated ("the eventual PIN-reauth modal must call the existing token-validation path in `discount-policy.ts`").

Auth stays server-side: `requireUser()` (already `'use server'`) gates the route in `page.tsx`, and the new `manager-override.ts` action verifies a manager's PIN server-side (bcrypt hash comparison must never ship to the browser). This mirrors the existing `pin-reauth.ts` action exactly.

This is a deliberate deviation from the action names `IMPLEMENTATION_PROGRESS.md`'s "Next Steps" section sketches (`finalize-sale.ts`, `hold-cart.ts`, `resume-cart.ts`, `open-shift.ts` as `'use server'` wrappers). Those wrappers would silently write to the wrong store per the evidence above. If a future card adds a real client↔server sync/replication layer, these client-side calls can be swapped for action calls (or kept and paired with a background sync worker) without changing the component API — `use-register-session.ts` and `pos-cart-context.tsx` are the seam.

### 2. Two storage regimes, not one

Auth/credentials (`AuthService`, `pinHash` on `User`) are verified exclusively through existing server actions (`requireUser`, `pin-reauth.ts`, and the new `manager-override.ts`) using `getServerStorageProvider()`. Sales/catalog/organization data is read/written client-side via `createDefaultStorageProvider()`. No file in this plan mixes the two: nothing under `_lib/` or `_components/` imports `@infra/storage/server` or anything under `@infra/auth/*`.

### 3. Bespoke `QtyStepper`, not the shared `NumberField`

`src/shared/components/ui/number-field.tsx:66-90` renders its increment/decrement buttons with `size="sm"` (`h-8`, 32px — `button.tsx:19`). That is below both the task's 44px floor and the design doc's 56px primary-control rule. Reusing `NumberField` as-is in a cart row would silently violate the register's own hit-target rule. `qty-stepper.tsx` is a new, small component matching `docs/design-direction-pos.md`'s Example 3 markup (56px buttons flanking a numeric span, `min-h-14 min-w-14`, `border-border bg-surface`, `--radius-input`). `NumberField`/`Input` remain fine for the two non-dense-row numeric fields in this feature (open-shift float amount, payment tendered amount) since those aren't inside the 64px row budget — but even there, the plan wraps them in `size="lg"`-equivalent classing (`min-h-11`) rather than accepting the default, to hold the 44px floor everywhere on this screen, not just in the cart list.

### 4. Minimal client-side outbox, not a full sync engine

No outbox/offline-sync infrastructure exists anywhere in the codebase today (`grep -rl "outbox|pendingSync|navigator.onLine" src` → zero hits before this card). The acceptance criteria requires a "pending-outbox count" in the header regardless. Building a full bidirectional sync engine is out of scope for a UI-adaptation card. `register-outbox.ts` is deliberately minimal: an in-memory + `localStorage`-mirrored FIFO array of `{ id, kind: 'finalize-sale' | 'hold-cart' | 'resume-cart', payload, attemptedAt }` entries. Because all writes already succeed synchronously against the local IndexedDB (decision 1), "pending" here means *not yet acknowledged by a future remote sync target* — today nothing consumes the queue, so it only grows/shrinks as a visible signal and is explicitly documented as a stub for the not-yet-built sync layer (cross-referenced from `docs/design-direction-pos.md`'s own "Known Risks" section). `useOnlineStatus` is used only to color the header pill and to decide whether to *attempt* a (currently no-op) flush — it does not gate whether a sale can be finalized, since finalize already writes to the local, always-available IndexedDB store.

### 5. Minimal inline shift bootstrap, not the full `/pos/shift/open` route

`IMPLEMENTATION_PROGRESS.md:139` lists `/pos/shift/open` (float entry form) as separate, not-yet-built UI. `FinalizeSaleService.finalize` requires a `shiftId` (`finalize-sale-service.ts:24`), so the register cannot be exercised end-to-end (and the acceptance criteria's "keyboard-only cashier can complete a sale" cannot be verified) without *some* way to open a shift. Rather than building the dedicated route (out of scope — different card, different URL, arguably owner/manager-facing float-count UI with more nuance than this card should absorb), `open-shift-panel.tsx` is a small inline form (float amount input + `ShiftService.openShift` call) shown by `RegisterShell` in place of the cart UI only when `use-register-session.ts` finds no open shift for the resolved register. It reuses the same register-size `Input`/`Button` components and token set, so it doesn't look like a bolted-on placeholder. This scope boundary is called out explicitly rather than silently expanding into a second route.

### 6. Register selection: per-device `localStorage` binding, not a cookie or picker UI

`SessionClaims` (`src/domains/auth/entities/session-claims.ts`) carries no `registerId`. A physical register in a real deployment is bound to a physical device, so `use-register-session.ts` stores the chosen `registerId` in `localStorage` (`pos-active-register-id`), scoped by `branchId`. On first load with none set, it calls `repos.organization.listRegistersForBranch(branchId)` and auto-selects the first `active: true` register (single-register branches — the common small-merchant case this app targets — need zero setup). This is a deliberate, documented default; a future card can add an explicit register picker if a branch has more than one active register and needs a chooser, without changing the storage shape.

### 7. Scan feedback is a dedicated always-mounted region, not the shared `Toast`

`src/shared/components/ui/toast.tsx` already has `success`/`danger`/`warning` variants and would be the natural reach, but it's built on `@radix-ui/react-toast`'s portal + `data-[state=open]:animate-in` mount transition (`toast.stories.tsx` confirms default 5s duration, animated entry). Portal mounting and CSS transition start are not guaranteed to land inside a 100ms budget, and the visual "flash" this task wants is a state *of the scan input's immediate surroundings*, not a queued notification. `scan-feedback-banner.tsx` is therefore a plain, always-in-the-DOM region whose background/icon/text swap via a single state variable — no mount/unmount, no animation library in the hot path (a CSS `transition-colors` using the existing `--motion-fast` token is fine for the *decay* back to idle, since that part isn't time-critical). Toast remains available and unused-by-default for this feature; it is not removed or altered.

## Implementation Steps

1. **Full-screen shell.** Build `layout.tsx` + `page.tsx`. Confirm `requireUser()` redirect works for an unauthenticated hit on `/pos/checkout` (manual check + covered by the keyboard e2e test's setup). Read `current-branch-id` cookie; if absent, redirect to `/app` (mirrors how the rest of the app expects a branch to be selected first — verify this cookie's origin at implementation time by reading whatever screen currently sets it, since no `/pos/checkout`-specific branch picker is in scope).

2. **`use-register-session.ts`.** Implement register auto-selection (Architecture Decision 6) and shift lookup (`ShiftService.getOpenShiftForRegister`). Returns a discriminated state: `{ status: 'loading' } | { status: 'needs-shift'; register } | { status: 'ready'; register; shift } | { status: 'error'; message }`.

3. **`RegisterShell`.** Wire `PosCartProvider` around the ready state. Render `OpenShiftPanel` for `needs-shift`, a full-screen error state for `error` (still full-screen and on-brand, not a raw thrown error), and the real register UI for `ready`.

4. **`pos-cart-context.tsx`.** Implement the reducer (see State Management Strategy) with pure functions, unit-testable without React. `RegisterShell` re-prices the cart via `CartPricingService.priceCart` on every line/discount change (debounced is unnecessary — pricing is a local, synchronous-fast IndexedDB read of a handful of variants, not a network call) and stores the priced result in context so `RunningTotal` and `CartList` never compute money themselves.

5. **`use-barcode-scanner.ts` + `scan-input.tsx`.** Implement the keystroke-timing heuristic: buffer characters; treat a run as scanner input if consecutive keydown events are <30ms apart; flush on `Enter` or 50ms idle after the last keystroke. On flush, call `repos.catalog.findVariantByBarcode(orgId, code)`. Keep the input focused via a `focusin`/`click` listener on `document` that refocuses it after 0ms, **except** when the newly focused element is inside a recognized "modal/editable" set (qty stepper's number input, discount amount field, any open `Dialog`/`Sheet`) — implement this as a small allowlist ref-registry (`data-scan-refocus-exempt` attribute) rather than a hardcoded element list, so new modals opt out without editing this hook.

6. **`use-scan-feedback.ts` + `scan-sound.ts` + `ScanFeedbackBanner`.** Wire the three outcomes:
   - **Success** — variant found → dispatch `SCAN_SUCCESS` to the cart reducer (increments existing line or adds new one) → banner shows item name + sage/`--success` treatment + optional short rising tone.
   - **Duplicate** — define precisely: "duplicate" means the *same barcode scanned twice within 1.5s* (guards against an accidental double-trigger of a physical scanner), not "item already in cart" (a cashier scanning the same product twice on purpose — e.g., two identical drinks — must increment normally, per the money/pricing model's per-line `quantity`). Banner uses `--warning`/amber treatment + a distinct flat tone.
   - **Not found** — `findVariantByBarcode` resolves `null` → banner uses `--danger`/brick treatment + a distinct low tone, and the raw scanned code is shown so the cashier can key it in manually or report it.
   All three states are visually distinguished by **shape/icon + text + color**, never color alone (per the design doc's WCAG 1.4.1 note, carried over from Accent Scarcity §5's rationale).

7. **`cart-row.tsx` + `qty-stepper.tsx` + `line-discount-popover.tsx`.** Build per the design doc's Example 3 markup (density, no muted text, inline 56px-class stepper). Wire qty edits to `SET_QTY` (typing a value directly, not just +/-, per the task's "fast qty edit"). Wire the discount popover to `DiscountPolicyService.validateDiscount`; on `ForbiddenError`, open `ManagerOverrideModal` and retry the same discount dispatch with the returned token once approved.

8. **`RunningTotal`.** Inverted block per §4, `text-display-2xl`, full width, no label, `tabular-nums`.

9. **`PaymentPanel`.** Tender method + amount → `FinalizeSaleService.finalize()` (client-side, per Decision 1) with the priced cart, any accumulated `managerOverrideToken`, and the shift ID from `use-register-session`. On success: clear the cart, push the write to `register-outbox` (immediately marked resolved, since the local write already succeeded — see Decision 4), and `router.push('/pos/receipt/${sale.id}')`.

10. **`VoidSaleModal`, `HoldResumeDrawer`, `EmptyCartState`, `RegisterHeader`.** Build per the "error recovery" and header acceptance criteria. `RegisterHeader` reads register/shift/cashier from context, `useOnlineStatus()`, and `useOutboxCount()`.

11. **`use-keyboard-shortcuts.ts` + `KeyboardShortcutsHelp`.** Implement the shortcut table below; wire `Escape` to close whatever `Dialog`/`Sheet` is open and refocus the scan input. This is the first shortcut map in the codebase — document it once in `keyboard-shortcuts-help.tsx` and in the usability-pass doc, not scattered as magic strings (define a single `SHORTCUTS` const map in `use-keyboard-shortcuts.ts` that both the hook and the help dialog import).

    | Key | Action |
    |---|---|
    | *(scanner input)* + `Enter` | Submit buffered barcode |
    | `F2` | Focus/open Payment panel |
    | `F3` | Hold current cart |
    | `F4` | Open Hold/Resume drawer |
    | `F6` | Open Void Sale modal |
    | `F8` | Open cart-level discount control |
    | `Ctrl+Z` | Undo last successful scan (pop most recent `SCAN_SUCCESS` from an undo stack capped at the last 10 scans) |
    | `Escape` | Close open modal/drawer, return focus to scan input |
    | `Shift+/` (`?`) | Toggle shortcuts help |

    `F1`, `F5`, `F11`, `F12`, `Ctrl+W`, `Ctrl+T` and other browser/OS-reserved combinations are deliberately avoided.

12. **`register-outbox.ts`.** Implement the minimal queue + `useOutboxCount`. Cover with a unit test that doesn't depend on any real network/sync behavior (there is none yet).

13. **`manager-override.ts` action + `AuthService.verifyManagerOverride`.** Implement, following `pin-reauth.ts`'s exact provider/hasher/clock wiring. Unit test role rejection, wrong PIN, and success paths.

14. **`docs/pos-checkout-usability-pass.md`.** Write last, after the screen is manually exercised (see Test Plan step 5) — it documents what was actually built and measured, not what was planned.

## State Management Strategy

- **Cart state:** `useReducer` in `pos-cart-context.tsx`. Actions: `SCAN_SUCCESS`, `SET_QTY`, `REMOVE_LINE`, `SET_LINE_DISCOUNT`, `CLEAR_LINE_DISCOUNT`, `SET_CART_DISCOUNT`, `CLEAR_CART_DISCOUNT`, `UNDO_LAST_SCAN`, `HOLD_CART` (snapshot + reset), `RESUME_CART` (load a `ParkedCart`'s lines back in), `RESET`. Pure reducer, no side effects inside it — all storage calls (pricing, finalize, hold/resume persistence) happen in the components/hooks that dispatch, keeping the reducer trivially unit-testable with plain objects.
- **Priced cart:** derived state, not reducer state — `RegisterShell` calls `CartPricingService.priceCart` in a `useEffect` keyed on the reducer's `lines`/`cartDiscount`, storing the `PricedCart` result in a second piece of context state (`PricedCartContext`). Keeping raw cart intent (reducer) separate from server/domain-computed money (priced cart) matches the codebase's own "server-authoritative pricing" principle (`IMPLEMENTATION_PROGRESS.md` Architecture Decision #6) even though "server" here resolves to the local domain service, not a network call.
- **Register/shift session:** `use-register-session.ts`, its own small state machine (see Implementation Step 2), provided via a separate context so `PosCartProvider` doesn't need to know about shift bootstrap.
- **Scan feedback:** local to `use-scan-feedback.ts`, not global context — it is inherently transient/local UI state (idle/success/duplicate/not-found + a timer), and nothing outside the scan-input/banner pair needs to read it.
- **Outbox count / online status:** each its own tiny hook/module-level emitter (`register-outbox.ts` follows the same subscribe/notify pattern already used by `src/shared/components/ui/use-toast.ts:14-27`, for consistency with an established in-repo pattern), not folded into the cart context, since they're unrelated concerns that would otherwise force unrelated re-renders.
- **No new dependency** (no Zustand/Redux/Jotai) — the existing codebase has zero client state library dependencies (`package.json` confirmed), and this feature's state is small enough that Context + `useReducer` (already the pattern `IMPLEMENTATION_PROGRESS.md:132` itself names for `pos-cart-context.tsx`) is sufficient.

## Accessibility Considerations

- **44px floor, 56px for primary controls, everywhere on this screen.** `Button` `size="sm"` (32px) and `size="md"` (40px) are not used anywhere under `src/app/pos/checkout/`; only `size="lg"` (48px, secondary chrome like the shortcuts-help trigger) and `size="register"` (56px, all primary controls: Pay, Hold, Void, qty stepper, Resume, discount apply/confirm) are permitted. This is stricter than the shared component library's defaults elsewhere in the app and is called out precisely because `NumberField`'s stock steppers would otherwise silently violate it (Architecture Decision 3).
- **No muted text.** `text-foreground-muted`/`--foreground-muted` must not appear in any file under `src/app/pos/checkout/`. Where secondary hierarchy is needed (e.g., a variant/modifier note under an item name), use type scale/weight only, per `docs/design-direction-pos.md` §1's explicit instruction. Add an ESLint-free but code-reviewable convention: a one-line comment at the top of `cart-row.tsx` stating this constraint, so it isn't silently reintroduced by a future edit (the doc itself has no lint rule for this — `docs/design-direction-pos.md`'s own "Known Risks: Doc/CSS drift" section flags exactly this gap).
- **Color is never the sole differentiator.** Scan feedback (success/duplicate/not-found) pairs color with distinct icon glyphs and distinct text, per Implementation Step 6. The Pay button is identified by accent color **and** label **and** position/size, matching §5's rationale.
- **Keyboard-only completability is the primary acceptance bar**, not an add-on: every interactive element (scan input, qty stepper, discount popover trigger and its fields, Pay, Hold, Void, Resume, manager-override PIN field) must be reachable via `Tab`/`Shift+Tab` in a logical order and operable via `Enter`/`Space`, in addition to being reachable via the dedicated shortcut keys in Implementation Step 11. Manual verification via the keyboard e2e test (Test Plan) is required before this card is considered done — type-checking and unit tests verify code correctness, not this.
- **Focus management on modals:** `Dialog`/`Sheet` (Radix-based, already handle focus trap + return-focus-on-close per Radix's built-in behavior) are used for `ManagerOverrideModal`, `VoidSaleModal`, `HoldResumeDrawer`, and `KeyboardShortcutsHelp` rather than hand-rolled overlays, so focus trapping doesn't need to be reinvented. After any of these closes, focus must return to the scan input specifically (not just "whatever Radix returns to," which would be the trigger button) — implement via each modal's `onOpenChange(false)` explicitly calling the scan input's ref-based `.focus()`, and cover this in the keyboard e2e test.
- **`aria-live` regions:** `ScanFeedbackBanner` uses `role="status" aria-live="polite"` (not `assertive` — a rapid-fire scanning cashier shouldn's have every scan interrupt a screen reader mid-sentence; `polite` queues). This mirrors the existing `role="alert" aria-live="polite"` pattern in `login/page.tsx:23`, adapted since this is a status update, not an error.
- **Sound is optional and user-controllable.** `scan-sound.ts` tones default **on** but are gated behind a single header-level mute toggle (persisted in `localStorage`, not global settings — this is a per-device/per-cashier preference, not an org policy) so a shared retail floor doesn't force sound on every register. Volume/tone choice is out of scope beyond "distinct."
- **`prefers-reduced-motion`:** the only genuinely time-based visual in this feature is the scan-feedback banner's decay back to idle and the qty-stepper's brief highlight on change; both use `--motion-fast`/`--motion-base` (already collapse to `0.01ms` under the media query per the base doc, inherited with no register-specific override needed, per `docs/design-direction-pos.md`'s "Implementation Guidance" §7).
- **Contrast:** every new component pulls colors exclusively from existing tokens (`--foreground`, `--surface`, `--background`, `--accent`, `--accent-strong`, `--success`, `--danger`, `--warning`, `--border`) — no new hex values, matching the "no new custom colors" item in the doc's Reviewer Checklist.

## Test Plan

1. **Reducer unit tests** (`pos-cart-context.test.tsx`, plain function tests, no rendering needed for the reducer itself): scan-add, scan-increment-existing, qty set to 0 removes line, line discount clamped to line subtotal, cart discount independent of line discounts, hold snapshots and clears, resume restores lines, undo-last-scan pops correctly and is a no-op past the 10-item cap.
2. **`use-barcode-scanner` unit tests** (jsdom + fake timers, `vitest`): fast keystrokes (<30ms apart) buffer and flush as one code on `Enter`; slow keystrokes (>30ms apart, simulating manual typing) do not falsely trigger scanner mode (define and test the actual boundary condition — this is the crux of the "HID timing heuristic" and needs explicit coverage, not just a happy path); 50ms idle flush fires without `Enter`; refocus-after-blur behavior respects the exemption registry.
3. **`use-scan-feedback` unit tests**: given a `findVariantByBarcode` mock, assert the dispatched state (`success`/`duplicate`/`not-found`) is correct for (a) new barcode, (b) same barcode twice within 1.5s, (c) same barcode after 1.5s (must be `success`, not `duplicate`), (d) unknown barcode. Assert the **latency budget itself**: measure `performance.now()` from "scan flush" to "feedback state committed" inside the test and assert it's under a generous CI-safe threshold (e.g., 50ms, leaving headroom below the 100ms product requirement to absorb CI jitter) — this is a proxy for the real-device number, not a replacement for the manual/e2e measurement in step 5.
4. **Component tests**: `CartRow` (qty edit fires `SET_QTY`, discount trigger opens popover, all interactive elements meet the 44px floor — assert via computed `min-height`/`min-width` in jsdom or via a snapshot of applied classes), `RunningTotal` (renders `bg-foreground text-surface`, never renders `foreground-muted`), `RegisterHeader` (renders register/cashier/shift/online-offline/outbox count from context, updates on `useOnlineStatus` change), `LineDiscountPopover` (over-limit discount opens `ManagerOverrideModal`; approved override lets the original discount through).
5. **`AuthService.verifyManagerOverride` tests**: correct manager PIN + correct role → token; correct PIN + non-manager role → `ForbiddenError`; wrong PIN → `ForbiddenError`; unknown email → `ForbiddenError` (not a different error shape that would leak whether an email exists).
6. **`e2e/pos-checkout-keyboard.spec.ts`** (Playwright, extends the seeding pattern the not-yet-written `e2e/pos-scan-to-receipt.spec.ts` was already scoped for in `IMPLEMENTATION_PROGRESS.md:163-168` — org/branch/register/user/product seeding, cashier login): drive the **entire flow with `page.keyboard` only, zero `page.mouse`/`page.click` calls** — log in, land on `/pos/checkout`, open shift via keyboard, tab to scan input (or confirm it's already focused), simulate a scan (`page.keyboard.type` fast enough to hit the heuristic, or expose a test-only direct-dispatch fallback if simulating true HID timing in Playwright proves flaky — document whichever approach is used, since this is exactly the kind of thing that looks trivial and isn't), adjust qty via keyboard, apply a discount requiring override via keyboard + manager PIN, pay via `F2` + keyboard form fill, land on the receipt page. This is the acceptance criteria's literal bar ("a keyboard-only cashier can complete a sale without a mouse") and must be an automated, re-runnable test, not just a manual note.
7. **`e2e/pos-checkout-scan-feedback.spec.ts`**: scan a known SKU → success banner + correct item in cart; scan the same barcode twice fast → duplicate banner, only one line added; scan an unregistered barcode → not-found banner, cart unchanged. Also captures a rough latency number via `page.evaluate(() => performance.now())` bracketing the simulated keystroke and the banner's DOM mutation (Playwright network/paint timing is not a substitute for a real-device measurement, so the usability-pass doc must state this test's number is a lower-bound sanity check, not the product's authoritative latency claim — the authoritative claim comes from manual on-device testing, documented in step 8 below).
8. **Manual verification pass** (documented in `docs/pos-checkout-usability-pass.md`, not automatable): run the dev server, physically exercise the golden path and the three error-recovery paths (mis-scan undo, void with reason, hold/resume) in a real browser per this repo's own engineering norm ("For UI or frontend changes, start the dev server and use the feature in a browser before reporting the task as complete"); record actual observed scan-to-feedback latency using browser devtools Performance panel on at least 5 scans; re-run `docs/design-direction-pos.md`'s "Reviewer Checklist" against the live screen, item by item.
9. **Standard gates:** `npm run typecheck`, `npm run lint`, `npm run test` (covers all unit/component tests above), `npx playwright test e2e/pos-checkout-keyboard.spec.ts e2e/pos-checkout-scan-feedback.spec.ts`.

## Risks and Edge Cases

- **The task brief assumes prior art that doesn't exist.** "Preserve the existing always-focused scan input and shortcut keys exactly" cannot be honored literally (Overview, fact 2). This plan originates a first, well-reasoned implementation instead of silently inventing a fictional "existing" behavior to preserve. Flagging this explicitly so the gap between brief and reality is visible to whoever reviews the implementation, not discovered later.
- **Two-storage-regime split is a judgment call, not something the codebase states outright.** It's derived from reading `default-provider.ts`, `server.ts`, `pin-reauth.ts`, and the broken precedent in `receipt/[saleId]/page.tsx` (Architecture Decisions 1–2). If a future card introduces a real sync backend that changes where auth or business data lives, this card's client/server split may need revisiting — the outbox stub (Decision 4) exists specifically to make that swap contained.
- **Manager-override token has no expiry enforcement beyond a client-held value.** `verifyManagerOverride` returns `expiresAt`, and `DiscountPolicyService.validateDiscount` already checks `new Date() > override.expiresAt` (`discount-policy.ts:44`), but the token itself is an unsigned plain object once it leaves the server action. Since it's short-lived (`pinReauthTtlSeconds`, default from `DEFAULT_PIN_REAUTH_TIMEOUT_MINUTES`) and only ever consumed locally to gate a local IndexedDB write (not a security boundary against a remote attacker in this local-first architecture), this is judged acceptable for this card; note it rather than over-engineer a signed-token scheme that the rest of this domain layer doesn't otherwise use.
- **"Void" in this card means abandoning an in-progress, unfinalized cart**, not refunding a completed/paid `Sale` (that's a distinct, larger refund flow touching `Payment`/`Refund` entities and is out of scope — the task brief's "void with reason" reads naturally as an in-progress-sale control given the rest of the acceptance criteria is entirely about the pre-payment cart experience). If the intent was post-sale voiding, that's a different card; this plan should not silently absorb that scope.
- **Duplicate-scan definition is a judgment call** (Implementation Step 6) — a 1.5s same-barcode window. Too short and a fast, deliberate double-scan of two identical items gets misflagged; too long and legitimate quick re-scans of the same item get blocked. 1.5s is a starting point to validate during the manual pass (step 8 of Test Plan), not a number derived from user research (none exists for this product yet).
- **Single-register auto-selection (Decision 6) will pick the wrong register if a branch has more than one active register and no per-device binding has been made yet.** Acceptable for this card (matches the target small-merchant use case), but the usability-pass doc should note it as a known limitation, not a solved problem.
- **HID timing heuristic constants (30ms/50ms) are estimates**, not measured against real scanner hardware (none available in this environment). The unit tests in Test Plan step 2 pin the *behavior* at these constants so they're easy to tune later without hunting through component code, but the constants themselves should be treated as provisional.
- **`register-outbox.ts` doing nothing but counting is intentionally underwhelming** — it would be a mistake to read the acceptance criteria's "pending-outbox count" as license to build a real sync engine in this card (that's a multi-card effort touching conflict resolution, retry backoff, and a server sync endpoint that doesn't exist). Resist scope creep here specifically.
- **`page.keyboard.type` may not faithfully reproduce real USB/HID scanner keystroke timing in Playwright/CDP.** If step 6 of the Test Plan proves flaky in practice, the fallback (a test-only direct-dispatch hook) must be clearly marked as test-only (e.g., gated behind `process.env.NODE_ENV === 'test'` or a Playwright-injected flag) so it can never be triggered in production and doesn't become an unintended second input path.
