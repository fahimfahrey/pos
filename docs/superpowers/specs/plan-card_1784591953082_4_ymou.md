# Plan: Base Component Library on Radix/shadcn Primitives (Token-Driven)

## Overview

Build a token-driven base component library at `src/shared/components/ui/` on top of unstyled Radix primitives, following the shadcn pattern (own the component source, not a black-box npm package). Every visual property — color, radius, shadow, spacing, type, motion — resolves through the CSS custom properties already established in `src/app/globals.css` and `@theme inline` (task: "Build the design token system wired to per-tenant theming", commit `2c7881d`). No component hardcodes a Tailwind color/gray/blue utility; only semantic token-backed utilities (`bg-surface`, `text-foreground`, `border-border`, `text-accent`, `rounded-[var(--radius-button)]`, etc.) or `cn()`-merged class names are used.

Scope: Button, Input, NumberField, Combobox, Select, Dialog, Sheet, Toast, Tooltip, Tabs, Table, DataGrid, Badge/Pill, Card, Skeleton, EmptyState, Money. Each ships:
1. A Radix-backed, token-styled implementation.
2. A colocated Storybook story file covering default/hover/focus/disabled/loading/error states, both themes (`data-theme='light'` / `'dark'`), and both fonts/locales (English/Inter, Bengali/Hind Siliguri via `--font-bengali`), rendered in the serif (`--font-display`, Fraunces) + sans (`--font-body`, Inter) pairing from task 1.
3. Keyboard operability and visible focus (relies on the existing global `*:focus-visible { outline: var(--focus-ring-*) }` rule in `globals.css` — components must never suppress focus-visible with `outline-none` unless they supply an equivalent token-driven ring).
4. ARIA labeling appropriate to the Radix primitive (most is free from Radix; components must not override or strip it).

This is new infrastructure — no existing screen currently imports a shared UI kit (confirmed: `src/app` and `src/domains` have zero pre-existing generic `Button`/`Card`/etc. components; `git grep` for `shared/components` returns nothing). No consuming screens are migrated to the new components in this task; that is explicitly out of scope (audit-and-build the library, not a rip-and-replace of `/login`, `/pos/checkout`, etc.). Do not touch existing app screens except where noted (font-loading verification only).

**Money component note:** currency/locale are **not** global — they are per-tenant, resolved from `ResolvedSettings.currency` / `ResolvedSettings.locale` (`src/domains/organization/entities/settings.ts:67-69`, defaults `'USD'` / `'en-US'`). The `Money` component therefore takes `currency`/`locale` as **required props** (or reads them from a lightweight formatting context provided by the caller) — it must not import from `src/domains/*` (the `shared` boundary layer only allows importing `shared`/`constants`, enforced by `eslint-plugin-boundaries` in `eslint.config.mjs:103-141`). Screens under `src/app` are responsible for threading the resolved tenant settings down as props.

## Architecture Decisions

1. **shadcn pattern, not a packaged design-system dependency.** Component source lives in-repo (`src/shared/components/ui/`) so every visual decision routes through project tokens and stays lint/type-checked with the rest of the app, per `AGENTS.md`'s instruction to verify against this fork's actual conventions rather than assume standard tooling behaves as expected.

2. **Placement under `src/shared/components/ui/`, not `src/components/ui/`.** The existing `eslint-plugin-boundaries` config (`eslint.config.mjs:103-118`) already includes `src/shared/**` as an allowed source for every layer (`app`, `actions`, `services`, ..., `shared`). Placing components anywhere else requires new boundary rules and risks breaking the existing `default: 'disallow'` posture. `shared` is also the only layer with zero domain knowledge, matching the requirement that Money/Table/etc. be pure presentational primitives.

3. **Money/locale/currency are props, not globals.** There is no global "current tenant" store for UI components to read from (theming is already wired per-tenant via `tenant-theme-provider.tsx` / `theme-resolver.ts`, but that only carries `logoUrl` today — no currency/locale). Rather than invent a second context provider that duplicates `ResolvedSettings`, `Money` takes `currency`/`locale` as explicit props. This keeps the `shared` boundary rule intact (no domain import) and keeps the component trivially testable/storyable without a provider tree.

4. **Combobox has no Radix primitive — build on `Popover` + ARIA listbox pattern.** Radix ships `Select` (for closed-choice pickers) but not an open-text-filter `Combobox`. Do not fake it by hacking `Select`; implement the documented WAI-ARIA 1.2 Combobox pattern manually on top of `@radix-ui/react-popover`, matching the same approach shadcn/ui itself uses (`Command`-style listbox). Keep it minimal — no fuzzy-search library dependency unless a consumer need arises later (YAGNI; the task only requires the primitive, not a search engine).

5. **Sheet is a variant of Dialog, not a separate Radix wiring.** Both are `@radix-ui/react-dialog` under the hood; `sheet.tsx` imports and composes `dialog.tsx`'s primitives with a slide-in `cva` variant rather than re-implementing focus trap / escape / overlay logic a second time.

6. **Theme and locale switching in Storybook via global toolbar, not per-story boilerplate.** A single `.storybook/preview.tsx` decorator reads Storybook `globals` (`theme`, `locale`) and sets `data-theme` + `lang`/`font-*` class on a wrapper. Individual stories don't each hand-roll four permutations; instead, `argTypes`/`parameters` combined with the global toolbar let a reviewer flip theme/locale live. **However**, the acceptance criterion asks for stories that show light/dark and en/bn "variants" explicitly — so in addition to the global toggle (for interactive exploration), each component's story file also exports explicit fixed-permutation stories (e.g. `DarkBengali`) that hardcode `parameters: { theme: 'dark', locale: 'bn' }` so the automated visual-regression baseline captures all 4 permutations without a human flipping toggles first. This satisfies both interactive review and deterministic CI capture.

7. **Visual regression via Playwright screenshots against the static Storybook build, not a paid third-party service.** The repo already depends on `@playwright/test`. Add a dedicated `playwright-visual.config.ts` (separate from any existing e2e Playwright config — check `playwright.config.ts` if one exists before adding a second, and namespace clearly) that: builds Storybook (`npm run build-storybook`), serves the static output, iterates `storybook-static/index.json` to discover every story ID, and calls `page.goto` on each story's iframe URL + `expect(page).toHaveScreenshot()`. First CI run commits the baseline PNGs under `tests/visual/__screenshots__/`; this satisfies "visual-regression baseline captured" without introducing a paid Chromatic dependency (not in `package.json`, and adding a paid external service is out of scope / not requested).

8. **Automated a11y via `@storybook/test-runner` + `axe-playwright`, run against the built Storybook**, not just the interactive `addon-a11y` panel (which is dev-time only and not "automated" in a CI sense). `test:storybook-a11y` script runs `test-storybook` which, per Storybook's documented pattern, executes an `axe` check in a `postVisit` hook for every story — this is what makes "automated a11y (axe) passes on all stories" true and CI-checkable rather than manually eyeballed.

9. **Tabular figures are enforced at the component level, not left to callers.** `NumberField`, `Table`/`DataGrid` numeric cells, and `Money` all hardcode the `tabular-nums` Tailwind utility (backed by the existing `--text-*` scale and `font-variant-numeric` default in Tailwind v4, confirmed present per `docs/design-direction.md:427`) inside the component itself — a consumer cannot forget to add it.

## State Management Strategy

These are presentational primitives; state is local and uncontrolled-by-default with controlled overrides, matching Radix's own convention:

- **Uncontrolled-first**: `Dialog`, `Sheet`, `Select`, `Tabs`, `Tooltip`, `Toast` all support Radix's `defaultOpen`/`open`+`onOpenChange` pair — components pass both through untouched so callers can choose uncontrolled (simple cases) or controlled (e.g. a checkout flow that must close a dialog programmatically after a server action resolves).
- **`Input`/`NumberField`/`Combobox`** follow standard React controlled-input convention (`value`/`onChange`), no internal state duplication of the value itself; `Combobox` internally manages only transient UI state (open/highlighted-index), never the selected value.
- **`Toast`** requires a single app-level `<ToasterProvider>` (Radix `Toast.Provider` + `Toast.Viewport`) mounted once near the root; `use-toast.ts` exposes an imperative `toast()` function (a small module-level event-emitter/store pattern, the same approach shadcn/ui uses) so any component can trigger a toast without prop-drilling — this is the one place a lightweight internal store is justified, since toast-triggering is inherently cross-tree.
- **No global theme/locale state inside components.** Theme (`data-theme`) is an ancestor DOM attribute the components read via CSS custom properties only — components never call `useState` for theme. Locale for `Money` is a prop, not context, per Architecture Decision 3.
- **`DataGrid` sort state**: controlled by the consumer (`sortColumn`/`sortDirection`/`onSortChange` props) — the component does not own a sorted copy of data internally, keeping it a pure display component and avoiding a second source of truth for order.

## Accessibility Considerations

- **Focus visibility**: rely on the existing global `*:focus-visible` rule (`globals.css:87-90`) driven by `--focus-ring-*` tokens; never add `outline: none` / `focus:outline-none` in any component without a compliant replacement ring. Verify Radix's own focus-trap/return-focus behavior (Dialog, Select, Toast, Tooltip, Popover) is not fighting this rule.
- **Keyboard operability per component**:
  - Button: Space/Enter activates; `loading` state must remain focusable but not re-triggerable (`aria-disabled` + `disabled` both considered — prefer `aria-disabled` if the button must stay in the tab order to announce state, otherwise plain `disabled`; decide per Radix/native semantics and document the choice in the component).
  - Input/NumberField: native tab order; NumberField Up/Down arrow keys adjust value in addition to spinner buttons.
  - Combobox: Arrow Up/Down moves highlight, Enter selects, Escape closes without selecting, Home/End jump to first/last option, typeahead jumps to matching option — full WAI-ARIA 1.2 Combobox keyboard spec, not a subset.
  - Select/Tabs/Tooltip/Dialog/Toast: inherited from Radix — verify (don't assume) via manual keyboard pass that Escape, Tab-cycling, and arrow navigation all work per Radix's shipped defaults before considering the component done.
  - DataGrid: sortable column headers must be real `<button>` elements (not `<div onClick>`) so they're keyboard-reachable, with `aria-sort="ascending"|"descending"|"none"` on the parent `<th>`.
- **ARIA labeling**:
  - Icon-only Button variants require a mandatory `aria-label` prop (TypeScript should make it required when `children` is absent — enforce via a discriminated union prop type, not just a runtime warning).
  - Input/NumberField/Combobox/Select: label association via `<label htmlFor>` or Radix's built-in `Label` primitive; `error` state wires `aria-invalid="true"` + `aria-describedby` pointing at the error message `id`.
  - Dialog/Sheet: `Dialog.Title` required (TS prop, not optional) — a dialog with no accessible name is a hard accessibility bug, not a style nit.
  - Toast: verify Radix's default `role`/`aria-live` mapping is correct for each variant (danger toasts likely warrant `aria-live="assertive"` — check Radix docs/source, don't guess).
  - Table/DataGrid: proper `<thead>/<tbody>`, `scope="col"` on header cells, caption or `aria-label` on the `<table>` when no visible heading precedes it.
  - Skeleton: `aria-hidden="true"` (decorative), loading state announced via `aria-busy` on the real content container instead, so screen readers aren't spammed with meaningless skeleton content.
  - EmptyState: heading structure must not skip levels relative to where it's mounted (component takes a `headingLevel` prop rather than hardcoding `<h2>`, so callers keep a valid document outline).
- **Color contrast**: no new colors are introduced — all components consume the already-AA-audited tokens from `docs/design-direction.md`'s contrast table and `src/shared/utils/color-contrast.ts`. If any component-specific state (e.g. disabled text) needs a new shade, it must be checked against `color-contrast.ts`'s existing test harness rather than eyeballed.
- **Reduced motion**: shimmer/spinner/transition animations must be plain CSS `animation`/`transition` (not JS-driven RAF loops) so the existing `prefers-reduced-motion` global rule (`globals.css:93-100`) actually suppresses them.
- **Bengali rendering**: verify Hind Siliguri actually renders Bengali script correctly in Storybook's `preview-head.html` font loading (don't assume the `<link>` tag works — visually confirm in the browser during step 3/5, since Storybook doesn't get Next's font optimization pipeline for free).

## Risks and Edge Cases

- **This Next.js fork has undocumented breaking changes** (per `AGENTS.md`). `@storybook/nextjs`'s webpack/Turbopack integration and `next/font` handling are exactly the kind of surface likely to diverge. Mitigation: step 1 explicitly reads the fork's own docs before wiring the framework integration; if `@storybook/nextjs` proves incompatible, fall back to the framework-agnostic `@storybook/react-vite` builder (Storybook doesn't strictly need Next's server, only the component tree + Tailwind/PostCSS pipeline) — flag this decision point rather than silently forcing a broken integration.
- **No `playwright.config.ts` currently confirmed to exist or not** — check before adding `playwright-visual.config.ts` so the two configs don't collide on `testDir`/`testMatch` and accidentally run visual tests as part of an unrelated e2e suite (or vice versa). Namespace `testDir: './tests/visual'` explicitly.
- **CI wiring is out of scope but the criteria imply CI-checkable gates.** `.github/workflows/**` is denied to this agent (global CLAUDE.md deny rule) — the three new scripts (`build-storybook`, `test:storybook-a11y`, `test:visual`) must be documented (e.g. in this plan and a short note in `docs/ui-audit.md`) as ready-to-wire but not actually added to a workflow file. Flag this clearly to the user as a manual follow-up.
- **`bn-BD` currency/locale correctness for `Intl.NumberFormat`** — Bengali is currently only a font token (`--font-bengali`), not a wired locale/currency anywhere in `ResolvedSettings` defaults (`'en-US'`/`'USD'`). The Money component and its Bengali story must use a real, valid BCP-47 locale (`bn-BD` or `bn-IN`) and ISO 4217 currency (`BDT`) — verify these are accepted by `Intl.NumberFormat` in the target Node/browser runtime rather than assuming.
- **`class-variance-authority` version compatibility with React 19 / Tailwind v4** — this repo is on bleeding-edge versions (`react@19.2.4`, `next@16.2.10`, Tailwind v4 via `@tailwindcss/postcss`); pin and test the exact `cva`/Radix package versions rather than accepting whatever `npm install` resolves to latest, since Radix packages have historically lagged React 19 support — verify each `@radix-ui/react-*` package's peerDependencies before locking them in.
- **Toast's imperative `toast()` API is the one piece of module-level mutable state in this otherwise stateless library** — must be SSR-safe (no `window`/module-level singleton access during a server render); guard appropriately since this is a Next.js App Router project where component modules can be evaluated on the server.
- **Icon-only Button `aria-label` enforcement** — TypeScript discriminated unions catch this at compile time for direct usage, but do not catch it if a consumer spreads props dynamically; document this as a known gap rather than over-engineering a runtime check for a case the codebase doesn't currently have.
- **DataGrid scope creep** — "DataGrid" in the task could be read as implying virtualization/pagination/filtering. Explicitly scoped down to "Table + sortable headers + loading/empty states" per Architecture Decision item on `data-grid.tsx`; if the user later needs virtualization for large registers of products, that's a separate follow-up task, not silently added here.
- **Existing screens (`/login`, `/admin/audit`, etc.) are NOT migrated to the new components in this task** — the ui-audit correctly lists them as still using ad hoc `bg-blue-600`/`bg-gray-*` classes; that migration is explicitly out of scope here (task is "standardize the base component library," not "roll it out"), but should be called out to the user as the natural next task.
