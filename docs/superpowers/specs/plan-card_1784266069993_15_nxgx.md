## Architecture Decisions

1. **SQLite engine choice: `better-sqlite3` (server-only), not `wa-sqlite` (in-browser WASM).** Justification to record verbatim in `docs/storage-adapters.md`:
   - `better-sqlite3`'s fully synchronous API makes atomic transactions trivial to reason about and implement correctly on the first attempt — directly relevant to the card's own "record how long the adapter took" ask, since `wa-sqlite`'s OPFS `SyncAccessHandle` + Worker-thread requirement (OPFS main-thread synchronous file access is disallowed in most browsers) would add a whole extra concurrency layer (message-passing to a worker) unrelated to proving the `StorageDriver` contract.
   - This repo's storage architecture already has a precedent for engine-per-runtime-context: `memory` for SSR/tests/tooling, `indexeddb` for the browser. A server-side `sqlite` engine slots into the same precedent as a third context (Node tooling, CI conformance parity, a future server-hosted/multi-terminal deployment mode) rather than displacing IndexedDB as the in-browser engine — this is architecturally cleaner than making SQLite compete with IndexedDB for the same (browser) runtime slot.
   - This does **not** claim SQLite becomes the app's live browser storage. If a future card wants a genuinely offline in-browser SQL engine, `wa-sqlite` remains the documented alternative — record this explicitly as a "not chosen, here's why" note so the decision isn't re-litigated blind later.

2. **`better-sqlite3` must never enter the client webpack bundle — this drives file layout, not just a build flag.** `src/infrastructure/storage/index.ts` today unconditionally imports `./adapters/indexeddb` and is reachable from client code (browser engine selection). If `./adapters/sqlite` were added there per the guide's literal step 4 ("Wire It Up — one config change"), webpack's client compiler would try to resolve `better-sqlite3` (a native `.node` binary requiring `fs`) for the browser bundle and fail the build. `serverExternalPackages` in `next.config.ts` only affects the **server** compiler — it does not stop the **client** compiler from choking on a module reachable from client-imported code. Fix: a new `src/infrastructure/storage/server.ts` entry point, marked with `import 'server-only'` (fails the build loudly, not silently, if a client component ever imports it), used exclusively by `scripts/migrate-engine.ts`, any future server actions needing the `sqlite` engine, and `tests/storage/sqlite-adapter.test.ts`/the conformance matrix (both run under Node/Vitest, not the webpack client graph, so importing the driver directly there is safe). **This is contract gap #2** (see Overview) — the fix goes into the guide as a new "Server-only adapters" section describing this pattern generally, since any future Postgres/MySQL/LevelDB adapter (all listed in the guide's own effort table) hits the identical problem.

3. **Transaction execution model — contract gap #1.** `StorageDriver.transaction()`'s `work: (tx) => Promise<T>` signature is agnostic about whether the underlying engine's transaction primitive is itself synchronous or asynchronous. IndexedDB/Dexie and the in-memory driver's staged-clone approach are naturally fine with arbitrary awaited work inside the callback. `better-sqlite3` transactions are synchronous C-binding calls — there is no way to literally "await" in the middle of one. The safe implementation is: issue `BEGIN IMMEDIATE` synchronously, run `await work(tx)` (all of `tx`'s own methods are synchronous SQLite calls wrapped in `Promise.resolve()` to satisfy the interface, so no real yield happens on the driver's own I/O), then `COMMIT`/`ROLLBACK` synchronously — and **serialize all `'readwrite'` transactions on a single mutex/write-queue**, exactly mirroring `MemoryStorageDriver`'s `writeQueue` field (`src/infrastructure/storage/adapters/memory/memory-driver.ts`). The contract gap: nothing in `driver.ts`'s JSDoc or the guide states the invariant "`work` must not perform I/O unrelated to the methods on `tx`" — an adapter author relying on a synchronous engine could otherwise be misled into thinking arbitrary async work mid-transaction is safe (it usually "works" by accident on a single-threaded event loop as long as nothing else touches the DB concurrently, which is exactly the kind of latent bug that survives testing and breaks under load). **Fix**: add this invariant to `driver.ts`'s `StorageDriver.transaction` JSDoc and to the guide's "Key Design Principles" list (new principle 6), not as an SQLite-specific caveat.

4. **`STORAGE_ENGINE` (server) vs `NEXT_PUBLIC_STORAGE_ENGINE` (client) stay separate env vars, not one.** The card's literal wording ("switching engines must be exactly `STORAGE_ENGINE=sqlite`") maps naturally onto a **new server-side var**, because `NEXT_PUBLIC_STORAGE_ENGINE` is inlined into the client bundle at build time (Next.js `NEXT_PUBLIC_*` convention) and is read by `defaultStorageEngine()` specifically to choose the browser's own engine (`indexeddb` vs `memory` fallback) — repurposing it for `sqlite` would either be a no-op (browser never resolves `sqlite`, since it's only registered in `storage/server.ts`) or, worse, would need `storage/index.ts` to conditionally import the server module, reintroducing the client-bundle problem from Decision 2. `STORAGE_ENGINE` (no `NEXT_PUBLIC_` prefix) is added to `serverSchema` in `src/shared/env.ts`, consumed by `scripts/migrate-engine.ts` and any future server-only code path that needs engine selection (e.g. a server action doing a scheduled backup). Document both vars side by side in the guide's Environment Variable Reference so this split doesn't look like an oversight.

5. **Single generic key-value table, not one SQL table per collection.** The `StorageDriver` contract (per its own "Key Design Principles") deliberately has no secondary-index or query-builder surface — `getAll()` scans the whole collection, records are small in practice. Mirroring this in SQL, `SqliteDriver` uses one physical table: `kv_store(collection TEXT NOT NULL, id TEXT NOT NULL, data TEXT NOT NULL, PRIMARY KEY (collection, id))`, storing each record as its own-serialized JSON in `data` (same JSON shape the memory/indexeddb drivers already hold in-process — no new codec). A `meta(key TEXT PRIMARY KEY, value TEXT)` table holds `schemaVersion`, mirroring IndexedDB's `__meta__` object store. This keeps the adapter to genuinely "one file" (no per-collection DDL migrations to maintain) and sidesteps `CollectionName` being a fully dynamic string set (`COLLECTIONS` in `core/schema.ts` — verify exact set at implementation time; do not hardcode a table-per-collection schema against a set that can grow via `schemaSteps` in `schema-versioner.ts`, since ordinary SQL `CREATE TABLE IF NOT EXISTS` per new collection would need to run in lockstep with every future `SchemaStep` — the flat-table design makes that entire migration category a no-op for this adapter, another argument for it belonging in the guide's principles rather than just being a local implementation shortcut).

6. **`open()` runs `PRAGMA journal_mode = WAL` and `PRAGMA foreign_keys = OFF`** (no foreign keys are declared — the KV design has none) for reasonable concurrent-read performance; not required for correctness given the write-queue mutex, but avoids `SQLITE_BUSY` surprises if something outside this driver ever opens the same file (e.g. an inspection tool during the runbook drill).

## State Management Strategy

- **SQLite driver state**: the `better-sqlite3` `Database` handle is held as a private field on `SqliteDriver`, opened lazily on first `open()` call (never at module import — matches the guide's "No SSR risk" principle). All record state lives in the `kv_store` table; the driver itself holds no in-memory cache of records (unlike nothing — the memory driver *is* the cache; the indexeddb driver has none either — this keeps `sqlite` consistent with `indexeddb`'s "durable engine, no driver-level cache" shape rather than the memory driver's "driver-level cache is the whole store" shape).
- **Write serialization state**: a single `writeQueue: Promise<void>` field per `SqliteDriver` instance, identical pattern to `MemoryStorageDriver`. This is process-local — if a future deployment ran multiple Node processes against the same SQLite file, cross-process write serialization would need OS-level file locking (SQLite's own locking already provides this at the file level; the in-process queue is purely to avoid `SQLITE_BUSY` retries within a single process, not a correctness requirement given SQLite's own transaction guarantees). Document this distinction in the adapter README so it isn't mistaken for a cross-process mutex.
- **Migration script state**: `scripts/migrate-engine.ts` holds no persistent state between runs — it is a pure export→import→verify pipeline over two already-stateful `StorageProvider` instances, consistent with how `exportAll`/`importAll` are used everywhere else in the codebase (backup/restore UI, conformance suite's own `suites/export-import.ts`). No new state-management pattern is introduced.
- **Service worker / PWA state**: the service worker caches the app shell (static assets) only, via the Cache Storage API — it does not touch IndexedDB/application data, keeping data state fully owned by the existing storage subsystem and avoiding a second source of truth. Install-prompt state (`beforeinstallprompt` event, dismissed-by-user flag) lives in a small client component's local state plus `localStorage` for the dismissal flag, matching the existing offline-auth pattern's use of `localStorage` for lightweight client-only flags (`src/domains/auth/client/offline-auth.ts`).
- **Sentry**: no app state — purely an observability side-channel initialized once at startup per the SDK's own lifecycle.

## Risks and Edge Cases

- **`.github/workflows/**` writes may be blocked by this environment's sandbox policy** (same risk card 14 already flagged, confirmed still true — no `.github/` directory exists in the repo today). Fallback: produce the exact CI YAML content in the PR description for a human to apply manually.
- **Card 14's own deliverables (audit log, seed script, existing CI file, `tsx` dependency) may or may not be implemented yet by the time this card starts** — `scripts/`, `tsx`, and `.github/workflows/` were all confirmed absent as of this plan's writing, meaning either card 14 hasn't been implemented, was implemented differently, or was descoped. **Before starting Implementation Step 9/15, re-check the current tree state** rather than assuming this plan's snapshot is still accurate — if `scripts/seed.ts` or a CI file already exist, integrate with them (e.g. add `migrate-engine.ts` alongside an existing `scripts/seed.ts`, add the `audit` job to an existing `ci.yml`) rather than treating this plan's file list as exhaustive truth.
- **`better-sqlite3` is a native addon** — CI runners and any containerized dev environment need matching Node ABI/prebuilt binaries or a working native toolchain. If the implementation environment can't install it (no build tools, non-standard Node build), this blocks Part 1 entirely; there is no pure-JS fallback for `better-sqlite3` specifically (that's what `wa-sqlite` would have been — if this blocks, the fallback decision is to switch engines to `wa-sqlite`, not to fake the adapter, and to update the "chosen and justified" section of the guide accordingly with the real reason).
- **The conformance suite's `export-import.ts` suite exercises cross-engine round-trips using `allAdapters`** — adding `sqlite` to `defaultConformanceAdapters` means every *existing* adapter's export/import test now also round-trips against `sqlite`. If the flat-KV-table design (Architecture Decision 5) has any subtle serialization mismatch (e.g. SQLite's `TEXT` affinity vs a numeric-looking ULID string, or JSON key ordering affecting a naive equality check rather than the suite's presumably-structural comparison — verify `normalize.ts`'s comparison strategy at implementation time), it would show up as a **failure in an existing adapter's test**, not just the new one — this is a feature (it's exactly the kind of gap the card wants surfaced) but budget time for it, since it's the most likely source of an actual contract gap beyond the two anticipated in this plan.
- **`fake-indexeddb` vs real IndexedDB availability under Node for `migrate-engine.ts`** — the script needs to run outside a browser (CLI), so `indexeddb` as a `--from`/`--to` engine choice may not be meaningfully supported there without `fake-indexeddb` (a devDependency, not meant for production data). Decide explicitly (per Implementation Step 9) whether the script supports `indexeddb` at all, or is scoped to `memory ↔ sqlite` (and browser-side import/export for `indexeddb` stays on the existing backup-UI path documented in `docs/backup.md`) — do not silently half-support it.
- **CSP breaking existing functionality is the single highest-risk item in Part 2** — Next.js dev-mode HMR, any inline `style` attributes from Tailwind's JIT output (verify whether Tailwind v4's engine emits any inline styles requiring `'unsafe-inline'` or a nonce), and third-party embeds (none currently found, but re-check) can all be silently broken by an overly strict policy that only fails in production builds, not dev. Verify the CSP against a **production build** (`next build && next start`), not just `next dev`, before considering Step 14 done.
- **Sentry DSN availability in this environment** — if no real Sentry project/DSN is available to the implementer, the "verify a deliberately-thrown error actually reaches Sentry" bar in Step 16 can't be fully met end-to-end; fall back to verifying the SDK correctly captures and would-transmit the event (mock/intercept the network call) and document this limitation explicitly in the launch checklist rather than silently claiming full verification.
- **"The application is production ready at the completion of this task" is the card's own closing claim** — treat `docs/launch-checklist.md` as the actual arbiter of that claim, not a rhetorical flourish. If any checklist line can't be genuinely backed by evidence by the end of implementation (e.g. Sentry DSN unavailable, `.github/**` blocked), the checklist must say so plainly with the real blocker, not mark the line done anyway.
