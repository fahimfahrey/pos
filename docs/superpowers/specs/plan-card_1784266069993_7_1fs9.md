# Plan: domains/catalog — Product, ProductVariant, Category, Price Lists + /app/catalog UI

## Overview

Build the `catalog` domain (Product / ProductVariant / Category / PriceList with effective
dates) and the `/app/catalog` admin UI on top of the existing storage subsystem
(`src/infrastructure/storage/core`, adapters in `src/infrastructure/storage/adapters/{memory,indexeddb}`).

The existing `domains/catalog` module (`CatalogItem` + `Category`, see
`src/domains/catalog/entities/catalog-item.ts` and `src/domains/catalog/repository.ts`) predates
this card and conflates "product" and "sellable unit" into one flat record with no `orgId`. This
plan **replaces** that model with the card's split: a `Product` is the non-sellable grouping
(name, category, description, image); a `ProductVariant` is the sellable unit that carries
SKU/barcode/price-relevant flags (decimal-quantity for weighed items). `Category` gains `orgId`.
`PriceList` + `PriceListEntry` are new, date-effective price records resolved per variant.

This is a **breaking change** to the `catalogItems`/`categories` collections (schema v3 → v4).
There is no production data to migrate (pre-launch product), so the migration simply drops
`catalogItems` and introduces `products`, `productVariants`, `priceLists`, `priceListEntries`.

New cross-cutting pieces required by the card, currently absent from the codebase (confirmed via
repo-wide search — no `EventBus`, no `FileStore`, no CSV library, no barcode library):

1. **`EventBus` port** (`src/shared/ports/event-bus.ts`) + in-process adapter, for catalog
   mutation invalidation events, swappable for a push/queue implementation later.
2. **`FileStore` port** (`src/shared/ports/file-store.ts`) + IndexedDB-blob adapter today,
   object-storage adapter later — same registry/swap pattern as `StorageDriver`
   (see `docs/storage-adapters.md`).
3. **Hand-rolled CSV parser and EAN-13/Code-128 encoders** — no new runtime dependencies. The app
   is an offline-first PWA (`public/sw.js`, `NEXT_PUBLIC_STORAGE_ENGINE=indexeddb` default); both
   are small, well-specified, pure-function algorithms, and the card explicitly asks for unit
   tests on "CSV parser and schema," which reads as first-party logic, not a wrapped library.

**Repository note (skills):** this repo has no `.agents/skills/` directory (verified via glob —
zero `SKILL.md` files under the project root), so no project-defined skill applies to this task.
Conventions below are instead reverse-engineered from the existing `system-enums` and
`organization` domains, which are the most recently built (and most consistent) reference
implementations in the codebase. Per `AGENTS.md`, Next.js conventions used below (server actions,
`useActionState`, route-group layouts) are taken from the app's own working code under `src/app/`
rather than assumed from training data, since `node_modules/next/dist/docs/` in this install only
ships top-level index/glossary pages with no API-reference detail beyond what's already visible in
the repo's own usage.

## Architecture Decisions

**1. Product vs. ProductVariant split, replacing `CatalogItem`.** The card is explicit:
"Product, ProductVariant (sellable unit, barcode, decimal-quantity flag for weight items)". The
current `CatalogItem` flattens both into one record with no `orgId` — incompatible with every
other domain's tenant-scoping convention (see `tenant-scoping.ts` conformance suite, and
`orgId` on `products`/`customers`/`orders`/etc. in `COLLECTION_INDEXES`). Rather than bolt fields
onto `CatalogItem`, cut over cleanly: new entities/collections, drop `catalogItems` in the v4
migration. There's no production data (pre-launch), so a destructive schema migration is the
correct trade — not a compatibility shim.

**2. Naming collision with `domains/inventory`'s `Product`.** `src/domains/inventory/entities/
product.ts` already defines a `Product` (id, name, sku, price, stock) used by
`InventoryService`/`DexieInventoryRepository`. This plan adds a *different* `Product` under
`domains/catalog` (name/category/description grouping, no price/stock — pricing lives in
`PriceList`, stock lives in `inventory`). TypeScript scopes these fine (different modules,
imported under distinct paths), but this is a real conceptual overlap the codebase will carry
forward: two "Product" concepts, unreconciled. **Do not** attempt to merge/rename the inventory
domain in this card — out of scope, and `inventory`'s actions/services/tests are unrelated to
this ticket. Flag it explicitly in code review and leave a one-line comment atop
`domains/catalog/entities/product.ts` noting the distinction, so a future consolidation ticket has
a pointer. Fixture-factory naming (`makeCatalogProduct` vs. the existing `makeProduct`) is the one
place this collision would silently break tests if not handled — called out above.

**3. Barcode hot-path indexing: optional `getByIndex` on `DriverTransaction`.** The
`StorageDriver`/`DriverTransaction` contract (`docs/storage-adapters.md`) is deliberately minimal
— `get/getAll/put/delete`, no query pushdown, "queries are simple predicates evaluated
in-process; the driver need not optimize them." Reworking that contract for every adapter is out
of scope and would violate the "one adapter = one file + one config change" guarantee the last two
cards built. Instead: add `getByIndex` as an **optional** method. `IndexedDBDriver` implements it
via Dexie's native `.where(index).equals(key)` against the compound `[orgId+barcode]` index
already declared in `COLLECTION_INDEXES`. `MemoryStorageDriver` does not implement it — `Collection
.findByIndex` transparently falls back to `getAll()+find()`, which is what `CoreCatalogRepository
.findVariantByBarcode` already does today for `catalogItems`. This delivers real indexed lookups
on the adapter that matters for a barcode scanner's hot path (IndexedDB, the default engine) while
staying 100% backward compatible with the existing driver contract and every other adapter/domain
repository. `findVariantByBarcode(orgId, barcode)` is intent-named per the card's requirement and
is the only call site that needs to know about the index.

**4. FileStore is its own port/registry, not layered onto `StorageDriver`.** Blobs don't fit the
`{ id: string }`-typed-record contract that `RepositorySet`/`Collection<T>` assume (and Dexie
IndexedDB puts already handle Blobs natively without needing the generic `DriverTransaction`
abstraction). Building a parallel, smaller port (`FileStore`) with its own tiny registry mirrors
the existing swap pattern (interface + `registerEngine`-style function + default-provider) without
forcing every domain repository to reason about binary payloads. This satisfies "same
swappability rule" literally — same *pattern*, deliberately not the same *type*.

**5. Image lives on `Product`, not `ProductVariant`.** The card lists "image handling" once,
generically, and Product is the natural catalog-browsing unit (a shirt has one photo; its
size/color variants share it). Adding per-variant images would be scope creep not requested by the
card; if a future card needs it, `imageFileId` can be added to `ProductVariant` additively without
touching this plan's schema.

**6. CSV parser and barcode encoders are hand-rolled, not third-party dependencies.** No CSV or
barcode library exists in `package.json` today. Adding one is a one-line `package.json` change,
but (a) the acceptance criterion explicitly asks for "CSV parser and schema unit tests" — testing
first-party parsing logic, which argues for owning the parser; (b) EAN-13 and Code-128B are fully
specified, compact algorithms (checksum + fixed symbol tables), well within a few hundred lines
total, and keep the offline PWA bundle free of an external rendering library's DOM/canvas
assumptions; (c) it keeps every artifact a pure, synchronously-testable function
(`text -> bar widths`), matching the SVG-based rendering approach used for the printable label.

**7. Event bus is synchronous and in-process only; no persistence.** Matches the card's own
framing — "in-process event bus (interface, so a push/queue implementation can replace it)". No
attempt is made to survive process restarts or fan out across server instances; that's explicitly
the reason the interface exists. Actions publish *after* a successful `provider.withTransaction`
commit (never inside it — a rollback must not have already notified subscribers).

**8. Repository/service/action layering follows `system-enums`, not the legacy `container.ts`
pattern.** `src/infrastructure/container.ts` wires `DexieInventoryRepository`/
`DexieSalesRepository` directly (bespoke per-domain Dexie classes, engine-locked). The
`system-enums`/`organization`/`auth` domains instead use `createDefaultStorageProvider()` +
`provider.withTransaction(repos => ...)`, which is engine-agnostic (works against `memory` or
`indexeddb` via the same `CoreCatalogRepository`) and is the pattern every commit since "Tenancy
and settings" has used. Catalog actions use this newer pattern exclusively; `container.ts` is left
untouched.

### `CatalogRepository` — final method list

```ts
export interface CatalogRepository {
  // Categories
  saveCategory(category: Category): Promise<void>
  findCategoryById(id: string): Promise<Category | null>
  listCategoriesByOrg(orgId: string): Promise<Category[]>
  deleteCategory(id: string): Promise<void>

  // Products (the grouping, non-sellable)
  saveProduct(product: Product): Promise<void>
  findProductById(id: string): Promise<Product | null>
  listProductsByOrg(orgId: string): Promise<Product[]>
  searchProductsByName(orgId: string, query: string): Promise<Product[]>
  deleteProduct(id: string): Promise<void>

  // Product variants (the sellable unit — barcode hot path)
  saveVariant(variant: ProductVariant): Promise<void>
  findVariantById(id: string): Promise<ProductVariant | null>
  findVariantBySku(orgId: string, sku: string): Promise<ProductVariant | null>
  findVariantByBarcode(orgId: string, barcode: string): Promise<ProductVariant | null>
  listVariantsByProduct(productId: string): Promise<ProductVariant[]>
  listVariantsByOrg(orgId: string): Promise<ProductVariant[]>
  deleteVariant(id: string): Promise<void>

  // Price lists (effective-dated)
  savePriceList(priceList: PriceList): Promise<void>
  findPriceListById(id: string): Promise<PriceList | null>
  listPriceListsByOrg(orgId: string): Promise<PriceList[]>
  savePriceListEntry(entry: PriceListEntry): Promise<void>
  listPriceListEntries(priceListId: string): Promise<PriceListEntry[]>
  findEffectivePrice(orgId: string, variantId: string, atDate: Date): Promise<PriceListEntry | null>
}
```

`findEffectivePrice` resolves in `CatalogQueryService`, not the repository, by: list active
`PriceList`s for the org where `effectiveFrom <= atDate && (effectiveTo == null || atDate <
effectiveTo)`, sorted by `effectiveFrom` descending, then find the first with a matching
`PriceListEntry` for `variantId` (latest-wins on overlapping ranges — call this out to the
implementer as the tie-break rule, since the card doesn't specify one).

## State Management Strategy

- **Server is the source of truth.** All catalog data is fetched in server components
  (`page.tsx` files) via server actions calling `createDefaultStorageProvider()` directly —
  no client-side data-fetching library, matching every other page in `src/app/`.
- **URL search params drive table state** (`q`, `categoryId`, `page`) so pagination/search/filter
  are shareable/bookmarkable and require no client state store. `search-box.tsx` and
  `category-filter.tsx` are the only client components on the list page; both just push a new URL
  via `useRouter().replace(...)` on change (debounced for the search box).
- **Forms use `useActionState`** (React 19, already the app's convention nowhere else explicitly
  shown in the files read, but is the only server-action-bound form hook compatible with Next 16 +
  React 19.2 in this repo's `package.json`; the existing `create-enum-value-form` referenced from
  `admin/enum-values/page.tsx` is presumed to already use this pattern — confirm by reading that
  component before copying its shape, since it wasn't in the files inspected for this plan) for
  optimistic pending/error state, with **client-side `safeParse` against the same
  `catalog.schema.ts`** for instant field-level feedback before submission, and the server action
  re-parsing the same schema as the authority (never trust the client parse for persistence).
- **CSV import is two-phase, no client state beyond the current step.** Step 1 (`preview`) posts
  the file to a server action that returns a plain serializable array of `{ row, status, errors,
  parsed }` — held in the page via `useActionState`'s returned state, not a separate client
  store. Step 2 (`commit`) re-submits the same parsed+validated payload (not re-uploading the raw
  file) to avoid re-parsing drift between preview and commit.
- **`catalogEventBus`** is server-side only (module singleton in a Node process) and is not wired
  to the browser in this card — it exists so a future subscriber (e.g., a barcode-scanner
  client-cache invalidation channel, or a background sync worker) can attach without changing any
  action's write path. `revalidatePath` remains the mechanism that actually refreshes the Next.js
  page cache for this card's UI.

## Accessibility Considerations

- **Table.** Use a semantic `<table>` with `<caption className="sr-only">`, `<th scope="col">` for
  column headers (Name, SKU, Barcode, Category, Active); row-selection checkboxes get a visible
  `<label>` (`sr-only` text: "Select {product name}") — never an icon-only checkbox with no
  accessible name. Pagination controls are real `<button>`/`<a>` elements with `aria-label="Next
  page"`/`aria-current="page"` on the active page indicator, not `<div onClick>`.
- **Forms.** Every input has a associated `<label htmlFor>`; validation errors render in a
  `role="alert"` region adjacent to the field and are also referenced via `aria-describedby` on
  the input; the submit button reflects pending state via `aria-busy` while `useActionState`'s
  `isPending` is true, not just a visual spinner.
- **CSV import preview table.** Per-row error state must not be color-only — pair the red/error
  styling with a text badge ("Error", "New", "Update") and an `aria-live="polite"` region
  announcing the summary count ("3 rows will be created, 1 row has errors") when preview results
  load, since this is an async update the user may not be looking at.
- **Category filter / search box.** Native `<select>` for the category filter (not a custom
  listbox) — gets full keyboard/screen-reader support for free. Search input has a visible
  `<label>` (can be `sr-only` if a placeholder-only design is required, but the label must exist
  in the DOM).
- **Barcode labels.** The rendered SVG barcode is decorative once paired with the visible
  human-readable text underneath it (a `<text>` element in the SVG, already required by both
  EAN-13 and Code-128 print conventions) — mark the SVG `aria-hidden="true"` and let the adjacent
  text carry the accessible content, rather than trying to give the bar pattern itself an
  `aria-label`.
- **Print view (`/app/catalog/labels`).** Verify with the browser's print preview (or Playwright's
  `page.emulateMedia({ media: 'print' })`) that `@media print` rules hide the app chrome
  (nav/header) and that label content reflows onto a reasonable label-sheet grid — this is a
  functional, not purely visual, requirement since it's the deliverable ("printable").

## Risks and Edge Cases

- **Schema v3→v4 is destructive for `catalogItems`.** Confirmed no production data exists yet, but
  double-check with the team before merging if any seeded/demo environment relies on the old
  collection — the migration silently drops it with no data-carryover step (per Architecture
  Decision 1).
- **Two `Product` types in the codebase** (`domains/catalog/entities/product.ts` vs.
  `domains/inventory/entities/product.ts`) is a standing source of confusion for future
  contributors and for anyone grepping `Product` — mitigated by the doc-comment called out in
  Architecture Decision 2, not eliminated. A follow-up ticket should decide whether `inventory`
  should reference `catalog.ProductVariant` by id instead of duplicating `name`/`sku`.
  This plan does **not** wire `inventory` to `catalog` — stock tracking stays keyed on its own
  `Product.sku` string, unrelated to the new `ProductVariant.sku`. If the two SKUs drift, nothing
  in this plan detects it; that reconciliation is explicitly out of scope.
- **Barcode uniqueness is enforced per-org, not globally.** Two different organizations (tenants)
  can register the same physical barcode independently (expected — they're different stores'
  catalogs). Within one org, `saveVariant` must reject a duplicate barcode (`ConflictError`) —
  make sure `catalog-service.ts` checks `findVariantByBarcode` before `saveVariant`, since the
  repository layer itself doesn't enforce uniqueness (the driver contract has no unique-constraint
  primitive).
  Barcode can also be left blank (`barcode?: string`) — `findVariantByBarcode('')`/`undefined`
  lookups must not match every blank-barcode variant; guard against empty-string index lookups
  explicitly in `CoreCatalogRepository`.
- **CSV upsert-by-SKU + unknown category.** Decide (and encode in `csv.test.ts`) whether an
  unrecognized `categoryName` column value auto-creates the category or is a per-row validation
  error. Auto-create is more convenient but can silently produce duplicate near-identical
  categories from typos; recommend **per-row error** ("unknown category — create it first") as the
  safer default, with the category dropdown/CSV template listing valid names to reduce friction.
- **Decimal-quantity variants and barcode label price.** Weight-item barcodes sometimes encode a
  price/weight in the barcode itself (retail weight-embedded EAN-13, prefix 2). This card only
  asks for standard EAN-13/Code-128 rendering of the variant's own barcode value — do **not**
  build weight-embedded barcode generation; flag it as a likely follow-up if the business needs
  scale-printed labels, but it's not in scope here.
- **`getByIndex` compound-key ordering.** Dexie's `[orgId+barcode]` index requires the query key
  array `[orgId, barcode]` in the exact declared field order — a swapped-order bug here fails
  silently (returns `undefined`, falls through to "not found" rather than throwing), so the
  conformance suite's hit-case test is the real safety net; make sure it's not skipped.
- **FileStore SSR safety.** Product image upload/display must never attempt `indexedDB` access
  during server rendering — `image-upload.tsx` must be a client component, and any server-rendered
  `<img>` referencing a stored file needs a URL-able access path (e.g., a route handler streaming
  the blob) since a raw client-side `IDBDatabase` blob can't be embedded directly in server-rendered
  HTML. Plan a `src/app/api/catalog/files/[fileId]/route.ts` route handler that reads via
  `createDefaultFileStore()` and streams the blob with the correct `Content-Type` — add this file
  to the Files list above if the image feature is implemented (noting it here since it wasn't
  caught until this risk pass).
- **`useActionState` CSV payload size.** Server actions have a body-size ceiling; a large CSV
  (thousands of rows) posted as `FormData` should be checked against Next's configured action body
  limit — if the default is too small for realistic catalog sizes (hundreds to low thousands of
  SKUs per `docs/storage-adapters.md`'s own stated runtime assumption), document the limit and/or
  chunk the preview parse client-side before sending only the parsed+validated rows (smaller
  payload than raw CSV text) to the server action.
