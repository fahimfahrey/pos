# Plan: Receipts — template rendering, print/PDF/ESC-POS dispatch, customer display

Card: `card_1784266069993_11_d5jq`

## Overview

The sales domain finalizes sales (`FinalizeSaleService`) and produces `Sale` + `SaleItem[]` + `Payment[]` rows, but nothing renders a receipt from them, and no printer abstraction exists. This card adds a **receipt domain** (`src/domains/receipts/`) that assembles a settings-driven `ReceiptDocument` view model from a finalized sale, plus an **infra-level Printer port** (`src/infrastructure/printing/`) that mirrors the two swappable-adapter subsystems already in the repo: `StorageDriver` (`docs/storage-adapters.md`) and `PaymentGateway` (`docs/payments.md`). Both follow the same shape — a minimal port interface, adapters registered into a module-level `Map` via a `register*()` call, and a side-effect-import `index.ts` that wires adapters in. This card repeats that shape for `Printer`.

Three concrete printer adapters ship: `browser-print` (default, always available, opens the OS print dialog against 80mm thermal CSS), `pdf` (renders the same document to a downloadable PDF), and `escpos` (WebUSB/WebSerial dispatch of raw ESC/POS byte sequences to real thermal hardware, feature-detected and gracefully absent when the browser/OS doesn't support it). Printing is dispatched through an in-process `JobRunner` port (`src/shared/ports/job-runner.ts`) — a `run(job): Promise<JobResult>` contract with exactly one implementation today (`InProcessJobRunner`, fire-and-await in the same tab) so a real queue/worker (Web Worker, SharedWorker, or eventually a server job engine) can replace it later without callers changing. Checkout (`FinalizeSaleService` / the eventual `finalize-sale.ts` action) never awaits a printer: it publishes a `sale.finalized` event on `salesEventBus` (a new bus, same pattern as `paymentsEventBus`/`catalog-event-bus`), and a receipt-print subscriber (wired at the app layer, e.g. in the checkout screen) fires the print job asynchronously. A printer failure is caught, logged, and surfaced as a toast/banner — it can never roll back or block the sale.

A second, independent feature — the customer-facing display (`/pos/display`) — mirrors the *live cart* (not the finalized sale) from the cashier's register to a second screen/browser tab via `BroadcastChannel`, keyed by `registerId` so multiple registers in the same browser profile don't cross-talk. This does not touch the storage layer at all; it's a pure browser-tab-to-tab pub/sub built on a form of the same `EventBus<E>` port already defined in `src/shared/ports/event-bus.ts`, backed by a `BroadcastChannel` instead of the in-process `InProcessEventBus`.

Settings that drive the receipt template (org logo, branch address, tax registration number, header/footer text, tax-breakdown visibility) require two additive fields not yet on any entity — `Organization.logoUrl` (or branch-level override) and `Branch.taxRegistrationNumber` — plus reuse of the existing `ReceiptTemplate` on `ResolvedSettings` (`headerText`, `footerText`, `showItemDescription`, `showTaxBreakdown`).

### Shared ports
- `src/shared/ports/job-runner.ts` — **new port**:
  ```ts
  export interface Job<TPayload = unknown, TResult = unknown> {
    id: string
    type: string
    payload: TPayload
  }
  export interface JobResult<TResult = unknown> {
    ok: boolean
    result?: TResult
    error?: string
  }
  export interface JobRunner {
    run<TPayload, TResult>(job: Job<TPayload, TResult>, handler: (payload: TPayload) => Promise<TResult>): Promise<JobResult<TResult>>
  }
  ```
  `run()` takes the handler as an explicit argument (rather than a job-type→handler registry) to keep this the same "minimal contract, no registry indirection" shape used by `FileStore`/`EventBus` — the caller already knows which handler it wants; only the execution strategy (now vs. queued vs. worker) is abstracted.
- `src/shared/ports/printer.ts` — **new port**:
  ```ts
  export interface PrintJobPayload {
    document: ReceiptDocument // from @domains/receipts/entities/receipt-document — allowed: shared may not import domains, so this type is duplicated/narrowed as a structural shape or the payload is typed `unknown` at the port level and cast by adapters. See Architecture Decisions #4.
  }
  export interface PrinterCapabilities {
    supportsBrowserPrint: boolean
    supportsPdf: boolean
    supportsEscPos: boolean
  }
  export interface Printer {
    readonly id: string
    isAvailable(): Promise<boolean>
    print(payload: PrintJobPayload): Promise<{ jobId: string }>
  }
  ```

Every attempt goes through `jobRunner.run()`.

## Architecture Decisions

1. **Printer port mirrors `PaymentGateway`/`StorageDriver`, not a new pattern.** Same `register()`/`resolve()` map, same side-effect `index.ts`, same per-adapter folder. Consistency with `docs/payments.md` and `docs/storage-adapters.md` means one mental model for "how do I add a hardware/output integration" across the whole repo.

2. **`JobRunner` is a real port with exactly one production implementation (`InProcessJobRunner`) today, on purpose.** The card asks for "an in-process JobRunner interface (so a real queue/worker can replace it when a server engine lands)." That means the *port* must be shaped so a future `WorkerJobRunner` or `ServerQueueJobRunner` can drop in — hence `run(job, handler)` returning a `Promise<JobResult>` rather than the caller directly `await`ing `printer.print()`. Every print call goes through `jobRunner.run()`, never calls `printer.print()` directly, even though today it's the same call stack synchronously. This is the one piece of this plan that is deliberately an abstraction for a not-yet-built future — justified because the card explicitly asks for it by name, unlike unrequested speculative abstractions elsewhere in the codebase's stated conventions.

3. **Checkout never blocks on printing — event-driven, not inline.** `FinalizeSaleService.finalize()` gains **no** printer/job-runner dependency (services stay I/O-port-only, no UI/hardware concerns — consistent with `boundaries/element-types`: `services` may only import `services, repositories, entities, shared, constants`, and `Printer`/`JobRunner` adapters live in `infrastructure`, which `services` cannot import). Instead: the print trigger lives at the **app/action layer**, one level up, subscribing to `salesEventBus`'s `sale.finalized` event and calling the orchestrator fire-and-forget (`.catch()`'d, never `await`'d in the checkout-completion path). This satisfies "checkout never blocks on printing" structurally, not just by convention — there is no code path from `finalize()` returning to a print call being awaited.

4. **`Printer`/`JobRunner` ports live in `src/shared/ports/` but must not import `@domains/receipts`.** `boundaries/element-types` disallows `shared → entities`/`shared → domains-anything` (`shared` may only import `shared, constants`). `ReceiptDocument` is a domain type. Resolution: `PrintJobPayload.document` is typed as a generic/structural shape at the port level (`unknown`, or a duplicated minimal structural interface defined in `shared/ports/printer.ts` itself — not imported from `receipts`), and the **infrastructure** adapters (which *are* allowed to import `entities`/`services` per the `infrastructure → services, repositories, entities, ...` rule) do the real typing and casting. This mirrors how `PaymentGatewayContext.repos: RepositorySet` — a repository-set type — is fine to reference from `domains/payments/ports/payment-gateway.ts` today (ports already sit in `domains/*/ports/`, which is `services`-tier and allowed to import `entities`); this new `Printer`/`JobRunner` port sits in `shared/ports/` specifically because it's cross-domain infrastructure (like `FileStore`, `EventBus`), so it takes the stricter boundary. Document this distinction explicitly in a code comment at the top of `printer.ts` since it's a non-obvious asymmetry with `payment-gateway.ts`'s port location.

5. **PDF adapter uses the browser's native print-to-PDF path, not a new dependency.** No PDF-generation library exists in `package.json` today (`dependencies` has no `pdf-lib`/`jspdf`/etc.). Adding one is out of scope unless required; the `pdf` adapter instead opens the same thermal-styled HTML in a hidden iframe/new window and calls `window.print()` with a CSS media query hint — modern browsers' native print dialog offers "Save as PDF" as a destination, which satisfies "PDF generation" without a new dependency or server-side rendering step. If true programmatic (non-interactive) PDF generation is later required (e.g., emailing receipts from a server), that's a separate card that would add a real PDF library server-side — noted here so the acceptance bar is "downloadable/printable PDF via the browser," not "headless PDF bytes."

6. **`CartSnapshot` is a new, display-only entity — not a reuse of `Sale`/`SaleItem`.** The cart doesn't exist as persisted rows until finalize; broadcasting the in-memory cart state requires its own lightweight shape. Keeping it in `src/domains/sales/entities/` (not `receipts/`) because it's conceptually "the sale before it's a Sale," and the (future) cart context that produces it lives under `sales` UI/state, not receipts.

7. **Display channel keyed by `registerId`, not `orgId`/`branchId`.** A branch can have multiple registers, each potentially paired with its own second screen; scoping broadcast channel names to `pos-cart-${registerId}` prevents cross-register bleed within the same browser profile (e.g., a manager with two POS tabs open for two registers).

8. **No new collections/schema version bump.** `logoUrl` and `taxRegistrationNumber` are additive optional fields on already-persisted entities (`Organization`, `Branch`) — Dexie/IndexedDB here stores whole objects per `COLLECTION_INDEXES`-declared indexes only; adding an unindexed optional field requires no migration block, matching how `Category.taxRuleId` was added in the prior card without a schema version bump (per `IMPLEMENTATION_PROGRESS.md`).

## State Management Strategy

- **`ReceiptDocument`** (`src/domains/receipts/entities/receipt-document.ts`) is a fully-resolved, immutable view model — no partial/loading states inside it. Shape:
  ```ts
  export interface ReceiptTaxBreakdownLine { rate: number; taxableAmount: number; taxAmount: number }
  export interface ReceiptLineItem { name: string; sku: string; quantity: number; unitPrice: number; discount: number; taxRate: number; total: number }
  export interface ReceiptPaymentLine { method: string; amount: number; tendered?: number; changeDue?: number }
  export interface ReceiptDocument {
    receiptNumber: string
    issuedAt: Date
    org: { name: string; logoUrl?: string }
    branch: { name: string; address?: string; taxRegistrationNumber?: string }
    lines: ReceiptLineItem[]
    subtotal: number
    discount: number
    taxMode: TaxMode
    taxBreakdown: ReceiptTaxBreakdownLine[]
    tax: number
    total: number
    payments: ReceiptPaymentLine[]
    changeDue: number
    footerText?: string
    showItemDescription: boolean
  }
  ```
  Built once, server-side (or at print-trigger time client-side from already-fetched data) and passed down as props — no client-side derivation/refetching inside `ReceiptView`.

- **Print job state** lives entirely inside `receipt-print-orchestrator.ts`'s call — it's a fire-once async function, not a store. UI-visible state is limited to a local `'idle' | 'printing' | 'succeeded' | 'failed'` `useState` in `receipt-actions.tsx`, reset per button click. No global print-queue UI in this card (single job at a time, matching "checkout never blocks" — there's no queue to visualize yet since `JobRunner` has one synchronous implementation).

- **Cart snapshot broadcast state** is the one genuinely event-driven piece: the display page holds `useState<CartSnapshot | null>(null)`, updated purely by the `subscribe()` callback from `useCartBroadcast`. No polling, no fetch — the display page has zero knowledge of the storage layer (it never touches a `SalesRepository`), only the broadcast channel. On mount with no message received yet, render an idle/"no active sale" state (handles the case where the display tab opens after the cashier's last broadcast, or before the cashier's cart-screen has published anything yet — since that screen doesn't exist yet in this codebase, the display route's idle state is also its *only* reachable state until the cart screen is built and wired to `useCartBroadcast(...).publish(...)`).
- `useCartBroadcast(registerId)` internally memoizes one `BrowserBroadcastChannel` instance per `registerId` for the component's lifetime, and calls `.close()` in a `useEffect` cleanup — channels must not leak across route navigations.

## Accessibility Considerations

- **Receipt preview/print route (`/pos/receipt/[saleId]`)**: semantic table markup for line items (`<table>` with `<caption>`/`<th scope="col">`) rather than div-grids, so screen readers can navigate rows/columns; monetary values include currency in accessible text (not just symbol-only) via `aria-label` where the visual format is symbol-only (e.g. `$12.00` visually, `aria-label="12 dollars 00 cents"` or simpler `"$12.00"` read literally — prefer just correct semantic markup over over-engineered ARIA since currency symbols are generally read correctly by modern screen readers; avoid unnecessary `aria-label` duplication per repo's "don't add validation/handling for scenarios that can't happen" ethos — only add explicit `aria-label`s where a screen reader would plausibly mis-read, e.g. avoid on plain `$12.00` text).
- **Print/Download buttons** (`receipt-actions.tsx`): real `<button>` elements (not `<div onClick>`), visible focus rings (Tailwind `focus-visible:` utilities, consistent with existing `_components` buttons like `logout-button.tsx` — check its focus-style convention and match it), `aria-live="polite"` region for the idle/printing/succeeded/failed status text so screen-reader users get non-visual feedback when a print job completes or fails without needing to re-focus the button.
- **Customer display (`/pos/display`)**: large-font by design (it's meant to be readable from a customer's viewing distance) — use relative units (`rem`) scaled up, not fixed small px; sufficient color contrast (test against WCAG AA 4.5:1, especially since this may render on cheap secondary monitors with worse color reproduction); the idle/"no active sale" state must be announced via `aria-live="polite"` when transitioning from a populated cart back to idle (sale just completed) so any assistive tech tracking the tab (unlikely primary use case, but the region should not be silently swapped) doesn't need to poll.
- **80mm thermal print CSS**: irrelevant to on-screen accessibility (it's a physical print medium), but must not be the *only* stylesheet — ensure `ReceiptView` renders sensibly on-screen (the preview page) independent of the `@media print` thermal rules, i.e. thermal-specific CSS (`width: 80mm`, monospace-only) must be scoped under `@media print` and not leak into the normal on-screen preview's readability/contrast.

## Risks and Edge Cases

- **WebUSB/WebSerial are Chromium-only** (no Safari/Firefox support as of current browser landscape) — `escpos` printer must never be assumed available; every call path already routes through `isAvailable()` + orchestrator fallback, but document this prominently in `docs/hardware.md` so cashiers on non-Chromium browsers aren't confused when "Connect Printer" doesn't appear.
- **WebUSB/WebSerial require a user gesture** to call `requestDevice()`/`requestPort()` (browser security requirement) — this means the *first* connection to a printer cannot be fully automatic; a "Connect Printer" button triggered from a settings screen (not built in this card, but the port/transport must support being invoked from one later) is required. Once granted, `getDevices()`/`getPorts()` can silently re-list previously-authorized devices on later page loads without a new gesture — the `escpos` adapter should attempt silent re-connection via `getDevices()` first and only prompt if empty.
- **Print job "success" for `browser-print` is not real completion.** `window.print()` returns as soon as the dialog opens, not when the user actually finishes/cancels printing. The orchestrator's `{ ok: true }` for browser-print means "dialog was invoked," not "paper came out." This is an inherent limitation of the Web platform, not a bug — document it in `docs/hardware.md` and do not claim stronger guarantees in UI copy (e.g. say "Print dialog opened" not "Printed successfully").
- **jsdom's `BroadcastChannel` support** — vitest's `jsdom` environment (`vitest.config.ts`) may or may not polyfill `BroadcastChannel` fully depending on jsdom version (`jsdom: ^23.2.0`, listed in `package.json`); verify at implementation time and fall back per Test Plan item 5 if needed.
- **Multiple sales domain event buses proliferating.** This is the second cross-domain `EventBus` instance added after `payments-event-bus.ts`/`catalog-event-bus.ts` — if a third or fourth domain needs one, consider whether a shared generic bus factory belongs in `src/infrastructure/events/` rather than one hand-copied file per domain; out of scope for this card (follow existing convention), but flag it as a candidate for the next refactor.
- **`FinalizeSaleService` currently hardcodes `orgId: 'unknown'`, `branchId: 'unknown'`, `createdBy: 'unknown'`** (visible in the current `finalize-sale-service.ts` source, marked "Caller should provide") — this plan's `sale.finalized` event payload needs real `orgId`/`branchId` to look up `Organization`/`Branch` for the receipt. Since the action-layer wrapper (`finalize-sale.ts`) that would set these fields correctly does not exist yet, this card's event-publish step is **specified but not wired into a real caller** — implementers should publish `sale.finalized` from wherever `FinalizeSaleService.finalize()` is actually invoked (currently only test code and, later, the real action) and must ensure `sale.orgId`/`sale.branchId` are populated by that point (a pre-existing gap in `finalize-sale-service.ts`, not introduced by this card, but this card is blocked on real values to build a correct receipt — call this out rather than silently building the receipt against placeholder `'unknown'` org/branch ids).
- **Idempotent re-print.** A cashier might click "Print" twice, or the display might replay the last broadcast on reconnect. `receipt-print-orchestrator.ts` has no dedup logic in this card (each click is a new job); acceptable since printing is a physical, cashier-visible, idempotent-in-practice action (reprinting a receipt is a normal, expected POS operation, unlike double-charging a payment) — explicitly not treated as a bug.
- **`ReceiptTemplate.showTaxBreakdown === false`**: `receipt-builder.ts` must still *compute* `taxBreakdown` (cheap, pure) but `ReceiptView` conditionally omits rendering it — keeps the builder simple/total (one shape always) and pushes the display decision to the view layer, consistent with settings like `showItemDescription` already being a render-time toggle rather than a data-shaping one elsewhere in the codebase.
- **Org-level vs. branch-level `logoUrl`/`taxRegistrationNumber` precedence**: spec puts `logoUrl` on `Organization` only (a chain typically has one logo) and `taxRegistrationNumber` on `Branch` only (tax registration is often per-branch/per-jurisdiction, especially for multi-country chains) — not both overridable at both levels, to avoid over-engineering a resolution chain (`settings-resolver.ts`'s org→branch merge pattern is for *settings*, not core entity identity fields; logo/tax-reg-number are identity fields, not settings, so they don't need `resolveSettings`-style merging).
