# Plan: domains/inventory — StockMovement ledger, StockLevel, reconciliation, adjustment/transfer/low-stock/stocktake UI

## Overview

Build the real `inventory` domain around an **append-only `StockMovement` ledger** and a
**materialized `StockLevel`** (per-org, per-branch, per-variant quantity), written together in one
`UnitOfWork` transaction so they can never diverge silently. Add a `ReconciliationService` that
recomputes `StockLevel` from the ledger and reports drift. Add UI for manual adjustment (mandatory
reason), inter-branch transfer, a low-stock view driven by reorder thresholds, and stocktake mode
(count session → variance report → approval that creates adjustment movements). Enforce
non-negative stock in the service layer, gated by an `allowOversell` toggle resolved from
org/branch settings. The concurrent-decrement conformance case is the acceptance gate: two
simultaneous last-unit sales must not both succeed when oversell is off.

**Key finding that reshapes this plan:** the repo already has a `src/domains/inventory` *stub*
(`Product` with an embedded `stock: { quantity, unit }`, a flat `StockMovement { productId,
movementType: 'in'|'out'|'adjustment' }`, `InventoryRepository.save/findById/listAll/findBySku/
recordMovement/listMovements`), wired through a **legacy, non-transactional path**
(`container.ts` → `DexieInventoryRepository` → raw `db.products`/`db.stockMovements`, used only by
`src/domains/inventory/actions/list-products.ts`). This legacy `Product`/`InventoryRepository` CRUD
shape is also reused, purely as a generic test fixture, by five **unrelated** conformance suites
(`crud.ts`, `ulid-keys.ts`, `tenant-scoping.ts`, `unit-of-work.ts`, `export-import.ts`) that have
nothing to do with real inventory business rules — they just needed *some* simple entity to drive
generic storage-engine mechanics (ULID keys, tenant scoping, rollback, export/import) and inventory
was the first stub scaffolded. Rewriting `InventoryRepository`'s CRUD surface would silently break
all five of those files for no product reason.

**Decision:** leave `Product`, and the CRUD methods `save/findById/listAll/findBySku`, untouched —
they keep serving those five generic suites. Real inventory business logic (this card) lives in
**new** entities/collections that are additive: `StockLevel` (new file, new collection) and a
**rewritten** `StockMovement` (ledger shape: `orgId`, `branchId`, `variantId` instead of
`productId`, expanded `movementType`, transfer/stocktake reference fields). Because `StockMovement`'s
shape changes, the one place that already exercises it as a generic fixture
(`crud.ts`'s "movements" test, `fixtures.ts`'s `makeStockMovement`) gets updated in the same
mechanical pass. `recordMovement`/`listMovements(productId)` on `InventoryRepository` are removed and
replaced by ledger-shaped methods (`appendMovement`, `listMovementsForVariant`,
`listMovementsForBranch`) — the only conformance file affected by this rename is `crud.ts`.

The legacy `container.inventoryService` / `DexieInventoryRepository` / `list-products.ts` path is
**not used** by the new domain logic — all new services/actions use
`createDefaultStorageProvider().withTransaction(...)`, the same pattern as `sales/actions/
complete-sale.ts` and every `catalog` action, because that is the only path that gives us one
driver transaction across `stockLevels` + `stockMovements` (the atomicity the card requires). The
legacy path is left in place untouched (out of scope — no regression, nothing deleted that other
code depends on).

**Second key finding — the actual concurrency bug this card must close:** `MemoryStorageDriver.
transaction()` (`src/infrastructure/storage/adapters/memory/memory-driver.ts`) clones the touched
collections into a **snapshot at call time**, runs the work, and swaps the snapshot back into the
live store **only after the callback resolves** — with no lock around any of it. Two overlapping
`withTransaction()` calls (e.g. two concurrent `Promise.all`-driven sales) both synchronously clone
the *same* pre-mutation state before either has committed (because the clone happens before the
first `await` inside the callback, and JS runs each call's synchronous prefix before yielding), then
each independently swaps its own copy back — last write wins, silently discarding the other's
decrement. This is exactly why `conformance/suites/concurrency.ts` currently has a
`capability-gated` branch (`adapter.capabilities.serializableTransactions`) that lets non-serializable
adapters get away with only "quantity ≥ 0", instead of the exact count the card demands. The
IndexedDB adapter does **not** have this bug — `IndexedDBDriver.transaction()` delegates directly to
Dexie/native IndexedDB, whose engine natively queues overlapping readwrite transactions (and
`StorageProvider.withTransaction` always scopes every transaction over *all* collections, so any two
concurrent transactions always overlap and always queue). So "IndexedDB: single-writer transactions"
from the card is already true and needs no code change; the memory adapter needs a real fix so it
match that guarantee (a future SQL adapter would get it via row locks, per the card). This plan adds
a write-queue mutex to `MemoryStorageDriver` and then removes the capability gate, since after the
fix both current adapters genuinely guarantee it.

- **Rewrite** `src/domains/inventory/services/inventory-service.ts` — drop the old
  `createProduct/findProduct/listProducts` (belonged to the stub `Product`/embedded-stock model,
  now superseded); implement:
  - `private async applyMovement(repos, input: { orgId, branchId, variantId, quantity, movementType, reason?, reference?, relatedMovementId?, createdBy, allowOversell }): Promise<{ level: StockLevel; movement: StockMovement }>` —
    reads `findStockLevel` (defaults to `{ quantity: 0 }` if none), computes
    `newQuantity = current + input.quantity`, throws `InsufficientStockError` if
    `newQuantity < 0 && !allowOversell`, else writes `saveStockLevel` and `appendMovement` **in that
    order, within the same transaction** (both calls happen inside the one `repos` passed by the
    caller's `withTransaction`, so they can't diverge — see Architecture Decisions).
  - `receivePurchase(repos, { orgId, branchId, variantId, quantity, reference?, createdBy, allowOversell })` — `applyMovement` with `movementType: 'purchase'`, `quantity` positive.
  - `recordSale(repos, { orgId, branchId, variantId, quantity, reference, createdBy, allowOversell })` — `applyMovement` with `movementType: 'sale'`, `quantity: -quantity`.
  - `recordReturn(repos, { orgId, branchId, variantId, quantity, reference?, createdBy, allowOversell })` — `movementType: 'return'`, `quantity` positive.
  - `adjustStock(repos, { orgId, branchId, variantId, quantity, reason, createdBy, allowOversell })` — `movementType: 'adjustment'`; throws `ValidationError` if `!reason?.trim()` (belt-and-suspenders; the zod schema at the action boundary already requires it).
  - `transferStock(repos, { orgId, variantId, fromBranchId, toBranchId, quantity, createdBy, allowOversellAtSource })` — validates `fromBranchId !== toBranchId` and `quantity > 0`; generates one `transferId` (via injected `IdGenerator`) as the shared `reference`; calls `applyMovement` twice in sequence *within the same transaction*: out-leg at `fromBranchId` (`quantity: -quantity`, `movementType: 'transfer'`, oversell-gated), in-leg at `toBranchId` (`quantity: +quantity`, `movementType: 'transfer'`, never oversell-gated since it's an increase); sets each movement's `relatedMovementId` to the other's `id` (ids are generated up front so both can reference each other) and returns both.
  - `listLowStock(repos, branchId)` → `repos.listLowStock(branchId)`.
  - Stocktake: `startStocktakeSession(repos, { orgId, branchId, createdBy })`; `recordCount(repos, { sessionId, variantId, countedQuantity, countedBy })` — session must be `open`; reads current `StockLevel` for `expectedQuantityAtCount` snapshot; upserts a `StocktakeCount` (one per `sessionId`+`variantId`, so a recount overwrites); `submitStocktakeSession(repos, sessionId)` — session `open → submitted`; `getVarianceReport(repos, sessionId)` — joins counts with **live** `StockLevel` (not the count-time snapshot) to compute `currentQuantity`, `countedQuantity`, `variance = countedQuantity - currentQuantity`, plus `expectedQuantityAtCount` for display context (see Edge Cases for why live, not snapshot, drives the number that matters); `approveStocktakeSession(repos, { sessionId, approvedBy, allowOversell })` — session must be `submitted`; for each count with non-zero live variance, calls `applyMovement` with `movementType: 'stocktake'`, `quantity: variance`, `reference: sessionId`, `reason: 'Stocktake session <id> approval'`; sets session to `approved`, stamps `approvedBy`/`approvedAt`. Because the target is the counted physical quantity (always ≥ 0), the oversell check inside `applyMovement` can only ever reject a stocktake correction if `allowOversell` is off **and** `countedQuantity` itself would be negative, which can't happen — no special-casing needed, it falls out of the shared `applyMovement` path.
  - Constructor keeps `(clock: Clock, ids: IdGenerator)` (repo is now passed per-call as `repos` from the caller's transaction, not injected once, since every method must run inside the specific `UnitOfWork`'s repository set — this is a deliberate deviation from `SalesService`'s constructor-injected-repo pattern; call out explicitly in Architecture Decisions since it's the one place this domain's service shape differs from its siblings).

- **Fix** `src/infrastructure/storage/adapters/memory/memory-driver.ts` — add a write-queue mutex so
  `readwrite` transactions are strictly serialized (root-caused above):
  ```ts
  export class MemoryStorageDriver implements StorageDriver {
    // ...
    private writeQueue: Promise<void> = Promise.resolve()

    async transaction<T>(collections, mode, work): Promise<T> {
      if (mode !== 'readwrite') {
        return this.runTransaction(collections, mode, work) // readonly: no queueing needed, see rationale below
      }
      const previous = this.writeQueue
      let release!: () => void
      this.writeQueue = new Promise((r) => { release = r })
      try {
        await previous
        return await this.runTransaction(collections, mode, work)
      } finally {
        release()
      }
    }
    // runTransaction = today's clone/work/swap-back body, unchanged
  }
  ```
  Readonly transactions don't join the queue: they clone from `this.store` and never write back, and
  because a readwrite commit is a single synchronous swap at the very end of `runTransaction`, any
  readonly clone happens either fully-before or fully-after a given commit — never mid-way — so
  interleaving readonly reads with the write queue is safe and doesn't need serializing.
- **Modify** `src/infrastructure/storage/core/schema-versioner.ts` (`schemaSteps`) — **pre-existing
  gap found during investigation**: `CURRENT_SCHEMA_VERSION` is `4` (bumped by the catalog card) but
  `schemaSteps` only goes up to version `3`, and `SchemaVersioner.migrate()` targets
  `steps[steps.length - 1].version`, not the `CURRENT_SCHEMA_VERSION` constant — so the generic
  (non-Dexie) migration path currently silently stops at v3. Add the missing step 4 (catalog:

## Architecture Decisions

1. **Signed-delta ledger.** `StockMovement.quantity` is a signed integer delta (+in, -out), not an
   unsigned magnitude + implied direction from `movementType`. One field, one meaning, applied
   uniformly by `applyMovement` as `newQuantity = current + delta` regardless of movement type. This
   makes the non-negative check, the reconciliation sum, and the stocktake "target the counted
   value" logic all the same arithmetic — no per-type branching for "does this type add or
   subtract."
2. **`StockLevel` is not embedded on `Product`/`ProductVariant`.** It's its own collection keyed by
   `(branchId, variantId)` via a compound index, mirroring the `catalogProductVariants`
   `[orgId+sku]` pattern already established. This is what makes "per-branch, per-variant" the unit
   of materialization the card asks for, independent of however many branches carry a given variant.
3. **Same-transaction write, not two-phase.** `applyMovement` calls `saveStockLevel` then
   `appendMovement` using the *same* `repos` object the caller received from one
   `provider.withTransaction(...)` call. Because `withTransaction` opens exactly one driver
   transaction scoped over every collection (see `storage-provider.ts`), both writes commit or roll
   back together — there is no window where one exists without the other. This is the literal
   mechanism behind "written in the same UnitOfWork so they can never diverge silently."
4. **Concurrency correctness comes from the driver, not from optimistic locking / version fields.**
   The service does read-check-write with no compare-and-swap. This is only safe because every
   `withTransaction` call is scoped over *all* collections (so any two concurrent transactions always
   overlap) and both current drivers now serialize overlapping readwrite transactions (native
   IndexedDB always did; the in-memory driver is fixed by this plan's write-queue). A future SQL
   adapter must provide the equivalent via row locks (`SELECT ... FOR UPDATE` on the `stockLevels`
   row) inside its transaction implementation — noted for whoever builds it, no code exists yet to
   change.
5. **`InventoryService` methods take `repos` as a parameter, not a constructor-injected
   repository.** This deliberately diverges from `SalesService`'s shape (`constructor(repo, clock,
   ids)`). Rationale: every method needs to run *inside the caller's specific transaction* (so its
   writes share atomicity with whatever else that action is doing, e.g. `recordSale` sharing a
   transaction with `completeSale`'s payment/order writes in a future integration) — a
   constructor-bound repo would tie the service to one `UnitOfWork` for its whole lifetime, which is
   wrong for a service instantiated once and reused. `clock`/`ids` stay constructor-injected (pure
   ports, no transaction coupling).
6. **Oversell toggle is resolved by the caller (action), not fetched by the service.** Actions
   already run inside `withTransaction`; they call `resolveSettings(org, branch)` using
   `repos.organization` reads in the same transaction, then pass the resulting boolean into the
   service call. Keeps `InventoryService` free of an `OrganizationRepository` dependency it doesn't
   otherwise need.
7. **Transfers are two ledger rows, not one.** A `transfer` movement type needs a "from" and a "to";
   modeling it as a single row with `fromBranchId`/`toBranchId` would break the
   `(branchId, variantId)` materialization key (`StockLevel` is per-branch, so one movement can't
   debit one branch and credit another in the same row without special-casing every ledger reader).
   Two rows, linked by `relatedMovementId` and a shared `reference` (the transfer id), keeps
   `applyMovement` — and every downstream reader (reconciliation, variance report) — ignorant of
   transfers being "special."
8. **Stocktake approval targets the counted value, computed against live `StockLevel`, not the
   count-time snapshot.** See Edge Cases for the reasoning; this is the one place the design
   deliberately avoids the naive "store the delta at count time" approach.

## State Management Strategy

- Server-side: no long-lived server state beyond what's persisted (`StockLevel`,
  `StockMovement`, `StocktakeSession`, `StocktakeCount`) — every action opens, uses, and closes its
  own `StorageProvider` per the existing per-request pattern; no server-side caching layer is
  introduced (out of scope, matches every other domain in this repo).
- Client-side: all new forms are `'use client'` components using `useActionState` bound to a single
  server action each (`adjustStockAction`, `transferStockAction`, etc.), exactly like
  `create-enum-value-form.tsx`. No client-side global store (Redux/Zustand/Context) — form state is
  local `useActionState` state; list views (`low-stock-table.tsx`, `variance-report.tsx`) are server
  components re-rendered via `revalidatePath` after each mutating action, same as
  `save-product.ts`'s `revalidatePath('/app/catalog')`.
- Stocktake count entry is the one place with meaningfully "session" state across multiple actions
  (`start` → repeated `recordCount` → `submit` → `approve`); this state lives entirely in the
  `StocktakeSession`/`StocktakeCount` records (status field + rows), not in client memory — reloading
  the `[sessionId]` page at any point reconstructs the full in-progress state from storage.

## Accessibility Considerations

- Adjustment/transfer/count forms: every input has an associated `<label htmlFor>` (matching
  `create-enum-value-form.tsx`'s existing pattern); the mandatory reason field additionally gets
  `aria-required="true"` and a visible `*` marked `aria-hidden` (the label text itself says
  "required"); validation errors from `useActionState`'s returned `error` string are rendered in an
  element with `role="alert"` so screen readers announce them without requiring focus to move there.
- Buttons disabled during `pending` (`disabled={pending}`) also get `aria-disabled={pending}` so
  assistive tech announces the disabled state consistently across browsers that don't fully expose
  native `disabled` semantics through every AT.
- Low-stock and variance-report tables use semantic `<table>`/`<thead>`/`<th scope="col">` (not
  div-grids), so column headers are announced when navigating cells; the variance sign (over/under)
  is conveyed by text ("+3", "-2") in addition to color, not color alone.
- Stocktake's Approve button: since it's hidden client-side for non-admins as a UX nicety only, make
  sure the *server* still enforces `requireAdminMembership` — never rely on the hidden button as the
  security boundary (already covered under Architecture, repeated here since it's also the
  accessibility-adjacent "don't gate on client-only signals" principle).

## Risks and Edge Cases

- **Concurrent stocktake approval + live sales.** `approveStocktakeSession` reads live `StockLevel`
  at approval time (Architecture Decision 8) specifically so intervening sales between count and
  approval aren't double-counted, but this means the variance approved can differ from the variance
  shown when the count was submitted, if enough time passes with activity in between. This is
  inherent to any stocktake process running against a live system, not a bug to "fix" — surfaced in
  the variance report by showing both `expectedQuantityAtCount` and a live-recomputed current
  quantity side by side, so the discrepancy is visible to the approver rather than hidden.
- **`resolveSettings`'s shallow merge and the new `inventory` key.** `{ ...DEFAULT_SETTINGS,
  ...org.settings, ...(branch?.settings ?? {}) }` replaces the whole `inventory: { allowOversell }`
  object if a branch sets *any* inventory setting, rather than merging inside it. Not a new problem
  (same footgun already exists for `loyalty`, `security`, etc.) — just make sure branch-level
  settings writers always write the full `InventorySettings` object, not a partial patch, until/unless
  `settings-resolver.ts` gets a deep-merge (out of scope for this card).
- **`schema-versioner.ts`'s missing step 4.** Documented above as a pre-existing gap this plan
  incidentally fixes while adding step 5. Worth flagging in review so it isn't read as unrelated
  scope creep, and so whoever reviews confirms the retroactive step-4 description (`createCollection`
  for the four catalog collections) matches what the catalog card actually shipped, since this plan's
  author is inferring it from `schema.ts`'s `COLLECTIONS` map, not from the catalog card's own service
  code paths.
- **`STOCK_MOVEMENT_TYPE` rename breaks anything importing the old `IN`/`OUT` members.** Grep during
  implementation confirmed only `stock-movement-type-labels.ts` and the admin enum-values UI
  reference the enum, and the UI reads labels generically via `STATIC_ENUM_VALUES`/the registry
  service rather than hardcoding `IN`/`OUT` — but re-grep at implementation time in case something
  new landed between this plan being written and being executed.
- **`InventoryRepository`'s split personality (generic `Product` fixture methods + real ledger
  methods on one interface)** is a deliberate, documented compromise to avoid a five-file blast
  radius in the conformance suite, not an oversight. If a future card wants to retire the
  `Product`/embedded-stock fixture from the five generic suites in favor of a neutral fixture entity,
  that's a separate, explicitly-scoped cleanup — flagged here so it isn't silently done as a
  "while I'm here" during this card's implementation.
- **No event bus / notification for low-stock or drift.** The card asks for a *view* and a *report*,
  not push notifications — `catalog`'s `EventBus` port exists and could be reused later, but wiring
  inventory into it now would be speculative (no consumer asks for it in this card).
- **No auto-correction from `ReconciliationService`.** It reports drift; it does not write a
  corrective movement. The card's acceptance line is "reconciliation detects an artificially injected
  drift," not "reconciliation repairs it" — auto-repair is a natural follow-up but is a distinct
  product decision (should it require approval? does it always trust the ledger over the
  materialized value, or vice versa?) that shouldn't be assumed.
- **Negative-adjustment vs. return direction.** `return` is modeled as always-positive (customer
  return increases branch stock). If the business also needs "return to supplier" (decreases stock),
  that's a distinct movement semantics question not raised by the card's list — flag for product
  clarification rather than guessing a second meaning onto the same enum value.
- **e2e coverage gap.** Called out under Test Plan — no Playwright flow added for the new pages in
  this pass.
