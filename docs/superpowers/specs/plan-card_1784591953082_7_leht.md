# Plan — Register Motion System (Micro-Feedback, Sheets/Dialogs, Optimistic Outbox, Sound)

Card: `card_1784591953082_7_leht`

## Overview

This card turns the register's existing, partially-tokenized motion into a coherent, restrained system built strictly on top of the task 1 token set (`2c7881d`, `src/app/globals.css:47-50`: `--motion-fast: 150ms`, `--motion-base: 300ms`, `--motion-ease: cubic-bezier(0.4, 0, 0.2, 1)`, aliased at `:96-98`... actually `:103-105` to `--duration-fast`, `--duration-base`, `--ease-base`). It covers four surfaces: micro-feedback on scan/add/remove, sheet/dialog transitions, optimistic-UI for the offline outbox, and settings-controlled sound — all gated by a single reduced-motion rule and a rule that motion never blocks a cashier action.

Three facts, verified directly against the live tree, drive the scope below:

1. **Sheet/Dialog/Toast/Select/Tooltip entrance-exit animations are currently inert.** `sheet.tsx`, `dialog.tsx`, `toast.tsx`, `select.tsx`, and `tooltip.tsx` all reference `animate-in`/`animate-out`/`fade-in-0`/`zoom-in-95`/`slide-in-from-*` utility classes. These classes are provided by the `tailwindcss-animate` (v3) / `tw-animate-css` (v4) plugin — **neither package appears anywhere in `package.json`** (`dependencies`/`devDependencies` checked directly; only `tailwindcss": "^4"` and `@tailwindcss/postcss` are present, no animate plugin, no manual `@keyframes`/`@utility` definitions for these class names anywhere in `globals.css`). Tailwind v4 does not synthesize these utilities on its own. Concretely: today, opening/closing the payment sheet and every dialog in the app is an instant show/hide with dead weight classes — there is no transition to make "fast" or to gate behind `prefers-reduced-motion`, because there is no transition at all. This is a pre-existing bug, not a preference, and this card cannot honestly ship "smooth but fast sheet and dialog transitions" without fixing it (Architecture Decision 1).
2. **The global `prefers-reduced-motion: reduce` CSS block already exists and is correct** (`globals.css:195-203`): it forces `animation-duration`/`transition-duration` to `0.01ms` and `scroll-behavior: auto` on every element. This handles all *CSS-driven* animation for free once (1) is fixed. It does **not** cover JS-driven timers (`setTimeout` dwell delays, auto-decay, auto-advance) — those need their own reduced-motion check. Exactly one such check exists today, ad hoc, in `src/app/pos/checkout/_lib/scan-sound.ts`'s private `shouldReduceMotion()`. This card centralizes that into one hook reused everywhere a JS timer exists (Architecture Decision 2).
3. **The offline outbox (`register-outbox.ts`) and its UI (`register-header.tsx:56-60`, "N pending" badge) already implement the optimistic-add / background-sync shape** — `getOutbox().addEntry(...)` is fire-and-forget from the caller's perspective, cart mutations happen synchronously in local state before/without any outbox involvement, and `useOutboxCount()` already renders live. What's missing is *feedback*, not architecture: the badge has no enter/exit transition (it just appears/disappears), and there is no "clear progress for anything over 400ms" affordance — a slow sync currently looks identical to a fast one. Scan feedback (`use-scan-feedback.ts`, `scan-feedback-banner.tsx`) is similarly optimistic already (barcode lookup is local-storage-backed, not network) and already plays a synthesized tone (`scan-sound.ts`) for `success`/`duplicate`/`not-found` — but has no mute setting and no `error`/`payment-complete` tones, which the card asks for explicitly.

**No `.agents/skills/` directory exists in this repository** (confirmed: `find . -maxdepth 6 -iname skills` returns only unrelated paths outside the project — `~/.claude/skills`, `~/.copilot/skills`, and other repos' `.agents/skills` — nothing under `/Users/faahmeey/Projects/pos`). This matches the same finding already recorded in sibling plans `plan-card_1784591953082_5_77la.md` and `_6_nkir.md`. There is no project skill to apply; implementation follows the conventions already established by the component library (`src/shared/components/ui/*`), the token system (`src/app/globals.css`), and the existing custom ESLint design-token rules (`eslint.config.mjs:150-172`).

### Motion utility layer (new)
- `src/shared/utils/motion.ts` (**new**) — the single source of truth for JS-driven motion decisions:
  - `export function prefersReducedMotion(): boolean` — reads `window.matchMedia('(prefers-reduced-motion: reduce)').matches`, `false`-safe on the server (`typeof window === 'undefined'`).
  - `export function useReducedMotion(): boolean` — React hook wrapping the above with a `matchMedia` `change` listener (so toggling the OS setting mid-session updates live, not just on mount — the current `scan-sound.ts` check is mount-time-only in effect since it's read per-call, which is actually fine for a one-shot sound but wrong for a hook driving conditional dwell timers).
  - `export const MOTION_TOKENS = { fast: 150, base: 300, slower: 500 } as const` — **numeric mirrors** of the CSS custom properties, needed because `setTimeout(ms)` cannot read a CSS var directly. Each value has a one-line comment pointing at its CSS source (`// mirrors --duration-fast in globals.css`) so drift is grep-able; a lint/test (Test Plan) asserts these three numbers match the parsed CSS file so they can never silently diverge.
  - `export const SLOW_OPERATION_THRESHOLD_MS = 400` — the exact "anything over 400ms" line from acceptance criteria. This is a **JS timing threshold**, not an animation duration, and is deliberately not a CSS token (there is nothing to transition *to* 400ms; it's a decision boundary for "should I show a progress affordance yet").
  - `export function useDelayedVisible(active: boolean, delayMs = SLOW_OPERATION_THRESHOLD_MS): boolean` — returns `true` only once `active` has been `true` continuously for `delayMs`; resets to `false` the instant `active` goes `false`. Used to gate "still working" / "syncing…" affordances so fast operations never flash a spinner. When `useReducedMotion()` is true, this hook still honors the 400ms timing (it's a *progress disclosure* threshold, not a decorative animation, so it is **not** collapsed to instant — the acceptance criterion "clear progress for anything over 400ms" must hold in the reduced-motion path too, just rendered as static text/icon instead of an animated spinner).

## Architecture Decisions

**1. Fix the dead animation classes with hand-rolled `@keyframes` in `globals.css`, not by adding the `tw-animate-css` package.** Adding the plugin (the officially recommended shadcn/ui-on-Tailwind-v4 fix) would restore *all* of `animate-in`/`fade-in-0`/`zoom-in-95`/`slide-in-from-*` in one line and fix `toast.tsx`/`select.tsx`/`tooltip.tsx` for free. Rejected for this card specifically because: (a) this card's stated surfaces are scan/add/remove, sheet/dialog, outbox, sound — not a general animation-system upgrade; pulling in a whole utility-class matrix to use four keyframes is more surface than the task asks for; (b) hand-rolled keyframes keep every duration/easing value textually anchored to `--duration-fast`/`--duration-base`/`--ease-base` by construction, which is easier to verify against "motion durations all sourced from tokens" than auditing a third-party plugin's defaults; (c) no new dependency to vet/install in a codebase that has added zero animation-related packages so far. Toast/Select/Tooltip remaining dead is flagged in Risks as a known, pre-existing, out-of-scope gap a future card should close (ideally by adopting `tw-animate-css` wholesale at that point, once the animation surface is broad enough to justify it).

**2. One shared `useReducedMotion()`/`prefersReducedMotion()` pair in `src/shared/utils/motion.ts`, not a re-check inside every consumer.** `scan-sound.ts` already proved the ad hoc pattern (private `shouldReduceMotion()`) works but duplicates on every new sound/timer site. Centralizing means: the CSS-media-query listener logic (for live OS-setting changes) is written once, and every JS timer in this card (`useDelayedVisible`, the cart-row exit-animation hook, the payment-sheet dwell states) consults the same source of truth, so "reduced-motion path verified" is one code path to test, not five.

**3. Sheet/Dialog use `--duration-base` (300ms), not a new "modal" token.** The 6-nkir sibling plan's payment-sheet redesign and the base motion doctrine (`docs/design-direction-pos.md:17`) both describe modal-adjacent motion as "smooth and confirming" — that's `--motion-base`'s stated role, while `--motion-fast` is for micro-interactions (hover, button press, the existing register button `duration-[var(--motion-fast)]` usage throughout `design-direction-pos.md`). Introducing a third top-level duration token purely for sheets would fragment the "restrained" system the card asks for; 300ms is still fast by any UI standard and satisfies "smooth but fast" without a new token to maintain.

**4. The sound mute toggle is a local (per-browser-tab-group, `localStorage`) device setting, not a tenant/org `ResolvedSettings` field.** `ResolvedSettings` (`src/domains/organization/entities/settings.ts`) is resolved server-side per org/branch and has no existing per-register or per-device axis, no settings-editing UI exists anywhere in the app (`find -iname "*settings*" -iname "page.tsx"` returns nothing), and "mute" is inherently a *this specific register, right now* preference (a loud retail floor vs. a quiet boutique register are the same tenant but want different defaults) — modeling it as an org-wide `ResolvedSettings.notifications.soundEnabled` field would require building settings-resolution plumbing and (nonexistent) admin UI this card doesn't ask for, to control something that's actually more correctly a client preference. "Settings-controlled" is satisfied by a real, persistent, user-facing toggle (not a hardcoded always-on); it does not require it to live in the tenant settings domain. Flagged as a scope decision, not an oversight, in case a future card wants to promote it to a real per-branch default.

**5. `OutboxEntry.attemptedAt` (already present) is sufficient for the 400ms progress threshold; no new "startedAt"/"syncing" field is added to the entry shape.** The consumer (`sync-progress-indicator.tsx`) computes age as `Date.now() - attemptedAt`, which is exactly "how long has this been pending" — identical to what a dedicated field would store, without touching the store's already-correct, already-tested shape.

**6. `useOldestPendingAge()` polls on a 250ms interval (only while `pendingCount > 0`) rather than pushing a live "now" clock through the outbox's own pub/sub.** The outbox's `notifyListeners()` only fires on actual state changes (add/resolve), so there is no natural event to hang a continuously-ticking "age" value off of. A 250ms poll is cheap (this queue is in-memory, typically 0-2 entries, per the file's own doc comment "Does not persist across page reloads"), self-terminating (the `setInterval` is cleared the moment `pendingCount` returns to 0, via a `useEffect` cleanup keyed on `pendingCount > 0`), and only exists to drive a threshold check, not a visual per-frame animation — a `requestAnimationFrame` loop would be over-engineering for a boolean flip at 400ms.

## State Management Strategy

- **No new global state.** Every piece of state introduced is either (a) already-existing store state consumed differently (`register-outbox.ts`'s existing pub/sub, extended with one derived selector `useOldestPendingAge`), or (b) local component/hook state scoped to exactly where it's used: `use-sound-settings.ts`'s `useState` mirrors `localStorage` and is read fresh by any component that mounts the hook (no context provider needed — the register is a single-page surface per tab, and the `storage` event listener handles cross-tab sync without a shared React context).
- **`use-exit-animation.ts`** owns a small piece of derived local state (`exitingEntries: Map<key, Line>`) inside `cart-list.tsx`'s render scope — it is *not* lifted into the cart's own reducer/context (wherever `state.lines` actually lives, e.g. `pos-cart-context.tsx`) because exit-animation bookkeeping is a pure rendering concern with no bearing on the authoritative cart data; keeping it local means it can be deleted with zero coupling if a future redesign changes how the cart list renders.
- **`useDelayedVisible`** is intentionally stateless from the caller's perspective beyond the boolean it returns — callers pass a raw `active: boolean` derived from whatever state they already have (`phase === 'charging'`, `pendingCount > 0`); the hook does not reach into the outbox or payment-sheet state itself, keeping it a generic, reusable primitive rather than a one-off.
- **Sound mute state is not part of any reducer.** It never affects sale/cart/payment logic, only whether `playFeedbackSound` is a no-op — treating it as anything more than a localStorage-backed toggle would be over-modeling a boolean.

## Accessibility Considerations

- The sound mute toggle is a real, focusable `Button` (not a raw `<div>`) with `aria-pressed` reflecting state and a descriptive `aria-label` that changes with state ("Mute sound" / "Unmute sound") — screen reader users get an accurate live label, not a static one.
- `sync-progress-indicator.tsx` and `payment-status-view.tsx`'s charging state both use `role="status" aria-live="polite"` (matching the existing pattern already established in `scan-feedback-banner.tsx:16-17`) so the ">400ms progress" affordance is announced, not just visually shown — a screen-reader cashier isn't left silently waiting.
- The cart-row exit animation must not remove the row from the accessibility tree before assistive tech has a chance to react to the prior DOM mutation event; using `onAnimationEnd` (rather than removing on the same tick as `onRemove`) keeps the node present (with `aria-hidden` applied for the exit duration, so it doesn't get focus during the brief exit window) until the animation frame actually completes, in the non-reduced-motion path. In the reduced-motion path it is removed immediately, matching what a screen reader user would expect (no dangling "removed but still announced" node).
- All new interactive elements (mute toggle, "Check status" style buttons already covered by `_6_nkir.md`'s plan) reuse the existing `Button` component, inheriting its already-correct `focus-visible:outline` treatment (`globals.css:113-116`) — no new focus-ring logic is written.
- The reduced-motion path is not merely "less flashy" — for JS-timed affordances (`useDelayedVisible`, dwell states in `payment-status-view.tsx`) the *information* (progress is happening, a sale succeeded) still needs to reach the cashier without relying on an animation to carry meaning; this card ensures the 400ms progress threshold and success/failure state text render identically whether or not motion is reduced, only the transition *style* differs (instant vs. animated), never the *content*.

## Risks and Edge Cases

- **Toast/Select/Tooltip remain on the same dead animation classes this card fixes for Sheet/Dialog.** Explicitly out of scope (Architecture Decision 1); do not silently "also fix while I'm here" beyond the two components named in the card, since that would pull three more files' visual behavior into an unreviewed change. Note it here so it isn't rediscovered as a surprise regression later — it was never animating to begin with.
- **The payment-sheet `catch` block added in this card is a minimal safety net, not the six-state payment sheet redesign described in sibling plan `_6_nkir.md`.** If `_6_nkir.md`'s work lands first or concurrently, `payment-sheet.tsx` may be replaced wholesale by `payment-sheet/payment-sheet.tsx` (a different file, per that plan) — in that case, this card's `catch`/sound-call additions should be re-applied to whichever file is authoritative at merge time; flagged so the two cards' overlap on this one file doesn't produce silently-dropped work.
- **`cart-list.tsx`'s key fix assumes one line per `variantId`.** If a future feature allows two independent lines for the same variant (e.g., serialized/lot-tracked items), the key collision would resurface the exact reconciliation bug this card fixes. Verified as true today (grep of all `CartLine` construction sites); not re-verified against any planned future feature.
- **`useOldestPendingAge`'s 250ms poll only runs while `pendingCount > 0`,** so it adds zero overhead in the (overwhelmingly common) steady state of an empty outbox — confirmed this doesn't need a "is the tab visible" guard beyond what already exists, since the interval self-clears.
- **Web Audio's `AudioContext` requires a user gesture to start in most browsers** (autoplay policy) — `scan-sound.ts` already handles this today only incidentally (the first scan is itself a user-driven action, e.g., a physical barcode scanner event or keyboard input, which browsers generally treat as sufficient interaction context; this is pre-existing behavior, not something this card changes, but worth noting as a non-blocking known limitation — if `AudioContext` construction throws, `getAudioContext()` already catches and returns `null`, and `playFeedbackSound` already no-ops on a `null` context).
- **`prefers-reduced-motion` and sound-mute are independent settings** — a cashier could have reduced motion on and sound on, or vice versa. Verified both gates are checked independently in `scan-sound.ts` (two separate early returns), not combined into one "quiet mode" flag, matching the card's explicit ask for two separate acceptance criteria.
- **The 400ms threshold constant (`SLOW_OPERATION_THRESHOLD_MS`) is defined once and imported everywhere**, not re-typed as a literal `400` in `sync-progress-indicator.tsx` or `payment-status-view.tsx` — a future change to the acceptance threshold is a one-line edit, not a multi-file hunt.
