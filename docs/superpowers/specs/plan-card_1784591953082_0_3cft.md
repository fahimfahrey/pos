## Overview

This card produces two things:

1. **`docs/design-direction.md`** — the canonical, human-readable design-direction document: aesthetic character, mood/inspiration references, color system (light + dark, with AA contrast verification), typography system (font pairing + licensing), shape/depth/motion rules, an explicit anti-goals list, and three worked examples (dashboard card, data table row, button set) rendered as annotated markdown/ASCII-labeled mockups referencing the token names defined in step 2. This is the document later formalization cards (a Tailwind theme layer, a component library, Storybook, etc.) will read as the source of truth.

2. **Seed design tokens** — the actual CSS custom properties that back the doc's color/type/shape claims, added to `src/app/globals.css` as a new token layer (light default + `[data-theme="dark"]` / `prefers-color-scheme` dark override), replacing the current stock Tailwind blue/black scaffold. This is a **seed**: it defines the primitive token set (`--color-*`, `--font-*`, `--radius-*`, `--shadow-*`, `--motion-*`) and wires it into Tailwind v4's `@theme inline` block so `bg-background`, `text-foreground`, `bg-accent`, etc. become usable utility classes immediately — but it does **not** refactor any existing page/component to use them (that is explicitly left to "later tasks that formalize" per the card title). The only behavioral change to existing UI is the ambient background/foreground/focus-ring color shift that flows from redefining the existing `--background` / `--foreground` variables and the `*:focus-visible` outline color, since those are already globally wired.

Non-goals (left to later cards): a Tailwind plugin/preset package, a component library (`Button`, `Card`, `Table` React components), Storybook, per-domain theming (POS terminal vs. admin dashboard), and any i18n routing changes for the Bengali font.

## Architecture Decisions

1. **Tokens live as CSS custom properties inside Tailwind v4's `@theme inline`, not a `tailwind.config.ts`.** This repo has no Tailwind config file — Tailwind v4's CSS-first config (`@import "tailwindcss"` + `@theme inline` in `globals.css`) is already the established pattern (confirmed by reading the current file). Introducing a `tailwind.config.ts` for this card would fork the config strategy the repo already committed to; stay consistent with the existing mechanism.

2. **Two-layer token naming: raw palette → semantic alias.** Define raw hex-backed primitives (`--palette-ivory`, `--palette-cream`, `--palette-near-black`, `--palette-taupe`, `--palette-warm-gray`, `--palette-terracotta`, `--palette-terracotta-dark`, `--palette-sage`, `--palette-brick`, `--palette-amber`, `--palette-charcoal`, `--palette-charcoal-deep`, `--palette-off-white`) then semantic aliases that consume them (`--background`, `--surface`, `--foreground`, `--foreground-muted`, `--border`, `--accent`, `--accent-foreground`, `--success`, `--danger`, `--warning`). Components and later cards should only ever consume semantic aliases (`bg-background`, `text-foreground-muted`, `bg-accent`), never raw palette variables directly — this is what lets the dark-mode override in step 4 swap the whole palette by only reassigning the alias layer. Document this rule explicitly in the direction doc's token section so later formalization cards don't regress it.

3. **Single accent, not a scale.** Per the brief ("never a rainbow of colors"), only one accent hue (terracotta-coral) is tokenized, with exactly two values: `--accent` (default) and `--accent-strong` (hover/pressed — a slightly deepened terracotta, derived, not a new hue) plus `--accent-foreground` (text/icon color placed on top of the accent, e.g. white or near-black depending on contrast math — verified in Implementation Steps §2). No blue, no purple, no multi-hue brand ramp.

4. **Dark mode via `prefers-color-scheme`, matching the existing mechanism — no new toggle infra.** The current `globals.css` already switches on `prefers-color-scheme: dark` with no JS toggle. This card keeps that mechanism (redefine the same media query block with the new dark palette) rather than introducing a `data-theme` attribute + toggle component, which is out of scope ("later tasks that formalize"). Note this constraint explicitly in the doc so a later card introducing manual theme switching knows to convert the media query to a class/attribute strategy without changing token *names*.

5. **Fonts via `next/font/google` with CSS variables, not `<link>` tags.** Next.js's built-in font optimization (self-hosts Google Fonts, zero layout shift, automatic `font-display`) is the framework-idiomatic approach and avoids a runtime request to Google Fonts (privacy + performance). Confirm current API shape against `node_modules/next/dist/docs/` per the AGENTS.md note before writing the import — the `next/font/google` variable-font pattern (`const fraunces = Fraunces({ variable: '--font-display', subsets: ['latin'] })`) is assumed but must be verified against the installed Next 16 docs, not assumed from training data.

6. **Font choice: Fraunces (display) + Inter (body/UI), both open-source/free with no attribution burden.** Fraunces (SIL OFL 1.1) is the more "editorial, warm, human" of the two serif options offered (vs. Newsreader/Source Serif 4 — more neutral/utilitarian) and pairs well with a POS's need for large, confident price/heading display. Inter (SIL OFL 1.1) is chosen over Geist for body/UI because it has broader Bengali-adjacent Unicode support in surrounding UI chrome and is already the more battle-tested choice for dense tabular UI (this is a POS with receipts/tables). Both ship via `next/font/google`, so no manual font-file vendoring or license-file bundling is needed — the direction doc still records the OFL-1.1 license explicitly per the acceptance criteria. Document the two runner-up choices (Newsreader/Source Serif 4; Geist) as noted alternates so a later card can swap without re-researching.

7. **Bengali font declared as a token, wiring deferred.** `Hind Siliguri` (also SIL OFL) is recorded as `--font-bengali` in the seed token set and the doc's typography section (satisfies "pair a warm, legible Bengali face" as a *documented decision*), but is **not** loaded via `next/font/google` in `layout.tsx` unless the implementer's repo scan (Implementation Steps §0) finds existing Bengali-script UI strings/pages. Loading a font family with zero current call sites would be dead weight the linter/bundler can't tree-shake away automatically for `next/font`. This is a scope guard, not a doc omission — the doc still fully specifies the family, stack, and rationale.

8. **Tabular figures via `font-variant-numeric`, tokenized as a utility-adjacent rule, not a new font.** "Money and quantities use tabular figures" is implemented as a documented CSS rule (`font-variant-numeric: tabular-nums;`) applied via a `.tabular-nums` utility (Tailwind v4 ships this utility by default — verify, don't reintroduce it) rather than a special numeral font, since Inter already has tabular lining figures. Document the utility class name in the worked examples (dashboard card price, table row amount) so later cards apply it consistently.

9. **Shape/depth/motion recorded as tokens, not just prose.** `--radius-card` (10px), `--radius-button` (10px), `--radius-input` (6px), `--shadow-sm` / `--shadow-md` (soft, low-spread, warm-tinted rgba shadows — not pure black, per "warm" brief), and `--motion-fast` / `--motion-base` durations + a single shared easing token `--motion-ease` (e.g. `cubic-bezier(0.4, 0, 0.2, 1)` — quiet, confirming, no bounce/overshoot) are added to the same `@theme inline` block so they're immediately consumable as Tailwind arbitrary-value-free utilities where Tailwind v4 supports it, and as raw `var(--token)` elsewhere.

**Step 1 — Draft `docs/design-direction.md`.** Section order:
1. **Aesthetic Character** — one paragraph, the five adjectives from the brief (warm, calm, editorial, human) plus a "feels like" analogy (e.g. "a well-run independent café's paper receipt, not a big-box terminal").
2. **Mood / Inspiration Board** — a bulleted list of reference points (no images fetched/embedded — this is a text-only doc; describe each reference in one line: e.g. "Anthropic.com's warm-neutral editorial palette", "a printed restaurant receipt on cream paper", "Basecamp's calm data tables", "an independent bookstore's shelf-talker cards"). Do not fetch external URLs — describe, don't link, to avoid unauthorized web fetches for a docs task.
3. **Anti-Goals** — explicit bulleted "NOT" list: NOT cold/clinical, NOT neon/saturated, NOT pure-white-and-blue (SaaS-generic), NOT cramped/dense-for-density's-sake, plus 2-3 more specific to a POS (NOT a casino-bright touchscreen, NOT skeuomorphic register buttons).
4. **Color System** — two subsections (Light, Dark), each a table: token name | hex | usage | contrast ratio vs. its paired surface | AA pass/fail. Computed in Implementation Step 2.
5. **Typography** — font pairing rationale, full stack strings (with generic-family fallbacks), license (SIL OFL 1.1) named for each, weight range used (e.g. Fraunces 400/500/600, Inter 400/500/600/700), and the tabular-figures rule.
6. **Shape & Depth** — corner radii table, shadow tokens with their rgba values and blur/spread, whitespace/line-height guidance (e.g. body line-height 1.6, headings 1.2).
7. **Motion** — duration tokens, easing token, one-sentence philosophy ("confirming, not decorative — motion should tell you the system heard you, never perform").
8. **Token Reference** — a single consolidated table mirroring exactly what's added to `globals.css` (name → value → CSS variable), so the doc and the CSS can't silently drift.
9. **Worked Examples** — three subsections, each with: a fenced code block showing the actual HTML/Tailwind-class markup (using only tokens/utilities defined above) and a one-paragraph annotation of which tokens are doing the work:
   - **Dashboard card** — e.g. a "Today's Sales" stat card: `bg-surface` container, `rounded-[--radius-card]`/`rounded-xl`, `shadow-sm`, serif large numeral in `font-display` with `tabular-nums`, muted label in `text-foreground-muted`, small accent-colored delta chip.
   - **Data table row** — a line-item/receipt row: `border-b border-border`, `font-body`, quantity + price right-aligned with `tabular-nums`, hover state using a subtle warm-tinted background shift (not the accent color — reserve accent for actionable elements per "used sparingly").
   - **Button set** — primary (solid `bg-accent`/`text-accent-foreground`), secondary (outlined/bordered, `border-border` `text-foreground`), destructive (uses `--danger`, brick-red, used only for genuinely destructive actions like voiding a sale), each at `rounded-[--radius-button]` with the shared `--motion-fast`/`--motion-ease` transition on hover/active/focus states, and a visible focus ring using `--accent` (not browser default blue) satisfying keyboard-accessibility.

**Step 2 — Compute and verify all color pairings before writing them into the doc.** For every token pair that will appear together as text-on-background in the app, compute the WCAG contrast ratio (relative luminance formula) and confirm it meets **AA** (4.5:1 for normal text, 3:1 for large text/UI components) — this is an explicit acceptance criterion, not optional:
   - Light: `--foreground` (#20201C) on `--background` (#F5F4EF) and on `--surface`.
   - Light: `--foreground-muted` (#6B675E) on `--background` and `--surface`.
   - Light: `--accent-foreground` on `--accent` (#CC785C) — verify whether white or the near-black text token passes AA on top of terracotta; pick whichever does (terracotta at #CC785C is mid-luminance, so this must actually be computed, not assumed — likely white text passes for large/bold button text at 3:1 but may fail 4.5:1 for small text, in which case either darken the accent slightly for text contexts or restrict accent-background text to large/bold sizes only — resolve this with real numbers, not a guess, and record the resolution in the doc).
   - Light: success/danger/warning tokens on `--background` and as small text/icon usage.
   - Dark: `--foreground` (#EDEAE0) on dark `--background`/`--surface`, muted text, accent-on-dark (desaturated coral, per brief), and status colors on dark surfaces.
   - Record every ratio and pass/fail explicitly in the Color System tables (Step 1 §4) — do not publish a token as "AA-verified" without a computed number next to it.
   - If any brief-suggested hex fails AA, adjust that token's lightness/chroma minimally (document the adjustment and the original brief value alongside it) rather than silently picking a different color family.

**Step 3 — Update `src/app/globals.css`.**
   - Replace `:root` block with the full light semantic + raw palette custom properties.
   - Replace the `@media (prefers-color-scheme: dark)` block with the full dark palette (raw + semantic reassignment).
   - Expand `@theme inline` to map every semantic token to a Tailwind-consumable `--color-*`, `--font-*`, `--radius-*`, `--shadow-*` variable (Tailwind v4 auto-generates utilities like `bg-accent`, `text-foreground-muted`, `rounded-card` from these).
   - Update `*:focus-visible { outline: 2px solid ...; }` to reference `var(--accent)` instead of the hardcoded `#0066cc`.
   - Update `body { font-family: ... }` to reference `var(--font-body)` (falling back to the existing Arial/Helvetica stack as the final fallback in the stack, not a replacement of the safety net).
   - Preserve the existing `prefers-reduced-motion` block verbatim — do not weaken it.

## Accessibility Considerations

- **AA contrast is a hard acceptance gate**, not aspirational — every text/background and icon/background pairing that appears in the doc's tables and worked examples must be computed and pass (Implementation Step 2). Any token that fails must be adjusted before publishing, with the adjustment documented.
- **Focus visibility**: the existing `*:focus-visible` outline rule is preserved and repointed to the new accent token — do not remove it, and confirm the new accent color still produces a visible ring against both light and dark backgrounds (compute contrast of `--accent` against `--background` in both themes, not just against foreground text, since a focus ring's job is to be visible against the surface it sits on).
- **Reduced motion**: the existing `prefers-reduced-motion: reduce` block must be preserved verbatim; new `--motion-*` tokens must not bypass it (they only define durations/easing consumed by transitions that are already collapsed to `0.01ms` under that media query).
- **Color independence for status**: success/danger/warning must not be the *only* signal in the worked examples — e.g. the button set's destructive button should be labeled "Void" in text, not identified by red color alone (WCAG 1.4.1). Note this explicitly as a rule in the doc's Color System section so later formalization cards (e.g. status badges, toasts) inherit the rule.
- **Tabular figures accessibility note**: tabular-nums is a visual/legibility aid (aligns digits in receipts/tables) with no accessibility downside — note this is a legibility feature, not a substitute for proper `<td>`/ARIA table semantics, which are out of scope for this doc-and-token card.
- **Skip link**: `layout.tsx`'s existing `Skip to content` link and `sr-only focus:not-sr-only` pattern must continue to work unchanged after the font/className edits — confirm the class list edit doesn't accidentally clobber existing utility classes.

## Risks and Edge Cases

- **Contrast failure on the accent color for small text.** #CC785C on white text may sit right at or below the 4.5:1 AA threshold for small text even though it's fine for large/bold text — must be resolved with a real computed number (Step 2), not assumed from the brief's approximate hex. Mitigation already specified: restrict small accent-background text usage or slightly darken the accent for text-bearing surfaces, and document which.
- **Dark-mode accent desaturation is under-specified in the brief** ("same coral accent slightly desaturated") — no exact hex given. Mitigation: derive it by reducing saturation ~15-20% and slightly lifting lightness from the light-mode accent, then re-verify AA against the dark background/surface; record the derivation method in the doc so it's reproducible, not arbitrary.
- **Tailwind v4 `@theme inline` cannot reference `prefers-color-scheme`-scoped variables inside the `@theme` block itself** — Tailwind's `@theme` block is typically declared once; the existing repo pattern (confirmed by reading the current file) already solves this correctly by keeping raw values in `:root`/`@media` and only *aliasing* through `@theme inline`, so utilities stay in sync automatically. The implementer must preserve this exact structural pattern rather than duplicating `@theme` blocks per color scheme (which would not work with Tailwind v4's model).
- **Bengali font scope creep** — risk of wiring `Hind Siliguri` into `next/font/google` speculatively with no actual Bengali content, adding unused font weight downloads to every page load. Mitigation: Step 0's scan gate; default to doc-only if no Bengali content is found, and say so explicitly in the doc rather than silently omitting it.
- **`next/font/google` API drift** — this repo's AGENTS.md explicitly warns the installed Next version may differ from training-data assumptions. Mitigation: mandatory doc read in Step 0/Architecture Decision 5 before writing the import; if the API differs materially, adjust the plan's Step 4 mechanics but not its intent (CSS-variable-based font loading).
- **Ambient regression risk**: because `--background`/`--foreground`/focus-ring are already globally wired into `body`/`*:focus-visible`, this "token seed" card is not purely additive — every existing page's base colors will visibly change the moment this ships. This is intended (it's the point of the card) but must be called out, not discovered by surprise in review; the manual visual check (Test Plan §4) across at least two routes exists specifically to catch any existing component that hardcodes a color assuming the old white/black scaffold instead of consuming the CSS variables.
- **Doc/CSS drift over time** — nothing enforces that `docs/design-direction.md`'s token table stays in sync with `globals.css` after this card lands (no lint rule ties markdown to CSS). Flag this as a known gap; a later formalization card could add a small script that diffs the doc's token table against the CSS custom properties, but building that is out of scope here.
- **No images/external references fetched** — the mood/inspiration board is written as text descriptions only; do not fetch or embed external URLs/screenshots for this doc, both because this is a plan/implementation task with no explicit user authorization to fetch external content and because committing hotlinked or scraped images would be a licensing risk.
