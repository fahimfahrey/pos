# Implementation Plan — Organization Domain (multi-tenant org / branch / register / membership / invite + shared settings resolver + onboarding UI)

Card: `card_1784266069993_4_kqb5`

## Overview

Build out `src/domains/organization/` on top of the already-built storage abstraction
(`RepositorySet`, `OrganizationRepository`, `Collection`, `withTransaction`, the
conformance matrix) into the real multi-tenant model the rest of the POS depends on:

- **Organization** — the top-level tenant (`plan`, `status`, `settings` JSON).
- **Branch** — a location under an org (own `settings` JSON). *A shop has one branch; a
  super-shop has many.* Branch **supersedes** the v1 placeholder `Store` entity (the
  `stores` collection already carries an `orgId` index — it was scaffolded to become this).
- **Register** — a terminal that belongs to a **branch** (`branchId`, `orgId`).
- **Membership** — a user's role inside an org, keyed by an **integer** `MembershipRole`
  (`OWNER=1`, `ADMIN=2` = branch manager, `MEMBER=3` = cashier, `VIEWER=4` =
  accountant/auditor). Integer values so role comparisons (`role <= ADMIN`) express a
  privilege ladder.
- **BranchAssignment** — scopes a membership (staff member) to specific branches.
- **Invite** — the flow to bring a user into an org with a role and branch scope.

Plus the two things that make the system configurable **without code changes**:

1. **The single shared settings resolver** — `resolveSettings(org, branch?)` returning
   `{ ...DEFAULTS, ...org.settings, ...branch.settings }` (the card's
   `{...org.settings, ...branch.settings}` with a defaults floor). Covers currency, locale,
   timezone, tax rules + tax-inclusive/exclusive mode, receipt template, rounding rules,
   business hours, barcode symbology, and loyalty rules.
2. **Onboarding UI** — create organization → first branch → first register (+ an OWNER
   membership for the creator), persisted through the storage provider.

**Acceptance gates:** settings-precedence unit tests (defaults → org → branch), and the
conformance tenant-scoping suite extended with **org/branch fixtures** (real scoped repo
queries, run against every adapter — memory + IndexedDB).

Because Branch replaces the v1 `Store`/`OrganizationSetting` placeholders, this card also
**migrates the storage collection set** (`stores`/`organizationSettings` → `organizations`
/`branches`/`memberships`/`branchAssignments`/`invites`, `registers` re-scoped to
`branchId`) and updates the conformance fixtures/CRUD/schema suites that referenced the old
names. This is in-scope: the card owns the organization domain end to end.

## Architecture Decisions

1. **Branch supersedes Store; migrate the collection set rather than run two parallel
   concepts.** The v1 `stores` collection already had an `orgId` index (`stores: 'id,
   orgId'`) — it was scaffolded to become the tenant-scoped location. Keeping both `Store`
   and `Branch` would be two names for one concept and confuse every downstream card. So
   `stores`→`branches`, `Register.storeId`→`Register.branchId`, and the obsolete
   key-value `organizationSettings` store is dropped (settings now live as JSON on the
   org/branch). The conformance CRUD/schema/fixtures that referenced the old names are
   updated in the same card (they are the storage contract's tests, and this card owns the
   org domain). Risk of breaking those suites is mitigated by updating them in lockstep and
   running the full matrix (Test Plan).

2. **Integer `MembershipRole` expresses a privilege ladder.** `OWNER=1 < ADMIN=2 <
   MEMBER=3 < VIEWER=4` (lower number = more privilege) lets authorization checks be
   `membership.role <= MEMBERSHIP_ROLE.ADMIN`. It is a domain role (org-scoped), **distinct
   from** the existing string `USER_ROLE` (`cashier`/`manager`/`admin`) on the auth `User`;
   the two are not merged. The mapping (OWNER/ADMIN≈admin, MEMBER≈cashier, VIEWER≈auditor)
   is documented in `membership-role-labels.ts` comments. Stored as the raw integer so
   IndexedDB can index/range it later.

3. **Expanded `OrganizationRepository` — flat, id-keyed, tenant-scoped list methods.**
   Following the existing repo idiom (`Collection.get/put/filter`, in-memory filtering per
   `docs/storage-adapters.md`):
   ```ts
   export interface OrganizationRepository {
     // organizations
     saveOrganization(org: Organization): Promise<void>
     findOrganizationById(id: string): Promise<Organization | null>
     findOrganizationBySlug(slug: string): Promise<Organization | null>
     listOrganizations(): Promise<Organization[]>
     // branches
     saveBranch(branch: Branch): Promise<void>
     findBranchById(id: string): Promise<Branch | null>
     listBranchesForOrg(orgId: string): Promise<Branch[]>
     // registers
     saveRegister(register: Register): Promise<void>
     findRegisterById(id: string): Promise<Register | null>
     listRegistersForBranch(branchId: string): Promise<Register[]>
     // memberships
     saveMembership(m: Membership): Promise<void>
     findMembership(orgId: string, userId: string): Promise<Membership | null>
     listMembershipsForOrg(orgId: string): Promise<Membership[]>
     listMembershipsForUser(userId: string): Promise<Membership[]>
     // branch assignments
     saveBranchAssignment(a: BranchAssignment): Promise<void>
     listAssignmentsForMembership(membershipId: string): Promise<BranchAssignment[]>
     listAssignmentsForBranch(branchId: string): Promise<BranchAssignment[]>
     deleteBranchAssignment(id: string): Promise<void>
     // invites
     saveInvite(i: Invite): Promise<void>
     findInviteByToken(token: string): Promise<Invite | null>
     listInvitesForOrg(orgId: string): Promise<Invite[]>
   }
   ```
   `CoreOrganizationRepository` holds six `Collection<T>` (`organizations`, `branches`,
   `registers`, `memberships`, `branchAssignments`, `invites`) and implements each method
   via `get`/`put`/`delete`/`filter`/`find` — same shape as the current class.

4. **One shared, typed settings shape + a defaults floor; the resolver is a shallow
   merge.** `ResolvedSettings` is the fully-populated shape; `OrganizationSettings` and
   `BranchSettings` are both `Partial<ResolvedSettings>` (an org or branch overrides only
   the keys it cares about). The resolver:
   ```ts
   export function resolveSettings(
     org: { settings: OrganizationSettings },
     branch?: { settings: BranchSettings } | null,
   ): ResolvedSettings {
     return { ...DEFAULT_SETTINGS, ...org.settings, ...(branch?.settings ?? {}) }
   }
   ```
   Precedence: **branch > org > defaults**. The merge is **shallow** (matches the card's
   literal `{...org.settings, ...branch.settings}`): a branch that sets `taxRules` replaces
   the org's array wholesale — it does not element-merge. This is deliberate and documented
   (top-level keys are the override unit). `ResolvedSettings` keys cover every configurable
   named in the card: `currency`, `locale`, `timezone`, `taxMode: TaxMode`
   (inclusive/exclusive), `taxRules: TaxRule[]`, `receiptTemplate: ReceiptTemplate`,
   `rounding: RoundingRule`, `businessHours: BusinessHours`, `barcodeSymbology:
   BarcodeSymbology`, `loyalty: LoyaltyRules`. `DEFAULT_SETTINGS` provides a sane floor
   (e.g. `currency:'USD'`, `locale:'en-US'`, `timezone:'UTC'`, `taxMode:'exclusive'`,
   `taxRules:[]`, `rounding:{ mode:'nearest', increment:0.01 }`, `loyalty:{ enabled:false }`).

5. **Settings are a pure domain concern; the resolver has no I/O.** It lives in
   `services/` (boundary `services` may import `entities`, `shared`, `constants` — not
   `infrastructure`), so it is trivially unit-testable and reusable by pricing/receipt
   code later. Callers load the `Organization` + active `Branch` from the repo and pass
   them in.

6. **IndexedDB structural migration v1→v2 via an explicit Dexie version chain.** Creating
   new object stores in an existing IndexedDB database **requires** a version bump — so
   `CURRENT_SCHEMA_VERSION` goes to `2` and `buildVersionChain(db)` declares the frozen v1
   spec **and** the v2 delta:
   ```ts
   db.version(1).stores(V1_STORES_SPEC)      // frozen historical snapshot
   db.version(2).stores({
     stores: null, organizationSettings: null,        // drop
     registers: 'id, orgId, branchId, active',        // re-index
     organizations: 'id, slug, status',
     branches: 'id, orgId',
     memberships: 'id, orgId, userId, [orgId+userId]',
     branchAssignments: 'id, orgId, branchId, membershipId, [membershipId+branchId]',
     invites: 'id, orgId, token, email, status',
     [META_STORE]: 'id',
   })
   ```
   A fresh DB opens straight at v2; a dev's local v1 `pos-storage` upgrades cleanly. The
   current/full spec (`buildStoresSpec()` from `COLLECTION_INDEXES`) is the v2 shape. Per
   the prior adapter card, the logical `SchemaVersioner` (`__meta__` version) is a separate
   axis — its v2 step is a no-op because there is no record-level data migration.

7. **Onboarding writes run in the browser, not on the server.** The storage engine is
   IndexedDB (browser-only); `defaultStorageEngine()` falls back to `memory` under SSR. So
   the wizard is a `'use client'` component that calls the onboarding action **in an event
   handler** (browser context, where `createDefaultStorageProvider()` resolves the real
   IndexedDB engine). The action file is a plain module (NOT `'use server'`) bundled into
   the client — it is classified `actions` only for the boundary graph, and the
   `app → actions → infrastructure` chain is allowed. `app` may **not** import `entities`,
   so the action's inputs/outputs are plain DTOs (strings/enums from `constants`), keeping
   entity types out of the UI layer.

8. **Onboarding is atomic.** `completeOnboarding` does org + branch + register + OWNER
   membership inside a single `provider.withTransaction(repos => …)` so a partial org (an
   org with no branch, or a branch with no owner) can never be persisted — commit-all or
   rollback-all, guaranteed by the storage core's UnitOfWork.

9. **Ids and timestamps come from the injected ports, not `Date.now()`/`crypto` inline.**
   The action uses `UuidIdGenerator` (`@infra/adapters/uuid-id-generator`) and `SystemClock`
   (`@infra/adapters/system-clock`) — the same ports the existing container wires — so the
   flow stays deterministic and testable (tests pass fakes). Invite `token` also comes from
   the id generator (opaque, unguessable-enough for a local-first app; not a security
   boundary — noted in Risks).

## State Management Strategy

- **Durable state** is the single storage provider (`createDefaultStorageProvider()` →
  IndexedDB in the browser, memory under SSR/test). No new global store; the org/branch/
  register/membership/invite records live in their collections and are read back through
  the repo. There is **no client cache layer** — the wizard reads nothing back mid-flow; it
  writes once and redirects.
- **Transaction/UnitOfWork state** is owned by the storage core: `completeOnboarding` and
  `acceptInvite` each run inside one `withTransaction`, so multi-record writes commit or
  roll back atomically (Decision 8).
- **Settings state is derived, never stored resolved.** Only the per-entity partials are
  persisted (`org.settings`, `branch.settings`); the effective config is computed on demand
  by `resolveSettings`. This keeps a branch's overrides authoritative and avoids stale
  denormalized copies — change an org default and every branch that didn't override it
  sees it immediately.
- **Wizard UI state** is local component state: `useState` for the current step index and
  an accumulating form-data object; `useTransition`/a `pending` boolean for the async
  submit; a `result`/`error` state for the outcome. The provider is created lazily inside
  the submit handler (not at module scope) to stay SSR-safe and avoid constructing
  IndexedDB during render. No context/Redux/Zustand — the flow is self-contained and
  linear.
- **Ids/timestamps** are produced by injected ports (`UuidIdGenerator`, `SystemClock`) so
  state transitions are deterministic under test.

## Accessibility Considerations

- **Semantic multi-step form.** Each step is a `<form>` with a `<fieldset>` + `<legend>`
  naming the step ("Organization details", "First branch", "First register"). Inputs use
  real `<label htmlFor>` associations; required fields set `aria-required` and, on error,
  `aria-invalid` + `aria-describedby` pointing at the message.
- **Progress is conveyed non-visually.** Render the step indicator with
  `aria-current="step"` on the active item and an accessible "Step 2 of 3" text (not
  color/position alone). WCAG AA contrast for all text and the primary button.
- **Focus management.** On advancing/going back, move focus to the new step's `<legend>`
  or first field (via a ref); on validation failure, move focus to an error summary
  region and to the first invalid field.
- **Status announced.** Submission outcome ("Organization created", errors) is rendered in
  an `aria-live="polite"` region (errors `assertive`), text-based — a screen-reader user
  learns success/failure without seeing color. The pending state sets `aria-busy` and
  disables the submit `<button>` (real button, keyboard-operable).
- **Keyboard-only path.** The entire wizard (all selects, next/back/submit) is operable by
  keyboard; native controls only, no `div` click handlers. Selects for plan/timezone/
  currency are real `<select>` (or an accessible listbox) with labels.
- **No entity jargon in labels.** User-facing copy ("Shop", "Branch", "Till/Register") is
  plain; role/plan labels come from the `*_LABELS` maps.

## Risks and Edge Cases

- **Breaking the CRUD/schema conformance suites during the rename.** `suites/crud.ts`,
  `suites/schema-versioning.ts`, `migration-ladder.ts`, `fixtures.ts`, `conformance/index.ts`
  all reference `stores`/`organizationSettings`/`makeStore`/`makeOrganizationSetting`.
  *Mitigation:* update every reference in the same change (enumerated in Steps 8–9) and run
  the full matrix before declaring done; the grep list in this plan is the exhaustive set.
- **Local IndexedDB upgrade v1→v2.** A dev who ran the prior card has a v1 `pos-storage`
  DB with `stores`/`organizationSettings` object stores and no new stores. *Mitigation:* the
  explicit `version(1)`+`version(2)` chain (Decision 6) upgrades it (drops old, adds new);
  without the version bump the new stores would never be created. Fresh DBs open at v2.
- **Shallow vs deep settings merge.** A branch overriding `taxRules`/`businessHours`
  replaces the whole value, not element-merges. This matches the card's literal spread and
  is intentional — but is a footgun if someone expects deep merge. *Mitigation:* documented
  in `settings.ts`/resolver JSDoc and asserted by a dedicated test (Step 4).
- **Onboarding cannot run on the server.** Server components / any `'use server'` action
  would hit the memory engine and lose data on the next request. *Mitigation:* wizard is
  `'use client'`, provider constructed in the browser event handler (Decision 7); do **not**
  add `'use server'` to `actions/onboarding.ts`.
- **`app` boundary forbids importing `entities`.** The wizard must not import
  `Organization`/`Branch` types. *Mitigation:* action DTOs use primitives + `constants`
  enums; role/plan come from `@constants`.
- **No auth session yet → OWNER membership owner id.** There is no session provider.
  *Mitigation:* `completeOnboarding` takes `ownerUserId` (defaulted to a documented
  placeholder / the first `auth` user if present); wire to a real session in the auth card.
  Note this in the action JSDoc so it isn't mistaken for a finished auth integration.
- **Invite token is not a security boundary.** Tokens are opaque ids from `UuidIdGenerator`
  for a local-first app; no server-side verification exists. *Mitigation:* documented;
  `expiresAt` + single-use (`status` flip on accept) enforced; don't present this as
  cryptographic security.
- **`noUncheckedIndexedAccess`.** `businessHours` day lookups and `taxRules[i]` are
  possibly-`undefined`. *Mitigation:* model `businessHours` as a fixed-key record with
  required day entries, guard array access, use `?? default`.
- **Schema-version coupling.** Bumping `CURRENT_SCHEMA_VERSION` to 2 changes the value
  `getSchemaVersion()` returns after `migrate()` (memory 0→2). The schema-versioning
  conformance suite asserts `<= 3` and uses its own `migrationLadder` on a raw driver, so it
  is unaffected — but verify the ULID/CRUD suites (which run at the migrated version) stay
  green.
- **Empty/duplicate slug.** `findOrganizationBySlug` + a unique `slug` index assume slug
  uniqueness; the wizard may omit slug. *Mitigation:* slug optional; derive from name or
  leave undefined; do not enforce DB-level uniqueness (the adapter index is non-unique).
