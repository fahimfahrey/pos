## Overview

Stand up a fresh **Next.js 15 (App Router, React 19)** application that is **local-first** and organized as a strict, one-directional layered architecture. The scaffold must:

- Use **TypeScript in strict mode** (plus extra strictness flags), **Tailwind CSS v4**, **ESLint 9 (flat config)** with **Prettier**, **Vitest + Testing Library** for unit/component tests, and **Playwright** for E2E.
- Create the directory skeleton: route groups under `src/app` (`(marketing)`, `(auth)`, `(dashboard)/app`, `admin`, `api`), plus `src/domains`, `src/infrastructure`, `src/shared` (`env.ts`, `logger.ts`, `errors/`, `utils/`), `src/constants` (`enums/`, `maps/`), and `src/middleware.ts`.
- Add **PWA plumbing** (web manifest + `public/sw.js` + a client-side registration component) because the app runs local-first.
- Enforce a **dependency direction** with `eslint-plugin-boundaries`:
  1. `app/` may import only `actions` (plus `shared`/`constants` and app-internal UI).
  2. `services` may import repository/port **interfaces** only — never a concrete storage implementation.
  3. **Nothing above `infrastructure`** may import a storage-engine package (`idb`, `dexie`, `fake-indexeddb`, `better-sqlite3`, `localforage`, …) directly.
- Ship npm scripts: `dev`, `build`, `lint`, `typecheck`, `test`, `test:e2e`.

**Acceptance:** a fresh clone boots (`npm install` → `npm run dev`/`npm run build` succeed with no required env), and a **fixture test** proves the boundary rule reports an error when a domain *service* imports an IndexedDB API. The env loader (`src/shared/env.ts`) throws at boot on invalid env but has safe defaults so a config-less clone still boots.

The reference layering (top imports downward only):

```
app  →  actions  →  services  →  repositories (interfaces) / entities
                                        ▲
infrastructure (Dexie/IndexedDB, concrete repos)  ── implements ─┘
shared / constants  ← importable by any layer, import nothing upward
```

## Architecture Decisions

1. **App Router + React 19 (Next 15).** Server Components by default; `"use client"` only where interactivity/browser APIs are needed (SW registration, forms). Server Actions are the single entry from `app` into business logic, which is what lets us enforce "app imports only actions".

2. **Layered / ports-and-adapters (hexagonal).** Business rules live in `domains/*/services`, depend on **interfaces** (`repositories`, `shared/ports`). Concrete IndexedDB adapters live in `infrastructure`. A **composition root** (`infrastructure/container.ts`) wires concrete adapters into services; server actions resolve wired services from the container's barrel. This keeps IndexedDB knowledge in exactly one layer.

3. **Boundary enforcement with `eslint-plugin-boundaries`.** Three complementary rules:
   - `boundaries/element-types` — the allowed cross-layer import matrix (below).
   - `boundaries/external` — restricts which **npm packages** each layer may import; storage engines allowed only in `infrastructure`. This is what the fixture test triggers (a service importing `idb`).
   - `boundaries/no-unknown` — every included source file must map to a declared element (catches misplaced files).
   - Plus `no-restricted-globals` for the bare `indexedDB`/`caches` globals in non-infra layers (belt-and-suspenders; the `import`-based path is the one the acceptance test asserts).

   **Element definitions** (`settings['boundaries/elements']`, first match wins, so order matters):
   | type | pattern |
   |---|---|
   | `app` | `src/app/**` |
   | `actions` | `src/domains/*/actions/**` |
   | `services` | `src/domains/*/services/**` |
   | `repositories` | `src/domains/*/repositories/**` |
   | `entities` | `src/domains/*/entities/**` |
   | `infrastructure` | `src/infrastructure/**` |
   | `shared` | `src/shared/**` |
   | `constants` | `src/constants/**` |

   **Allowed import matrix** (`boundaries/element-types`, `default: 'disallow'`):
   | from ↓ / allow → | app | actions | services | repositories | entities | infrastructure | shared | constants |
   |---|---|---|---|---|---|---|---|---|
   | app | ✅ | ✅ | | | | | ✅ | ✅ |
   | actions | | ✅ | ✅ | | ✅ | ✅ (barrel only) | ✅ | ✅ |
   | services | | | ✅ | ✅ | ✅ | ❌ | ✅ | ✅ |
   | repositories | | | | ✅ | ✅ | | ✅ | ✅ |
   | entities | | | | | ✅ | | ✅ | ✅ |
   | infrastructure | | | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
   | shared | | | | | | | ✅ | ✅ |
   | constants | | | | | | | | ✅ |

   Notes: `services → infrastructure` is **disallowed** (rule 2 — services see interfaces only). `actions → infrastructure` is allowed but constrained by `boundaries/entry-point` to the `infrastructure/index.ts` barrel (container) so actions cannot deep-import a Dexie file. `boundaries/include` is scoped to `src/{app,domains,infrastructure,shared,constants}/**` so framework files (`src/middleware.ts`) don't trip `no-unknown`.

   **External-package rule** (`boundaries/external`, `default: 'allow'`): for `from: [app, actions, services, repositories, entities, shared, constants]`, `disallow: ['dexie','idb','idb-keyval','localforage','fake-indexeddb','better-sqlite3','sql.js','@sqlite.org/sqlite-wasm']` with message *"Storage engine '${dependency}' may only be imported from src/infrastructure."* `infrastructure` is omitted from `from`, so it stays free to import them.

4. **ESLint flat config (ESLint 9).** Compose `eslint-config-next` (`next/core-web-vitals`, `next/typescript`) via `FlatCompat` if the installed version still ships eslintrc-style config, otherwise its native flat export; then the boundaries block; then `eslint-config-prettier` **last** to disable stylistic conflicts. `lint` script uses the ESLint CLI (`eslint .`) so our custom flat config (with the boundaries plugin) is authoritative rather than `next lint`'s managed subset.

5. **The fixture is excluded from normal builds, linted only by the test.** The violating file lives at a real *services* path (so boundaries classifies it as `services`) inside a `__boundary_fixtures__/` folder that is: ignored by `eslint.config.mjs` (`ignores`), excluded by `tsconfig.json` and by `next.config` build, and excluded by Vitest's source globs. The boundary test lints it with the ESLint Node API using `ignore: false`, so the rule still evaluates. This keeps `npm run lint`/`build`/`typecheck` green on a fresh clone while proving the rule fires on demand — satisfying both "fresh clone boots" and the fixture acceptance.

6. **Strict TypeScript.** `strict: true` plus `noUncheckedIndexedAccess`, `noImplicitOverride`, `noFallthroughCasesInSwitch`, `forceConsistentCasingInFileNames`, `verbatimModuleSyntax`, `exactOptionalPropertyTypes` (evaluate; relax if it fights Next types), `moduleResolution: "bundler"`, `noEmit: true`.

7. **Constants as `as const` unions, not TS `enum`.** Avoids `enum` runtime/`isolatedModules` pitfalls and pairs cleanly with `Record<Union,…>` label maps under `constants/maps`.

8. **Env validated at boot, but non-blocking for a clean clone.** Every var is optional with a default (`NODE_ENV`, `LOG_LEVEL` default `info`, `NEXT_PUBLIC_APP_NAME` default `POS`). Invalid *provided* values throw a formatted error; absent values fall back. This demonstrates boot-time validation without breaking "fresh clone boots".

9. **Storage engine: Dexie over raw IndexedDB.** Dexie is the single storage-engine dependency; the fixture instead imports `idb` to represent "an IndexedDB API" and is caught by the same external rule (both `dexie` and `idb` are on the disallow list). Local-first means all reads/writes go through Dexie in the browser; server actions operate on request-scoped data and delegate to repositories (Node-side persistence can be added later without changing the layering).

10. **Tailwind v4** (CSS-first config via `@import "tailwindcss"` + `@theme`, `@tailwindcss/postcss`) — current default for Next 15 scaffolds; no `tailwind.config.js` required.

### 5. `src/shared/env.ts` — boot-time validation
```ts
import { z } from 'zod'

const serverSchema = z.object({
  NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),
  LOG_LEVEL: z.enum(['fatal','error','warn','info','debug','trace','silent']).default('info'),
})
const clientSchema = z.object({
  NEXT_PUBLIC_APP_NAME: z.string().min(1).default('POS'),
})

const parsedServer = serverSchema.safeParse(process.env)
const parsedClient = clientSchema.safeParse(process.env)
if (!parsedServer.success || !parsedClient.success) {
  const issues = [
    ...(parsedServer.success ? [] : parsedServer.error.issues),
    ...(parsedClient.success ? [] : parsedClient.error.issues),
  ]
  // Throw at boot — fail fast with a readable message.
  throw new Error(`Invalid environment variables:\n${issues.map(i => ` - ${i.path.join('.')}: ${i.message}`).join('\n')}`)
}
export const env = { ...parsedServer.data, ...parsedClient.data } as const
```

## State Management Strategy

- **Server state / persistence (local-first):** IndexedDB via **Dexie**, encapsulated entirely in `src/infrastructure`. Domain **services** own business rules and depend only on repository **interfaces**; the **container** injects Dexie-backed implementations. No component or action touches storage directly.
- **Data flow into UI:** `app` (Server/Client Components) → Server **Actions** → **services** → **repositories** → Dexie. This single funnel is what the boundary rules enforce.
- **Client component state:** React local state (`useState`/`useReducer`) and Server Actions for mutations; no global client store added in this scaffold (add Zustand/Context later inside `app` only if needed). `revalidatePath`/router refresh for post-mutation UI updates.
- **Cross-cutting singletons:** `env`, `logger`, and the infrastructure `container` are module-level singletons; `Clock`/`IdGenerator` ports make services deterministic and testable.
- **Offline/PWA state:** the service worker caches the app shell for offline; Dexie holds domain data locally, keeping the app usable offline (local-first). Sync-to-server is out of scope for this scaffold but the layering leaves room for a sync adapter in `infrastructure`.
- **Result-typed errors:** services can return `Result<T,E>` (or throw `AppError`); actions map to serializable responses via `to-error-response`.

## Risks and Edge Cases

1. **Pino on the Edge runtime.** Pino targets Node; `middleware.ts` runs on the Edge runtime and must **not** import `logger.ts`. Mitigation: middleware uses `console`/edge-safe logging; add `serverExternalPackages: ['pino','pino-pretty']` so Node server components/actions bundle Pino correctly. Route handlers should declare Node runtime if they need Pino (`export const runtime = 'nodejs'`).
2. **ESLint 9 flat config × `eslint-config-next` shape.** If the installed version still ships eslintrc config, `FlatCompat` is required (as written); if it ships a native flat config, import it directly. Verify after install; adjust the one `extends`/import line. Also confirm `eslint-plugin-boundaries` v5+ (flat-config compatible) and that `eslint-import-resolver-typescript` resolves the `@/*` aliases (needed for boundaries to classify aliased imports).
3. **`boundaries/no-unknown` false positives.** Any `src/**` file not matching an element pattern errors. Mitigation: `boundaries/include` is scoped to the five known roots so framework files (`middleware.ts`, `next-env.d.ts`) are outside boundary analysis. Re-check when new top-level `src` folders are added.
4. **Fixture leaking into normal builds.** If the `__boundary_fixtures__` path is not excluded from ESLint ignores, `tsconfig`, and Vitest globs, `lint`/`typecheck`/`build`/`test` would fail on a fresh clone (breaking the "boots" criterion). Triple-exclude as specified; the boundary test re-includes it via `ignore: false`.
5. **`indexedDB` global vs import.** `boundaries/external` only catches module **imports** (`idb`, `dexie`). A raw `indexedDB.open()` uses a global, not an import, so `no-restricted-globals` covers that path (disabled in `infrastructure`). The acceptance fixture uses an **import** to hit the primary rule.
6. **Service worker caching in dev.** Registering the SW in development can serve stale assets. Mitigation: register only when `NODE_ENV === 'production'`; version the cache and clean old caches on `activate`; exclude `/sw.js` from middleware and from any caching that would prevent updates.
7. **Env "throw at boot" vs "fresh clone boots".** These conflict if any var is required. Mitigation: every var defaulted → absence never throws; only *invalid provided* values throw. Provide `.env.example`. The throw path is proven by the unit test, not by breaking boot.
8. **Server Actions importing the infrastructure container.** Actions pull the wired `container` from `@/infrastructure`. Ensure this stays a barrel-only import (optionally enforced by `boundaries/entry-point`) so actions cannot deep-import Dexie files, and so Dexie (browser IndexedDB) code isn't pulled into a server bundle inappropriately. If server-side persistence is later required, add a Node storage adapter behind the same repository interface rather than reusing the Dexie one.
9. **`exactOptionalPropertyTypes` / `noUncheckedIndexedAccess` friction.** These can clash with some Next/third-party types. If they cause noise beyond the app's own code, relax `exactOptionalPropertyTypes` first (keep `noUncheckedIndexedAccess`). Document any relaxation in `tsconfig` comments.
10. **Tailwind v4 content detection.** v4 auto-detects sources, but with `src/` + route groups verify utilities aren't purged; add `@source` directives in `globals.css` if any directory is missed.
11. **Playwright browsers not installed.** `test:e2e` fails on a clone until `npx playwright install`. Document in README; do not add it to `postinstall` (slow, and blocked-download policy sensitivity).
12. **Policy constraints during implementation.** Global installs, `curl`/`wget`, `git push`, and CI-file edits are denied by the environment. All installs must be local; no network fetch scripts; no `.github/**` changes. Keep the scaffold self-contained.
13. **`create-next-app` interactive flags in headless mode.** Some flags may still prompt; pass explicit flags for every option and pre-create files rather than relying on interactive selection.
