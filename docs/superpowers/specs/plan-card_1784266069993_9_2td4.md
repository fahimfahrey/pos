# Plan: domains/sales + /pos checkout screen

Card: `card_1784266069993_9_2td4`

## Overview

The repo already has a **placeholder** sales domain (`src/domains/sales/**`) built before the catalog and inventory domains existed: `Order`/`OrderLine` entities, an `orders`/`orderLines` collection pair, a thin `SalesService`, and a `completeSale` action that only attaches a payment to an existing order. Two facts in the current code confirm this was always meant to become the real sales/POS domain, not a parallel concept:

- `ENUM_REGISTRY_KEY.SALE_STATUS` is already registered and its static values are `Object.values(ORDER_STATUS)` (`src/constants/enums/registry.ts`).
- The Dexie index comment literally says `payments: 'id, orderId, ...' // orderId is the saleId per the card` (`src/infrastructure/storage/adapters/indexeddb/schema.ts`).

This plan **replaces** the `Order`-based scaffold with a real `Sale`/`SaleItem` model and builds the full checkout domain on top of it: `Shift`, `Sale`, `SaleItem`, `ParkedCart`, `ReceiptCounter`, a pure `money.ts` calculation library, finalize/void/hold/resume/shift server actions, and the full-screen `/pos` tablet UI (barcode capture, debounced search, cart editing, discounts with manager override, tax, customer attach, hold/resume, payment, receipt).

Finalization reuses the storage layer's existing whole-database transaction (`StorageProvider.withTransaction(fn)` → `driver.transaction(allCollections, 'readwrite', ...)`) as the UnitOfWork — this already gives atomic multi-collection commits (proven by the existing `unit-of-work.ts` conformance suite), so no new transaction abstraction is needed. What's new is the domain logic that runs inside it: idempotent sale creation by client ULID, price/tax snapshotting, gapless per-branch receipt numbering via a counter row read-modified-written in the same transaction, and stock decrement via the existing `InventoryService`.

## Architecture Decisions

1. **Rename `Order`→`Sale` instead of adding a parallel model.** The registry key `saleStatus` already aliases `ORDER_STATUS` and the Dexie schema comment already says "orderId is the saleId per the card." Keeping `Order` and adding `Sale` alongside it would create two competing concepts in the same collection group. The rename touches: entities, repository, core repository impl, indexeddb schema/version, `Payment.orderId`→`saleId`, and every fixture/test that used `makeOrder`. This is a larger diff than a purely additive approach but avoids permanent duplication.

2. **`SaleStatus` reuses `ORDER_STATUS` values (`draft|open|paid|void`) rather than a new enum.** POS finalize only ever produces `paid` directly (finalization is one atomic step, not a multi-stage draft-then-pay flow at the persistence layer — the "draft" stays client-side as cart state / `ParkedCart` until checkout). `void` is used by `void-sale.ts`. `draft`/`open` remain valid enum values (unused by the POS flow) so the existing `ENUM_REGISTRY_KEY.SALE_STATUS` admin UI and any future non-POS order flow keep working unchanged.

3. **No new UnitOfWork abstraction.** `StorageProvider.withTransaction(fn)` already opens one `driver.transaction(allCollections, 'readwrite', ...)` covering every collection, and the existing `unit-of-work.ts` conformance suite already proves rollback-on-throw across repositories. `FinalizeSaleService` is a plain class that receives an already-open `RepositorySet` and does all of its reads/writes through it — exactly the pattern `InventoryService` and `CatalogService` already use (`repos: { inventory: InventoryRepository }` parameter, no ambient transaction).

4. **Idempotent finalize by client ULID.** The client generates the `Sale.id` before calling `finalizeSale` (offline/tablet networks retry). `FinalizeSaleService.finalize` starts by doing `const existing = await repos.sales.findSaleById(input.saleId)`. If `existing && existing.status === SALE_STATUS.PAID`, return `{ sale: existing, receiptNumber: existing.receiptNumber, alreadyFinalized: true }` immediately — no counter increment, no second stock movement, no second payment. This satisfies "same ULID finalized twice yields one sale" without needing a unique-constraint primitive on the driver (none exists — see `CatalogService`'s SKU-uniqueness precedent, which does the same check-then-act-inside-one-transaction pattern).

5. **Gapless receipt numbers via a counter row inside the same transaction.** `ReceiptCounter` is keyed by `branchId`. `getAndIncrementReceiptCounter(branchId)` does `tx.get` → default `{nextNumber: 1}` if absent → `tx.put({nextNumber: n+1})` → returns `n`. Because this happens inside the exact same `driver.transaction(...)` as the `Sale`/`SaleItem`/`Payment`/`StockMovement` writes, a rollback (e.g. `InsufficientStockError` thrown after the counter read but before commit) rolls back the counter increment too — no gap is created by a failed sale. This is the mechanism the "conformance rollback test" (`finalize-sale-atomicity.ts`) verifies directly.

6. **Server-authoritative pricing and tax at finalize time.** The client cart carries `variantId` + `quantity` + any requested discounts; it does **not** dictate `unitPrice` or `taxRate`. `FinalizeSaleService` re-resolves `unitPrice` via `CatalogQueryService.resolveEffectivePrice` and `taxRate` via the new `tax-resolver.ts` at finalize time, and snapshots those resolved values onto `SaleItem`. This is what "sale items snapshotting price/tax/name" means operationally: the snapshot is the source of truth from the moment of sale forward, immune to later catalog/price-list edits, but it is *computed* server-side, not merely copied from client input (prevents a compromised/buggy client from checking out at an arbitrary price).

7. **`money.ts` lives at `src/domains/sales/lib/money.ts`, not `src/shared/`.** It has zero I/O and zero repository/React dependencies (pure functions over `Money`, `DiscountType`, `TaxMode`, `RoundingRule` — all already-existing plain types), so it is testable in complete isolation exactly like `barcode-encoder.ts` in the catalog domain. It is domain-owned because discount/tax/rounding semantics are POS business rules, not generic shared utilities — but nothing prevents `payments`/`purchasing` from importing it later via `@domains/sales/lib/money` if refunds need the same rounding rules.

8. **Money arithmetic is done in integer minor units internally.** `Money.amount` stays a `number` of major units at the interface boundary (matching the existing `Money` type used everywhere else in the codebase), but every `money.ts` function immediately converts to integer cents (`Math.round(amount * 100)`), does all arithmetic in integers, and converts back on return. This eliminates float drift (`0.1 + 0.2` class bugs) for the rounding/discount/tax test matrix required by the acceptance criteria. Documented limitation: assumes 2-decimal currencies (USD-like); a 0-decimal or 3-decimal currency would need a currency-aware minor-unit factor — out of scope for this card, noted in Risks.

9. **Cart-level discount is distributed across lines by proportional share with largest-remainder rounding**, not applied only to the total. This is required because tax must be computed per-line (mixed tax rates), so a cart discount has to reduce each line's taxable base before that line's tax is computed, not just be subtracted from the grand total at the end. Largest-remainder allocation guarantees the per-line discount amounts sum exactly to the requested cart discount (no penny leaks), which is what the "discounts" unit test matrix checks.

10. **Manager override is a scoped approval token, not a session role upgrade.** Reusing `AuthService.reauthWithPin` as-is would upgrade the *cashier's own session* if the cashier enters their own PIN — wrong actor model. Instead `manager-override.ts` verifies the entered PIN belongs to a membership with `role <= MANAGER` in the current org (via `OrganizationRepository.findMembership` + `hasAtLeast`), and if valid returns `{ approvedBy: managerUserId, expiresAt: now + 2min }` **without touching the cashier's session cookie**. The client attaches this token to the specific discount; `FinalizeSaleService` re-validates the token's `approvedBy` role and expiry server-side before honoring an over-limit discount (never trust a client-side "approved" flag alone).

11. **`Category.taxRuleId` is an additive optional field**, not a new tax-mapping table. Given no existing entity currently maps products to tax rates, and building a full tax-rule assignment UI is out of scope for this card, the minimal correct addition is one optional field on `Category` plus a fallback to `settings.taxRules[0]`. This is enough to exercise "mixed tax rates" end-to-end (two categories with different `taxRuleId`s) without inventing new domain surface area.

12. **Barcode capture uses a hidden, globally-refocused `<input>` with an inter-keystroke timing heuristic**, not the Web Serial/HID API (USB "HID scanners" in retail almost always enumerate as a keyboard wedge, not a raw HID device the browser can claim). The hook listens on the hidden input's `onKeyDown`/`onChange`, buffers characters, and treats a burst where consecutive keystrokes arrive under a `30ms` threshold and terminate in `Enter` as a scan; anything slower is treated as human typing and ignored (the visible search box handles human typing). The hidden input is refocused on `blur` and on `window focus`, *except* while a modal (`discount-modal`, `manager-override-modal`, `payment-modal`) is open, which claims focus for its own inputs — the hook exposes `pause()`/`resume()` for this.

- **Cart state lives in a `useReducer` + React Context** (`pos-cart-context.tsx`), scoped to the `/pos` route tree only. Actions: `ADD_LINE`, `SET_QUANTITY`, `REMOVE_LINE`, `SET_LINE_DISCOUNT`, `SET_CART_DISCOUNT`, `ATTACH_CUSTOMER`, `RESET_CART`, `LOAD_PARKED_CART`. The reducer is pure and holds only *inputs* (`variantId`, `quantity`, requested discounts) — never computed totals; totals are always derived by calling `price-cart.ts` (server action) after each mutation, debounced 150ms for rapid quantity edits, so the cart never displays a locally-computed price that could drift from server truth.
- **A ref-held client ULID is generated once per new cart** (on mount / on `RESET_CART`) via `generateUlid()` and stored in reducer state as `Sale.id`-to-be. This is the value sent to `finalizeSale`; regenerated only when the cart is explicitly cleared or a parked cart is resumed (which carries its own prior id forward, so resuming and finalizing a previously-attempted hold is also idempotent).
- **Cart survives a reload** via `sessionStorage` (register-scoped key `pos-cart:${registerId}`), written on every reducer transition, hydrated on mount. This is distinct from the explicit `ParkedCart` hold feature — `sessionStorage` is crash/reload recovery for the *current* in-progress cart, `ParkedCart` is the deliberate "come back to this customer later" feature that shows up in the hold/resume drawer for *other* registers/shifts too.
- **Shift/register context** is server-fetched once in `pos/page.tsx` and passed down as props / a small non-reducer Context (it doesn't change within a session except by navigating to close-shift, which is a full navigation).
- **Manager-override token** is held in the reducer alongside the specific discount it approved (`{ discountId, approvedBy, expiresAt }`), cleared automatically when that discount is removed or edited (a new discount value requires a fresh override).

## Accessibility Considerations

- **Hidden barcode input is not `display:none`** (browsers can drop focus/events on fully hidden elements in some scanner-emulation modes) — use visually-hidden-but-focusable styling (`sr-only`-equivalent: `absolute w-px h-px overflow-hidden` with `opacity-0`, not `hidden`), with `aria-hidden="true"` and `tabIndex={-1}` so it's invisible to screen readers and never part of the normal tab order, while remaining a valid focus target for the imperative `.focus()` calls the hook makes.
- **All primary actions are reachable by keyboard**: product search results navigable with arrow keys + Enter, cart lines editable via Tab, quantity steppers have visible `+`/`-` buttons *and* accept direct numeric typing, discount/payment/override modals trap focus (first focusable element on open, `Escape` closes where safe, focus returns to the triggering control on close).
- **Touch targets ≥ 44×44px** for all `/pos` buttons (quantity steppers, numeric keypad keys, payment tender buttons) per WCAG 2.5.5 — tablet-first requirement from the card.
- **Live regions**: cart total changes and error/validation messages use `aria-live="polite"` (matching the existing `role="alert" aria-live="polite"` pattern already used in `create-enum-value-form.tsx`); the manager-override PIN pad uses `aria-live="assertive"` only for the final success/failure result, not per-keystroke.
- **Color is never the only signal** — insufficient-stock, discount-limit-exceeded, and payment-error states pair color with an icon + text, not color alone (contrast checked against the existing `globals.css` focus-ring/theme tokens).
- **`*:focus-visible` outline already defined globally** (`src/app/globals.css`) — no `/pos` component should suppress it; if a custom-styled control needs a different focus treatment it must still meet a 3:1 contrast ratio against its background.
- **Receipt/print view** (`receipt-view.tsx`) uses semantic table markup for line items (not divs) so screen readers and print CSS both get sensible structure for free.

## Test Plan

### `money.ts` unit tests (`src/domains/sales/lib/money.test.ts`)
- **Rounding**: nearest/up/down at increments `0.01`, `0.05`, `0.10`; negative amounts; already-exact amounts are no-ops.
- **Discounts**: percentage discount on a line; fixed-amount discount clamped so it never exceeds the line subtotal (can't go negative); cart-level percentage/fixed discount distributed across N lines with largest-remainder allocation summing exactly to the requested amount (assert `Σ line.discountAmount === cartDiscount.amount`, cent-exact); zero-subtotal line with a discount requested (no-op, no divide-by-zero).
- **Tax — exclusive mode**: tax computed on (line subtotal − line discount − allocated cart discount), added on top; total = net + tax.
- **Tax — inclusive mode**: given a tax-inclusive line total, the net/tax split is extracted correctly (`net = total / (1+rate)`), and re-adding net+tax reproduces the original total to the cent.
- **Mixed tax rates**: a cart with lines at 0%, 5%, and 15% rates produces correct per-rate tax buckets and a correct summed `taxTotal`; changing rounding rule doesn't change which bucket a cent lands in incorrectly.
- **Currency mismatch guard**: mixing two different `Money.currency` values in one cart throws.
- **Change-due calculation**: tendered amount − total, clamped at 0 when tendered < total (should be prevented upstream, but the pure function itself must not return a negative "change due").

## Risks and Edge Cases

- **Schema rename is a breaking migration for any existing dev/local IndexedDB data.** The v6 Dexie migration drops `orders`/`orderLines` outright (matching how v4 dropped `catalogItems`) — any existing local dev data in those stores is lost. Acceptable pre-launch (no production data), but call it out in the PR description.
- **Two-decimal-currency assumption in `money.ts`.** If the org's `settings.currency` is ever a 0- or 3-decimal currency (e.g. JPY, KWD), the `×100`/`÷100` minor-unit conversion is wrong. Out of scope for this card (org currency is fixed at USD-style throughout the current codebase), but document the limitation directly in `money.ts`'s module comment so it's not silently wrong later.
- **`Category.taxRuleId` fallback to `settings.taxRules[0]`** means an org with zero configured `taxRules` (the `DEFAULT_SETTINGS` default is `taxRules: []`) resolves every sale to 0% tax with no explicit warning. Acceptable default behavior, but `tax-resolver.test.ts` must assert this explicitly so it's a documented, tested default rather than an accident.
- **Manager override token is purely time-boxed (2 min), not tied to a specific `saleId`/discount amount beyond what's re-validated at finalize.** A cashier could in principle get one override and reuse it for a second, different over-limit discount within the window if the client doesn't clear it correctly. Mitigated by clearing the token in the reducer whenever the discount it approved changes (see State Management Strategy) *and* by finalize-time re-validation, but a determined client could bypass the UI-level clearing — not a real risk since finalize re-checks the approver's role and expiry server-side regardless of what discount amount is presented, but it does **not** re-check that the token was issued for *this specific* discount value. Flag as a known gap; closing it fully would require binding the token to a hash of the discount request, which is a reasonable follow-up but adds scope beyond this card.
- **Barcode-vs-typing heuristic (30ms threshold) is inherently a heuristic.** Slow/cheap scanners or Bluetooth (as opposed to USB HID keyboard-wedge) scanners can inject characters slower than 30ms apart, misclassified as typing. Make the threshold a constant in `use-barcode-scanner.ts` so it's a one-line tuning knob, and note in the hook's comment that this assumes USB HID keyboard-wedge scanners specifically (matches the card's explicit "USB/HID barcode scans" wording).
- **Weight entry has no scale integration** — the card says "weight entry," not "scale integration"; this plan implements manual decimal quantity entry only for `isDecimalQuantity` variants. If a connected-scale requirement surfaces later, it's additive (another input source feeding the same `SET_QUANTITY` reducer action).
- **`sessionStorage` cart recovery is per-tab.** If a tablet's browser is fully closed (not just backgrounded), `sessionStorage` is cleared and the in-progress cart is lost — this is a deliberate scope boundary versus the explicit `ParkedCart` hold feature, which does persist durably; document this distinction for support/training purposes.
- **`payments.saleId` rename is a breaking field rename** on an entity another domain (`refunds`, keyed by `paymentId`, is unaffected) already touches — grep for every remaining `orderId` reference across `src/` before considering the rename step done, since a stale reference would fail silently (TypeScript will catch renamed-field misses at compile time given `strict: true`, but stringly-typed Dexie index strings in `indexeddb/schema.ts` will not be caught by the compiler — verify by running the conformance suite, not just `tsc`).
- **`pin-reauth.ts` (existing file) calls `provider.getRepositorySet(tx)` inside `provider.withTransaction(async (tx) => ...)`**, but `StorageProvider`'s `withTransaction` callback signature is `(repositories: RepositorySet) => Promise<T>` and no `getRepositorySet` method exists on `StorageProvider` per `src/infrastructure/storage/core/storage-provider.ts`. This looks like a pre-existing bug unrelated to this card. Do **not** copy this pattern for `manager-override.ts` — use the verified-working `provider.withTransaction(async (repos) => ...)` pattern from `complete-sale.ts` instead. Flag the `pin-reauth.ts` issue to the team but do not fix it as part of this card (out of scope).
