## Overview

This card lays the **engine-swappability foundation** — interfaces and one reference implementation, **no UI**. Three things get built:

1. **Repository interfaces for every bounded context** — `catalog, inventory, sales, payments, customers, purchasing, promotions, organization, auth/users, audit` — each in a canonical `src/domains/<context>/repository.ts`, with **narrow, intent-named methods** and **plain-TypeScript entity types** (no Dexie/IndexedDB/engine types anywhere). `sales` and `inventory` already have interfaces under `repositories/*-repository.ts`; those are **relocated** to `repository.ts` to establish one convention.

2. **Engine-agnostic storage core** in `src/infrastructure/storage/core/`:
   - **`StorageProvider`** — the bound result of the factory: a repository set + transaction/UnitOfWork + migrate + export/import + close, all for the engine named in config.
   - **`createStorageProvider(config)`** — the factory: resolves an engine name (`env.NEXT_PUBLIC_STORAGE_ENGINE`, e.g. `indexeddb`) to a registered adapter and returns a `StorageProvider`.
   - **`UnitOfWork`** — `begin/commit/rollback` semantics; **all repository writes happen inside a transaction scope** (buffered write-set, committed atomically).
   - **`SchemaVersioner`** — ordered, **engine-agnostic migration steps** (declarative `SchemaStep`s) that each engine adapter interprets (with a generic default interpretation in core).
   - **Canonical JSON export/import contract** — `exportAll()` / `importAll()` with **per-entity versioned envelopes**, lossless for `Date`, used for backup **and** for migrating data between engines.

3. **An IN-MEMORY reference adapter** (`src/infrastructure/storage/adapters/memory/`) implementing the full contract, plus **`docs/storage-adapters.md`** (what a new engine must implement + effort target), and a **first conformance test** the memory adapter passes.

**Key architectural lever (how "one adapter, one config change, nothing else" is achieved):** the adapter surface is deliberately tiny — a new engine implements only a **`StorageDriver`** primitive (a per-collection key/value store + an atomic `transaction` + a persisted schema-version slot). **Core** builds *everything domain-facing* generically on top of that primitive: the concrete `RepositorySet` (all 10 interfaces), the `UnitOfWork`, `transaction()`, `migrate()`, and `exportAll/importAll`. So a new engine is one `StorageDriver` implementation + one registry line.

**Acceptance:** the in-memory adapter implements the full `StorageProvider` contract and passes `tests/storage/memory-adapter.test.ts` (a reusable conformance suite exercising CRUD across contexts, transaction commit/rollback atomicity, UnitOfWork, schema migration, and export/import round-trip incl. `Date` fidelity).

Layering (unchanged from card 0; storage core lives entirely in `infrastructure`):

```
app → actions → services → repositories (INTERFACES: src/domains/*/repository.ts) / entities
                                   ▲
infrastructure/storage  ── implements the interfaces generically over a per-engine StorageDriver ─┘
shared / constants  ← importable by any layer
```

## Architecture Decisions

1. **Canonical interface location `src/domains/<ctx>/repository.ts`.** Honors the task literally and unifies the two pre-existing interfaces (`sales`, `inventory`) that lived under `repositories/*-repository.ts`. Relocation is mechanical (5 import sites, all enumerated). The `repositories` boundary element pattern is extended to an array so both the historical folder form and the new file form classify correctly and `boundaries/no-unknown` stays green.

2. **Entities stay in `src/domains/<ctx>/entities/` (plain TS), `repository.ts` imports them.** "Entity types (no engine types)" is interpreted as *plain-TypeScript, engine-free* types — the qualifier is about purity, not location. Keeping entities in the existing `entities` element preserves the boundary model (`repositories → entities` allowed) and matches sales/inventory. Every entity type is a plain interface of primitives / `Money` / enum unions / `Date` — **no Dexie `Table`, no IndexedDB types**.

3. **Tiny adapter surface (`StorageDriver`), fat core.** The single biggest design choice. A new engine implements only:
   ```ts
   interface CollectionStore {
     get(collection: string, id: string): Promise<unknown | undefined>
     getAll(collection: string): Promise<unknown[]>
     put(collection: string, record: unknown): Promise<void>
     delete(collection: string, id: string): Promise<void>
     clear(collection: string): Promise<void>
   }
   interface StorageDriver {
     readonly store: CollectionStore
     ensureCollection(name: string, keyPath: string): Promise<void>
     transaction<T>(work: (store: CollectionStore) => Promise<T>): Promise<T> // atomic: rollback on throw
     getSchemaVersion(): Promise<number>
     setSchemaVersion(version: number): Promise<void>
     close(): Promise<void>
     applySchemaStep?(step: SchemaStep): Promise<boolean> // return true if handled; else core applies generically
   }
   ```
   Everything the domain sees — the concrete `RepositorySet` (10 interfaces), `UnitOfWork`, `transaction()`, `migrate()`, `exportAll/importAll` — is built once, generically, in core. This is what makes the effort target ("one adapter, one config change, nothing else") real.

4. **Generic repositories over `CollectionStore`.** `create-repositories.ts` implements all 10 interfaces by mapping intent-named methods to store ops. Simple queries (`listOpen`, `findByEmail`, `findByCode`, `listByEntity`) are `getAll` + a typed JS predicate. This is correct for *any* engine; engines may later optimize inside their `CollectionStore` using native indexes without changing the repositories. Records are stored as the **native entity** (with `Date` objects) — canonicalization happens *only* at the export/import JSON boundary.

5. **`UnitOfWork` + `transaction()` via a buffered write-set — engine-agnostic atomicity.** `BufferedCollectionStore` wraps the driver's live store: reads fall through (overlaid with buffered writes), writes/deletes accumulate in-memory. `transaction(work)` runs `work` against repositories bound to a buffer, then flushes the buffer inside a single `driver.transaction`; a throw in `work` discards the buffer (**rollback**) and never touches the store. `createUnitOfWork()` exposes the same machinery imperatively (`begin` = start buffer, `commit` = flush atomically, `rollback` = discard). This satisfies "begin/commit/rollback semantics with all repository writes inside a transaction scope" on **every** engine, including those whose native transactions are callback-scoped (IndexedDB) — the driver only needs a "apply these writes atomically" primitive.

6. **`SchemaVersioner` = ordered declarative `SchemaStep`s each adapter interprets.** A `Migration` is `{ version, name, steps: SchemaStep[] }`. `SchemaStep` is a closed union: `createCollection`, `dropCollection`, `renameCollection`, `addIndex`, `dropIndex`, `backfill` (carries a **pure** `(record: JsonRecord) => JsonRecord`). `createSchemaVersioner` validates versions are unique, ascending, and start at 1. `migrate()` reads `driver.getSchemaVersion()`, takes `pending(current)`, and for each step calls `driver.applySchemaStep?(step)`; if the driver returns `false`/undefined, core applies a **generic default** (`createCollection → ensureCollection`; `backfill → transaction(getAll → map → put)`; `addIndex/dropIndex → no-op` — indexes are optional perf hints), then `setSchemaVersion(migration.version)`. Steps are pure data + pure functions → genuinely engine-agnostic; adapters that *want* native behavior (IndexedDB object stores/indexes) override via `applySchemaStep`.

7. **Canonical JSON export/import with per-entity versioned envelopes.**
   ```ts
   const EXPORT_FORMAT_VERSION = 1
   interface EntityEnvelope<T = JsonValue> { entity: string; version: number; records: T[] }
   interface StorageExport {
     formatVersion: number      // envelope container format
     engine: StorageEngineName  // producer (informational)
     schemaVersion: number      // overall schema version at export time
     exportedAt: string         // ISO
     entities: EntityEnvelope[] // one envelope per ENTITY_REGISTRY collection
   }
   interface ImportOptions { mode?: 'replace' | 'merge'; migrate?: boolean } // defaults: 'replace', true
   ```
   `exportAll()` iterates `ENTITY_REGISTRY`, reads each collection, and emits an envelope stamping that entity's current `version`, with `records` run through `toCanonical` (`Date → {$date}`). `importAll()` validates with `storageExportSchema`, rejects `formatVersion > EXPORT_FORMAT_VERSION` or any envelope `version >` the current registry version (forward-incompatible), `fromCanonical`-revives records, then (`replace`) `clear`+`put` or (`merge`) `put`, all inside one transaction. Because envelopes are plain JSON keyed by canonical entity names, **`exportAll()` from engine A → `importAll()` into engine B** is a data migration — the same path used for backup. Per-entity `version` lets a future entity bump trigger a targeted transform on import.

8. **`Date` fidelity via `canonical-json`.** Entities carry `createdAt`/`updatedAt`/`expiresAt` as `Date`. `JSON.stringify` would silently degrade these to strings and break the entity contract on round-trip. `toCanonical` recursively rewrites `Date` → `{ "$date": iso }`; `fromCanonical` reverses it (and guards against a literal `$date` collision on non-date objects). Storage-internal reads/writes keep native `Date` (memory: `structuredClone` preserves it; IndexedDB: structured-clone-native).

9. **SSR safety / lazy engine binding.** `createStorageProvider` is an **async factory that instantiates nothing at module load**. The `indexeddb` engine (future card) is browser-only; nothing in this subsystem may open a DB at import time. The memory engine is universal (used by tests + as a safe default fallback). No storage provider is wired into the app `container` in this card — the subsystem is parallel and self-contained; a later card swaps `container.ts` onto `createStorageProvider()` and adds an IndexedDB `StorageDriver`.

10. **`STORAGE_ENGINE` config knob = `NEXT_PUBLIC_STORAGE_ENGINE`.** Local-first: the engine choice is a **client** concern (IndexedDB lives in the browser), so it must be a `NEXT_PUBLIC_` var to be readable in the browser bundle. Default `'indexeddb'` per the task's example; the conformance test passes `'memory'` **explicitly**, so the default never affects tests, and since no app code resolves the provider this card, the default can't break boot. When the IndexedDB driver lands, it registers in `adapters/registry.ts` — no other change.

11. **`strict` + `noUncheckedIndexedAccess` + `verbatimModuleSyntax` compliance.** All type-only imports use `import type`. `Map.get` / array indexing returns `T | undefined` and is narrowed before use. No TS `enum` (reuse existing `as const` unions: `OrderStatus`, `PaymentMethod`, `UserRole`).

12. **Append-only `audit`.** `AuditRepository` exposes `append`, `getEntry`, `listRecent`, `listByEntity` — **no update/delete** — encoding audit-log immutability in the interface.

- **`create-storage-provider-from-driver.ts`** — assembles:

  - `migrate()`: loop `schema.pending(await driver.getSchemaVersion())`, apply each step (driver hook else generic), `setSchemaVersion(version)`.

### 8. Memory adapter
- **`memory-driver.ts`**: `collections: Map<string, Map<string, unknown>>`; `version: number`. `ensureCollection` lazily creates a `Map`. `store.get/getAll/put/delete/clear` operate on the inner map, using `structuredClone` on write and on read (isolation from callers). `transaction(work)`: snapshot = `structuredClone`-deep-copy of `collections` (+ version); run `work(this.store)`; on throw, restore snapshot then rethrow. `get/setSchemaVersion` read/write `version` (persisted in a dedicated `__meta__` collection so `exportAll` need not special-case it — or keep it a plain field; field is simpler, exclude from registry).
- **`memory/index.ts`**: `export const createMemoryStorage: StorageAdapterFactory = async ({ schema }) => { const driver = new MemoryStorageDriver(); const provider = createStorageProviderFromDriver('memory', driver, { schema: schema ?? schemaVersioner }); await provider.migrate(); return provider }`.

- **What an adapter is** — one `StorageDriver` implementation + one line in `adapters/registry.ts`. Nothing else. Repositories, UnitOfWork, migrations, and export/import are provided by core.
- **The `StorageDriver` contract** — annotated interface (each method's obligations: `store` isolation/serialization expectations; `ensureCollection` idempotent; `transaction` atomic with rollback-on-throw; `getSchemaVersion` returns `0` when unset; optional `applySchemaStep` for native schema ops).

## State Management Strategy

- **Persistence is fully behind `infrastructure/storage`.** Domains depend only on `src/domains/*/repository.ts` interfaces; the concrete `RepositorySet` is produced generically by core over an engine `StorageDriver`. No domain/app/action code references an engine.
- **Transactional writes.** All multi-write operations go through `provider.transaction(repos => …)` or a `UnitOfWork`; both use a buffered write-set flushed atomically via `driver.transaction`, giving real commit/rollback on every engine. A throw inside the scope leaves the store byte-for-byte unchanged.
- **Schema state.** The persisted schema version lives in the engine (driver `get/setSchemaVersion`). `migrate()` advances it through ordered declarative migrations; adapters interpret steps (native) or fall back to core's generic interpretation.
- **In-memory reference state.** `Map<collection, Map<id, entity>>` with `structuredClone` isolation; transactions snapshot/restore the whole map — deterministic, dependency-free, ideal for tests and as the safe default engine.
- **Portability/backup state.** The canonical `StorageExport` (per-entity versioned, `Date`-lossless envelopes) is the single serialization used for both backup and engine-to-engine migration; it is engine-independent by construction.
- **Env/config singletons.** `env` (incl. `NEXT_PUBLIC_STORAGE_ENGINE`) and `STORAGE_ADAPTERS` are module-level; the provider itself is created lazily via the async factory (no import-time engine binding — SSR-safe). No React/global client store is introduced (no UI in scope).

## Accessibility Considerations

**No UI, components, routes, or user-facing markup are produced by this card**, so there are no direct WCAG/ARIA obligations. The choices that keep future UI accessible:

- **Structured, typed errors over strings.** `createStorageProvider` throws precise, human-readable messages (`Storage engine "x" is not registered`; import validation errors from the Zod schema) so a future UI can surface them via `aria-live`/inline field errors rather than opaque failures.
- **Deterministic, labelable enums.** Entity states (`PaymentStatus`, `PurchaseOrderStatus`, `PromotionKind`, reused `OrderStatus`/`PaymentMethod`/`UserRole`) are `as const` unions — they map cleanly to the existing `constants/maps/*-labels.ts` pattern for accessible, localizable display text when UI arrives.
- **Backup/restore UX readiness.** `exportAll/importAll` return/accept plain, serializable data and report progress/failure explicitly, enabling an accessible, announceable backup flow later.
- When storage is eventually surfaced (settings/backup screens), apply the `dataviz` skill for any storage/usage charts and standard form-a11y (labels, `aria-describedby`, focus management) — out of scope here.

## Risks and Edge Cases

1. **Relocating `sales`/`inventory` interfaces breaks imports.** Mitigation: all 5 sites enumerated (grep-verified); update in the same change; `typecheck` catches a miss.
2. **`boundaries/no-unknown` on the new `repository.ts`.** Without the pattern change, `src/domains/*/repository.ts` matches no element → error. Mitigation: extend the `repositories` element to an array pattern (Step 2); confirm via `npm run lint` and the boundary test.
3. **`Date` silently degraded on export.** A naive `JSON.stringify` turns `Date` into a string, breaking entity types on import. Mitigation: `canonical-json` `$date` tagging + revival; covered by a dedicated test and the round-trip case. Edge: an entity legitimately storing `{ $date: 'literal' }` as data — guard `fromCanonical` to only revive objects whose **sole** key is `$date` **and** whose value is a valid ISO string; document the reserved key.
4. **Callback-scoped native transactions (IndexedDB, future).** An imperative `UnitOfWork.begin/commit` can't straddle a scoped IDB transaction across awaits. Mitigation: the buffered write-set — `commit` flushes the whole buffer inside a single `driver.transaction`; the driver only needs "apply these writes atomically," which every engine supports.
5. **Snapshot cost / aliasing in the memory driver.** Deep `structuredClone` on every read/write and per-transaction snapshot is O(n) but fine for a reference/test engine; the isolation prevents callers from mutating stored state (tested). Note `structuredClone` availability (Node ≥ 17 / modern browsers — satisfied by `engines.node >= 18.18`).
6. **`STORAGE_ENGINE` default `'indexeddb'` with no IDB adapter yet.** Resolving the default before that adapter exists throws `is not registered`. Mitigation: no app code resolves the provider this card (parallel subsystem); the conformance test passes `'memory'` explicitly; documented that registering the IDB driver is the only change needed later. If a cleaner default is preferred, set `'memory'` — but the task's example is `indexeddb`, so keep it and rely on the clear error.
7. **`env` change breaking `env.test.ts`.** Adding a **defaulted** client var keeps empty-env parsing valid. If that test asserts the exact `env` object shape, update it to include `NEXT_PUBLIC_STORAGE_ENGINE`. (Its current form parses valid/invalid inputs — a default won't fail it.)
8. **`NEXT_PUBLIC_` bundle exposure.** The engine name is non-secret and intentionally client-visible (browser picks the engine) — acceptable.
9. **Cross-context entity coupling.** New contexts share `Money` via `src/shared/types` (allowed `entities → shared`), not by importing another domain's entities — avoids illegal `entities → entities` cross-context edges and keeps contexts decoupled.
10. **Generic `getAll`+filter query performance.** Correct for all engines but O(n). Acceptable for a foundation; engines may later optimize inside their `CollectionStore` using native indexes without touching repositories (that's the point of the split). `addIndex` steps exist in the schema vocabulary precisely to enable this later.
11. **`verbatimModuleSyntax`/`isolatedModules`.** Every type-only import must be `import type`; re-exporting types from barrels must use `export type`. `typecheck` enforces.
12. **`noUncheckedIndexedAccess`.** `Map.get`, first-element access, and envelope lookups return `T | undefined`; narrow before use (memory driver, `create-repositories`, import path).
13. **Import atomicity.** `importAll` clears+writes many collections; a mid-import failure must not leave a half-imported store. Mitigation: perform the whole import inside one `driver.transaction`; validate the payload **before** any mutation.
14. **Conformance suite not double-collected.** `conformance-suite.ts` has no `.test.ts` suffix, so Vitest's `include` won't run it standalone (only via the `*-adapter.test.ts` importer).
