# Plan — UI/interaction audit doc (`docs/ui-audit.md`)

## Overview

Task 3 in the design-direction series (follows task 1: `docs/design-direction.md` + seed tokens in `src/app/globals.css`; task 2: `docs/design-direction-pos.md`, the register adaptation of task 1). This card is **documentation-only**: produce `docs/ui-audit.md`, a single inventory of every existing screen, component, interaction pattern, and keyboard shortcut in the codebase, cross-checked against the two design-direction docs, written for two personas — the **cashier** (register/floor, tablet-class device) and the **owner/manager** (back office, desktop-class device).

Research for this plan already walked the live tree (`src/app/**`, `src/domains/**`, `IMPLEMENTATION_PROGRESS.md`, `IMPLEMENTATION_STATUS.md`, `IMPLEMENTATION_PAYMENTS.md`) and confirmed two facts that must anchor the audit and must not be glossed over:

1. **No register/checkout screen exists yet.** `src/app/pos/` contains only `display/` (customer-facing second-screen mirror) and `receipt/[saleId]/` (printable receipt view) plus a shared `_lib/use-cart-broadcast.ts` hook. There is no `/pos/checkout`, no `cart.tsx`, no `discount-modal.tsx`, no `manager-override-modal.tsx`, no barcode-scanner hook. `IMPLEMENTATION_PROGRESS.md` (lines 140–154, 190–197) lists all of these explicitly as **not yet implemented** UI. `docs/design-direction-pos.md`'s own "What This Document Does NOT Cover" section says the same: "no `/pos/checkout` route exists yet in the codebase."
2. **No screen in the app currently uses the new design tokens.** `login/page.tsx`, `signup/page.tsx`, `(dashboard)/app/layout.tsx`, `admin/layout.tsx`, and `pos/display/_components/display-cart-view.tsx` all use raw Tailwind utility classes (`bg-blue-600`, `text-blue-600`, `bg-gray-50`, `border-gray-300`, `bg-gray-900`) — the exact "generic SaaS blue" the anti-goals in `docs/design-direction.md` reject. Only `src/app/globals.css` (token definitions) and `src/app/layout.tsx` (Fraunces/Inter font wiring via `next/font/google`) have been touched by tasks 1–2. Every screen is therefore a "gap to direction" by default; the audit's job is to say so per-screen with specifics, not to assume partial adoption.

The audit doc must make both facts impossible to miss, because they change what "preserve" means for the four conventions named in the task (scan-input focus, checkout shortcut keys, hold/resume flow, discount permission gates): all four exist only as **domain/service logic or data models today, not as UI**. The audit must document them as architectural commitments to honor when the UI is built, not as existing interaction patterns to leave untouched.

No `.agents/skills/` directory exists anywhere in this repository (verified via `find . -iname "*skills*"` from repo root — zero hits, and no `find-skills` tool is registered in this environment). There is therefore no project skill whose guidance applies to this card's execution; the implementer should rely on the research method already used to build this plan (`ctx_compose`/`ctx_search`/`ctx_tree` from the lean-ctx MCP server, or plain `grep`/`find` if lean-ctx is unavailable) rather than skip verification.

## Files To Create or Modify

- **`docs/ui-audit.md`** (new) — the sole deliverable. One file, not a per-screen split, so the cross-check against the two design docs stays a single readable pass for reviewers. Structure detailed in Implementation Steps.

No other file is created or modified. This is documentation; nothing under `src/`, `e2e/`, or `docs/design-direction*.md` changes.

## Architecture Decisions

- **One file, table-driven.** Screens, components, and conventions are large enumerable sets — tables (screen | route | persona | current styling | gap-to-direction) are more scannable than prose and easier to keep accurate on the next design-direction revision. Follow the table conventions already used in `docs/design-direction.md` and `docs/design-direction-pos.md` (pipe tables, a rationale/number column) so the three docs read as one series.
- **Ground every "preserve" claim in a file:line citation.** Given finding #1 above, several of the "conventions to preserve" named in the task brief do not exist as UI. Each entry in the "Conventions to Preserve" section must state its current implementation status as one of three levels — **UI-implemented** (cite the component), **domain-only** (cite the service/entity, note no UI wiring), or **not yet built** (cite the `IMPLEMENTATION_PROGRESS.md` line marking it pending) — rather than asserting a single blanket status for all four.
- **Persona tagging per screen, not a separate persona section.** The task asks the audit to "reference" the cashier and owner/manager personas. Rather than writing a standalone persona narrative (redundant with what a screen inventory already shows), tag every screen row with which persona it serves (Cashier / Owner-Manager / Both / Public) so the persona lens is baked into the inventory itself.
- **Anti-goals reused verbatim, not re-derived.** The "gap to direction" column must cite the specific anti-goal from `docs/design-direction.md`'s Anti-Goals list (e.g., "NOT pure white and blue") that each screen currently violates, rather than writing new subjective gap language. This keeps the audit traceable to task 1's source of truth.
- **No proposed fixes.** This card is an inventory, not a redesign; each gap note states *what's wrong relative to the direction docs*, not *what to change it to*. Prescribing colors/spacing here would pre-empt a future implementation card and risks drifting from whatever that card actually decides.

## Implementation Steps

1. **Re-verify the live tree immediately before writing** (state may have moved since this plan was drafted — Next.js route groups and `_components` folders are easy to miss with a shallow listing). Run:
   - `ctx_tree` (or `find src/app -type f -name "*.tsx"`) over `src/app` to enumerate every `page.tsx`, `layout.tsx`, and `_components/*.tsx`.
   - `ctx_search`/`grep -r` for `onKeyDown|keydown|autoFocus|useHotkeys|shortcut` across `src/` — confirm the zero-shortcut finding still holds before stating it as fact in the audit.
   - `grep -rl "bg-blue\|bg-gray\|text-blue\|text-gray" src/app` to catalog every screen still on raw Tailwind grays/blues, cross-referenced against `bg-accent|bg-surface|bg-background|text-foreground` usage (the token-adopting screens, if any exist by the time this runs).

2. **Section 1 — Purpose & Scope.** One paragraph: what this doc is, that it's a point-in-time inventory (not living documentation — flag it needs a re-audit whenever `docs/design-direction*.md` changes materially), and the two personas it's written for.

3. **Section 2 — Personas.** Two short persona blocks, cashier and owner/manager, each with: primary device (tablet-class landscape 1024×768 vs. desktop-class browsing per `docs/design-direction-pos.md`'s stated split), primary screens used (link forward to Section 3 rows), and the one line from `docs/design-direction-pos.md`'s Purpose section describing why their needs diverge ("a cashier... needs to read prices at a glance... a dashboard visitor can afford to browse and linger").

4. **Section 3 — Screen Inventory (table).** One row per route. Columns: Screen/Route | Persona | Files (page.tsx + layout.tsx + `_components/*`) | Current Styling Summary | Gap to Direction (cite anti-goal or "not yet built"). Populate from the re-verified tree in step 1. At minimum, based on the current tree, include:
   - `/` marketing (`src/app/(marketing)/page.tsx`, `layout.tsx`) — Public
   - `/login`, `/signup` (`src/app/(auth)/login/page.tsx`, `signup/page.tsx`, `layout.tsx`) — Public/both personas pre-auth
   - `/onboarding` (`src/app/(onboarding/)/onboarding/page.tsx` + `_components/onboarding-wizard.tsx`) — Owner/manager (org setup)
   - `/app` dashboard home (`src/app/(dashboard)/app/page.tsx`, `layout.tsx`) — Owner/manager
   - `/admin`, `/admin/audit` (+`audit-log-table.tsx`), `/admin/enum-values` (+`enum-values-table.tsx`, `create-enum-value-form.tsx`), `admin/layout.tsx` — Owner/manager
   - `/pos/display` (`page.tsx` + `_components/display-cart-view.tsx`) — Cashier-adjacent (customer-facing mirror, driven by the not-yet-built register via `use-cart-broadcast.ts`)
   - `/pos/receipt/[saleId]` (`page.tsx` + `_components/receipt-view.tsx`, `receipt-actions.tsx`) — Cashier/customer (post-sale)
   - Root chrome: `src/app/layout.tsx`, `error.tsx`, `not-found.tsx`, `loading.tsx`, `manifest.ts` — Both
   - Explicitly list `/pos/checkout` (or equivalent register screen) as a **planned, not-yet-existing** row — do not omit it just because there's no file to cite; its absence is itself the single biggest finding of this audit and the task brief asks for the register screen by name.

5. **Section 4 — Component & Interaction Pattern Inventory.** Catalog reusable patterns actually present in code (not aspirational ones), e.g.: `useActionState` + server-action form pattern (`login/page.tsx`, likely `signup/page.tsx`, `create-enum-value-form.tsx`); `requireSession()` layout guard (`(dashboard)/layout.tsx`, `admin/layout.tsx`); `LogoutButton` (`_components/logout-button.tsx`); `ServiceWorkerRegister` (`_components/service-worker-register.tsx`); `useCartBroadcast` cross-tab `BroadcastChannel` sync (`pos/_lib/use-cart-broadcast.ts` + consumer `DisplayCartView`); skip-to-content link + `focus-visible` pattern in root `layout.tsx`; ARIA live-region error pattern (`role="alert" aria-live="polite"` in `login/page.tsx`). For each, cite file:line and note whether it already uses design tokens or still needs migration.

6. **Section 5 — Conventions to Preserve (with rationale).** This is the section the task's acceptance criteria weighs most heavily ("every preserved convention is listed with its rationale"). For each of the four named conventions, use the three-tier status model from Architecture Decisions:
   - **Scan-input focus behavior** — status: not yet built (no barcode-scanner hook or component exists; confirmed via the step-1 grep). Rationale to preserve once built: USB/HID barcode scanners emulate keyboard input and require a persistently-focused hidden text field to work without a mouse click; this is an industry-standard POS pattern referenced in `IMPLEMENTATION_PROGRESS.md`'s acceptance criteria ("Hidden barcode input with USB/HID scanner timing heuristic — Not yet started"). Cite that line.
   - **Checkout shortcut keys** — status: not yet built (zero `onKeyDown`/`shortcut` matches anywhere in `src/`). Rationale: none exists to preserve; the audit should say so plainly rather than inventing a convention, and flag it as an open design decision for the eventual register-implementation card, informed by `docs/design-direction-pos.md`'s 56px-target / accent-scarcity rules for whatever primary actions get bound to keys.
   - **Hold/resume flow** — status: domain-only. `ParkedCart`/`ParkedCartLine` entities exist (`src/domains/sales/entities/parked-cart.ts`) and the repository interface has parked-cart CRUD methods (`src/domains/sales/repository.ts`), but `hold-cart.ts`/`resume-cart.ts` actions and any hold/resume UI are listed as not-yet-implemented in `IMPLEMENTATION_PROGRESS.md` (lines 134–135, 152, 197). Rationale to preserve the *data shape* (registerId/branchId/orgId scoping already on `ParkedCart`) once actions/UI land: it's already threaded through multi-branch/multi-register tenancy scoping used elsewhere in the domain layer.
   - **Discount permission gates** — status: domain-only. `DiscountPolicyService` (`src/domains/sales/services/discount-policy.ts`) enforces cashier percentage/fixed-amount limits and validates a time-boxed manager-override token (`ManagerOverrideToken`), fully unit-tested (`discount-policy.test.ts`), with default limits wired into `ResolvedSettings.discountLimits` (`src/domains/organization/entities/settings.ts`: `cashierMaxPercent=10, cashierMaxFixedAmount=20`). No `discount-modal.tsx` or `manager-override-modal.tsx` exists yet, and no `pin-reauth`/`manager-override` UI wiring was found (confirmed via grep in step 1). Rationale to preserve when UI lands: the override is explicitly "a scoped token... not a session upgrade" (`IMPLEMENTATION_PROGRESS.md` architecture decision #8) — the eventual PIN-reauth modal must call the existing token-validation path in `discount-policy.ts` rather than re-implementing authorization in the component layer.
   Also note `sessionTtlSeconds`/`pinReauthTtlSeconds` (`src/domains/auth/services/session-timeout.ts`) as the policy source the eventual PIN-reauth UI must read from.

7. **Section 6 — Cross-Check Against Design Direction (gap table).** Reuse the Section 3 "Gap to Direction" column but roll it up: one row per anti-goal from `docs/design-direction.md` ("NOT cold or clinical," "NOT neon or oversaturated," "NOT pure white and blue," "NOT cramped or dense," "NOT casino-bright or skeuomorphic") plus the register-specific rules from `docs/design-direction-pos.md` (no muted text on register, 56px targets, accent-scarcity, inverted running-total block), and list every screen currently violating it. Expect "NOT pure white and blue" to list nearly every existing screen (see Overview finding #2) — state this plainly rather than softening it.

8. **Section 7 — Open Questions / Follow-ups for Future Cards.** Non-exhaustive list of things this audit surfaces but does not resolve: dark-mode register variant (explicitly out of scope per `design-direction-pos.md`), phone/small-screen register layout (also explicitly out of scope), doc/doc and doc/CSS drift risk (already flagged as "Known Risks" in `design-direction-pos.md` — cross-reference rather than duplicate).

9. **Write the file, then self-check against the Acceptance Checklist below before considering the card done.**

## State Management Strategy

Not applicable in the runtime-state sense (no React/server state introduced). The relevant "state" here is *document currency*: `docs/ui-audit.md` is a point-in-time snapshot, not living documentation, so Section 1 must say explicitly when it should be re-run (whenever a design-direction doc changes, or whenever a new `/pos/*` screen is implemented — most importantly the eventual register/checkout screen, at which point every "not yet built" row in Sections 3/5 needs to flip to "UI-implemented" with new citations). Do not attempt to make the doc self-updating (no lint rule, no generated-from-source table) — that's explicitly called out as future/unscoped work in `design-direction-pos.md`'s "Known Risks" section and would be scope creep for this card.

## Accessibility Considerations

These are audit *content* requirements, not implementation requirements (no code ships in this card):

- Every accessibility-relevant pattern already in code must be captured in Section 4, not skipped because it's "just markup": the skip-to-content link (`src/app/layout.tsx`), `focus-visible` outline usage, `role="alert" aria-live="polite"` error announcements (`login/page.tsx`), `aria-busy` on pending form submits, and `disabled` state handling during `pending`.
- Section 5's discussion of scan-input focus and checkout shortcuts must note the accessibility tension `docs/design-direction-pos.md` already flags: 56px hit targets exceed WCAG 2.5.5 AA by design, and color is never the sole differentiator for actions (Pay is accent + label + position, not accent alone) — the audit should carry this constraint forward as a requirement for whatever future card builds the shortcut/scan UI, not re-litigate it.
- Section 6's gap table should include a callout that no screen currently in the tree has been checked against the design docs' WCAG AA contrast tables (that verification work is out of scope for *this* card — it belongs to whichever future card actually migrates a screen's classes — but the audit should say the check hasn't happened yet rather than implying it has).

## Risks and Edge Cases

- **Doc goes stale fast.** The codebase is mid-build on the POS domain (`IMPLEMENTATION_PROGRESS.md` shows active work). If the register screen ships between this plan being written and the audit being executed, Section 3/5 rows for `/pos/checkout` and the four named conventions will be wrong the moment they're written. Mitigate by re-running Implementation Step 1's verification immediately before writing, not by trusting this plan's file listing as ground truth.
- **Conflating "domain-only" with "not built."** It would be inaccurate to say discount permission gates or hold/resume "don't exist" — the service/entity layer is real, tested, and fairly complete. The three-tier status model (UI-implemented / domain-only / not yet built) exists specifically to avoid this under- or over-statement; do not collapse it to a binary.
- **Route-group folder naming.** `src/app/(onboarding/)/onboarding/` has an unusual literal `)` character in the directory name (visible in the tree listing) — double check this isn't a typo'd Next.js route group before citing it in the doc; if it's a real folder-naming bug, note it as an aside in Section 7 rather than silently "correcting" it in the citation (the audit reports what exists, it doesn't fix bugs).
- **Tailwind gray/blue usage may be partially intentional scaffolding.** Some raw-Tailwind screens may be intentionally left unstyled as functional-only scaffolding pending a later design pass — the audit should report the gap factually (screen X uses bg-blue-600, anti-goal Y says otherwise) without assuming malice or oversight; that judgment belongs to whoever prioritizes the follow-up implementation card.
- **Over-scoping into a redesign.** Because the gaps are so widespread (nearly every screen), there's a temptation to start proposing fixes inline. Resist this — per Architecture Decisions, this card is inventory only.
