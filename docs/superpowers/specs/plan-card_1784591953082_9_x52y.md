## Overview

The `(dashboard)/app` segment is currently a placeholder: a generic header with three hardcoded links (Home/Admin/Logout), a home page (`src/app/(dashboard)/app/page.tsx`) that reads a non-existent `session.branchId` field (a latent bug — `SessionClaims` has no `branchId` property; the codebase's actual active-branch mechanism is a `current-branch-id` cookie, used correctly by `src/app/pos/checkout/page.tsx` and `src/app/layout.tsx` but never wired into the dashboard shell), and zero catalog/inventory/purchasing/customer routes anywhere in `src/app`. Every table screen this card asks for (catalog, inventory, purchasing, customers) is a **net-new route**, not a redesign of an existing one.

This card delivers, within the existing warm-editorial design language (`docs/design-direction.md` — cream/ivory surfaces, terracotta accent, Fraunces display + Inter body, `--radius-*`/`--shadow-*` tokens, calm Basecamp-style tables):

1. **A real app shell** (`src/app/(dashboard)/layout.tsx` redesign) with: a persistent left nav (icon+label, collapsible on narrow viewports) whose item set is computed per-request from the signed-in user's **role context** (see Architecture Decisions — the combination of operational `UserRole` and org `MembershipRole` is not currently unified anywhere in the codebase); a header with a **branch switcher** and, where relevant, a **register indicator**, both scaling from 1 branch (switcher hidden, static label) to 20+ branches (searchable combobox, keyboard-navigable, grouped/filterable); and a fixed top-level "skip to content" + breadcrumb region for orientation.
2. **A genuinely useful home overview** (`src/app/(dashboard)/app/page.tsx` rewrite) with four glanceable editorial cards: Today's Sales (reusing/extending the existing sales aggregation), Low Stock (new — `inventory.listLowStock`), Open Shifts (new — shift-service/sales-repository), Recent Activity (new — audit-service, org-scoped, branch-filtered). Fixes the `session.branchId` bug by switching the home page onto the `current-branch-id` cookie, and turns the current self-redirect bug into a proper **branch-selection landing** for users with no active branch or multiple eligible branches.
3. **Four new consistent list/table routes** — catalog (`/app/catalog`), inventory (`/app/inventory`), purchasing (`/app/purchasing`), customers (`/app/customers`) — built on one shared, hardened `DataTable` composition (evolving the existing `DataGrid` primitive) that adds: URL-driven sorting/filtering/pagination (so state survives refresh/back-button and is shareable), row-selection + bulk actions toolbar, and full roving-tabindex keyboard navigation (arrow keys between cells/rows, Home/End, Enter to activate, Space to toggle selection) — none of which exist in the current `DataGrid`.
4. **Role-aware landing + gating**: each of cashier / manager / owner / auditor is redirected to (or restricted from) the surfaces appropriate to their role, verified with a permission matrix and Playwright coverage.

Non-goals (explicitly out of scope): building the actual catalog/inventory/purchasing/customer domain CRUD forms (create/edit product, create PO, etc.) — this card is the **shell, navigation, home overview, and list/table screens** (read + bulk-action surfaces) only, backed by existing repository read methods. No new persistence schema. No changes to the POS register/checkout flow itself (`src/app/pos/**`) beyond making sure the shell's branch cookie contract stays compatible with it. No i18n. No new payment/inventory business logic — only presentation and read-path aggregation of what repositories already expose.

## Architecture Decisions

**1. Unifying the two role systems into one `Persona`.** The codebase has two independent role concepts that were never reconciled:
   - `UserRole` (`src/constants/enums/user-role.ts`): `cashier | manager | admin`, stored as `User.roles: string[]` and carried in `SessionClaims.roles`. This is the *operational* role — what a person does at the register/back office.
   - `MembershipRole` (`src/constants/enums/membership-role.ts`): numeric `OWNER=1 | ADMIN=2 | MEMBER=3 | VIEWER=4`, stored per-org on `Membership`, looked up via `organization.findMembership(orgId, userId)`. This is the *organizational* role — what a person may do across the org's branches. `src/app/admin/audit/page.tsx` already treats `VIEWER` as "Auditor" in its UI copy — this card formalizes that precedent instead of inventing a third role enum.

   Precedence rule for `resolveRoleContext()`: **membership role wins when it grants broader org-level access than the operational role implies; operational role wins for register/shift-shaped surfaces.** Concretely: `MEMBERSHIP_ROLE.OWNER` → persona `'owner'` (sees everything, unrestricted branch switcher). `MEMBERSHIP_ROLE.VIEWER` → persona `'auditor'` (read-only everywhere, no bulk actions, no register indicator). Otherwise, fall through to `UserRole`: `'manager'` → persona `'manager'` (branch-scoped via `BranchAssignment`, full read + most bulk actions within their assigned branches, sees Admin only if `MembershipRole <= ADMIN`). `'cashier'` → persona `'cashier'` (home + register only, no catalog/inventory/purchasing/customers list screens — those are back-office surfaces; a cashier's "catalog" is the POS product grid, already served by `/pos/checkout`). This mapping is the single riskiest design call in this plan — flag it in the PR description so a human confirms it matches actual business intent before merge (see Risks).

**2. Active branch is a cookie, not a session claim — the existing pattern is correct, just unused by the dashboard.** `SessionClaims` is a signed JWT and should not be re-issued on every branch switch. The `current-branch-id` cookie (already read in 2 places, never written anywhere in current app code) is the right mechanism — this card adds the missing write path (`switch-branch.ts` server action) and the missing read path for the dashboard shell (`getActiveBranch()`), and fixes the home page's incorrect `session.branchId` reference. Register selection (`RegisterIndicator`) stays branch-scoped local storage as `use-register-session.ts` already does for the POS register itself — the dashboard shell only *displays* the open shift/register, it does not own register selection state (that remains POS's job).

**3. `StatCard` lives in `src/shared/components/ui/`, not `app/_components/`.** It directly implements a named Worked Example from the design-direction doc (a system-level primitive, not a one-off), and other future dashboards (e.g., a manager's branch-comparison view) will want it. This triggers the mandatory Storybook-coverage test — budget for it up front rather than discovering the CI failure.

**4. `DataTable` is additive, not a `DataGrid` rewrite-in-place.** `DataGrid` is simple and may have existing consumers (verify via `ToolSearch`/grep for `DataGrid` imports before touching it — at minimum it's used in its own `.stories.tsx`). Introduce `DataTable` as the new, richer component for the four list screens; leave `DataGrid` alone unless the implementer confirms zero other production consumers, in which case folding `DataGrid`'s simple case into `DataTable` (as a "no toolbar, no selection" mode) and deleting `DataGrid` is preferable to maintaining two table components long-term — but that consolidation is optional cleanup, not required for acceptance.

**5. Bulk actions are scoped to what existing domain actions already support (plus CSV export).** This card explicitly excludes inventing new mutations (e.g., a "bulk deactivate products" domain action) unless the implementer confirms during Step 4 that no equivalent already exists **and** it's a trivial wrapper over an existing single-item action already present in `src/domains/*/actions/` (e.g., if `catalog/actions/` has `updateProduct`, a bulk toolbar action can call it N times client-side via `Promise.allSettled` — this is presentation-layer looping, not new business logic). Where no safe existing action exists, the bulk toolbar offers CSV export only (via `src/domains/reporting/services/csv-export.ts`, which already exists and is domain-agnostic-ish — verify its signature before assuming it accepts arbitrary row shapes).

**6. URL-driven table state, not component state.** Sort/filter/page state in `useSearchParams` (read) + `router.replace(pathname + '?' + params, { scroll: false })` (write) keeps tables deep-linkable and back-button-safe, and lets the Server Component page re-fetch with the right slice server-side when practical — though since most `list*ByOrg`/`list*ByBranch` repository methods return **unbounded arrays with no server-side pagination params** (confirmed: `listProductsByOrg(orgId)`, `listSuppliers()`, etc. take no limit/offset), pagination/filtering/sorting for these four screens is implemented **client-side over the full fetched list** for this card (matches the existing `listSalesByBranch(branchId, limit, offset)` precedent where it exists, but most list methods don't support it yet). Document this as a known scale limit (see Risks) rather than silently deciding to add repository-level pagination, which is a larger cross-cutting change out of this card's scope.

**7. Responsive nav: rail + drawer, not a full sidebar always visible.** Desktop: persistent icon+label rail (~15-18% width capped, matching `docs/design-direction.md`'s "whitespace is a feature" spacing principle — do not cram to save width). Narrow viewports (POS terminals are often tablet-width, back-office might be phone-width): nav collapses into the existing `Sheet` primitive (`side="left"`) triggered by a header hamburger button — reuse `Sheet`, do not build a second drawer primitive.

## State Management Strategy

- **Server state (source of truth)**: session (JWT cookie, unchanged), active branch (`current-branch-id` cookie, httpOnly, written only via the `switch-branch` Server Action — never via client-side `document.cookie`), role/membership (re-derived server-side per request from `orgId` + `userId`, **not cached client-side** — Server Components re-fetch `resolveRoleContext` on every navigation, which is correct for a permissions surface: staleness here is a security bug, not a UX inconvenience).
- **URL state**: table sort/filter/page (`useSearchParams`), active nav highlight (`useSelectedLayoutSegments`, derived, no state needed).
- **Client-only ephemeral state**: `DataTable`'s roving-tabindex `{row, col}` focus position and `selectedIds: Set<string>` (selection is intentionally *not* persisted to the URL — refreshing a table should not resurrect a stale bulk-selection, since bulk actions are destructive-ish; this matches how most dense admin tables behave). Register selection stays in `localStorage` exactly as `use-register-session.ts` already does — this card does not touch that mechanism, only *reads and displays* its result.
- **No new global client state library** — everything above is either Server-Component-derived, URL-derived, or component-local `useState`/`useReducer`. Do not introduce Context for anything except what's already there (`TenantThemeProvider` pattern) — if `AppShell` needs to pass persona/nav down to deeply nested client children, prefer prop-drilling 1-2 levels (shell → header/nav are siblings, not deeply nested) over adding a new Context.

## Accessibility Considerations

- **Skip link**: the root `layout.tsx` already has one (`<a href="#content" className="sr-only focus:not-sr-only">`) — preserve the `#content` id on `AppShell`'s `<main>` exactly; do not let the new shell introduce a second competing `id="content"`.
- **Nav landmarks**: left nav is a `<nav aria-label="Main">`; breadcrumb is a second `<nav aria-label="Breadcrumb">` — two `<nav>` elements need distinct accessible names or screen readers announce them identically.
- **Branch switcher**: must expose `role="combobox"` semantics correctly (Radix/the existing `Combobox` component should already handle this — verify with an accessibility tree inspection, don't assume) and announce the currently-selected branch via `aria-activedescendant` or equivalent, matching the existing `Combobox` implementation's pattern rather than inventing new ARIA wiring.
- **`DataTable` keyboard nav is the acceptance-critical accessibility surface**: implement per WAI-ARIA APG "Grid" pattern — `role="grid"` on the table (not the default implicit `table` role, since plain `table` role doesn't support the 2D roving-tabindex/arrow-key model), `role="row"`/`role="gridcell"`/`role="columnheader"` on the relevant elements, `aria-selected` on selected rows, `aria-sort` on sortable headers (the existing `DataGrid` already sets `aria-sort` correctly on its sort button — carry that forward). Only the single active cell has `tabIndex={0}`; every other cell has `tabIndex={-1}` (classic roving-tabindex). Verify with keyboard-only navigation (unplug the mouse mentally) through: header sort activation, row selection via Space, row activation via Enter, pagination controls, and bulk-action toolbar buttons — the whole flow must be operable without a pointer.
- **Focus visibility**: reuse the existing `focus-visible` token-based outline (`docs/design-direction.md`'s Focus Ring section) — do not add competing focus styles on table cells.
- **Reduced motion**: nav drawer (`Sheet`) open/close and any new stat-card entrance animation must respect the existing global `prefers-reduced-motion` block in `globals.css` — verify new motion utilities are the existing `motion-*` token classes, not ad-hoc `transition-*` durations.
- **Color/contrast**: any new badge variants (low-stock warning, open-shift status) must reuse existing `badge.tsx` variants (`warning`, `success`, `danger`) which are already AA-verified per the design-direction doc's contrast table — do not introduce new raw colors (the design-token ESLint rule will fail the build if you do).
- **Screen-reader-only row/selection counts**: the bulk-action toolbar's "N selected" must be in a `aria-live="polite"` region so screen reader users get selection-count feedback without needing to visually scan.

## Risks and Edge Cases

- **Persona-mapping is a judgment call, not a spec.** The `owner`/`auditor` ↔ `MembershipRole` and `cashier`/`manager` ↔ `UserRole` mapping (Architecture Decision 1) is inferred from existing code precedent (`admin/audit/page.tsx`'s OWNER/ADMIN/VIEWER-as-auditor gate), not an explicit product spec. Flag this prominently in the PR description; if a human reviewer disagrees with the precedence rule, it's a one-function fix in `role-context.ts`, not a rearchitecture — isolating this logic in one pure function is exactly why Architecture Decision 1 keeps it there.
- **No server-side pagination on most repositories** (`listProductsByOrg`, `listSuppliers`, `listVariantsByOrg`, etc. take no limit/offset). Client-side pagination over a fully-fetched list is fine at realistic seed-data scale but will degrade at, say, 50,000 products. This card does not fix that (out of scope per Architecture Decision 6) — document it as a known follow-up, don't silently paper over it with an unbounded `.slice()` that looks like real pagination but isn't.
- **`current-branch-id` cookie has no existing writer** — confirm during Step 3 there isn't a writer hiding somewhere this grep missed (e.g., in a client component using `document.cookie` directly) before assuming `switch-branch.ts` is the only place it's ever set; a second silent writer would cause confusing races.
- **Redirect-loop risk** between `/pos/checkout` ↔ `/app` ↔ `/app/select-branch` — explicitly covered in Test Plan; don't skip that regression spec, this is exactly the kind of bug that's invisible until a real user with zero branches hits it in production.
- **`/admin` currently has no role gate at all** — adding one is a real (small) behavior change: any authenticated user can currently reach `/admin`. After this card, only owner/admin personas can. Confirm this isn't relied upon anywhere (e.g., a manager currently using `/admin/enum-values` for a legitimate reason) before locking it down — grep any Playwright/manual test coverage that logs in as non-owner and hits `/admin`.
- **Bulk actions with no safe existing single-item action to loop over** (Architecture Decision 5) — if catalog/purchasing/customers domains turn out to have zero existing actions suitable for bulk-wrapping, ship the bulk toolbar with **CSV export only** for that route rather than blocking the whole card on writing new domain mutations; note in the PR which routes got full bulk actions vs. export-only.
- **20-branch test data** may not exist in any seed script — building temporary fixture data to verify the "usable at 20 branches" acceptance criterion is the implementer's responsibility; don't mark that criterion verified against a 2-branch dev database.
- **Storybook-coverage test is a hard CI gate** — every new file under `src/shared/components/ui/` (StatCard, DataTable, DataTableToolbar) needs its story committed in the same change, or CI fails immediately; sequence commits so this never lands red.
